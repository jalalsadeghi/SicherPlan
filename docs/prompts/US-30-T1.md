# Codex Prompt — US-30-T1

## Task title

**Implement subcontractor invoice checks and line-level variance analysis.**

## Objective

Build the finance-owned subcontractor settlement backbone so period-based invoice checks and line-level variance records can compare released subcontractor work with approved finance actuals in a traceable, auditable structure.

## Source context

- Updated proposal: section 5.4 includes subcontractor history, controlling, invoice verification, and working-time evidence; section 5.6 calls for subcontractor settlement by comparing assigned shifts, attendance evidence, and actual hours before approval; workflow 6.3 ends with visible subcontractor invoice status and performance reports.
- Implementation specification: section 4.8 defines `finance.subcontractor_invoice_check` and `finance.subcontractor_invoice_check_line`; section 5 keeps the subcontractor chain and finance result chain stable from partner release through assignment, actuals, and invoice checking.
- Earlier uploaded subcontractor-management material emphasized shift controlling, invoice verification, time-evidence visibility, and role-scoped portal status for subcontractors.
- Sprint reference: `docs/sprint/sprint-10-payroll-billing-and-settlement.md`.
- Cross-cutting rules: `AGENTS.md`, especially finance-bridge discipline, least-privilege portal exposure, and audit-safe control workflows.

## Dependencies

- `US-14-T4` should already provide partner-side read contracts for subcontractor workers, readiness, and released-work context.
- `US-19-T1` through `US-19-T3` should already provide releases, assignments, and subcontractor staffing context that invoice checks summarize.
- `US-26-T2` and `US-27-T1` through `US-27-T4` should already provide approved actuals and finance reconciliation status used by invoice checks.
- `US-13-T1` should already provide subcontractor finance-profile context consumed later by commercial comparison logic.

## Scope of work

- Implement migrations, models, Pydantic schemas, services, and APIs for subcontractor invoice-check headers and line items in the finance bounded context.
- Support period-based invoice-check creation per subcontractor with explicit fields for assigned hours, actual hours, approved hours, expected cost, submitted invoice reference/amount, overall variance amount, and workflow status.
- Generate line-level records linked to actual records and/or assignments, carrying expected quantity, actual quantity, approved quantity, expected amount, approved amount, and initial variance-reason classification where available.
- Allow finance users to capture submitted invoice number/amount and regenerate or refresh line-level analysis in an auditable way according to repository conventions.
- Keep header totals and line records consistent and reproducible from underlying assignment/actual data rather than manual spreadsheet-only logic.
- Add automated tests for period uniqueness, line-generation linkage, subtotal consistency, submitted-invoice capture, and tenant/subcontractor scope safety.

## Preferred file targets

- `modules/finance/` subcontractor-control models, schemas, services, routers, and tests.
- Alembic migrations for invoice-check and invoice-check-line tables, indexes, and workflow-status support.
- Optional internal finance-admin endpoints only if the repository already exposes a controller workspace that later UI tasks will consume.
- Tests covering generation, totals consistency, and line-level provenance.

## Hard constraints

- Do not calculate subcontractor settlement from raw time events; invoice checks must derive from approved finance actuals and released assignment context.
- Invoice-check headers and lines are finance-owned summaries and workflow artifacts; do not mutate partner master truth or actual records directly from them.
- Each line must stay traceable to assignment and/or actual-record context where the normalized schema expects it.
- Do not expose internal invoice-check detail to subcontractor portal users in this task; portal release belongs to `US-30-T4`.
- Keep period uniqueness deterministic per subcontractor and tenant.
- Do not reduce the variance model to a single opaque total if line-level explainability is required by the story.

## Expected implementation outputs

- Subcontractor invoice-check header/line persistence and generation services.
- Auditable capture of submitted invoice references and amounts.
- Reproducible line-level variance structures linked to assignments and actuals.
- Tests proving uniqueness, provenance, and totals consistency.

## Non-goals

- Do not build the controller UI here; that belongs to `US-30-T3`.
- Do not expose invoice-check results in the subcontractor portal here; that belongs to `US-30-T4`.
- Do not embed partner rate-resolution ownership inside this task beyond the minimal seams needed for later `US-30-T2` work.
- Do not implement payment runs or financial booking integration in this task.

## Verification checklist

- [ ] Line items remain linked to source assignments and/or approved actuals.
- [ ] Header totals and variance values are reproducible from persisted line/context data.
- [ ] Submitted invoice reference/amount capture is auditable and test-covered.
- [ ] Tests cover period uniqueness, provenance, totals consistency, and tenant safety.
