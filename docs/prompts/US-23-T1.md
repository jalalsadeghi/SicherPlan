# Codex Prompt — US-23-T1

## Task title

**Build the information portal feed and mandatory notice read confirmation in web and mobile.**

## Objective

Expose the information-portal feed to the real consuming clients so employees and eligible internal users can see their notices, open attachments and links, and create durable read/acknowledgement evidence from both Vben-based web screens and the Prokit-based mobile app.

## Source context

- Updated proposal: section 5.7 internal information portal, read confirmation and control, internal communication, and audit-ready confirmation evidence; section 5.1 communication backbone; phase 4 emphasis on mobile/field execution.
- Implementation specification: section 4.2 `info.notice`, `info.notice_audience`, `info.notice_read`, and `info.notice_link`; docs attachments via `docs.document_link`; section 5 information/QM chain; reporting guidance in section 7.
- Sprint reference: `docs/sprint/sprint-08-employee-app-and-field-i.md`.
- Cross-cutting rules: `AGENTS.md`, especially Vben Admin for web patterns, Prokit for mobile, DE/EN rules, and durable read evidence.
- Earlier prompt context: `US-6-T3` should already provide the notice backbone; this task turns it into real web/mobile consumption flows.

## Dependencies

- `US-6-T3` must already provide the notice, audience, read-confirmation, link, and attachment foundation.
- `US-22-T1` should already provide authenticated employee mobile-shell access.
- `US-3-T1`, `US-3-T2`, and `US-3-T4` should already provide the web/mobile shells and localization baseline.
- `US-5-T3` should already enforce tenant- and role-scoped access rules on APIs.
- If optional blocking-until-read behavior is introduced, it must remain configuration-driven and reuse the existing notice/read foundation rather than inventing a second acknowledgement mechanism.

## Scope of work

- Implement scoped notice-feed APIs and client consumption flows for the users who should see notices in web and mobile.
- Support publish-window filtering, audience resolution, attachment visibility, curated links, first-open tracking, and mandatory acknowledgement.
- Build Vben-based web views for the notice feed and acknowledgment actions, plus corresponding Prokit-style mobile screens or cards.
- Where the repository already supports it, surface optional blocking state for mandatory unread items in a controlled, configuration-driven way without globally locking the product.
- Keep read/ack events durable and idempotent so repeated opens do not create duplicate evidence rows.
- Add tests for scoped feed visibility, publish windows, unique acknowledgements, and attachment/link access.

## Preferred file targets

- `modules/platform_services` API/query/service files for notice feed and read/ack behavior.
- Web notice feed views/components under the existing Vben workspace.
- Mobile notice feed screens/components under the Flutter app structure.
- Tests for read/ack idempotency, audience scoping, and publish-window behavior.

## Hard constraints

- Notice feed queries must remain tenant-safe and audience-scoped; no broad notice browsing is allowed.
- Read/acknowledgement evidence must stay durable and unique per notice/user according to the approved schema.
- Attachments must continue to flow through the docs service; do not add bespoke file columns to notice records.
- Optional blocking semantics must be configuration-driven and must not become a hardcoded global lockout rule.
- Keep DE default / EN secondary behavior across web and mobile notice surfaces.

## Expected implementation outputs

- Scoped notice-feed APIs and client screens for web and mobile.
- Durable open/acknowledgement handling for mandatory and non-mandatory notices.
- Attachment/link access integrated with the existing docs/notices structures.
- Tests covering audience scoping, publish windows, and acknowledgement idempotency.

## Non-goals

- Do not build the later QM reporting dashboards for notice statistics here.
- Do not create a chat, comment, or reaction system on top of notices.
- Do not bypass the notice/read tables with ad hoc local-only client tracking.
- Do not force customer/subcontractor watchbook participation or external portals into this task.

## Verification checklist

- [ ] Web and mobile clients show only notices within the acting user’s tenant and audience scope.
- [ ] Mandatory acknowledgement produces one durable evidence row per notice/user.
- [ ] Publish-window and attachment/link behavior work consistently across clients.
- [ ] Optional blocking state is configuration-driven and not hardcoded globally.
- [ ] Tests cover scope, publish-window filtering, and idempotent acknowledgement handling.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
