# AI-48 — Expert Answer Calibration and EN/DE Golden QA Evaluation

## Purpose

Even if RAG retrieval is technically present, the assistant must answer like a SicherPlan expert. This task creates a strict golden QA evaluation loop for English and German and forces the backend to improve retrieval/prompting until the assistant passes.

The aim is not to hardcode answers. The aim is to enforce answer quality using expected criteria.

---

## Prompt for Codex

You are working on the SicherPlan repository.

Implement **AI-48 — Expert Answer Calibration and EN/DE Golden QA Evaluation**.

### Step 1 — Build Golden QA Dataset

Create a dataset file, for example:

```text
backend/tests/fixtures/assistant/golden_qa_en_de.json
```

Each item must include:

```json
{
  "id": "de_product_overview",
  "language": "de",
  "question": "Bitte erklären Sie mir kurz und bündig, was diese Software genau macht.",
  "expected_intent": "product_overview",
  "expected_pages": [],
  "required_concepts": [
    "multi-tenant",
    "security operations platform",
    "customers",
    "employees",
    "subcontractors",
    "planning",
    "field execution",
    "billing",
    "reporting"
  ],
  "forbidden_patterns": [
    "Ich bin nur dafür vorgesehen",
    "out-of-scope"
  ],
  "requires_source_basis": true,
  "min_content_bearing_sources": 2
}
```

### Step 2 — Required Golden Questions

Include at least these questions:

#### Product overview

```text
Bitte erklären Sie mir kurz und bündig, was diese Software genau macht.
Please briefly explain what this software does.
```

#### Customer

```text
Bitte erklären Sie mir genau, wie ich einen Kunden registrieren kann.
How do I create a new customer?
```

#### Employee

```text
Ich möchte nun einen neuen Sicherheitsmitarbeiter auf der Plattform registrieren. Was muss ich tun?
How do I create a new security employee?
```

#### Contract/document

```text
Wie registriere ich einen neuen Vertrag?
How do I register a new contract?
```

#### Order/planning

```text
Wie erstelle ich einen neuen Auftrag für einen Kunden?
How do I create a new order for a customer?
```

#### Employee-to-shift

```text
Wie weise ich einen Mitarbeiter einer Schicht zu?
How do I assign an employee to a shift?
```

#### Shift release/app visibility

```text
Wie gebe ich eine Schicht für die Mitarbeiter-App frei?
How do I release a shift to the employee app?
```

### Step 3 — Evaluation Criteria

For each answer, evaluate:

1. provider called;
2. OpenAI/fake provider received grounding context;
3. intent correct;
4. top sources relevant;
5. content-bearing source count meets minimum;
6. source_basis returned;
7. answer language matches question language;
8. required concepts are present;
9. forbidden generic/refusal patterns absent;
10. confidence is not high unless source basis is sufficient;
11. no invented standalone pages/actions;
12. links are allowed only if permission permits.

### Step 4 — Fake Expert Provider for Tests

To avoid real OpenAI calls in tests, create a fake provider that:

- receives grounding context;
- fails if grounding context is empty;
- returns a structured answer derived from grounding facts;
- exposes what sources it received;
- never returns canned production answer text.

This fake provider is for tests only.

### Step 5 — Add Real Prompt Capture Test

Add tests that capture the exact provider request and assert:

- user message is not the only meaningful content;
- grounding context includes content-bearing sources;
- source basis candidates are present;
- system instructions require source-grounded answers.

### Step 6 — Improve Prompt if Golden Questions Fail

If golden QA fails, Codex must inspect why and fix the relevant layer:

```text
classification failure -> fix classifier
no content sources -> fix corpus/retrieval
wrong sources -> fix reranking/query expansion
generic answer -> fix prompt/source instructions
missing source basis -> fix structured schema/validation
missing UI details -> fix page-help coverage
```

Codex must not mark the task done if golden QA still fails.

### Step 7 — Optional Runtime Evaluation Endpoint

Add an admin-only endpoint if useful:

```text
POST /api/assistant/evaluation/golden-run
```

or a CLI/test command:

```bash

```

The output should include:

```text

```

### Step 8 — Expected Answer Quality Guidelines

Do not use these as runtime templates, but the answers should satisfy these criteria:

#### Product overview
Must explain SicherPlan as:
- multi-tenant security operations platform;
- for security companies;
- covers customers, employees, subcontractors, planning, field execution, actuals, billing/payroll, reporting, portals;
- role/permission/tenant-aware.

#### Customer registration
Must explain:
- use Customers Workspace;
- create/maintain customer master data;
- complete contacts, addresses, billing profile, invoice parties, rate cards/surcharge rules;
- portal/privacy/employee blocks if relevant;
- permissions may be needed.

#### Contract registration
Must explain:
- no verified standalone Contract module unless the corpus confirms one;
- contracts are treated as documents/attachments linked to customer/order/subcontractor context if verified;
- PS-01 handles documents/version/linking;
- C-01/P-02/S-01 may be relevant depending on contract type;
- ask a clarifying question if needed.

#### Employee registration
Must explain:
- Employees Workspace;
- employee file;
- identity/personnel number;
- qualifications/readiness/availability/access link;
- permissions.

### Acceptance Criteria

This task is complete only when:

1. Golden QA dataset exists.
2. All required questions pass.
3. Bad observed answers fail the test suite.
4. Product overview no longer returns out-of-scope.
5. Customer registration no longer returns only route/path context.
6. Contract question receives source-grounded explanation/clarification.
7. Source basis exists.
8. No final answer templates are introduced.
9. Tests pass.
