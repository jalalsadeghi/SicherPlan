# AI-37 — Diagnose and Fix OpenAI Provider 503 in RAG Assistant

## Purpose

The assistant is now configured with:

```env
SP_AI_ENABLED=true
SP_AI_PROVIDER=openai
SP_OPENAI_API_KEY=<present>
SP_AI_RESPONSE_MODEL=gpt-4o
SP_AI_STORE_RESPONSES=false
```

but every new assistant message still fails with:

```text
POST /api/assistant/conversations/{conversation_id}/messages -> 503 Service Unavailable
```

The structured logs show:

```json
{
  "provider_mode": "openai",
  "model_name": "gpt-4o",
  "openai_request_attempted": true,
  "provider_returned_successfully": false,
  "input_tokens": null,
  "output_tokens": null,
  "provider_latency_ms": null
}
```

This means the assistant runtime reached the OpenAI provider path, but the provider wrapper fails before producing a structured response. The current logging does not reveal the actual provider exception, so the root cause is hidden.

This task must identify and fix the real OpenAI provider failure. Do not guess. Add safe diagnostics, verify the OpenAI SDK call shape, add tests, and make the RAG assistant return a real model-generated response.

```text
backend/app/modules/assistant/openai_client.py

```

Known implementation details to inspect:

1. `OpenAIResponsesProvider.generate()` currently calls something like:

```python
client.responses.parse(
    model=self.runtime.model_name,
    input=self._build_input(request),
    text_format=AssistantProviderStructuredOutput,
    store=self.runtime.store_responses,
    tools=request.available_tools,
)
```

2. `_map_provider_exception()` currently maps many unknown exceptions to a generic `AssistantProviderUnavailableError`, which becomes a 503 without exposing the actual exception type or safe error reason.

3. The log shows no token usage and no latency, suggesting either:
   - local SDK/client validation failure;
   - invalid request shape;
   - invalid tool schema;
   - incompatible `responses.parse` usage;
   - missing/old `openai` Python SDK;
   - structured output parsing failure;
   - unsupported combination of parse + tools;
   - API key/project/model permission failure;
   - network/API status failure with hidden details.

Codex must determine the real cause through code and tests.

### Step 2 — Add Safe Provider Exception Diagnostics

Improve provider error visibility without leaking secrets.

When OpenAI provider fails, log a structured sanitized event containing:

```json
{
  "event": "assistant_provider_error",
  "provider_mode": "openai",
  "model_name": "...",
  "request_id": "...",
  "conversation_id": "...",
  "exception_class": "...",
  "provider_error_type": "...",
  "provider_error_code": "...",
  "http_status": 400,
  "safe_message": "short sanitized message",
  "openai_request_attempted": true
}
```

Rules:

- Do not log API keys.
- Do not log full raw request payload.
- Do not log full grounding context.
- Do not log PII or sensitive business data.
- Safe message may include exception class and sanitized provider message.
- For `APIStatusError`, log the status code and safe body/message if available.
- For local validation errors, log the exception class and short message.

### Step 3 — Return More Specific Backend Error Codes

Do not convert every provider failure to generic `assistant.provider.unavailable`.

Map errors more precisely:

```text
configuration missing SDK/API key/model -> assistant.provider.configuration_error -> 503
invalid OpenAI request shape/schema/tools -> assistant.provider.invalid_request -> 502 or 500
OpenAI authentication/project permission -> assistant.provider.authentication_failed -> 503 or 502
OpenAI rate limit -> assistant.provider.rate_limited -> 503
OpenAI timeout -> assistant.provider.timeout -> 504
OpenAI unavailable/API status >= 500 -> assistant.provider.unavailable -> 503
structured output invalid -> assistant.provider.invalid_response -> 502
```

Frontend can still show a safe user-facing message, but backend logs must be useful for development.

### Step 5 — Fix the OpenAI Responses API Call

After diagnostics reveal the issue, fix the provider wrapper.

Codex must validate the correct call shape against the installed `openai` SDK.

Possible areas to fix:

1. `responses.parse` may not support the current parameters or tool schema.
2. `tools=request.available_tools` may be incompatible with `parse`.
3. `text_format=AssistantProviderStructuredOutput` may need to be replaced by the SDK-supported structured output format.
4. If tool calls are returned, the provider wrapper may need to return requested tool calls before expecting `output_parsed`.
5. If no tool calls are needed, the wrapper must parse the final response into `AssistantProviderStructuredOutput`.
6. If structured output with Pydantic is unstable with tools, use `responses.create` with a JSON schema response format supported by the current SDK, then validate the JSON with Pydantic server-side.

Do not hardcode model output. The final answer must still come from OpenAI.
