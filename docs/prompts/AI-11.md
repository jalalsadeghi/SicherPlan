This task is **AI-11 — OpenAI Client Wrapper**.

## Goal

Implement the real backend-only OpenAI provider wrapper for the SicherPlan AI Assistant, using the provider interface created in AI-10.

The browser must never call OpenAI directly. All OpenAI requests must go through the backend assistant provider wrapper. The wrapper must use the assistant settings from AI-02, the structured response schema from AI-07, the language policy from AI-08, the out-of-scope policy from AI-09, and the provider abstraction from AI-10.

The real OpenAI provider should be production-safe, testable without network calls, privacy-conscious, and isolated from the rest of the assistant service behind the provider interface.

## Mandatory Official Documentation Check

Before coding, verify the current OpenAI Python SDK and Responses API syntax from the official OpenAI documentation.

Relevant official docs to check:

```text
https://developers.openai.com/api/reference/responses/overview/
https://developers.openai.com/api/reference/resources/responses/methods/create/
https://developers.openai.com/api/docs/guides/migrate-to-responses
https://developers.openai.com/api/docs/guides/structured-outputs
https://developers.openai.com/api/docs/guides/function-calling
https://developers.openai.com/api/docs/guides/conversation-state
```

Do not guess SDK method names. If the repository already has an OpenAI dependency, use the installed version and its supported syntax. If not, add the dependency following the project’s `pyproject.toml` or dependency management conventions.

## Mandatory Constraints

- Do not call OpenAI from the frontend.
- Do not expose `SP_OPENAI_API_KEY` to frontend code, logs, test snapshots, API responses, or prompts.
- Do not implement business write actions.
- Do not execute model-generated SQL.
- Do not implement unrestricted tool execution.
- Do not bypass the provider interface from AI-10.
- Do not bypass same-language behavior from AI-08.
- Do not bypass out-of-scope refusal from AI-09.
- Do not send more context than needed to OpenAI.
- Do not send JWTs, password hashes, reset tokens, API keys, database URLs, or raw secrets to OpenAI.
- Do not persist provider-side state unless explicitly configured.
- Keep `mock` provider mode fully functional.
- All tests must be runnable without a real OpenAI API key.

## Implementation Scope

### 1. Add OpenAI Provider Implementation

Create or update:

```text
backend/app/modules/assistant/openai_client.py
```

or, if this is cleaner with the AI-10 provider layout:

```text
backend/app/modules/assistant/providers/openai_provider.py
```

The implementation must conform to the provider interface from AI-10.

Suggested class name:

```python
class OpenAIResponsesProvider(AssistantProvider):
    ...
```

Required behavior:

- Initialize OpenAI client backend-side only.
- Use configured API key from backend settings.
- Use configured response model from backend settings.
- Use configured timeout and retry settings where supported by the SDK.
- Use structured output mode with the assistant response schema from AI-07.
- Use `store` according to `SP_AI_STORE_RESPONSES`; default should be privacy-conservative.
- Return provider result through the AI-10 provider result object.
- Convert provider-specific errors into safe assistant provider errors.
- Include request/correlation ID in internal logs if available, without secrets.

### 2. Preserve Mock Provider

`SP_AI_PROVIDER=mock` must continue to work exactly as AI-10 implemented it.

No OpenAI dependency, API key, or network call should be required when mock mode is active.

### 3. Provider Factory Update

Update the provider factory so:

```text
SP_AI_PROVIDER=mock
  returns MockAssistantProvider

SP_AI_PROVIDER=openai
  returns OpenAIResponsesProvider

unknown provider
  fails safely
```

If AI is disabled, provider creation should not force OpenAI initialization unnecessarily.

### 4. API Key Validation

Behavior must be environment-safe:

```text
SP_AI_PROVIDER=mock
  SP_OPENAI_API_KEY may be empty.

SP_AI_PROVIDER=openai
  SP_OPENAI_API_KEY is required.
```

If the key is missing in OpenAI mode, return a safe configuration error, for example:

```text
AI provider is not configured.
```

Do not include the name, value, length, or partial value of the key in user-facing errors.

### 5. Structured Output Handling

The real provider must request structured output matching the assistant response schema.

The exact API parameter names must match the current SDK. Use the official docs and installed SDK.

The final response must be validated by the backend after receiving it. If the provider returns invalid structured output:

- do not return raw provider output to the user;
- log a safe internal event;
- return a safe provider error or fallback response according to existing service patterns.

### 6. Tool Calling Preparation Without Final Tool Loop

This task may prepare the real provider to accept tool definitions from the provider request, but it must not implement the complete tool registry loop. The full registry and tool execution policy belongs to AI-15.

Acceptable in this task:

- define how tool definitions would be passed to OpenAI;
- parse provider tool call requests into the provider result object;
- return requested tool calls to the assistant service.

Not acceptable in this task:

- execute tools directly inside the OpenAI provider;
- call repository/database code from the provider;
- implement diagnostic tools;
- execute write actions.

### 7. Privacy and Conversation State

The SicherPlan backend is the source of truth for conversation persistence. Do not rely on provider-side conversations unless explicitly configured later.

Default behavior:

```text
SP_AI_STORE_RESPONSES=false
```

Requirements:

- Use the configured store/privacy behavior.
- Do not send all historic messages unbounded.
- Respect `SP_AI_MAX_INPUT_CHARS` and any recent-message cap already implemented.
- Do not include raw tool outputs containing sensitive fields.
- Do not include secrets in prompt/instructions/input.

### 8. Error Handling

Convert provider errors into safe internal exception types, for example:

```text
AssistantProviderConfigurationError
AssistantProviderTimeoutError
AssistantProviderUnavailableError
AssistantProviderRateLimitError
AssistantProviderStructuredOutputError
AssistantProviderError
```

Map errors safely:

- missing API key -> configuration error;
- timeout -> temporary unavailable/timeout;
- rate limit -> user-friendly retry later message;
- invalid structured output -> safe provider response error;
- network issue -> provider unavailable.

Do not expose SDK stack traces to the API client.

1. `SP_AI_PROVIDER=openai` selects the real OpenAI provider.
2. `SP_AI_PROVIDER=mock` still selects the mock provider and requires no API key.
3. OpenAI API key is read only from backend settings/environment.
4. No frontend file contains the OpenAI API key or OpenAI client initialization.
5. The real provider uses the Responses API or current officially documented equivalent.
6. The real provider requests/validates structured output according to the AI-07 schema.
7. Provider-side storage is controlled by configuration and defaults to privacy-conservative behavior.
8. Provider timeout and retry behavior are configured or safely wrapped.
9. Provider errors are mapped to safe application errors.
10. No raw provider payload containing sensitive information is returned to the frontend.
11. Tests can run without a real OpenAI API key by monkeypatching/faking the OpenAI SDK client.
12. Existing mock provider tests still pass.
13. Existing assistant tests from AI-04 to AI-10 still pass.
