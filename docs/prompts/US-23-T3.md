# Codex Prompt — US-23-T3

## Task title

**Generate revision-safe daily watchbook PDF outputs and portal-facing read models.**

## Objective

Turn watchbook evidence into durable customer- and operations-facing outputs by generating revision-safe daily PDFs, linking them into the docs backbone, and exposing safe read models for downstream portals and clients without duplicating the write-side watchbook truth.

## Source context

- Updated proposal: section 5.2 online watchbook requires revision-safe daily PDF output and customer/subcontractor visibility according to release scope; section 5.5 released outputs; deliverables/acceptance section requiring revision-safe PDF outputs and reproducible evidence.
- Implementation specification: `field.watchbook.pdf_document_id`, docs tables in section 4.2, watchbook tables in section 4.7, portal visibility chain in section 5, reporting/read-model guidance in section 7, and generated-output rule in section 10.
- Appendix D of the updated proposal: watchbook entries and billing/reporting output chain.
- Sprint reference: `docs/sprint/sprint-08-employee-app-and-field-i.md`.
- Cross-cutting rules: `AGENTS.md`, especially docs-linked generated outputs, customer-facing name-hiding by default, and no second write-side reporting schema.

## Dependencies

- `US-23-T2` must already provide watchbook data and entry review flows.
- `US-6-T1` must already provide docs/document-version linkage for generated PDFs.
- `US-9-T2` should already provide customer-portal read-model patterns for released outputs.
- `US-21-T1` and `US-21-T4` should already define release/visibility semantics and downstream client consumption patterns.
- If customer/subcontractor participation is enabled later in `US-23-T4`, keep the read models extensible without assuming it already exists.

## Scope of work

- Implement revision-safe daily PDF generation for watchbooks, including deterministic ordering and enough metadata to reproduce or audit the result.
- Persist generated PDF outputs as `docs.document` / `docs.document_version` rows and link them back to the owning watchbook.
- Build portal- and client-facing read models for released watchbook outputs and summaries, derived from the watchbook truth rather than copied into new write-side tables.
- Apply actor-specific visibility filters so internal users, customers, and subcontractors only see outputs that are explicitly released to them.
- Ensure customer-facing output shaping respects the platform rule that personal names are hidden by default unless explicit release rules permit them.
- Add tests for deterministic PDF generation hooks, docs linkage, visibility filtering, and default name-hiding behavior.

## Preferred file targets

- `modules/field_execution` services for watchbook output generation and released read models.
- `modules/platform_services` or docs helpers only where generated-output registration needs a narrow adapter.
- Portal/client query DTOs or endpoints for released watchbook outputs.
- Tests for PDF-link registration, released-only gating, and visibility-safe output shaping.

## Hard constraints

- Generated watchbook PDFs must be represented as docs rows linked back to the watchbook; do not leave them as detached files.
- Portal-facing read models must be derived from watchbook truth, not from a second write-side shadow schema.
- Customer-facing outputs must hide personal names by default unless explicit release rules say otherwise.
- Only released watchbook outputs may reach portal or mobile consumers.
- Keep generated-output behavior reproducible and auditable.

## Expected implementation outputs

- Revision-safe watchbook PDF generation pipeline and docs linkage.
- Released watchbook read models for downstream portals/clients.
- Visibility-safe output shaping with default name hiding for customer-facing results.
- Tests covering docs linkage, release filters, and safe output shaping.

## Non-goals

- Do not implement full reporting dashboards or analytics here.
- Do not create a separate reporting database/schema for watchbook outputs.
- Do not bypass the docs service for PDF storage.
- Do not release watchbook data broadly just because a PDF exists.

## Verification checklist

- [ ] Each generated watchbook PDF is linked back to the owning watchbook through the docs backbone.
- [ ] Portal/client read models are derived from released watchbook truth and respect actor scope.
- [ ] Customer-facing outputs hide personal names by default unless explicitly released.
- [ ] Generated-output behavior is deterministic enough to support audit/reproducibility.
- [ ] Tests cover docs linkage, release filters, and output shaping.

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
