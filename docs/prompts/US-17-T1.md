# Codex Prompt — US-17-T1

## Task title

**Implement customer orders, equipment lines, requirement lines, and attachments.**

## Objective

Create the operational/commercial order root that connects customer-owned commercial scope to the planning module without collapsing the separation between CRM pricing truth, planning structure, and downstream finance evidence.

## Source context

- Updated proposal: section 5.5 common project structure and project-details block; section 6.1 customer-order-to-report-and-invoice workflow steps 1–3; Appendix D customer order/project entity.
- Implementation specification: section 4.6 `ops.customer_order`, `ops.order_equipment`, and `ops.order_requirement_line`; order-to-planning split note; section 9 `PlanningAggregate`.
- Sprint reference: `docs/sprint/sprint-06-ops-planning-foundation.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-7-T1` must already provide customer master truth and stable customer identifiers.
- `US-8-T1` and `US-8-T2` should already provide customer billing/commercial settings that orders will reference indirectly, not duplicate.
- `US-16-T1` must already provide requirement-type and equipment-item catalogs.
- `US-6-T1` must already provide the docs backbone for order attachments.
- Sprint 4 HR catalogs should already exist for optional function/qualification references on requirement lines.

## Scope of work

- Create migrations, SQLAlchemy models, Pydantic schemas, services, routers, and tests for `ops.customer_order`, `ops.order_equipment`, and `ops.order_requirement_line`.
- Implement the order root with tenant-scoped `order_no`, customer FK, title, requirement type, service category, security concept text, date window, release state, notes, archive-safe lifecycle handling, and `version_no` if the repo uses optimistic locking on mutable roots.
- Implement equipment lines as normalized child rows referencing `ops.equipment_item`, with required quantity and note fields and uniqueness per `(order_id, equipment_item_id)`.
- Implement high-level requirement lines with optional function/qualification references, `min_qty`, `target_qty`, and notes, enforcing `0 <= min_qty <= target_qty`.
- Wire order attachments through the centralized docs service and `docs.document_link`/repo-equivalent patterns; keep the order as the owning business entity for those links.
- Add backend list/filter/detail APIs suitable for later Vben order/planning screens and later finance/timesheet/invoice references.
- Add tests for date validation, child-row uniqueness, attachment ownership, customer scoping, and archive-safe listing/query behavior.

## Preferred file targets

- Alembic migrations for order-root and child tables plus supporting indexes.
- `modules/planning/` or repo-equivalent models, schemas, services, routers, and tests.
- Docs-link helpers only where they preserve planning ownership and reuse the centralized docs model.
- No major UI work in this task beyond contract support for `US-17-T3`.

## Hard constraints

- Preserve the spec’s order -> planning-record split; `ops.customer_order` is not a disguised planning-record table.
- Keep customer commercial settings CRM-owned; do not copy rate cards, surcharge rules, invoice layouts, or invoice parties into the order schema unless the repository already has an approved snapshot pattern.
- Attachments must flow through the docs backbone, not through direct file paths or side tables.
- Do not implement mode-specific planning detail tables here; that belongs to `US-17-T2`.
- Do not implement shifts, staffing, field evidence, or finance actuals here.
- Any user-visible validation content introduced here must preserve German default and English secondary behavior.

## Expected implementation outputs

- A normalized order root with child equipment and requirement lines.
- Stable docs-backed attachment linkage owned by the order aggregate.
- Backend tests proving order validation, child uniqueness, attachment ownership, and tenant/customer scoping.

## Non-goals

- Do not implement planning-record mode details here; that is `US-17-T2`.
- Do not build the admin UI here; that is `US-17-T3`.
- Do not implement finance linkage logic beyond the clean references needed now; that is `US-17-T4`.
- Do not implement shift plans, templates, or concrete shifts here; those belong to `US-18-*`.

## Verification checklist

- [ ] Orders can be created, updated, archived, and listed with customer, equipment, requirement, and attachment data in tenant scope.
- [ ] Equipment-line uniqueness and requirement-line quantity validation are enforced consistently.
- [ ] Order attachments are represented through the docs service and remain linked to the order entity.
- [ ] Order dates and release-state transitions follow the approved validation logic.
- [ ] Tests cover the most important customer-scope, uniqueness, validation, and attachment behaviors.
