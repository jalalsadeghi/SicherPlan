# AI-18 — Employee Read-only Diagnostic Tools

## Task Objective

Implement read-only employee diagnostic tools for the SicherPlan AI Assistant.

These tools allow the assistant to safely inspect employee records, operational readiness, self-service/mobile access state, and employee-related blockers within the current user’s tenant and permission scope.

This task is a prerequisite for the Markus shift visibility diagnostic, but it must not implement the full Markus diagnostic flow yet. That belongs to `AI-21`.

The tools must never expose HR-private or payroll-sensitive fields unless existing repository permissions explicitly allow such access. For this sprint, keep outputs operational and redacted by default.

### 3. Tool: `employees.search_employee_by_name`

Purpose:

Find visible employees by name within the current authenticated user’s scope.

Input schema:

```json
{
  "name": "string",
  "limit": 5,
  "include_archived": false
}
```

Output schema:

```json
{
  "matches": [
    {
      "employee_ref": "safe-reference-or-uuid",
      "display_name": "Markus ...",
      "personnel_no": "optional-redacted-or-visible",
      "status": "active|inactive|archived|unknown",
      "is_active": true,
      "visibility_scope": "visible_to_current_user",
      "match_confidence": "exact|high|medium|low"
    }
  ],
  "match_count": 1,
  "truncated": false,
  "safe_message_key": null
}
```

Rules:

- Search only in current tenant/scope.
- Never search across tenants.
- If no visible employee is found, do not say whether the employee exists elsewhere.
- If multiple matches exist, return safe matching options that the current user may see.
- Apply limit.
- Redact fields according to role/permission.
- Do not include birth date, tax ID, bank data, social insurance, health insurance, religion, private HR notes, or payroll data.

### 4. Tool: `employees.get_employee_operational_profile`

Purpose:

Return a minimal operational employee profile needed for support/diagnostics.

Input schema:

```json
{
  "employee_ref": "uuid-or-safe-reference"
}
```

Output schema:

```json
{
  "found": true,
  "employee": {
    "employee_ref": "uuid-or-safe-reference",
    "display_name": "Markus ...",
    "personnel_no": "visible-or-null",
    "status": "active",
    "is_active": true,
    "default_branch_ref": "redacted-or-safe-reference-null",
    "default_mandate_ref": "redacted-or-safe-reference-null",
    "has_user_link": true,
    "user_status": "active|inactive|locked|unknown|null",
    "operational_notes_available": false
  },
  "redactions": ["hr_private_profile", "payroll_sensitive_fields"]
}
```

Rules:

- Return only operational fields.
- Do not include HR-private profile data.
- Do not include unrestricted notes unless the current role can see them and they are operationally safe.
- Do not expose full IAM account details; only status needed for diagnosis.

### 5. Tool: `employees.get_employee_mobile_access_status`

Purpose:

Check whether the employee appears ready for Employee Self-Service / mobile app access.

Input schema:

```json
{
  "employee_ref": "uuid-or-safe-reference"
}
```

Output schema:

```json
{
  "found": true,
  "mobile_access": {
    "has_linked_user_account": true,
    "linked_user_status": "active|inactive|locked|unknown|null",
    "self_service_enabled": true,
    "mobile_context_available": true,
    "can_receive_released_schedules": true,
    "blocking_reasons": [
      {
        "code": "missing_access_link|inactive_user|self_service_disabled|unknown",
        "severity": "blocking|warning|info",
        "message": "string"
      }
    ]
  },
  "missing_permissions": []
}
```

Rules:

- Use existing employee access-link/self-service logic where available.
- Do not create, attach, detach, or reconcile user accounts.
- If the current user lacks permission to inspect access-link details, return a safe missing-permission result.
- Do not expose username, email, tokens, password reset state, or credentials unless already generally visible in the existing UI and explicitly permitted. Prefer status-only output.

### 6. Tool: `employees.get_employee_readiness_summary`

Purpose:

Return operational readiness information relevant to planning visibility and assignment diagnostics.

Input schema:

```json
{
  "employee_ref": "uuid-or-safe-reference",
  "date": "YYYY-MM-DD|null",
  "include_qualifications": true,
  "include_absences": true,
  "include_availability": true
}
```

Output schema:

```json
{
  "found": true,
  "readiness": {
    "employee_status": "active",
    "has_active_absence_on_date": false,
    "availability_summary": "available|unavailable|unknown|not_checked",
    "qualification_summary": {
      "has_expired_qualifications": false,
      "has_missing_required_qualification": false,
      "details_redacted": true
    },
    "credential_summary": {
      "has_active_credential": true,
      "has_expired_credential": false,
      "details_redacted": true
    },
    "blocking_reasons": []
  },
  "missing_permissions": []
}
```

Rules:

- Use existing employee qualification/absence/availability/credential models if present.
- If no date is supplied, return general readiness only and mark date-specific checks as `not_checked`.
- For this task, do not evaluate assignment-specific required qualifications from demand groups; that belongs to planning diagnostics.
- Avoid exposing certificate numbers, document filenames, private notes, or sensitive HR data unless already permitted and needed. Prefer summary counts/statuses.

### 7. Safe Employee Reference Handling

The tools may use UUIDs internally, but model-facing output should use the repository’s existing safe reference approach if available.

Rules:

- Do not accept employee refs from another tenant.
- Every `employee_ref` must be resolved under current tenant and current scope.
- If inaccessible, return safe not-found-or-not-permitted response.
- Do not reveal whether an employee exists outside the current user’s scope.

### 8. Redaction Policy

Add or extend redaction rules for employee outputs.

Always redact or exclude:

```text
tax_id
social_insurance_no
health_insurer_name
health_member_no
bank fields
IBAN/BIC
religion
marital status if treated as HR-private
private HR notes
password hashes
reset tokens
JWTs
raw credentials
raw document storage keys
certificate numbers unless explicitly allowed
payroll amounts unless payroll permission exists
```

Prefer summaries:

```text
has_missing_documents: true/false
has_expired_qualification: true/false
has_active_absence_on_date: true/false
linked_user_status: active/inactive/unknown
```
