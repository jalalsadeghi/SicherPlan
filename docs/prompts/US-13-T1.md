# Codex Prompt — US-13-T1

## Task title

**Implement subcontractor master, contacts, scope assignments, and finance profile**

## Objective

Build the core SubcontractorAggregate so tenant administrators can manage subcontractor company master data, contacts, operational scope, and commercial profile from a normalized, tenant-safe backend foundation that later planning, portal, and finance flows can consume.

## Source context

- Updated proposal: section 5.4 subcontractor master file, including portal access, legal/company data, managing director, contacts, geolocation, assigned mandates/branches, tax and bank details, commercial terms, and function-based rates; section 6.3 step 1 of the subcontractor collaboration workflow.
- Implementation specification: section 4.5 `partner.subcontractor`, `partner.subcontractor_contact`, `partner.subcontractor_scope`, and `partner.subcontractor_finance_profile`; Phase C migration guidance in section 8; `SubcontractorAggregate` ownership in section 9.
- Uploaded Dienstplanmacher subcontractor source material: the Subunternehmerverwaltung sheet describes special function-based rates, geocoordinates, multi-mandate usage, object assignment intent, invoice options, and an own-login model for the subcontractor.
- Sprint reference: `docs/sprint/sprint-05-subcontractor-management.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-4-T1`, `US-4-T2`, and `US-4-T3` should already provide `core.tenant`, branch/mandate foundations, lookup domains, numbering/settings seams, and reusable address support.
- `US-5-T1` and `US-5-T3` should already provide `iam.user_account`, role-scope patterns, and authorization conventions needed for optional portal-user linkage.
- Use the current docs backbone from `US-6-T1` only where a master-data entity must be future-ready for attachments; do not front-load history attachment behavior here because that is `US-13-T3`.

## Scope of work

- Create migrations, SQLAlchemy models, Pydantic Create/Update/Read/List schemas, services, and API endpoints for `partner.subcontractor`, `partner.subcontractor_contact`, `partner.subcontractor_scope`, and `partner.subcontractor_finance_profile`.
- Support tenant-scoped subcontractor numbers, legal name, display name, legal form/status codes, managing director, notes, active/archive semantics, address linkage through `common.address`, and explicit latitude/longitude fields when geo search or map preview is enabled.
- Implement one or more subcontractor contacts with primary-contact handling, phone/email/mobile fields, optional `user_id` linkage to `iam.user_account`, and a portal-enable flag or equivalent portal-readiness marker.
- Implement subcontractor scope rows for branch and mandate assignment now, with date-effective windows and overlap-safe validation. If `ops.site` already exists in the repo, wire the optional site FK cleanly; otherwise keep the Phase C design compatible with a later Phase E site extension instead of inventing fake ops tables in this task.
- Implement the one-to-one finance/commercial profile with tax/VAT identifiers, payment terms, bank fields, invoice options, and invoice-status mode codes using lookup-backed fields rather than ad-hoc JSON blobs.
- Where the repository already expects explicit function-based partner pricing, add the minimal `partner.subcontractor_rate_card` / `partner.subcontractor_rate_line` seam now, or otherwise leave a clean extension point that does not bury rate logic inside the finance-profile row.
- Add service validation for duplicate subcontractor numbers, one-primary-contact rules, active-scope overlap rules, one finance profile per subcontractor, optional user-link safety, and optimistic locking where the repo already uses `version_no`.
- Add backend tests for CRUD success cases, validation failures, tenant scoping, contact linkage, and branch/mandate scope behavior.

## Preferred file targets

- Alembic migrations for partner master-data tables and related indexes/constraints.
- `modules/subcontractors/` or the repo equivalent for models, schemas, services, routers, validators, and tests.
- Shared lookup/address helpers only where needed to preserve module boundaries.
- No major UI work in this task except minimal contract notes or fixtures if the repo needs them.

## Hard constraints

- Subcontractors are scoped actors inside the tenant security model; do not model them as separate tenants or create a parallel auth store.
- Keep address truth in `common.address`; do not duplicate postal fields across the subcontractor master unless the repo has an established immutable snapshot pattern.
- Respect Phase C before Phase E: do not create fake site/object master tables merely to satisfy future scope assignment ideas. Use a deferred seam where operational location tables do not yet exist.
- Do not push history-entry, attachment timeline, or status-audit depth into this task beyond the base columns needed for mutable aggregate roots; those belong to `US-13-T3` and `US-13-T4`.
- Any user-visible validation content introduced here must preserve German default and English secondary behavior.

## Expected implementation outputs

- A normalized subcontractor master-data backend with contacts, operational scope, and finance profile.
- Stable tenant-safe API contracts reusable by admin UI, partner portal scope logic, planning releases, and finance checks.
- Migration and service/API tests covering the most important master-data constraints.

## Non-goals

- Do not build the admin screens here; that is `US-13-T2`.
- Do not implement subcontractor history/attachments or lifecycle-control UI depth here; that is `US-13-T3`.
- Do not finalize permission policy matrices or audit-event emission depth here beyond leaving clean seams; that is `US-13-T4`.
- Do not implement partner workforce tables here; that is `US-14-*`.

## Verification checklist

- [ ] New partner migrations apply cleanly on a fresh database and remain compatible with the Phase C-before-Phase E sequencing in the implementation spec.
- [ ] A subcontractor can be created, edited, archived, and queried with contacts, finance profile, and dated branch/mandate scope rows.
- [ ] One-primary-contact, one-finance-profile, and overlap/uniqueness rules are enforced consistently.
- [ ] Tenant scoping prevents reading or mutating another tenant's subcontractor records or linked contacts.
- [ ] Optional user-account linkage and portal-enable metadata remain tenant-safe and do not create orphan references.
