# Codex Prompt — US-13-T2

## Task title

**Build admin UI for company data, scope, and portal enablement**

## Objective

Deliver the subcontractor-management screens in the web admin so authorized tenant users can search, create, edit, archive, and portal-enable subcontractor companies through an idiomatic Vben Admin experience.

## Source context

- Updated proposal: section 5.4 subcontractor master file and collaboration portal intent, including portal access, contacts, legal/company data, geolocation, branch/mandate assignments, and commercial settings.
- Implementation specification: section 4.5 partner master tables and section 2 lifecycle/optimistic-lock conventions.
- Uploaded Dienstplanmacher subcontractor source material: the Subunternehmerverwaltung sheet shows master-data maintenance, location coordinates, multiple mandate use, and portal/login-oriented collaboration setup.
- Sprint reference: `docs/sprint/sprint-05-subcontractor-management.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-13-T1` must land first so the subcontractor API contracts and persistence rules are stable.
- `US-3-T1`, `US-3-T3`, and `US-3-T4` should already provide the Vben shell, theme tokens, and DE/EN localization baseline.
- `US-5-T3` should already provide route/action guard patterns that the admin pages can reuse.
- `US-4-T4` can be reused for branch/mandate selector patterns and archive-state UI conventions where appropriate.

## Scope of work

- Build Vben Admin list/detail pages for subcontractor company data, contacts, dated scope rows, commercial profile fields, and portal-enable controls on top of the backend contracts from `US-13-T1`.
- Provide searchable/filterable list views with subcontractor number, display/legal name, status, primary contact, branch/mandate summary, portal-enabled indicator, and archive-state filters.
- Create tabbed or sectioned forms for general company data, contacts, scope assignments, and finance/portal settings, with optimistic-lock-aware save flows and clear validation rendering.
- Support archive/reactivate and status-control actions with confirmation UX, reason capture if the backend exposes it, and permission-aware action visibility.
- Expose geolocation fields or a map-picker helper only through the repo's established map/input pattern; if map widgets are not standardized yet, use precise numeric fields and a future-safe display seam.
- Handle future site/object scope gracefully: if operational location tables are not present yet, the UI must not fabricate fake selectors. Use placeholders, help text, or disabled future sections instead of breaking Phase C sequencing.
- Add German and English resources for all new menu items, labels, empty states, action confirmations, and validation messages.
- Add frontend tests for list/detail flows, permission guards, archive/reactivate actions, localization, and portal-enable toggles.

## Preferred file targets

- Vben routes, menus, list/detail pages, composables, stores, API clients, and locale resources under the actual web-admin path.
- Shared subcontractor form components for company data, contacts, scope rows, and commercial fields.
- Frontend tests near the subcontractor admin pages.

## Hard constraints

- Use Vben Admin conventions for routing, list/detail layout, forms, drawers/modals, and reusable CRUD workflows; do not create an unrelated shell.
- Do not implement partner-portal login or external session flows here; those belong to `US-15-T1` and the IAM layer.
- Do not create fake site/object selectors just because the proposal mentions assigned objects; respect the implementation sequence and leave explicit future seams where ops data is not available yet.
- Status and portal-enable actions must rely on backend permission and optimistic-lock rules rather than client-only assumptions.
- All user-facing strings introduced here must ship in German and English and should use the shared light/dark design tokens rather than hardcoded colors.

## Expected implementation outputs

- A usable subcontractor-management admin UI with list/detail/create/edit/archive flows.
- Portal-enable and contact-management screens aligned with the approved Vben Admin shell.
- Frontend test coverage for core CRUD, permission, localization, and state-change flows.

## Non-goals

- Do not implement the history/attachment timeline here; that is `US-13-T3`.
- Do not implement the deeper access-policy or audit-event matrix here; that is `US-13-T4`.
- Do not build partner workforce maintenance screens here; that is `US-14-T2`.
- Do not build the subcontractor self-service portal here; that is `US-15-*`.

## Verification checklist

- [ ] Authorized users can search, create, edit, archive, and reactivate subcontractor records through the admin UI.
- [ ] Contact and scope-row editing behaves correctly and surfaces backend validations clearly.
- [ ] Portal-enable controls are permission-aware and do not assume login/session behavior that belongs to later portal tasks.
- [ ] German renders by default and English renders correctly when selected.
- [ ] Future site/object scope is represented safely without fake data models or broken selectors.

- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
