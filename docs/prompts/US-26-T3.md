# Codex Prompt — US-26-T3

## Task title

**Surface discrepancy flags for missed checkout, duplicates, and manual corrections.**

## Objective

Make attendance and actual derivation problems visible and actionable by surfacing structured discrepancy flags for incomplete, conflicting, or corrected evidence without breaking the raw append-only model or hiding why downstream finance results need review.

## Source context

- Updated proposal: section 5.6 and the three-stage reconciliation flow assume that deviations, absences, delayed relief, pauses, and other differences are visible before finance finalization.
- Appendix D and Appendix E describe time-record differences and validation/reporting expectations.
- Implementation specification: raw time events, attendance summaries, and `actual_record` remain separate layers, which means discrepancy visibility must connect them without inventing a hidden correction truth.
- Dienstplanmacher reconciliation materials explicitly describe employee notes, delayed handover remarks, and visible differences before final confirmation.
- Sprint reference: `docs/sprint/sprint-09-time-capture-and-actuals.md`.

## Dependencies

- `US-26-T1` and `US-26-T2` must already produce attendance summaries and actual bridge records.
- `US-20-T1` may already provide reusable validation/discrepancy result structures.
- If a Vben finance/operations review area already exists, reuse that shell instead of creating a parallel discrepancy UI module.

## Scope of work

- Implement discrepancy classification and persistence/read-model behavior for cases such as missed checkout, duplicate start/stop, invalid sequence, unresolved actor/context, and manual-correction-needed outcomes.
- Expose discrepancy data through backend contracts and, if the repository already has an operator review surface, through a narrow Vben list/detail review experience.
- Link discrepancies back to the raw events, attendance summaries, and actual bridge records that caused them, so reviewers can see provenance without opening unrelated modules.
- Support comments or operator notes only where the repository already has an approved audit-safe pattern; do not overwrite the original evidence to annotate it.
- Add tests for discrepancy classification, read-model correctness, and UI/API visibility if frontend work is included.

## Preferred file targets

- `modules/field_execution` or `modules/finance` discrepancy services/read models, depending on the repository’s current ownership split.
- Vben review/list screens only if there is already an operator-facing actuals/discrepancy area.
- Tests around discrepancy detection, linkage, and display contracts.

## Hard constraints

- Discrepancy surfacing must not become a second correction ledger that silently replaces attendance or actual truth.
- Reviewer notes or correction markers must stay auditable and separate from immutable raw event payloads.
- Do not implement full finance approval or adjustment line logic here; this task is about visibility and review readiness.
- Keep DE default / EN secondary behavior for any user-facing labels, statuses, and helper text introduced.
- Do not leak HR-private or customer-hidden data while surfacing discrepancy context.

## Expected implementation outputs

- Structured discrepancy classification and linkage for attendance/actual review.
- Backend contracts and optional narrow UI support for discrepancy visibility.
- Tests proving discrepancy detection and provenance links.

## Non-goals

- Do not finalize corrections, approvals, or payroll exports here.
- Do not rewrite raw events or attendance summaries when a discrepancy is found.
- Do not build the full reporting dashboard set for actuals in this task.

## Verification checklist

- [ ] Missed checkout, duplicate, and similar discrepancy states are surfaced through structured outputs.
- [ ] Each discrepancy links back to source raw events / attendance / actual context.
- [ ] Reviewer-facing visibility stays scoped and auditable.
- [ ] Immutable raw evidence remains untouched.
- [ ] Tests cover classification, linkage, and any review surface added.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
