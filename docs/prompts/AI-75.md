# AI-75 — Draggable, Dockable, and Discoverable SicherPlan Assistant Launcher

## Purpose

The Assistant launcher currently has a fixed bottom-right position and can overlap important UI actions such as `Apply demand groups`, `Next`, `Save`, sticky wizard footers, and right-aligned primary buttons.

The user must be able to move the Assistant launcher easily with mouse/touch and understand intuitively that it is movable.

Known current behavior:
- `SicherPlanAssistantWidget.vue` wraps `AssistantPanel` and `AssistantLauncher`.
- The widget is currently fixed via CSS at `right: 1rem; bottom: 1rem; z-index: 1050`.
- `AssistantLauncher.vue` is a normal button with click-to-toggle only.
- There is no drag, dock, smart placement, persisted position, or safe-area avoidance.

---

## Non-Negotiable Requirements

1. The Assistant launcher must remain visible and usable.
2. It must not permanently cover important bottom-right page actions.
3. User must be able to reposition it with mouse and touch.
4. The feature must be discoverable without confusing the user.
5. Clicking the launcher must still open/close the Assistant.
6. Dragging must not accidentally open/close the Assistant.
7. Position must be persisted per browser/device.
8. Position must be clamped inside the viewport.
9. Keyboard accessibility must remain intact.
10. Mobile behavior must be safe and compact.
11. Do not break Assistant panel behavior, conversation persistence, or API calls.

### Step 2 — Add Floating Position State

Add local UI state for assistant floating position.

Suggested shape:

```ts
type AssistantDockSide = 'bottom-right' | 'bottom-left' | 'top-right' | 'top-left' | 'custom';

interface AssistantFloatingPosition {
  x: number;
  y: number;
  dockSide: AssistantDockSide;
  userPinned: boolean;
  lastMovedBy: 'system' | 'user';
  updatedAt: string;
}
```

Store it in local storage, for example:

```text
sicherplan.assistant.floatingPosition.v1
```

Rules:
- Default position: bottom-right, same visual location as today.
- Persist position after drag ends.
- Restore on reload.
- If stored position is outside viewport after resize, clamp it.
- Provide a reset-to-default action.

### Step 3 — Implement Dragging

The launcher should support:
- mouse drag;
- touch drag;
- pointer events preferred;
- no text selection while dragging;
- click suppression after drag.

Recommended:
- Use `pointerdown`, `pointermove`, `pointerup`, `pointercancel`.
- Start drag only after movement threshold, e.g. 6px.
- If movement is below threshold, treat as normal click.
- If movement exceeds threshold, do not toggle panel on release.

### Step 4 — Add Drag Handle / Visual Discoverability

The user should understand it can be moved.

Add a subtle handle to the launcher, for example:
- grip icon `lucide:grip-horizontal` or `lucide:move`;
- tooltip/title: `Drag to move`;
- cursor: `grab` / `grabbing`;
- on first sessions, a subtle hint bubble:
  ```text
  You can drag me
  ```

### Step 5 — Docking / Snap Behavior

After dragging ends:
- If near an edge, snap to that edge.
- Keep at least `safeMargin = 12px`.
- Prefer the nearest horizontal edge: left or right.
- Keep vertical position where user placed it unless near top/bottom.
- On mobile, optionally snap to bottom-right or bottom-left only.

Examples:
- Drag from bottom-right to bottom-left → dock bottom-left.
- Drag upward along right edge → remain right side at new y position.
- Drag near center → snap to nearest side to avoid covering central content.

### Step 6 — Panel Placement Relative to Launcher

When the Assistant panel is open:
- Panel should open near the launcher.
- If launcher is on right side, panel aligns to right.
- If launcher is on left side, panel aligns to left.
- Panel must fit inside viewport.
- Panel must not go off-screen.
- The launcher should not be hidden behind the panel.

### Step 7 — Keyboard Accessibility

Provide keyboard-friendly movement.

Minimum:
- Launcher remains a real button.
- Add `aria-label` containing both open action and drag hint if appropriate.
- Add keyboard shortcuts when focused:
  - Arrow keys move launcher by 16px.
  - Shift + Arrow moves by 48px.
  - Home/End or a reset action can restore/dock if easy.
- Position changes via keyboard are persisted.
- Do not interfere with normal Enter/Space toggle.

### Step 8 — Reset Position

Add an unobtrusive way to reset position:
- context menu on right-click;
- small reset control in panel header;
- or double-click drag handle;
- choose the least intrusive approach.

Expected:
- User can restore default bottom-right position without clearing all app storage.
