# Codex Prompt — US-16-T3

## Task title

**Build admin UI and import tools for operational locations and route master data.**

## Objective

Deliver a Vben-based admin experience for ops master data so dispatchers and tenant administrators can manage sites, venues, fairs, routes, equipment, and requirement catalogs through searchable, localized CRUD screens with safe import workflows.

## Source context

- Updated proposal: section 5.5 planning modes and operational tools; technical architecture section 7 for the web stack; delivery section 8 Phase 3 planning engine.
- Implementation specification: section 4.6 ops master-data tables; Phase E sequencing; package map for `PlanningAggregate`.
- Sprint reference: `docs/sprint/sprint-06-ops-planning-foundation.md`.
- Cross-cutting rules: `AGENTS.md`, especially Vben Admin, DE/EN, and tenant/role-scope rules.

## Dependencies

- `US-16-T1` and `US-16-T2` must already expose stable backend APIs and validation behavior for all relevant ops master entities.
- `US-3-T1`, `US-3-T3`, and `US-3-T4` should already provide the Vben shell, theme tokens, and DE/EN localization baseline.
- `US-5-T3` should already provide permission/route guard patterns.
- If the repo already has an import-job pattern from `US-6-T4`, reuse it instead of inventing a second import mechanism.

## Scope of work

- Build Vben Admin routes, list pages, detail/edit pages, form dialogs/pages, and API bindings for requirement types, equipment catalog items, sites, event venues, trade fairs with zones, and patrol routes with checkpoints.
- Use Vben CRUD conventions for filters, table state, status badges, breadcrumbs, detail drawers/pages, permission keys, and archive-state handling.
- Support customer-aware filtering and nested editing flows where appropriate, such as trade-fair zones inside fair detail and checkpoints inside route detail.
- Provide geo-aware input/display behavior for coordinates and optional map preview hooks if the repository already has a compatible map component; otherwise keep the UI map-ready through structured coordinate components.
- Implement import tooling for operational masters using the repository’s approved upload/job pattern. If imports are asynchronous, surface job status, validation errors, and result summaries in the admin UI rather than hiding them in logs.
- Add German and English resources for all new labels, actions, hints, empty states, and validation messages.
- Add frontend tests and, where the repo supports them, end-to-end coverage for CRUD permissions, nested child editing, and import behavior.

## Preferred file targets

- Vben pages/components and route registrations under the web-admin workspace, for example planning/ops-master sections or repo-equivalent paths.
- Web API client bindings and composables/stores for ops master-data CRUD and import-job interaction.
- Locale resources in the repo’s existing i18n structure.
- Frontend tests for list/detail/forms, route guards, and import-state handling.

## Hard constraints

- Follow Vben Admin patterns instead of inventing a separate admin UI style or state-management approach.
- German must be the default locale and English must exist for every new user-visible string.
- Do not expose broad tenant-wide data without the intended permission and customer filters.
- Reuse the backend validation model; do not fork business rules into a contradictory frontend-only interpretation.
- If imports create result artifacts or error files, represent them through the approved docs/job flow rather than unmanaged browser downloads or temp-file shortcuts.
- Do not turn this task into a planning board or shift editor; this story is about master-data administration only.

## Expected implementation outputs

- Localized Vben CRUD screens for ops master data and nested route/fair child structures.
- Safe import tooling integrated into the repo’s approved job/upload pattern.
- Frontend tests for the most important permission, CRUD, nested-edit, and import flows.

## Non-goals

- Do not implement shift, staffing-board, or release UI here; that belongs to `US-17-*`, `US-18-*`, and Sprint 7.
- Do not implement live map routing, patrol capture, or field mobile workflows here.
- Do not create a second backend import pipeline outside the repository’s existing integration/job conventions.
- Do not expose finance, customer-portal, or field-execution data in these admin screens.

## Verification checklist

- [ ] Authorized users can create, edit, archive, and search ops master data through localized Vben screens.
- [ ] Nested route checkpoints and trade-fair zones can be maintained without breaking backend ordering or uniqueness rules.
- [ ] Import flows surface validation results and job status clearly and safely.
- [ ] German is default and English is present for all newly introduced UI strings.
- [ ] Frontend tests cover route guards, CRUD basics, nested child editing, and import-state behavior.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
