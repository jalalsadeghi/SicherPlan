## 4. Widget Placement

Mount the widget once near the app root so it is not destroyed during route navigation.

Preferred pattern:

```vue
<template>
  <ConfigProvider ...>
    <App>
      <RouterView />
      <SicherPlanAssistantWidget />
    </App>
  </ConfigProvider>
</template>
```

Adapt this to the actual `app.vue` structure.

Rules:

- The widget must not replace `RouterView`.
- The widget must not break existing providers.
- The widget must not be mounted inside a page component that gets destroyed on route change.
- The widget should hide itself based on capabilities/auth/public-page checks, not by being unmounted unnecessarily.

## 5. Visibility Rules

The widget should be visible only when all are true:

1. Assistant feature is enabled from capabilities.
2. Current user can chat (`can_chat === true` or equivalent).
3. User is authenticated.
4. Current route is not public unless the assistant is explicitly allowed there.
5. The assistant is not in a hard error state that requires hiding.

If exact auth/public-route detection is not straightforward, use the safest existing project pattern. If there is uncertainty, implement capability-based display and document the auth detection gap.

## 6. Launcher Requirements

The launcher must:

- appear in the bottom-right corner;
- use fixed positioning;
- not block major page actions more than necessary;
- have a high but controlled z-index;
- display a clear assistant/chat icon or label;
- show a loading/disabled state when capabilities are loading;
- open the panel on click;
- be keyboard accessible.

Suggested behavior:

```text
Collapsed:
  round or rounded button at bottom-right

Expanded:
  panel opens above/near launcher
```

## 7. Panel Requirements

The panel must:

- show a header with assistant title and close/minimize button;
- show a short description or empty-state message before first message;
- show message list;
- show message input;
- show loading state while a response is pending;
- show safe error state;
- render structured diagnosis/evidence/links;
- support internal scrolling for long conversations;
- remain usable on common desktop screen sizes;
- degrade reasonably on narrower screens.

Suggested size:

```text
desktop width: 380px–460px
desktop max height: 70vh–80vh
mobile/narrow width: calc(100vw - 24px)
```

Use existing design tokens/classes where possible. Do not introduce a heavy new UI dependency.

## 8. Message Rendering

Render:

### User message

- right-aligned or clearly marked as user;
- plain text, safely escaped;
- no markdown execution unless the project already has a safe renderer.

### Assistant message

- left-aligned or clearly marked as assistant;
- `answer` text;
- diagnosis/evidence if present;
- missing permissions if present;
- next steps if present;
- safe navigation links if present;
- confidence may be shown subtly if useful.

Do not render raw JSON to normal users.

If the structured response is missing, fall back to the message `content`.

## 9. Evidence Rendering

Create `AssistantEvidenceList.vue` or equivalent.

Render diagnosis items:

- severity: `info`, `warning`, `blocking`;
- finding;
- evidence summary, if present.

Rules:

- Keep evidence human-readable.
- Do not render hidden/raw tool payloads.
- Blocking findings should be visually distinguishable but not alarming.

## 10. Link Card Rendering

Create `AssistantLinkCard.vue` or equivalent.

For every backend-provided safe link:

- show label;
- show reason if present;
- click navigates through Vue Router;
- external navigation must not be used for internal app paths;
- no arbitrary HTML from backend is rendered.

If route name exists and current router supports it, use `router.push({ name, query })` only if path/params can be safely derived. Otherwise use path-based navigation.

Do not generate links on the frontend that were not returned by the backend.

## 11. Message Input Requirements

The input must:

- be a textarea or multi-line input;
- bind to the assistant store `draftInput`;
- send on button click;
- optionally send on Enter and insert newline on Shift+Enter, only if consistent with UX conventions;
- trim and prevent empty messages;
- disable while sending;
- update route context before sending;
- keep draft if send fails;
- clear draft after successful send.


Use existing i18n if available. Otherwise use neutral English fallback and rely on backend same-language answers.

## 12. Route Context Updates

Before sending a message:

1. Build current route context using the helper from `AI-22`.
2. Store it as `lastRouteContext`.
3. Include it in the `sendMessage` request.

The widget must not trust route context for authorization. It is only contextual information for the backend.

## 13. Accessibility and UX

Minimum requirements:

- launcher button has accessible label;
- close button has accessible label;
- input has accessible label or placeholder;
- focus enters panel reasonably after opening;
- keyboard users can send and close;
- loading state is announced visually;
- panel does not trap focus in a broken way;
- color-only severity should be avoided; include text/icon/label.

## 14. Security Requirements

- Do not render raw HTML from assistant responses.
- Do not use `v-html` for model-generated content.
- Do not execute links or scripts from assistant text.
- Do not generate unauthorized links on frontend.
- Do not expose provider details or internal errors to the user.
- Do not store secrets in component state.
- Do not add write-action buttons.
