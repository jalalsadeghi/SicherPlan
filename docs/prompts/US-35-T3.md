# Codex Prompt — US-35-T3

## Task title

**Complete German/English text review, print-template signoff, and accessibility QA.**

## Objective

Close the rollout-readiness gap by reviewing critical user-facing web, mobile, portal, notification, and print-output text for DE/EN completeness and by running accessibility QA on the highest-value flows before business signoff.

## Source context

- Updated proposal: assumptions state that UI labels, print templates, localization details, and final report layouts are finalized during design workshops; the deliverables include revision-safe PDFs and go-live support.
- Cross-cutting rules: `AGENTS.md`, especially German default, English secondary, exact approved theme tokens, and the ban on hardcoded mixed-language user-facing strings.
- Sprint reference: `docs/sprint/sprint-12-migration-uat-and-go-live.md`.
- This task closes quality on user-visible surfaces that were implemented throughout earlier sprints and must now be reviewed as one coherent product.

## Dependencies

- `US-34-T2` and `US-34-T3` must already provide finalized template metadata and representative generated outputs.
- `US-3-T3`, `US-3-T4`, and `US-22-T4` must already provide the baseline theme and localization approach across web and mobile.
- Key web/portal/mobile/print surfaces from earlier sprints must already be present so the review can target the real integrated product.
- `US-35-T1` should already expose the highest-value flows that need localization/accessibility review before signoff.

## Scope of work

- Audit critical web, mobile, and portal screens for missing translations, inconsistent terminology, hardcoded strings, validation messages, notification text, generated-output labels, and locale-sensitive formatting.
- Review representative print and PDF templates for translated labels, correct date/number formatting, required business fields, branding consistency, and signoff suitability using actual generated outputs.
- Run accessibility QA on the highest-value flows, including authentication, list/filter/detail pages, form submission, portal downloads, employee mobile actions, and printed-output readability.
- Add automated locale-completeness checks or linting where the repository supports it, and document a manual accessibility checklist for issues that cannot be automated.
- Record approved terminology or glossary decisions plus unresolved localization/accessibility issues that must block or follow go-live.
- Keep fixes narrow and review-oriented: clean up missing strings, labels, and accessibility defects without turning the task into a full UX redesign.

## Preferred file targets

- Locale resource files under the web, portal, mobile, and backend message areas already used by the repository.
- Generated-template configuration or docs areas used for print/PDF outputs.
- `docs/qa/` or a repository-equivalent area for localization and accessibility evidence/checklists.
- UI lint/test areas if the repository already supports locale or accessibility checks.

## Hard constraints

- German must remain default and English secondary everywhere user-facing text is introduced or corrected.
- Do not sign off print templates without reviewing real generated outputs.
- Accessibility findings must be recorded honestly even if they become release blockers or follow-up backlog items.
- Do not widen the task into a full product copy rewrite or full design-system rebuild.
- Preserve the approved theme tokens and avoid introducing new ad hoc styling while fixing accessibility issues.

## Expected implementation outputs

- Localization fixes or completeness checks for critical web/mobile/portal/notification/print surfaces.
- Print-template signoff evidence based on real outputs.
- Accessibility QA evidence, checklists, and any resulting issue list.
- A terminology/glossary or equivalent decision record if needed by the repository.

## Non-goals

- Do not add support for languages beyond German and English.
- Do not run a certification-grade accessibility program beyond the scoped QA needed for go-live readiness.
- Do not redesign broad user flows unless a narrow accessibility defect makes it unavoidable.
- Do not replace business acceptance with localization/accessibility-only signoff.

## Verification checklist

- [ ] No blocker missing translations remain on critical flows or generated outputs.
- [ ] Print/PDF templates were reviewed as real outputs, not only as config files.
- [ ] Accessibility QA evidence exists for the highest-value go-live flows.
- [ ] Terminology and formatting are consistent across DE/EN surfaces.
- [ ] Outstanding issues are captured honestly and linked to release readiness decisions.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
