# Codex Prompt — US-14-T1

## Task title

**Implement subcontractor workers, qualifications, document proofs, and credentials**

## Objective

Create the partner-owned workforce foundation so subcontractor workers, their qualifications, their proof documents, and their operational credentials can be managed cleanly before planning, field, and finance flows reference them.

## Source context

- Updated proposal: section 5.4 mandatory documents and qualification checks; section 5.4 IDs, attendance, and time evidence; section 6.3 step 3 where the subcontractor uploads staff master data, qualifications, and mandatory documents.
- Implementation specification: section 4.5 `partner.subcontractor_worker`, `partner.subcontractor_worker_qualification`, and `partner.subcontractor_worker_credential`; validation guidance in section 6; `SubcontractorAggregate` in section 9.
- Uploaded Dienstplanmacher subcontractor source material: subcontractor login pre-creates staff, mandatory documents must be uploaded before deployment, planning qualifications are checked, and worker IDs/badges can carry barcode or QR support.
- Sprint reference: `docs/sprint/sprint-05-subcontractor-management.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-13-T1` must already provide the subcontractor company master and ownership root.
- `US-6-T1` must already provide document metadata/link infrastructure for qualification proofs and worker documents.
- `US-12-T1` and `US-12-T3`, or the repo equivalent, should already provide shared qualification-type and credential conventions that partner workers can reuse rather than redefining.

## Scope of work

- Create migrations, SQLAlchemy models, Pydantic schemas, services, and API endpoints for subcontractor workers, worker qualifications, worker proof-document linkage, and worker credentials.
- Support worker master data such as `worker_no`, names, birth date where approved, contact fields, status, and notes, all tenant-safe through the owning subcontractor.
- Implement qualification rows using shared qualification types where available, including certificate number, issued date, validity date, status, and proof-document linkage through the docs backbone.
- Implement credential rows for badge/token/credential identifiers used in entry control or time capture, including credential type, credential number, encoded value, validity dates, and status, with tenant-safe uniqueness rules.
- Model the data so later badge/QR/barcode print generation can be added without schema rework, but do not build the full print-output feature in this task.
- Add validation for duplicate worker numbers per subcontractor, credential uniqueness, valid qualification date ordering, and document-link ownership safety.
- Write backend tests for CRUD, proof linking, qualification validity edge cases, credential uniqueness, and tenant/subcontractor ownership rules.

## Preferred file targets

- Partner workforce migrations, models, schemas, services, routers, and tests under `modules/subcontractors/` or the repo equivalent.
- Minimal shared helpers for qualification/document/credential reuse only when they preserve bounded-context ownership.

## Hard constraints

- Partner worker master data stays owned by the partner module; do not let planning or field modules become the write source for worker truth.
- Reuse shared qualification and credential taxonomies where they already exist; do not fork a second global qualification dictionary inside the partner module.
- Proof files must go through `docs.document` / `docs.document_link`; do not add raw file paths or ad-hoc blob columns to partner tables.
- Do not overreach into worker import/export or compliance dashboards here; those belong to `US-14-T2` and `US-14-T3`.
- Keep personal data lean and operationally justified; do not clone the full internal-employee private profile into subcontractor-worker records.

## Expected implementation outputs

- A normalized partner workforce backend with worker, qualification, proof, and credential entities.
- Stable APIs and service contracts for admin screens, portal edits, planning validation, field actor lookup, and finance references.
- Tests covering the core partner-worker constraints and document/credential behavior.

## Non-goals

- Do not build bulk import/export or workforce admin screens here; that is `US-14-T2`.
- Do not implement readiness dashboards or compliance completeness scoring here; that is `US-14-T3`.
- Do not create cross-module planning projections here; that is `US-14-T4`.

## Verification checklist

- [ ] Workers can be created and updated only inside their owning subcontractor scope.
- [ ] Qualification rows preserve type, certificate, validity, and proof-link semantics correctly.
- [ ] Credential numbers are unique and validity/date rules are enforced.
- [ ] Proof documents are linked through the docs backbone and remain tenant-safe.
- [ ] The resulting contracts are stable enough for admin UI, portal editing, and later planning/field/finance references.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
