# AI-20 — Field Released Schedule Visibility Tool

## Task Objective

Implement the read-only field/self-service visibility tool that determines whether a released schedule or assignment should be visible in the Employee Self-Service / mobile app.

This task connects planning assignments and shift release state to the employee-facing schedule visibility logic. It must remain read-only and must not implement the full Markus diagnostic orchestration. The full diagnostic service belongs to `AI-21`.

The tool must answer questions such as:

- “Given this employee and assignment, should the assignment appear in the employee app?”
- “Is this shift released in a way that the employee self-service schedule can show it?”
- “Is the schedule hidden because release, assignment status, employee access, or field/self-service visibility is incomplete?”

### 3. Tool Input Schema

The tool must support multiple safe entry points because callers may know only the employee and date, or may already have assignment/shift references.

Input schema:

```json
{
  "employee_ref": "uuid-or-safe-reference|null",
  "assignment_ref": "uuid-or-safe-reference|null",
  "shift_ref": "uuid-or-safe-reference|null",
  "date": "YYYY-MM-DD|null",
  "date_from": "YYYY-MM-DD|null",
  "date_to": "YYYY-MM-DD|null",
  "target_channel": "employee_self_service|mobile_app|null"
}
```

Validation rules:

- At least one of `assignment_ref`, `shift_ref`, or `employee_ref + date/date range` must be provided.
- Do not accept tenant ID.
- Do not accept user ID as authoritative.
- If the current user is an employee self-service user, ignore any supplied `employee_ref` that does not resolve to the current user’s own employee record.
- If multiple assignments match, return a bounded list and mark ambiguity.

### 4. Tool Output Schema

Output schema:

```json
{
  "found": true,
  "visibility": {
    "target_channel": "employee_self_service|mobile_app",
    "employee_ref": "uuid-or-safe-reference|null",
    "assignment_ref": "uuid-or-safe-reference|null",
    "shift_ref": "uuid-or-safe-reference|null",
    "visible": true,
    "visibility_state": "visible|not_visible|ambiguous|unknown",
    "confidence": "low|medium|high",
    "schedule_window": {
      "starts_at": "timestamp|null",
      "ends_at": "timestamp|null"
    },
    "checked_rules": [
      {
        "code": "employee_active|linked_user_active|self_service_enabled|assignment_active|shift_released|shift_plan_released|planning_record_released|employee_visibility_enabled|released_schedule_query_matches",
        "status": "passed|failed|warning|not_checked|unknown",
        "severity": "info|warning|blocking",
        "summary": "safe summary"
      }
    ],
    "blocking_reasons": [
      {
        "code": "employee_not_active|missing_access_link|inactive_linked_user|self_service_disabled|assignment_cancelled|assignment_archived|shift_not_released|shift_plan_not_released|planning_record_not_released|visibility_flag_disabled|stealth_mode|schedule_not_in_released_query|ambiguous_assignment|unknown",
        "severity": "blocking|warning|info",
        "message": "safe summary"
      }
    ],
    "missing_permissions": []
  },
  "matches": [],
  "redactions": []
}
```

Rules:

- Return `visible=true` only when the tool can verify all required conditions.
- Return `not_visible` when a blocking condition is verified.
- Return `unknown` when the repository lacks the required implementation or permission.
- Return `ambiguous` if multiple assignments/shifts match and the tool cannot determine the intended one.
- Do not expose HR-private, payroll-sensitive, document-sensitive, or credential-secret fields.

### 5. Visibility Rules to Check

The tool must check as many of the following as repository data allows.

#### Employee / Account Rules

Use employee tools/services from `AI-18` where possible.

Check:

```text
employee exists in current scope
employee is active
employee has linked self-service/mobile user account
linked user account is active
employee self-service/mobile context is available
current employee self-service user is checking only own schedule
```

#### Assignment Rules

Use planning tools/services from `AI-19` where possible.

Check:

```text
assignment exists in current scope
assignment actor is internal employee, not subcontractor worker
assignment belongs to the employee being checked
assignment is active/offered/confirmed according to app visibility rules
assignment is not cancelled/archived/draft-hidden
assignment date is inside requested window
```

#### Shift / Planning Release Rules

Check:

```text
shift exists and is active
shift is not cancelled or archived
shift release state allows employee visibility
shift plan release state allows downstream visibility
planning record release state allows downstream visibility
shift visibility flags do not suppress employee display
stealth mode does not suppress employee display if applicable
```

#### Released Schedule Query Rule

If the repository has an actual Employee Self-Service released schedules service or query, use it as the authoritative final visibility check.

Preferred behavior:

```text
Call/reuse the same service/query that powers GET /api/employee-self-service/me/released-schedules, but do not call HTTP from inside the backend if a service/repository exists.
```

If the only implementation is in a router and not reusable, refactor carefully into a read service only if it does not break existing behavior. Otherwise, create a small internal helper that matches the existing query logic and report the limitation.

### 6. Permission Behavior

For internal users:

- They may inspect employee app visibility only if they have `assistant.diagnostics.read` plus the relevant planning/employee read permissions.
- If they lack employee mobile access permission, return `missing_permissions` and `unknown` for account/mobile-specific rules.
- If they lack planning permission, do not inspect assignments/shifts.

For employee self-service users:

- They may inspect only their own released schedule visibility.
- They must not inspect another employee by name/ref.
- They must not see internal release diagnostics, hidden planning notes, or other employees’ assignments.

For customer portal users:

- They must not use this employee schedule visibility diagnostic unless the repository has a deliberate customer-facing support permission. Default: deny safely.

For subcontractor portal users:

- They must not use employee self-service schedule visibility tools for internal employees. Subcontractor worker visibility should be future work unless already explicitly supported.

### 7. Redaction Policy

Always exclude:

```text
HR-private profile fields
payroll amounts
bank/tax/social/health/religion fields
password hashes
reset tokens
JWTs
raw credential values
raw document storage keys
private HR notes
unrestricted internal planning notes
other employees’ identifiers for self-service users
```

Prefer safe summaries:

```text
linked user active: yes/no/unknown
assignment active: yes/no/unknown
shift released: yes/no/unknown
schedule query matched: yes/no/unknown
```
