# Codex Prompt — US-16-T4

## Task title

**Integrate address reuse, geo coordinates, and watchbook/patrol cross-links.**

## Objective

Harden the ops master-data foundation with shared-address reuse, consistent geo-coordinate handling, and forward-compatible cross-links to watchbook and patrol contexts so later field-execution modules can attach without schema rework.

## Source context

- Updated proposal: object/site planning with watchbook linkage; patrol/route planning with route map and route-oriented execution; employee app shift detail with meeting point/route via maps.
- Implementation specification: `common.address` in section 4.1; section 4.6 ops master tables; section 4.7 `field.watchbook` and `field.patrol_round_event` relationships; section 3.2 cross-service write discipline.
- Sprint reference: `docs/sprint/sprint-06-ops-planning-foundation.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-4-T1` must already provide `common.address`.
- `US-16-T1` and `US-16-T2` must already provide the operational master tables that will participate in these cross-links.
- Keep Sprint 8 and Sprint 9 field-execution tasks in mind, but do not implement their raw evidence tables here.

## Scope of work

- Normalize and finalize address-link behavior across sites, venues, fairs, and route-linked locations so the planning module always reuses `common.address` for postal truth.
- Clarify and implement entity-level geo-coordinate rules where the operational point is distinct from the shared postal address, for example route checkpoints, patrol meeting points, or venue/site operational coordinates.
- Add service/query helpers and DTO fields that expose address + geo data cleanly for later map, watchbook, patrol, and shift-detail consumers.
- Wire the explicit cross-links already implied by the spec, such as `ops.site.watchbook_enabled`, optional `ops.patrol_route.site_id`, and route/checkpoint references needed later by patrol-round capture.
- Where the current repository already has minimal watchbook or patrol context helpers, align them with the ops master model instead of letting them drift into parallel location truth.
- Add tests proving that address reuse, entity-specific geo points, and route/site cross-links behave consistently and remain tenant-safe.

## Preferred file targets

- `modules/planning/` services, projection helpers, validators, and API contracts.
- Shared geo/address utilities only where they do not dilute bounded-context ownership.
- Small migration adjustments only if the current repo drifted away from the approved address/geo model.
- No major UI work in this task.

## Hard constraints

- `common.address` remains the source of postal truth; do not reintroduce duplicated address columns into ops master tables.
- Distinguish postal-address data from operational-point coordinates when both are needed; do not overwrite one with the semantics of the other.
- Do not create `field.watchbook`, `field.patrol_round`, or other field-execution tables here just to “complete” the cross-link story.
- Cross-links must stay explicit and foreign-key-friendly; avoid polymorphic “owner object” shortcuts.
- Preserve tenant/customer scoping on every helper, projection, and lookup path.
- Any user-visible error or metadata labels introduced here must keep German default and English secondary behavior.

## Expected implementation outputs

- Consistent address/geo handling across ops masters.
- Stable forward-compatible cross-link helpers for later watchbook, patrol, and route-map workflows.
- Tests that reduce future schema drift between planning masters and field-execution references.

## Non-goals

- Do not implement watchbook creation, patrol rounds, or time-capture flows here; those land in later sprints.
- Do not build map-heavy admin or mobile screens here.
- Do not redesign the customer/address model owned by CRM or core; consume it correctly from planning.
- Do not broaden this task into order, planning-record, or shift implementation.

## Verification checklist

- [ ] Sites, venues, fairs, and routes reuse `common.address` correctly where postal truth is needed.
- [ ] Entity-specific geo coordinates remain available where operational map points differ from the shared postal address.
- [ ] Route-to-site and site-to-watchbook seams are explicit, tenant-safe, and queryable for later modules.
- [ ] No duplicate or conflicting location truth is introduced by helper DTOs or API projections.
- [ ] Tests cover the most important cross-link and geo/address edge cases.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
