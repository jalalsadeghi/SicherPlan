# Codex Prompt — US-19-T1

## Task title

**Implement demand groups, teams, team members, and assignment entities.**

## Objective

Build the normalized staffing-entity backbone inside the planning bounded context so each concrete shift can express required demand, team structure, and filled staffing rows for internal employees and subcontractor workers.

## Source context

- Updated proposal: section 5.5 shift and staffing model, operational tools, and Appendix D entities for demand group, assignment, and team; Appendix E makes clear that minimum coverage, qualification rules, and override logging all depend on a stable staffing backbone.
- Implementation specification: section 4.6 `ops.demand_group`, `ops.team`, `ops.team_member`, and `ops.assignment`; section 5 customer/employee/subcontractor relationship chains; section 9 `PlanningAggregate` and board-oriented planning ownership.
- Sprint reference: `docs/sprint/sprint-07-staffing-engine.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-18-T1` and `US-18-T2` must already provide planning records, shift plans, shifts, and the shift-generation backbone.
- `US-12-T1` must already provide function and qualification references used by demand groups.
- `US-14-T4` should already provide read-safe subcontractor-worker projections that planning can reference without writing partner truth.
- `US-5-T3` should already provide RBAC and tenant/branch/mandate scoping patterns for admin APIs.
- Keep `US-19-T3` in mind: this task should leave a clean seam for subcontractor releases without front-loading that workflow here.

## Scope of work

- Create Alembic migrations, SQLAlchemy models, Pydantic Create/Update/Read/List/Filter schemas, services, routers, and tests for `ops.demand_group`, `ops.team`, `ops.team_member`, and `ops.assignment`.
- Implement demand groups as normalized required-staffing rows per shift with `function_type_id`, optional `qualification_type_id`, `min_qty`, `target_qty`, `mandatory_flag`, and note/remark support.
- Implement teams as named operational groups that can belong to a planning record and/or a specific shift, with team-member rows that support exactly one actor (`employee_id` or `subcontractor_worker_id`) plus validity windows and optional role labels.
- Implement assignments as concrete filled staffing rows attached to one shift and one demand group, with employee-or-worker exclusivity, assignment status/source metadata, offer/confirm timestamps, team linkage, and remarks.
- Add list/query endpoints needed by later staffing-board and release work, but keep them intentionally domain-focused rather than prematurely optimized for every UI permutation.
- Add service validation for actor exclusivity, team-member exclusivity, target/min quantity sanity, one-active-assignment-per-shift-and-person behavior, and tenant/customer/plan ownership consistency.
- Emit business audit events through the existing audit backbone where the repository already has that pattern for sensitive planning mutations.

## Preferred file targets

- Alembic revisions for the new `ops.*` staffing tables, indexes, and constraints.
- `modules/planning/` (or the repo-equivalent planning package) for models, schemas, services, routers, validators, and tests.
- Shared query helpers only where they preserve planning ownership and do not move staffing truth into CRM, HR, or partner modules.
- No substantial web UI work in this task beyond API fixtures or DTO support needed by later Sprint 7 prompts.

## Hard constraints

- Planning owns demand groups, teams, and assignments. HR and partner modules remain read-only sources of people, qualifications, availability, and compliance facts.
- Use explicit foreign keys and check constraints; do not hide staffing structure in JSON blobs or polymorphic pseudo-links.
- `ops.assignment` and `ops.team_member` must preserve exactly-one-actor semantics for employee vs subcontractor worker.
- If the repository prefers adding `subcontractor_release_id` only once `US-19-T3` lands, keep the assignment design compatible with that future FK instead of forcing a half-finished release workflow here.
- Do not implement the validation engine, staffing-board orchestration, release workflow, or portal/mobile exposure in this task.
- Any user-facing enum labels or API message keys introduced here must preserve German default and English secondary behavior.

## Expected implementation outputs

- A migration-safe normalized schema for demand groups, teams, team members, and assignments.
- Planning-domain CRUD and query services that later validation, staffing-board, release, field, and finance work can reuse.
- Tests covering exclusivity rules, uniqueness behavior, tenant scoping, and the main planning ownership constraints.

## Non-goals

- Do not implement `ops.subcontractor_release` business behavior here; that belongs to `US-19-T3`.
- Do not build assign/unassign/substitute orchestration here; that belongs to `US-19-T2`.
- Do not build coverage dashboards or traffic-light UI here; that belongs to `US-19-T4`.
- Do not implement validation overrides or release-state gating here; those belong to `US-20-*` and `US-21-*`.

## Verification checklist

- [ ] The new staffing migrations apply cleanly on top of Sprint 1–6 schema waves.
- [ ] Demand groups, teams, team members, and assignments can be created, updated, archived, and queried within tenant scope.
- [ ] Exactly-one-actor constraints are enforced for team members and assignments.
- [ ] Assignments are linked explicitly to shifts and demand groups, with no hidden cross-context writes into HR or partner ownership.
- [ ] Tests cover exclusivity, uniqueness, scoping, and the most important invalid-shape failures.
