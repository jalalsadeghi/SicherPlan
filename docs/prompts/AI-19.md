# AI-19 — Planning Read-only Diagnostic Tools

## Task Objective

Implement read-only planning diagnostic tools for the SicherPlan AI Assistant.

These tools allow the assistant to safely inspect planning records, shift plans, concrete shifts, demand groups, assignments, assignment validations, release state, and visibility state within the current user’s tenant and permission scope.

This task provides the planning-side data needed for the Markus shift visibility diagnostic, but it must not implement the full diagnostic orchestration yet. The full diagnostic flow belongs to `AI-21`.

The tools must never mutate planning data. They must not create, update, assign, unassign, substitute, release, change visibility, generate outputs, queue dispatch messages, execute imports, or create validation overrides.

### 3. Tool: `planning.find_shifts`

Purpose:

Find visible shifts inside the current user’s planning scope by date, date range, employee reference, customer/order/planning-record reference, or free-text hints.

Input schema:

```json
{
  "date": "YYYY-MM-DD|null",
  "date_from": "YYYY-MM-DD|null",
  "date_to": "YYYY-MM-DD|null",
  "employee_ref": "uuid-or-safe-reference|null",
  "planning_record_ref": "uuid-or-safe-reference|null",
  "shift_plan_ref": "uuid-or-safe-reference|null",
  "customer_ref": "uuid-or-safe-reference|null",
  "limit": 10,
  "include_archived": false
}
```

Output schema:

```json
{
  "matches": [
    {
      "shift_ref": "uuid-or-safe-reference",
      "shift_plan_ref": "uuid-or-safe-reference|null",
      "planning_record_ref": "uuid-or-safe-reference|null",
      "customer_ref": "uuid-or-safe-reference|null",
      "starts_at": "2026-05-01T08:00:00+02:00",
      "ends_at": "2026-05-01T16:00:00+02:00",
      "status": "active|cancelled|archived|draft|unknown",
      "release_state": "draft|below_released|released|customer_visible|subcontractor_visible|unknown",
      "employee_visible": true,
      "customer_visible": false,
      "subcontractor_visible": false,
      "location_label": "redacted-or-visible-label|null",
      "match_reason": "date|employee|planning_record|shift_plan|customer|combined"
    }
  ],
  "match_count": 1,
  "truncated": false,
  "missing_permissions": []
}
```

Rules:

- Never accept tenant ID from the model or user message.
- Tenant and scope must come from backend auth context.
- If `date` is supplied, treat it as the business date in the client/user timezone when appropriate, but store/query against UTC timestamps consistently with repository conventions.
- If both `date` and range are supplied, use the stricter valid range or return a validation error if ambiguous.
- Apply row limit.
- Exclude archived/cancelled shifts by default unless `include_archived=true` and permission allows it.
- Return only fields useful for operational diagnostics.
- Do not expose internal-only planning notes unless already visible to the current user.

```json
  "assignment_status": "active|offered|confirmed|cancelled|archived|draft|null",
  "limit": 10,
  "include_archived": false
}
```

Output schema:

```json
{
  "matches": [
    {
      "assignment_ref": "uuid-or-safe-reference",
      "shift_ref": "uuid-or-safe-reference",
      "employee_ref": "uuid-or-safe-reference|null",
      "subcontractor_worker_ref": "uuid-or-safe-reference|null",
      "actor_type": "employee|subcontractor_worker|unknown",
      "assignment_status": "active|offered|confirmed|cancelled|archived|draft|unknown",
      "source_code": "manual|staffing_board|subcontractor_portal|import|unknown|null",
      "offered_at": "timestamp|null",
      "confirmed_at": "timestamp|null",
      "starts_at": "timestamp|null",
      "ends_at": "timestamp|null",
      "shift_release_state": "draft|below_released|released|unknown",
      "is_visible_candidate_for_employee_app": true
    }
  ],
  "match_count": 1,
  "truncated": false,
  "missing_permissions": []
}
```

Rules:

- Search only current tenant/scope.
- If `employee_ref` is supplied, resolve it under current tenant/scope using existing safe reference logic or employee tools/services.
- Do not reveal subcontractor-worker details unless current user has permission to see that worker.
- If an assignment points to a subcontractor worker, clearly mark `actor_type=subcontractor_worker`; it is not an employee app assignment.
- A cancelled/archived assignment must be returned only when requested and permitted, with clear status.

### 10. Safe Reference and Scope Rules

All tools must follow these rules:

- Never trust tenant ID, role, permission, or scope from tool input.
- Resolve all refs through current auth context.
- If a referenced entity is inaccessible, return a safe not-found-or-not-permitted result.
- Do not confirm whether inaccessible planning records, shifts, or assignments exist.
- Do not expose internal database IDs if the project has a safer reference format. If only UUIDs are used internally, return them only as safe refs according to existing API behavior.
