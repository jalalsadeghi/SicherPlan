# AI-52 — Assistant “Processing Request” Loading State for Message Submission

## Purpose

When the user submits a question, the assistant currently gives little or no immediate feedback for a few seconds, then the answer appears suddenly. The user should immediately see that the assistant is processing the request.

The project already has a visual pattern similar to a small “Processing request” pill. If an existing loading component/style exists, reuse it. If not, create a compact assistant-specific typing/processing indicator.

---

Codex must inspect current files before implementation:

```text
web/apps/web-antd/src/store/assistant.ts
web/apps/web-antd/src/components/sicherplan/assistant/SicherPlanAssistantWidget.vue

web/apps/web-antd/src/components/sicherplan/*
web/apps/web-antd/src/views/*
```

Known current behavior:
- `AssistantPanel.vue` shows `loadingConversation` spinner only when loading an existing conversation.
- `AssistantMessageInput.vue` changes button text to `Send...` when `sendingMessage` is true.
- `assistant.ts` adds the optimistic user message only after `startConversationIfNeeded()` completes, so a first message can appear delayed while the conversation is created.
- There is no clear assistant “thinking” bubble inside the message list.

---

## Requirements

1. User sees immediate feedback after pressing Send.
2. The user’s message appears immediately, even if the conversation must first be created.
3. A visible assistant processing/loading indicator appears while the backend is generating the answer.
4. The indicator disappears when the assistant answer arrives.
5. If the request fails, the indicator disappears and the draft/question is restored or a clear error is shown.
6. The input and send button remain disabled while sending.
7. Message list scrolls to the pending indicator.
8. The loading indicator should not be persisted as a real conversation message.
9. Loading indicator must be accessible.
10. Use existing styling/loading pattern if available, otherwise create a compact indicator.

## Prompt for Codex

You are working on the SicherPlan repository.

Implement **AI-52 — Assistant “Processing Request” Loading State for Message Submission**.

### Step 1 — Inspect Current Send Flow

Before coding, report:

1. Exact order of operations in `sendMessage()`.
2. Whether optimistic user message is added before or after conversation creation.
3. Whether `sendingMessage` is passed to message list.
4. Whether there is an existing loading/pending component or CSS class to reuse.
5. Current behavior on failed message submission.

### Step 2 — Move Optimistic User Message Earlier

Update the store so that after validation:

```text
user presses send
-> immediately add local user message
-> clear draft
-> show pending assistant indicator
-> create conversation if needed
-> send message to backend
-> replace pending indicator with assistant response
```

If the backend request fails:

```text
-> remove pending indicator
-> optionally remove optimistic user message or mark it as failed
-> restore draft input
-> show safe error
```

Choose the behavior that best matches current UX, but the user must not be left without feedback.

### Step 3 — Add Pending Assistant Indicator

Either:

#### Option A — Pseudo-message

Add a local UI-only message:

```ts
{
  id: "local-assistant-pending-...",
  role: "assistant",
  content: "",
  pending: true
}
```

Update `AssistantUiMessage` type with `pending?: boolean`.

#### Option B — MessageList prop

Pass `sendingMessage` and render a separate pending bubble at the bottom.

Preferred: Option A or B is acceptable if tests pass and no pending indicator is persisted to backend.

Indicator text examples:
- English: `Processing request`
- German: `Anfrage wird verarbeitet`
- Persian: `در حال پردازش درخواست`

Use existing i18n patterns.

### Step 4 — Visual Design

Suggested indicator:

```text
Assistant
[spinner] Processing request
```

or a pill similar to the existing platform loading pattern.

Requirements:
- compact;
- visible inside the assistant message list;
- compatible with dark mode;
- not visually overwhelming;
- works in the small panel.

### Step 5 — Accessibility

The pending indicator should have:

```html
role="status"
aria-live="polite"
```

The spinner should not be the only signal; include text.

### Step 6 — Button/Input State

Keep existing disabled behavior. Improve button if useful:

```text
Send -> Sending...
```

or localized equivalent.

The user should not be able to submit duplicate messages while `sendingMessage=true`.
