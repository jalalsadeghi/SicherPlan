# AI-50 — Inline Allowed Page Links and Hide Internal Page Codes in Assistant Answers

## Purpose

Improve the user-facing assistant answer UX.

The assistant currently writes page/workspace references like:

```text
Dies können Sie im Mitarbeiter-Workspace (E-01) überprüfen.
Dies erfolgt auf der Schichtplanungsseite (P-03).
```

This is not suitable for normal users. Page IDs such as `E-01`, `P-03`, `P-04`, `P-05`, `C-01`, `PS-01`, etc. are internal documentation/canonical page identifiers. The user should see human-readable page names, linked to the actual allowed route.

Expected user-facing style:

```text
Dies können Sie im Mitarbeiter-Workspace überprüfen.
```

where **Mitarbeiter-Workspace** is an inline link to the allowed page.

No extra button is needed in the text. Use an inline link.

## Non-Negotiable Requirements

1. Do not expose internal page IDs in the main answer text for normal users.
2. Do not remove page IDs from backend traces, source basis, or admin/debug payloads; only hide them from the main user-facing answer and normal source display when appropriate.
3. Inline links must only use allowed links from the backend response.
4. The model must not invent routes.
5. The frontend must not turn arbitrary URLs into links.
6. If a page has no allowed link, render the human-readable page name as plain text.
7. Keep source_basis available for QA/debug, but normal message text should be clean.

## Prompt for Codex

You are working on the SicherPlan repository.

Implement **AI-50 — Inline Allowed Page Links and Hide Internal Page Codes in Assistant Answers**.

### Step 1 — Audit Current Answer and Link Structures

Before coding, inspect and report:

1. Current `AssistantStructuredResponse` fields.
2. Current `links` schema.
3. Current `source_basis` schema.
4. How `page_id` and `path` are generated.
5. Whether `links[]` includes all pages mentioned in answers.
6. Whether `source_basis[]` has enough page labels to map page IDs to human names.
7. Whether the answer text currently contains page IDs generated by OpenAI or by backend post-processing.

### Step 2 — Backend Prompt Change

Update the assistant prompt instructions so OpenAI is told:

```text
For normal user-facing answers, do not show internal page IDs such as E-01, P-03, P-04, P-05, C-01, PS-01.
Use human-readable page/workspace names.
If an allowed navigation link exists for a page, include that page in the structured links list.
Do not invent page routes.
Use page IDs only inside source_basis/debug fields, not in the main answer text.
```

This prompt change is not enough alone; frontend/backend must still sanitize the final answer.

### Step 3 — Backend Sanitization / Normalization

Add a safe post-processing step after provider response and before returning to frontend:

```python
normalize_user_facing_answer_links(response, allowed_links, page_catalog)
```

Responsibilities:

1. Detect internal page IDs in the main answer text:
   - `E-01`
   - `P-03`
   - `P-04`
   - `P-05`
   - `C-01`
   - `PS-01`
   - `FI-01`, etc.
2. Remove visible page ID suffixes such as:
   - `(E-01)`
   - `(P-03)`
   - `— E-01`
   - `- E-01`
3. Preserve the human-readable page/workspace name.
4. Ensure `links[]` contains allowed link entries for pages referenced in the answer when available.
5. If link is not allowed, keep plain text only and do not expose the internal code.
6. Do not alter source_basis/debug data.

If this is easier to do entirely in frontend from `links[]`, Codex may implement it in frontend, but backend still must instruct model not to expose page IDs and should not rely only on model compliance.

### Step 4 — Frontend Inline Link Rendering

Replace plain answer rendering with a safe renderer that supports inline links only from `structured_response.links`.

Acceptable approaches:

#### Option A — Answer Segments
Add a structured field:

```ts
interface AssistantAnswerSegment {
  type: 'text' | 'link';
  text: string;
  link?: AssistantLink;
}
```

Backend returns `answer_segments`.

#### Option B — Frontend Linkifier
Frontend builds answer segments from:
- answer text;
- `structured_response.links`;
- known page labels/aliases;
- stripped page IDs.

This must never create links from arbitrary markdown/URLs. It must only link to `AssistantLink.path` values already returned by backend.

Preferred minimum behavior:
- link exact occurrences of `link.label`;
- if answer contains `Mitarbeiter-Workspace (E-01)` and an allowed link for `E-01` exists, display `Mitarbeiter-Workspace` as link and hide `(E-01)`;
- if answer contains `Schichtplanung (P-03)`, display `Schichtplanung` as link;
- if only `P-03` appears, replace it with the allowed link label if available.

### Step 5 — Link Behavior

Inline links should:

1. Use Vue Router navigation, not full page reload.
2. Use the same `open-link` flow as `AssistantLinkCard`.
3. Be styled like a normal text link.
4. Be keyboard accessible.
5. Have `aria-label` when helpful.
6. Not render as buttons.

### Step 6 — Source Basis Normal Display

In normal visible source basis, avoid leading with internal page IDs. For example:

Current:
```text
E-01 - employee_readiness - ...
```

Better:
```text
Mitarbeiter-Workspace — employee_readiness — ...
```

Page IDs may remain in `title`/tooltip/debug if needed, but should not be the primary visible label for normal users.

### Step 7 — Tests

Add tests:

Required cases:

   - UI renders `Mitarbeiter-Workspace` as inline link.
   - UI does not show `(E-01)` in answer text.
2. Answer contains `P-03`.
   - UI replaces/links it to the allowed label if link exists.
3. No allowed link exists.
   - UI hides page code but does not create a link.
4. Model-generated arbitrary URL is not linked.
5. Source basis still exists and debug data is not destroyed.
