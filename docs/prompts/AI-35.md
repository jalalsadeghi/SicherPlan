# AI-35 — Strict RAG Answer Pipeline: All In-Scope Answers Must Be OpenAI-Generated From Retrieved Context

## Purpose

Convert the SicherPlan Assistant from a deterministic/FAQ-like helper into a true **RAG — Retrieval-Augmented Generation** assistant.

The assistant must retrieve relevant SicherPlan-specific context at runtime and pass that context to the OpenAI model. The OpenAI model must generate the final natural-language answer. Backend code may collect facts, enforce permissions, execute tools, and validate output, but it must not generate canned final answers for in-scope user questions.

---

## Core Product Requirement

For every in-scope SicherPlan question:

```text
User question
  -> classify safety/scope/language
  -> build retrieval plan
  -> retrieve relevant context from SicherPlan knowledge and tools
  -> package grounding facts
  -> send grounding context to OpenAI
  -> OpenAI generates final structured answer
  -> backend validates links/permissions/output
  -> UI renders the result
```

The backend must not return prewritten/canned answers for:

- customer creation
- employee creation
- order creation
- plan creation
- shift planning
- assignment
- release
- app visibility
- finance/billing/payroll questions
- reporting questions
- portal questions

Direct backend responses are allowed only for:
- assistant disabled
- provider unavailable
- permission denied
- rate limit
- obviously unsafe request
- obviously out-of-scope request

## Prompt for Codex

You are working on the SicherPlan repository.

Implement **AI-35 — Strict RAG Answer Pipeline**.

### Step 1 — Audit All Canned Answer Paths

Search the assistant backend for any function that directly builds final user-facing answer text for in-scope platform questions.

Inspect at minimum:

```text
backend/app/modules/assistant/service.py
backend/app/modules/assistant/page_help.py
backend/app/modules/assistant/workflow_help.py
backend/app/modules/assistant/diagnostics/*
backend/app/modules/assistant/language.py
backend/app/modules/assistant/provider.py
backend/app/modules/assistant/prompt_builder.py
```

Known examples to inspect:
- `render_verified_employee_create_answer`
- `render_shift_visibility_answer`
- `no_verified_ui_label_message`
- workflow seed helpers that act like answer templates
- any `[MOCK RAG]` response path
- any code path that returns `answer` without provider generation for an in-scope question

### Step 2 — Refactor Tools to Return Facts, Not Final Answers

Tools and deterministic helper modules may return **facts**, not final answers.

Correct tool output pattern:

```json
{
  "source_type": "ui_action",
  "page_id": "E-01",
  "page_title": "Employees Workspace",
  "action_label": "Create employee file",
  "location": "top-right toolbar",
  "required_permissions": ["employees.employee.write"],
  "verified": true,
  "allowed": true
}
```

Incorrect tool output pattern:

```json
{
  "answer": "برای ایجاد یک Employee جدید، وارد Employees Workspace شوید..."
}
```

Refactor these domains so they only return facts:
- page help
- UI action catalog
- workflow help
- navigation links
- diagnostics
- knowledge retrieval

### Step 3 — Build a Single RAG Orchestrator

Create or refactor a central orchestrator, for example:

```text
backend/app/modules/assistant/rag_orchestrator.py
```

Responsibilities:

1. Receive user message, route context, actor context, classification, and language.
2. Build a retrieval plan.
3. Retrieve context from:
   - knowledge chunks
   - page route catalog
   - page help / UI action catalog
   - workflow facts
   - accessible pages
   - diagnostic tools when needed
   - current auth/permission context
4. Create a `grounding_context` package.
5. Call the provider.
6. If provider requests more tool calls, execute allowed tools and call provider again.
7. Return only the provider-generated final structured answer.

### Step 4 — Grounding Context Schema

Add or formalize a grounding schema such as:

```json
{
  "query": "user message",
  "detected_language": "fa",
  "response_language": "fa",
  "route_context": {},
  "auth_summary": {
    "tenant_scope": "tenant",
    "role_keys": [],
    "permission_keys": []
  },
  "retrieval_plan": {
    "intent_candidates": [],
    "likely_page_ids": [],
    "required_sources": []
  },
  "sources": [
    {
      "source_id": "string",
      "source_type": "knowledge_chunk|ui_action|page_route|workflow|diagnostic|auth_context|current_route",
      "source_name": "string",
      "page_id": "string|null",
      "module_key": "string|null",
      "title": "string|null",
      "content": "short redacted content",
      "facts": {},
      "relevance_score": 0.0,
      "verified": true,
      "permission_checked": true
    }
  ],
  "missing_context": [],
  "missing_permissions": []
}
```

### Step 5 — Prompt Builder Must Enforce RAG

Update `prompt_builder.py` so the OpenAI model receives clear instructions:

- Answer in the user's language.
- Use only the provided grounding context and allowed tool outputs for SicherPlan-specific facts.
- Do not invent button names, routes, workflows, field names, or permissions.
- If grounding context is insufficient, say what is missing.
- The final answer must be natural and expert-like, not a template.
- The model may explain uncertainty when context is incomplete.
- The model should use exact verified labels only when present in grounding sources.
- Never reveal inaccessible data.
- Never claim a record exists unless a backend tool returned it.

### Step 6 — Provider Must Be Called for In-Scope Questions

Modify `AssistantService.add_message` or the equivalent flow so that:

- in-scope platform questions never return final answer before provider call;
- UI how-to questions go through RAG + OpenAI;
- workflow questions go through RAG + OpenAI;
- diagnostic questions collect tool facts first, then go through OpenAI for final response;
- provider fallback to mock is not allowed unless explicitly configured by AI-34.

### Step 7 — Preserve Structured Output

The OpenAI final response must still use the existing structured schema:

```json
{
  "answer": "string",
  "confidence": "low|medium|high",
  "out_of_scope": false,
  "diagnosis": [],
  "links": [],
  "missing_permissions": [],
  "next_steps": [],
  "tool_trace_id": "string|null"
}
```

The `answer` field must be model-generated.

### Step 8 — Validate Links After Model Response

Even though OpenAI generates the final response, the backend must validate links before returning them.

Rules:
- Links must come from allowed navigation tools or known safe route catalog entries.
- Unauthorized links must be removed.
- If model invents a path, reject or sanitize it.
- Final response should include only allowed links.

### Step 9 — Required Regression Tests

Add tests proving these user messages call the provider and do not return canned answers:

```text
چطور میتونم یک پلن جدید برای مشتری ثبت کنم؟
چطور باید مشتری ثبت کنم؟
چطوری یک قرارداد جدید ثبت کنم؟
I want to create a new order. How can I do it?
How can I assign a new employee to a shift?
Wie kann ich einen neuen Auftrag erstellen?
```

For each test:
- use fake provider, not mock provider;
- assert provider `generate()` was called exactly once or more;
- assert grounding context was non-empty;
- assert response is not `[MOCK RAG]`;
- assert response did not come from `render_*_answer` helper;
- assert relevant sources include expected page candidates when available.

Expected page/source direction:
- customer creation -> `C-01`
- customer plan / planning for customer -> `C-01`, `P-02`, optionally `P-03`
- order creation -> `P-02`
- employee-to-shift assignment -> `E-01`, `P-03`, `P-04`, optionally `P-05`
- contract/document registration -> likely `PS-01` and/or customer/order context; if ambiguous, the model should ask a clarifying question instead of inventing a workflow.

### Step 10 — Remove or Quarantine Canned Rendering

Canned rendering functions may remain only if:
- they are used in tests to create synthetic fixtures; or
- they are renamed clearly as mock/test helpers; and
- they are never used in production `add_message` flow for in-scope questions.

Recommended:
- delete `render_verified_employee_create_answer` or move it to tests;
- delete deterministic final-answer generation in diagnostic service;
- retain only fact extraction and formatting utilities.

## Acceptance Criteria

This task is complete only when:

1. Every in-scope user message calls the provider.
2. Provider receives a non-empty `grounding_context`.
3. UI how-to answers are model-generated from retrieved facts.
4. Workflow answers are model-generated from retrieved facts.
5. Diagnostic answers are model-generated from retrieved tool facts.
6. Backend no longer returns canned final answers for in-scope questions.
7. Tests prove the provider is called for Persian, German, and English questions.
8. Tests prove `[MOCK RAG]` is not returned in OpenAI/fake-provider mode.
9. Links are validated after model generation.
10. If context is insufficient, the answer says so instead of guessing.
