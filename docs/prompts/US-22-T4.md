# Codex Prompt — US-22-T4

## Task title

**Ensure the mobile UI supports German default, English secondary, and the approved light/dark theme tokens.**

## Objective

Finish the employee-app experience so every Sprint 8 mobile screen follows the shared SicherPlan localization and theme rules, with German as default, English as secondary, and the approved light/dark primary colors consistently applied across the employee shell, schedule, document, notice, watchbook, and patrol surfaces.

## Source context

- Updated proposal: section 7 technical architecture and deliverables, plus the assumption that localization details are finalized during design workshops.
- Cross-cutting rules in `AGENTS.md`: German default, English secondary, exact primary colors light `rgb(40,170,170)` / dark `rgb(35,200,205)`, Prokit mobile baseline, and no hardcoded mixed-language UI.
- Earlier prompt context: `US-3-T3` and `US-3-T4` define the shared theme-token and i18n baseline; `US-22-T1` to `US-22-T3` introduce the Sprint 8 mobile screens that must now comply.
- Sprint reference: `docs/sprint/sprint-08-employee-app-and-field-i.md`.
- Brand assets and prior shell work from Sprint 1 remain the visual baseline for this task.

## Dependencies

- `US-3-T3` must already define the shared light/dark theme-token baseline.
- `US-3-T4` must already define the DE/EN resource structure and localization rules.
- `US-22-T1`, `US-22-T2`, and `US-22-T3` should already have introduced the relevant mobile surfaces that now need full compliance.
- If watchbook and patrol screens land in the same branch, apply the same i18n/theme rules there rather than deferring them.

## Scope of work

- Audit all Sprint 8 employee-app screens for localization completeness and remove hardcoded strings in widgets, validators, empty states, toasts, and action labels.
- Apply the approved light/dark primary colors and shared semantic token usage to employee mobile components so the app remains visually consistent with the web baseline.
- Ensure typography, badges, traffic-light indicators, QR/barcode containers, notice cards, and action buttons stay legible and coherent in both light and dark modes.
- Add locale-switch and theme-switch behavior or verify the shell wiring already provides them; fill any gaps needed for a demonstrable mobile result.
- Add regression tests or snapshot/widget coverage where feasible to keep i18n and theme drift from reappearing in later sprints.
- Document any remaining design-token or localization conventions that later field/mobile work must follow.

## Preferred file targets

- Flutter theme files, localization resources, and widget/component files touched by Sprint 8 mobile work.
- Shared mobile design-token helpers and localization utilities.
- Optional docs such as `docs/ux/theme-tokens.md` or a mobile localization note if the repo already uses such documentation.

## Hard constraints

- Use the exact required primary colors: light `rgb(40,170,170)` and dark `rgb(35,200,205)`.
- German must remain the default language and English the secondary language.
- Do not leave newly added Sprint 8 screens partially localized or styled with raw ad hoc color values.
- Keep theme and localization definitions centralized and reusable rather than scattering exceptions through widgets.
- Preserve readability, contrast, and field-operations usability in both light and dark modes.

## Expected implementation outputs

- Sprint 8 mobile screens fully wired to the shared DE/EN localization resources.
- Consistent light/dark theme application using the approved SicherPlan tokens.
- Regression coverage or documented checks that reduce future i18n/theme drift.
- Optional design/localization notes where the repository benefits from them.

## Non-goals

- Do not redesign navigation or introduce a new design system unrelated to the approved tokens.
- Do not rework business logic that already functions correctly unless a theme/i18n defect makes it necessary.
- Do not postpone string/token cleanup by leaving TODO placeholders in production UI.
- Do not change the agreed light/dark primary colors.

## Verification checklist

- [ ] All Sprint 8 employee mobile screens use localized DE/EN resources rather than hardcoded strings.
- [ ] The approved primary colors are applied consistently in both light and dark modes.
- [ ] Widgets remain legible and usable in field-oriented scenarios under both themes.
- [ ] Theme and i18n definitions stay centralized and reusable for later mobile work.
- [ ] Regression checks or widget/snapshot tests cover the highest-risk screens.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
