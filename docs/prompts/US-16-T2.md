# Codex Prompt — US-16-T2

## Task title

**Implement trade fair zones and patrol checkpoints with map/geolocation data.**

## Objective

Extend the ops master-data backbone with the fine-grained route and fair structures needed for hall/stand grouping, checkpoint-based patrol control, and future field evidence capture, while keeping the schema normalized and tenant-safe.

## Source context

- Updated proposal: section 5.5 trade-fair planning with hall/stand grouping and repeated-shift copy; patrol/route planning with route map, alarm objects, and travel-time logic; Appendix D control-round context.
- Implementation specification: section 4.6 `ops.trade_fair_zone` and `ops.patrol_checkpoint`; section 4.7 note that patrol checkpoints are later referenced by `field.patrol_round_event`.
- Sprint reference: `docs/sprint/sprint-06-ops-planning-foundation.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-16-T1` must already provide `ops.trade_fair` and `ops.patrol_route`.
- Shared lookup/status conventions from `US-4-T1` should already exist.
- Keep later field-execution tasks (`US-23-*`, `US-24-*`) in mind, but do not implement their append-only evidence tables here.

## Scope of work

- Create migrations, models, schemas, services, routers, and tests for `ops.trade_fair_zone` and `ops.patrol_checkpoint`.
- Implement trade-fair zones as explicit child records of one trade fair with normalized fields such as `zone_type_code`, `zone_code`, and label, plus any additional notes/status fields that align with the repository conventions.
- Implement patrol checkpoints as ordered child records of one patrol route with `sequence_no`, `checkpoint_code`, label, latitude, longitude, `scan_type_code`, expected token/value, and minimum dwell seconds as defined by the implementation spec.
- Add validation for unique zone tuples within one fair, unique sequence and checkpoint-code tuples within one route, safe sequence ordering, and valid geo/token fields.
- Add service/query helpers so later map previews, route editors, patrol apps, and fair-planning shift detail screens can consume the data through stable contracts.
- Where the repo already supports import-ready nested writes, allow batch create/update semantics for checkpoints and zones; otherwise implement clean child-collection endpoints that preserve ordering and uniqueness.
- Add tests for child CRUD, uniqueness, ordering, geo-field handling, and future-safe compatibility with later field-execution references.

## Preferred file targets

- Alembic revisions for zone/checkpoint tables and supporting indexes.
- `modules/planning/` or equivalent for child models, schemas, nested services, routers, and tests.
- Reusable geo/ordering validators only where they can be shared without pulling planning logic into a generic utilities module.
- No major UI work here; that belongs to `US-16-T3`.

## Hard constraints

- Keep patrol checkpoints and fair zones as real child tables with explicit foreign keys; do not collapse them into free-form JSON on routes or fairs.
- Do not build `field.patrol_round_event`, watchbook tables, or any append-only evidence records here.
- Do not bypass tenant scoping through nested child endpoints or list queries.
- Geo data must remain compatible with later map and patrol use cases; avoid lossy string-only coordinate storage.
- Preserve the distinction between route definition data and later patrol execution evidence.
- Any user-visible validation or list labels introduced here must keep German default and English secondary behavior.

## Expected implementation outputs

- Normalized trade-fair-zone and patrol-checkpoint tables with stable CRUD/query APIs.
- Ordering, uniqueness, and geo/token validation suitable for later map, patrol, and trade-fair planning work.
- Backend tests that prove the child structures behave predictably inside tenant scope.

## Non-goals

- Do not build the Vben editor screens or import tooling here; that is `US-16-T3`.
- Do not implement watchbook/patrol cross-link projection helpers beyond the clean FK seams needed for later work; that is `US-16-T4`.
- Do not implement live patrol capture, QR/NFC scanning workflows, or mobile offline sync here; those land in Sprint 8 and Sprint 9.
- Do not implement shift-level fair-zone or route-stop assignment logic here; that comes with planning and field tasks later in the plan.

## Verification checklist

- [ ] Zones can be created, edited, listed, and archived under a specific trade fair without duplicate `(trade_fair_id, zone_type_code, zone_code)` tuples.
- [ ] Checkpoints can be created, reordered, and queried under a specific patrol route with enforced sequence/code uniqueness.
- [ ] Latitude/longitude, scan type, expected token, and dwell-time fields validate correctly and survive round-trip API serialization.
- [ ] Nested child endpoints or batch operations remain tenant-safe and route/fair scoped.
- [ ] Tests cover ordering rules, uniqueness failures, and future-safe child references.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
