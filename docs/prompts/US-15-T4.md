# Codex Prompt — US-15-T4

## Task title

**Support partner updates to staff data, documents, and confirmations inside allowed scope**

## Objective

Finish the partner self-service loop so approved subcontractor portal users can maintain their own worker data, upload required documents, and confirm participation within the exact scope the prime company allows.

## Source context

- Updated proposal: section 5.4 subcontractor collaboration portal and section 6.3 steps 3 and 4, where the subcontractor maintains staff data, uploads documents, and confirms participation.
- Implementation specification: partner worker tables in section 4.5, docs-link rules in section 4.2, and portal visibility chain in section 5.
- Uploaded Dienstplanmacher subcontractor source material: own login for subcontractors, pre-creation of worker master data, mandatory document upload, and confirmation of assigned shifts before field execution.
- Sprint reference: `docs/sprint/sprint-05-subcontractor-management.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-15-T1` must already provide portal auth/navigation and effective subcontractor scope.
- `US-14-T1` and `US-14-T3` should already provide worker CRUD and readiness outputs.
- `US-6-T1` must already provide the centralized document backbone.
- `US-15-T3` should already provide the allocation/confirmation command seam if participation confirmations are part of the same workflow in the repo.

## Scope of work

- Build the subcontractor-portal pages and forms that let partner users create or update their own worker base records inside the allowed scope.
- Support document uploads and proof maintenance for partner workers through the centralized docs service, with clear visibility of missing or expiring required items.
- Expose allowed confirmation actions for participation or shift acceptance where the planning/release contract supports them, and surface the resulting status back to the portal user.
- Keep edits constrained to the logged-in subcontractor's own company and own workers. Where certain fields are admin-only, render them read-only or omit them in the portal rather than reusing admin forms blindly.
- Reuse the partner-worker APIs/contracts from `US-14-*` and add portal-specific adapters only when the portal UX genuinely needs different payload shapes.
- Add frontend tests for self-service worker edits, document upload visibility, permission boundaries, and confirmation flows.

## Preferred file targets

- Portal pages/components/forms, API clients, and locale resources under the web application path.
- Minimal backend DTO or endpoint adjustments only where the portal requires scoped payloads distinct from internal admin screens.
- Frontend tests and any necessary scope-specific backend tests.

## Hard constraints

- Portal users may edit only their own subcontractor and their own workers; never expose cross-partner or broad tenant maintenance capabilities.
- Use the centralized docs service for uploads and proof management; do not add a portal-only file-storage path.
- Do not reuse unrestricted internal admin forms in a way that accidentally exposes finance, audit, or other admin-only fields.
- Confirmation behavior must follow the approved planning/release contract; do not invent side-channel confirmation flags disconnected from operational truth.
- All new portal UI strings and status messages must be available in German and English and follow the shared theme tokens.

## Expected implementation outputs

- Partner self-service screens for worker maintenance and document/proof updates.
- Scoped confirmation flows where allowed by the underlying planning/release contract.
- Tests proving the portal remains limited to the partner's own data and actions.

## Non-goals

- Do not implement internal admin workforce management here; that is `US-14-T2`.
- Do not redesign the allocation engine here; that is `US-15-T3`.
- Do not add unrestricted finance-profile or company-level admin editing to the portal unless explicitly supported by the approved visibility model.

## Verification checklist

- [ ] A portal user can create or update only workers belonging to the linked subcontractor.
- [ ] Document upload/proof maintenance uses the docs backbone and reflects readiness gaps clearly.
- [ ] Confirmation actions are visible only where permitted and are written through the approved workflow boundary.
- [ ] Read-only versus editable field boundaries are enforced consistently for portal users.
- [ ] German default and English secondary coverage exists for all new self-service UI strings.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
