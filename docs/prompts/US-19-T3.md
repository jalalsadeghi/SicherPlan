# Codex Prompt — US-19-T3

## Task title

**Support subcontractor releases before worker-level assignment and team lead rules.**

## Objective

Implement the release-before-allocation workflow for subcontractor demand and the related team-lead rules so planning can reserve released staffing quantities for partners before concrete worker assignments are made.

## Source context

- Updated proposal: section 5.4 subcontractor collaboration portal, section 6.3 subcontractor workflow steps 2–4, and section 5.5 shift/staffing model all require released positions before concrete subcontractor-worker allocation.
- Implementation specification: section 4.6 `ops.subcontractor_release`, `ops.team`, and `ops.assignment`; section 5 subcontractor relationship chain; section 6 minimum-staffing and compliance enforcement points.
- Sprint reference: `docs/sprint/sprint-07-staffing-engine.md`.
- Cross-cutting rules: `AGENTS.md`, especially planning-vs-partner write discipline.

## Dependencies

- `US-19-T1` must already provide the core staffing entities and assignment model.
- `US-14-T1` and `US-14-T4` should already provide subcontractor-worker, qualification, and readiness truth that planning can read safely.
- `US-15-T3` should already define or anticipate the partner-side self-allocation contract that consumes released demand.
- `US-5-T3` must already enforce role scope on release-management APIs.

## Scope of work

- Create the migration, model, schemas, services, routers, and tests for `ops.subcontractor_release` with quantity, status, timestamps, and optional demand-group linkage as defined by the implementation spec.
- Implement planning-service commands for creating, amending, revoking, and listing subcontractor releases per shift/demand group/subcontractor combination.
- Enforce the business rule that released demand exists before worker-level partner allocation, and protect against over-release or worker assignment beyond released quantity.
- Integrate release-state data into assignment queries/commands so a subcontractor worker can only occupy capacity actually released to that subcontractor.
- Implement or tighten team-lead rules so a team can have no more than one lead reference and the selected lead actor is consistent with the team membership and shift/planning scope.
- Expose the minimal read-model/API seam that later subcontractor portal flows can consume without duplicating planning truth.
- Add backend tests for release quantity controls, cross-subcontractor denial, team-lead validity, and the main lifecycle/status transitions.

## Preferred file targets

- Alembic revisions and planning-domain files under `modules/planning/`.
- Small contract DTOs or adapters reused by subcontractor-portal flows, but still owned by planning for release truth.
- Tests covering release quantities, actor scope, and team-lead constraints.

## Hard constraints

- Planning owns released-demand truth. Partner modules and portals may consume it, but they must not own or back-write it.
- Released quantity must remain an explicit controlled value; do not infer it from downstream worker assignments alone.
- A subcontractor may only fill positions released to that subcontractor. Cross-partner and internal-staff misuse must be blocked.
- Do not blend invoice-check, attendance, or finance semantics into release rows. This task stays in the planning bounded context.
- Team-lead logic must remain compatible with both employee and subcontractor-worker actors without allowing dual-lead ambiguity.
- Any user-visible state labels or validation texts introduced here must preserve German default and English secondary behavior.

## Expected implementation outputs

- A normalized released-demand model and service flow for subcontractor staffing.
- Clear team-lead enforcement compatible with mixed internal/partner teams.
- Tests proving release-quantity integrity, partner scope safety, and lead-rule correctness.

## Non-goals

- Do not build the subcontractor portal UI here; that is handled by portal-facing tasks and later consumers.
- Do not implement invoice verification, actuals, or attendance evidence here.
- Do not build the general validation engine here, beyond the release-quantity and lead-shape checks required by this workflow.
- Do not broaden this task into customer-visible publication or employee mobile exposure; that belongs to `US-21-*`.

## Verification checklist

- [ ] Released subcontractor demand can be created and managed per shift/demand group without breaching tenant scope.
- [ ] Worker-level partner assignments cannot exceed or bypass released quantity.
- [ ] Team-lead rules prevent ambiguous or invalid lead states for mixed teams.
- [ ] Release data is exposed through planning-owned contracts, not partner-owned write models.
- [ ] Tests cover over-release, cross-subcontractor misuse, and invalid team-lead scenarios.
