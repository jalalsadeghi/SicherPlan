This task is **AI-09 — Out-of-Scope Classifier and Refusal Policy**.

## Goal

Implement a lightweight out-of-scope classifier and same-language refusal policy so the assistant only answers questions related to the SicherPlan platform.

The assistant is not a general chatbot. It must reject unrelated questions politely and in the same language as the user. Platform-related questions should continue through the normal assistant flow, even if the full provider/knowledge/tool implementation is not connected yet.

This task must not implement OpenAI provider logic, knowledge retrieval, tool registry, diagnostic tools, navigation links, or frontend UI. It only adds intent/scope classification and refusal behavior at the backend service level.

#### Out-of-Scope Signals

Classify as out-of-scope when there is no platform signal and the message asks about unrelated topics.

Do not make this too aggressive. A vague question like:

```text
"Why is this not visible?"
```

may be platform-related if the current route context is a SicherPlan page. Use route context as a weak signal, but never as an authorization source.

### 4. Unsafe Request Response

For obvious prompt-injection or unsafe requests, return a refusal in the same language. The message can be the same as out-of-scope or slightly more security-specific.

Suggested English:

```text
I cannot follow instructions that bypass SicherPlan security rules, permissions, tenant scope, or assistant policy. Please ask a question about the platform within your current access rights.
```

Suggested Persian:

```text
من نمی‌توانم دستورهایی را اجرا کنم که قوانین امنیتی SicherPlan، سطح دسترسی، محدوده Tenant یا سیاست‌های Assistant را دور بزند. لطفاً سوالی درباره پلتفرم و در محدوده دسترسی فعلی خود مطرح کنید.
```

Suggested German:

```text
Ich kann keine Anweisungen ausführen, die SicherPlan-Sicherheitsregeln, Berechtigungen, Tenant-Scope oder Assistant-Richtlinien umgehen. Bitte stellen Sie eine Frage zur Plattform innerhalb Ihrer aktuellen Zugriffsrechte.
```

### 5. Integrate into Message Flow

Update assistant service message flow:

```text
1. Store user message according to AI-06 behavior.
2. Detect language according to AI-08.
3. Classify message.
4. If out-of-scope or unsafe, return structured refusal response immediately.
5. Do not call provider, knowledge, tools, or diagnostics for refused messages.
6. If platform-related, continue to current placeholder/in-scope response until later tasks connect provider/knowledge/tools.
```

The structured refusal response must include:

```json
{
  "out_of_scope": true,
  "confidence": "high" or "medium",
  "diagnosis": [],
  "links": [],
  "missing_permissions": [],
  "next_steps": []
}
```

For unsafe prompt-injection, `out_of_scope` may be true and diagnosis may remain empty. Do not include sensitive security internals.

### 6. Persist Classifier Result Safely

If structured payload is stored, include non-sensitive classifier metadata such as:

```json
{
  "classification": {
    "category": "out_of_scope",
    "reason": "no_sicherplan_platform_signal"
  }
}
```

Do not store raw hidden policy text or detailed security internals.
