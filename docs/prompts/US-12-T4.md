# Codex Prompt — US-12-T4

## Task title

**Build self-service APIs for availability, free wishes, applications, and controlled profile updates**

## Objective

Expose authenticated employee-facing APIs that are ready for later Prokit-based mobile consumption, allowing employees to manage their own availability/free wishes, submit applications where supported, and request controlled profile updates without ever gaining unrestricted write access to HR-private master data.

## Source context

- Updated proposal: section 5.3 employee app and self-service scope, including availability/free wishes, event applications, document access, confirm/decline flows, and controlled self-service updates; workflow 6.2 step 5 where the employee provides availability, free wishes, and applications after onboarding.
- Implementation specification: `hr.employee_availability_rule`, `hr.employee_event_application`, employee/user linkage through `hr.employee.user_id`, and planning/HR write-discipline rules in section 3.2.
- Uploaded Dienstplanmacher employee-app and employee-file material: employee login, mobile schedule context, availability/free-wish entry, event applications, and the expectation that master-data changes are controlled rather than unrestricted.
- Sprint reference: `docs/sprint/sprint-04-recruiting-and-employee-core.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-11-T3` must already provide employee-to-user linkage so the authenticated actor can resolve to exactly one employee context.
- `US-12-T2` must already provide availability, absence, and any event-application persistence that is currently feasible.
- `US-5-T3` must already enforce authenticated role scope and own-record authorization.
- Because planning records land later, event-application endpoints must be wired only to the real planning contract when it exists; otherwise leave an explicit forward-compatible seam instead of inventing fake planning entities.

## Scope of work

- Implement authenticated self-service APIs scoped to the acting employee record for availability rules and free wishes.
- Implement controlled profile-update APIs for a whitelisted subset of operational fields such as mobile, email, preferred name, and current-address updates, with audit-safe handling and without exposing HR-private profile edits.
- When current-address updates are allowed, preserve address history correctly by closing/opening address-history windows rather than overwriting the past.
- Implement employee application submit/cancel/read endpoints only against the real event/planning contract when available; otherwise define forward-compatible request/response/service seams without fake planning data.
- Return stable, mobile-ready DTOs and error responses suitable for later Flutter/Prokit consumption, including DE default / EN secondary message resources.
- Add tests for own-record authorization, forbidden HR-private updates, address-history behavior on profile updates, and safe handling of the cross-phase event-application dependency.

## Preferred file targets

- Employee self-service API routers/controllers and service modules under `modules/employees/*`.
- Auth context resolvers that map the acting IAM user to the correct employee record.
- Schema files for self-service request/response DTOs and tests for own-record authorization.

## Hard constraints

- Employees may act only on their own allowed self-service surface.
- Do not expose or allow edits to HR-private profile fields such as tax, social-insurance, religion, health-insurance, or bank details through self-service APIs.
- Profile updates must use a whitelist and audit-safe handling; do not open unrestricted writes to the employee aggregate.
- Do not invent temporary planning records or fake event tables to support applications early.
- All new API-facing messages and validation errors must preserve German default and English secondary localization.

## Expected implementation outputs

- Own-record-scoped self-service APIs for availability/free wishes.
- Controlled profile-update APIs with address-history-safe behavior.
- Event-application API contracts that are either wired to the real planning contract or explicitly forward-compatible.
- Tests for self-scope authorization, field whitelisting, and address-history correctness.

## Non-goals

- Do not build the employee mobile UI here; that comes in later mobile sprints.
- Do not implement schedule reads, shift confirm/decline, document download, or push notifications here.
- Do not expose HR-private payroll or banking data to self-service.
- Do not create fake planning/event data models just to satisfy endpoint completeness.

## Verification checklist

- Authenticated employees can only read or mutate their own self-service data.
- Whitelisted profile updates cannot touch HR-private fields.
- Address updates preserve history correctly instead of overwriting prior addresses.
- Availability/free-wish APIs return stable, localized, mobile-ready DTOs.
- Event-application handling is either connected to the real planning contract or left as an explicit forward-compatible seam without fake dependencies.

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
