This task is **AI-10 — Provider Interface and Mock Provider**.

## Goal

Create the provider abstraction layer for the SicherPlan AI Assistant and implement a deterministic mock provider for local development, tests, and CI.

This task must **not** implement the real OpenAI API client yet. The real client belongs to AI-11. In this task, the goal is to define the stable backend contract that the assistant service will use later for both mock and real providers.

The result must let the assistant service call a provider in `mock` mode and receive a valid structured assistant response without a real API key, without network calls, and without non-deterministic behavior.

## Mandatory Constraints

- Do not call OpenAI or any external AI provider in this task.
- Do not add frontend code in this task.
- Do not implement the final tool registry in this task; that belongs to AI-15.
- Do not implement diagnostic tools in this task; those belong to later tasks.
- Do not implement knowledge retrieval in this task; only accept provider input that can include future knowledge snippets.
- Do not change unrelated business modules.
- Do not bypass the out-of-scope policy from AI-09.
- Do not bypass the same-language policy from AI-08.
- Do not store secrets, JWTs, access tokens, password reset tokens, or raw provider payloads in logs.
- Keep the implementation read-only.

## Implementation Scope

### 1. Add Provider Abstraction

Create or update:

```text
backend/app/modules/assistant/provider.py
```

Use repository conventions, but the provider layer should contain a clear abstraction similar to:

```python
class AssistantProvider(Protocol):
    async def generate(self, request: AssistantProviderRequest) -> AssistantProviderResult:
        ...
```

The exact implementation can use `Protocol`, `ABC`, or the project’s preferred style.

The provider request should be independent of the concrete AI vendor and should include at least:

```text
conversation_id
user_message
system_instructions or prompt/instructions
response_language
detected_language
route_context
recent_messages
knowledge_chunks
available_tools
max_tool_calls
max_input_chars
metadata / correlation_id / request_id
```

The provider result should include at least:

```text
final_response: structured assistant response or structured response payload
raw_text: optional debug-safe text only
requested_tool_calls: empty list for mock provider in this task
usage: token-like usage object, optional
provider_name
provider_mode
model_name
latency_ms or duration_ms, optional
finish_reason, optional
```

Keep the type names aligned with existing schemas. If `AssistantResponse`, `AssistantMessageResponse`, `AssistantDiagnosisItem`, `AssistantLink`, or similar schemas already exist from AI-07, reuse them instead of creating duplicates.

### 2. Add Provider Errors

Add explicit safe exceptions or error result types, for example:

```text
AssistantProviderError
AssistantProviderConfigurationError
AssistantProviderTimeoutError
AssistantProviderUnavailableError
AssistantProviderStructuredOutputError
```

Requirements:

- Exceptions must not expose secrets.
- Exceptions must be mapped by the assistant service to safe user-facing error messages.
- Internal logs may include provider mode and request ID, but must not include raw secrets or full sensitive payloads.

### 3. Implement Mock Provider

Create or update:

```text
backend/app/modules/assistant/provider.py
```

or, if cleaner:

```text
backend/app/modules/assistant/providers/mock.py
backend/app/modules/assistant/providers/__init__.py
```

The mock provider must be deterministic.

Required behavior:

- It must never call the network.
- It must never require `SP_OPENAI_API_KEY`.
- It must return a valid structured assistant response matching the AI-07 schema.
- It must respect `response_language` supplied by the assistant service.
- It must preserve technical terms such as `SicherPlan`, `Tenant`, `Shift Plan`, `Assignment`, `Release`, `Employee Self-Service`, and `RBAC`.
- It must support a simple platform-related response for normal questions.
- It must support provider simulation for error tests if useful, for example through a test-only flag or injected behavior.

### 5. Integrate Provider into Assistant Service

Update:

```text
backend/app/modules/assistant/service.py
```

The assistant message flow should be:

```text
1. Verify assistant feature flag and permission.
2. Load/create current conversation as implemented in AI-06.
3. Detect language as implemented in AI-08.
4. Run out-of-scope classifier as implemented in AI-09.
5. If out of scope, return refusal without provider call.
6. If in scope, build a provider request.
7. Call the configured provider.
8. Validate provider output against structured response schema.
9. Store assistant message.
10. Return structured response.
```

In this task, the provider request may include empty `knowledge_chunks` and empty `available_tools` because knowledge retrieval and tool registry are implemented later.
