# Codex Prompt — US-21-T4

## Task title

**Expose released schedule read models to portals and mobile clients.**

## Objective

Create the released-schedule read-model contracts that downstream customer portal, subcontractor portal, and employee mobile clients will consume, with strict scope filters and contract-first DTOs that remain stable as later field and finance modules land.

## Source context

- Updated proposal: employee app schedule scope in section 5.3; customer/subcontractor portal visibility and released outputs in sections 5.2, 5.4, and 5.5; operational note that customer-facing views hide personal names by default.
- Implementation specification: portal visibility chain and released-planning relationship guidance in section 5; reporting/read-model rules in section 7; release/visibility fields in section 4.6.
- Sprint reference: `docs/sprint/sprint-07-staffing-engine.md`.
- Cross-cutting rules: `AGENTS.md`, especially portal scoping, mobile-read discipline, and no shadow write-side schemas.

## Dependencies

- `US-21-T1` must already provide release-state and visibility semantics.
- `US-9-T2` should already provide customer-portal released-read patterns.
- `US-15-T2` should already provide subcontractor-portal released-read patterns.
- `US-12-T4` and the mobile shell work should already provide employee-facing self-service/read seams.
- `US-6-T1` should already provide docs-linked output metadata where released schedules include document references.

## Scope of work

- Implement query services, DTOs, and API endpoints for released schedule data consumed by customer portal, subcontractor portal, and employee mobile clients.
- Keep the contracts actor-specific where necessary, but derive them from one planning-owned released-data backbone instead of duplicating write-side truth.
- Support the released schedule fields that downstream clients need now: timing, location/meeting point, role or demand context, assignment/confirmation status, related released documents where appropriate, and visibility-safe display values.
- Apply tenant scope plus local actor scope on every query path. Customer-facing views must hide personal names by default unless explicit release rules already authorize otherwise.
- Design the employee/mobile contract so it is easy for the Prokit-style Flutter client to consume later, but do not implement the mobile UI itself in this task.
- Add tests for customer, subcontractor, and employee scope filters; released-only behavior; and the default name-hiding rule for customer-facing schedules.

## Preferred file targets

- `modules/planning/` query services, read DTOs, and API endpoints.
- Shared portal/mobile contract packages only where the repository already has an approved pattern for read DTO reuse.
- Tests focused on scope filtering, released-only gating, and visibility-safe field shaping.

## Hard constraints

- Do not create new write-side portal/mobile tables. Read models must come from released planning truth.
- Every query path must apply tenant scope plus actor-local visibility scope.
- Customer-facing views hide personal names by default unless an approved explicit release rule says otherwise.
- Released schedule reads must not accidentally expose draft, unreleased, finance-sensitive, or HR-private data.
- Keep DTOs explicit and versionable so later field/actuals/pay-data additions do not break clients.
- Do not implement the mobile or portal UI here; this task is about backend read contracts.

## Expected implementation outputs

- Released-schedule query services and API contracts for customer portal, subcontractor portal, and employee mobile clients.
- Visibility-safe DTOs that respect actor scope and default name-hiding rules.
- Tests proving released-only gating, actor-specific scope filters, and stable contract behavior.

## Non-goals

- Do not implement actuals, payroll, pay-data, or invoice visibility here.
- Do not implement watchbook or patrol read models here unless the repo needs a tiny already-approved contract seam.
- Do not build mobile or portal UI screens here.
- Do not create shadow write-side schedule copies for external clients.

## Verification checklist

- [ ] Customer, subcontractor, and employee clients each receive only their own released schedules.
- [ ] Customer-facing schedule reads hide personal names by default unless explicitly released.
- [ ] The read models are contract-first and derived from planning truth rather than duplicated tables.
- [ ] No draft, hidden, finance-sensitive, or HR-private data leaks through the released schedule endpoints.
- [ ] Tests cover actor scope, released-only behavior, and stable DTO shaping.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
