# Codex Prompt — US-32-T2

## Task title

**Implement QM views for notice read stats, free Sundays, sickness/vacation, and inactivity.**

## Objective

Extend the compliance/QM reporting layer so controller and QM users can inspect mandatory-notice read evidence, free-Sunday compliance, sickness/vacation visibility, and inactivity indicators through reproducible read models grounded in information, HR, finance, and IAM truth.

## Source context

- Updated proposal: section 5.7 requires read tracking/statistics for QM plus compliance reporting for free Sundays, sickness, vacation, and inactive users; Appendix E repeats notice-read, free-Sunday, sickness/vacation, threshold, and inactivity outputs.
- Implementation specification: `rpt.compliance_status_v` is built from compliance rules, qualification tables, document links, `info.notice_read`, HR absence tables, and IAM status; section 5 also identifies the information/QM chain as `info.notice -> notice_audience -> notice_read -> compliance/QM report views`.
- Sprint reference: `docs/sprint/sprint-11-reporting-and-hardening.md`.
- Cross-cutting rules: `AGENTS.md`, especially durable read evidence, tenant-safe role scoping, and reporting as read-only views/materialized views.

## Dependencies

- `US-23-T1` must already provide mandatory notice publication, audience resolution, and read-confirmation truth.
- `US-12-T2` and downstream HR/finance work must already provide absences, leave/balance context, and schedule/actual data needed for free-Sunday and sickness/vacation reporting.
- `US-31-T1`/`US-31-T2` and `US-32-T1` should already establish the reporting-package conventions and view/service/filter seams used here.
- IAM work from `US-5-*` should already provide account status and login/last-activity fields needed for inactivity signals.

## Scope of work

- Extend `rpt.compliance_status_v` or add repository-equivalent QM/compliance subviews so mandatory notice-read statistics, free-Sunday status, sickness/vacation visibility, and inactivity signals can be queried cleanly.
- Implement notice-read metrics from `info.notice`, `notice_audience`, and `notice_read`, including audience counts, read completion, overdue/unread state, and durable evidence references where the repository already stores them.
- Implement free-Sunday reporting using approved actuals, schedules, absences, or repository-equivalent attendance/work patterns in a way that matches the business interpretation already encoded in normalized data and settings.
- Implement sickness/vacation reporting from HR absence truth, including counts, status windows, and filters by person/group/function where repository conventions already support them.
- Implement inactivity signals using IAM status and repository-available last-activity indicators; if threshold settings are not yet configurable, create an explicit seam and document the default rather than hardcoding magic numbers everywhere.
- Expose read-only services/endpoints and tests covering notice-read completion, free-Sunday calculations, absence visibility, and inactivity thresholds or defaults.

## Preferred file targets

- `modules/reporting/` compliance/QM view definitions, query services, schemas, routers, and tests.
- Optional small shared helpers for threshold windows or evidence summarization if the repository already uses them.
- No controller/QM export screen implementation in this task.

## Hard constraints

- Do not mutate notice-read, absence, schedule, or IAM status data from the reporting layer.
- Keep the reporting logic reproducible from normalized source tables and explicit settings/default seams.
- Free-Sunday and inactivity logic must be documented and testable; avoid untraceable business heuristics hidden inside controller code.
- Preserve tenant isolation, role scoping, and HR/privacy boundaries. These are internal QM/controller views, not general tenant-wide unrestricted user lists.
- Do not broaden this task into security-activity auditing or scheduled export delivery.

## Expected implementation outputs

- QM/compliance reporting support for notice-read statistics, free Sundays, sickness/vacation, and inactivity.
- Read-only endpoints and filter/result schemas grounded in normalized information, HR, finance, and IAM truth.
- Tests covering notice-read completion, free-Sunday logic, absence windows, inactivity detection, and scope safety.

## Non-goals

- Do not implement security-activity views here; that belongs to `US-32-T3`.
- Do not build Vben export screens or scheduled delivery here; that belongs to `US-32-T4`.
- Do not change absence, notice, or IAM lifecycle rules in this reporting task.
- Do not invent a full HR analytics module beyond the scoped QM outputs.

## Verification checklist

- [ ] Notice-read stats, free-Sunday status, sickness/vacation, and inactivity can all be queried reproducibly.
- [ ] Threshold/default behavior for inactivity or free-Sunday interpretation is explicit and test-covered.
- [ ] Scope and privacy boundaries remain correct for HR/QM-sensitive data.
- [ ] No source-of-truth mutation or hidden duplication occurs.
- [ ] Tests cover representative normal, edge, and empty-state periods.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
