# Codex Prompt — US-28-T3

## Task title

**Build payroll export batches/items and provider-specific file/API adapters.**

## Objective

Generate payroll outputs in a controlled, auditable, adapter-based way so approved finance actuals and effective pay settings become provider-ready payroll export batches without direct side effects in the same domain transaction.

## Source context

- Updated proposal: section 5.6 requires gross pay export/API support and section 6.2 finishes the employee lifecycle with payroll export/import and optional payslip archiving.
- Implementation specification: section 4.8 defines `finance.payroll_export_batch` and `finance.payroll_export_item`; section 10 says generated files must be represented as `docs.document` rows and external side effects must use integration endpoints/jobs/outbox patterns.
- The broader technical architecture in the proposal explicitly keeps payroll import/export in the integration layer alongside messaging, maps, terminals, and scanner workflows.
- Sprint reference: `docs/sprint/sprint-10-payroll-billing-and-settlement.md`.
- Cross-cutting rules: `AGENTS.md`, especially the outbox rule, finance-bridge rule, and document-linkage requirement for generated files.

## Dependencies

- `US-28-T1` and `US-28-T2` should already provide tariff/profile resolution and employee payroll hooks.
- `US-26-T2` and `US-27-T1` through `US-27-T3` should already provide the approved finance actuals and reconciliation lines that exports must consume.
- `US-6-T1` should already provide document storage/linking for generated export files.
- `US-6-T4` should already provide integration endpoints, job records, and transactional outbox processing patterns that payroll adapters can reuse.

## Scope of work

- Implement or complete migrations, models, schemas, services, and APIs for payroll export batches and export items, including batch lifecycle status, provider targeting, export timestamps, and source lineage.
- Build generation commands that select only eligible approved actuals for a chosen period/provider, resolve pay basis using tariff tables and employee pay profiles, and create itemized export rows with pay codes, quantities, amounts, currency, and source references.
- Generate provider-specific file content or outbound API payloads through adapter interfaces that can be versioned and tested independently from domain write logic.
- Represent generated export artifacts as `docs.document` rows linked to the export batch and keep provider dispatch asynchronous via integration jobs or outbox events rather than inline network calls.
- Make export generation idempotent and retry-aware so duplicate provider submissions, partial reruns, or stale-state races are detectable and controlled.
- Add automated tests for eligibility filtering, export item provenance, idempotency, adapter serialization, outbox/job handoff, and failure/retry behavior.

## Preferred file targets

- `modules/finance/` export services, schemas, routers, and tests.
- Integration adapter and outbox/job wiring under `modules/platform_services/` or the repository-equivalent infrastructure package.
- Alembic migrations for batch/item tables or status/index adjustments if the schema is not yet present.
- Optional engineering notes under `docs/engineering/` if the repo benefits from documenting provider adapter boundaries.

## Hard constraints

- Do not export from raw time events, attendance summaries, or draft actuals; exports must derive from approved `finance.actual_record` plus effective payroll settings.
- Do not call external payroll providers directly from the same transaction that persists export batches/items.
- Generated export files and provider payload lineage must remain traceable back to the batch, employee, and source actual records.
- Keep export reruns idempotent or explicitly versioned; do not silently overwrite historical export truth.
- Do not bury provider-specific logic inside generic finance domain models when it belongs in adapter or integration layers.
- Protect provider credentials and generated payload metadata according to existing integration/security patterns.

## Expected implementation outputs

- Payroll export batch/item persistence and domain services.
- Provider-adapter interfaces plus at least one implementation seam for file or API delivery.
- Document-linking and outbox/job handoff for asynchronous export delivery.
- Tests proving eligibility, provenance, serialization, and idempotent delivery behavior.

## Non-goals

- Do not implement payslip import/archive in this task; that belongs to `US-28-T4`.
- Do not implement statutory payroll calculation engines beyond the approved tariff/profile/allowance basis already defined in the repository.
- Do not ship every production payroll provider in depth if the current branch only needs a stable adapter contract and one representative implementation.
- Do not expose raw provider secrets or payloads through general admin logs or broad API reads.

## Verification checklist

- [ ] Only approved eligible actuals are selected into payroll export batches.
- [ ] Every export item is traceable to employee, actual-record, pay code, quantity, amount, and provider batch context.
- [ ] Generated export artifacts are linked to `docs.document` and provider delivery happens asynchronously through integration patterns.
- [ ] Idempotency/retry behavior is explicit and test-covered.
- [ ] Tests cover serialization, failure paths, and safe rerun behavior.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
