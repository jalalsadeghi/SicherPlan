# Codex Prompt — US-29-T1

## Task title

**Implement timesheets and timesheet lines from approved actual records.**

## Objective

Create the customer-facing timesheet backbone so released timesheets can be generated reproducibly from approved finance actuals, scoped customer/order/planning context, and billing granularity rules without breaking audit linkage back to operational evidence.

## Source context

- Updated proposal: section 5.6 customer billing includes billing by shift/object/event/patrol and released timesheets for the customer portal; workflow 6.1 ends with released report packages, timesheets, and invoices after planned-vs-actual reconciliation.
- Implementation specification: section 4.8 defines `finance.timesheet` and `finance.timesheet_line`; section 5 keeps the customer order chain and finance result chain stable end to end; customer-facing result views should hide personal names by default unless explicitly released.
- Earlier uploaded customer-management material established invoice parties, commercial rules, layouts, and portal visibility patterns that the updated proposal carries into customer billing.
- Sprint reference: `docs/sprint/sprint-10-payroll-billing-and-settlement.md`.
- Cross-cutting rules: `AGENTS.md`, especially finance-bridge discipline, document linkage for generated outputs, role-scoped visibility, and least-privilege customer-facing data.

## Dependencies

- `US-26-T2` and `US-27-T1` through `US-27-T4` should already provide approved `finance.actual_record` rows and approval status needed for customer-facing timesheets.
- `US-17-T1` through `US-17-T4` should already provide order, planning-record, and mode-specific operational context that timesheets may reference.
- `US-8-T1` through `US-8-T4` should already provide customer billing profile, invoice parties, and commercial-rule settings consumed later by invoicing.
- `US-21-T1` and `US-21-T4` should already provide release/visibility semantics that timesheet publication can align with.

## Scope of work

- Implement migrations, models, schemas, services, and APIs for `finance.timesheet` and `finance.timesheet_line` if they are not already fully present in the repository.
- Generate timesheets from approved actual records for a customer/order/planning scope and period, enforcing deterministic logical uniqueness according to the chosen billing granularity.
- Create line items that link back to `finance.actual_record` and `ops.shift`, carrying planned minutes, actual minutes, billable minutes, and customer-readable descriptions appropriate to event, object/site, fair, or patrol planning modes.
- Support release/status transitions and customer visibility flags so a timesheet can move from draft/internal review to released/customer-visible state without losing provenance.
- Prepare the document seam for later PDF generation by linking a generated document placeholder or completed PDF document where repository conventions already support it.
- Add automated tests for timesheet-scope uniqueness, source-actual linkage, release gating, name-hiding or customer-safe description shaping, and tenant/customer scope filtering.

## Preferred file targets

- `modules/finance/` billing-timesheet models, schemas, services, routers, and tests.
- Alembic migrations for timesheet/timesheet-line tables, indexes, and release-status support where needed.
- Optional document-generation seam under the docs service if a timesheet PDF placeholder or artifact is produced in this task.
- Reporting/query helper files only if they support deterministic timesheet generation without violating finance ownership.

## Hard constraints

- Do not generate timesheets from raw time events or draft attendance summaries; timesheets must derive from approved `finance.actual_record` rows.
- Each timesheet line must remain traceable to its source actual and operational context.
- Do not expose draft/unreleased timesheets or internal finance notes through customer-facing contracts.
- Customer-facing descriptions and result views should hide personal names by default unless an explicit release rule already allows them.
- Do not collapse timesheet generation into invoice rendering or commercial dispatch behavior in this task.
- Keep logical timesheet scope uniqueness deterministic for the chosen customer/order/planning period.

## Expected implementation outputs

- Timesheet and timesheet-line persistence plus generation services.
- Release/status and visibility behavior aligned with finance/customer workflows.
- Stable linkage from timesheet lines back to actual records and shifts.
- Tests proving reproducibility, scope safety, and customer-safe output shaping.

## Non-goals

- Do not implement invoice numbering, tax totals, PDF invoice rendering, or dispatch logic here; those belong to later tasks in this story.
- Do not create broad customer-portal UI surfaces here unless a narrow backend contract seam needs a minimal read endpoint.
- Do not duplicate commercial settings inside finance tables that already live in CRM.
- Do not expose sensitive internal reconciliation details in customer-facing timesheet data.

## Verification checklist

- [ ] Timesheet generation selects only approved eligible actual records for the requested customer scope and period.
- [ ] Logical uniqueness prevents duplicate timesheets for the same intended scope.
- [ ] Timesheet lines remain linked to actual records and operational context with customer-safe descriptions.
- [ ] Release and visibility flags gate customer-facing exposure correctly.
- [ ] Tests cover generation, duplication prevention, source linkage, and name-hiding or safe-description behavior.
