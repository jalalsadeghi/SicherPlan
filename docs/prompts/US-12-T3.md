# Codex Prompt — US-12-T3

## Task title

**Implement time accounts, allowances, advances, and ID credentials/badges**

## Objective

Create the normalized compensation-support and credential foundation so time-value balances, allowances, advances, and system-issued badges can be maintained once in HR and later consumed by finance, field execution, and access-control workflows.

## Source context

- Updated proposal: section 5.3 absence/HR administration and IDs/document outputs, including time-value accounts, allowances, advances, company ID cards, work badges, QR/barcode support, and operational verification use cases.
- Implementation specification: `hr.employee_time_account`, `hr.employee_time_account_txn`, `hr.employee_allowance`, `hr.employee_advance`, `hr.employee_note`, and `hr.employee_id_credential`; generated-output guidance through docs rows.
- Uploaded Dienstplanmacher employee and payroll material: time-value account handling, advances, flexible allowances by employee/function/qualification, PDF badge generation, barcode/QR usage for entry control and time capture.
- Sprint reference: `docs/sprint/sprint-04-recruiting-and-employee-core.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-11-T1` must already provide the employee master foundation.
- `US-12-T1` should already provide function and qualification catalogs for allowance scoping.
- `US-6-T1` must already provide docs storage/linkage if badge outputs or generated files are produced.
- Keep future finance and field consumers in mind, but do not hand ownership of these HR rows to later modules.

## Scope of work

- Implement normalized time-account tables and append-only time-account transaction rows for employee balance movement.
- Implement employee allowance rows with effective dates, basis codes, and optional function/qualification references, including overlap validation on the approved basis tuple.
- Implement employee advances with status and outstanding-amount semantics suitable for later payroll export consumption.
- Implement employee ID credential records with credential number, encoded value, validity windows, and status aligned with badge/time-capture use cases.
- If the repo already has document-generation infrastructure, add a minimal badge/ID output path that represents generated badge files as docs rows linked to the employee; otherwise create a stable generation seam without inventing a parallel file model.
- Write tests for time-account transaction aggregation, allowance overlap rules, advance lifecycle basics, credential uniqueness, and generated-output linkage where applicable.

## Preferred file targets

- Employee bounded-context models, services, and schemas under `modules/employees/*`.
- Docs/output integration helpers for badge generation or issuance records.
- Tests for temporal overlap, append-only transactions, and credential uniqueness.

## Hard constraints

- Time-account transactions must be append-only; corrections should add new rows, not rewrite history.
- Allowance temporal overlap checks must be centralized and test-covered.
- Credential numbers must remain tenant-unique and suitable for later QR/barcode/NFC consumers.
- Do not bypass the docs service if badge or ID files are generated.
- Do not implement final payroll calculation, time-capture ingest, or terminal workflows here.

## Expected implementation outputs

- Normalized time-account, allowance, advance, and credential models/services.
- Append-only transaction behavior for time accounts.
- Credential/badge issuance foundation ready for later field/time-capture workflows.
- Tests for temporal integrity, transaction correctness, and credential uniqueness.

## Non-goals

- Do not implement full payroll export or payslip logic here.
- Do not build terminal/device ingest or raw time-capture workflows here.
- Do not build a full badge-design studio or unrestricted template editor.
- Do not let finance own HR allowance or advance master truth.

## Verification checklist

- Time-account movements are append-only and balances derive correctly from transactions.
- Allowance rows enforce effective-date overlap rules on the approved basis tuple.
- Advances track status and outstanding amount consistently.
- Credential numbers are unique per tenant and validity/status rules are enforced.
- If badge outputs are generated, they are represented through docs/document links.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
