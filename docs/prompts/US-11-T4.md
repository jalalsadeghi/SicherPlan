# Codex Prompt — US-11-T4

## Task title

**Enforce audit logging for HR-sensitive changes and archival lifecycle**

## Objective

Make the employee aggregate compliance-safe by ensuring HR-sensitive edits, archive/reactivate transitions, and access-affecting changes are fully auditable and lifecycle-controlled without destructive deletes.

## Source context

- Updated proposal: section 5.3 HR data protection requirement, where sensitive HR data must be restricted to authorized roles and all critical HR changes remain logged for audit and compliance.
- Implementation specification: audit foundations, aggregate lifecycle conventions (`status`, `archived_at`, `version_no`), and `EmployeeAggregate` ownership.
- Uploaded employee-management source material: role-differentiated visibility, administrative control over employee file lifecycle, and the expectation that personnel evidence remains traceable.
- Sprint reference: `docs/sprint/sprint-04-recruiting-and-employee-core.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-11-T1` must already provide the employee/private/address foundation.
- `US-11-T2` and `US-11-T3` should already provide at least some real write paths that now need audit/lifecycle coverage.
- `US-5-T4` must already provide the audit-event infrastructure.

## Scope of work

- Define and implement audit coverage for HR-sensitive employee changes, including private-profile edits, address-history changes, status changes, user-linkage changes, archive/reactivate actions, and other high-risk personnel updates handled in Sprint 4.
- Implement archive-safe lifecycle services for employee records using approved status and `archived_at` semantics rather than hard deletes.
- Ensure archived employees drop out of default operational queries where appropriate while remaining available for authorized historical/audit access.
- Coordinate archival behavior with linked app-access and credential visibility rules without deleting IAM or document evidence.
- Add optimistic-lock or version-aware checks where the repo pattern already uses `version_no` on mutable aggregate roots so concurrent HR edits do not silently overwrite each other.
- Write tests for audit-event emission, archive/reactivate flows, default query filtering, and protection against hard-delete shortcuts.

## Preferred file targets

- Employee lifecycle services and repositories under `modules/employees/*`.
- Audit integration/adapters under `modules/iam` or `modules/platform_services/audit/*` depending on repo layout.
- Tests for audit emission, lifecycle transitions, and archived-query behavior.

## Hard constraints

- Do not hard-delete employee aggregates, private profiles, or linked evidence to simulate archiving.
- Audit rows must remain append-only and attributable to the acting user.
- Do not expose archived employees accidentally in default operational lists intended for active workforce management.
- Do not treat archive/reactivate as UI-only toggles; lifecycle logic belongs in backend services.
- Keep DE/EN user-facing messages aligned where lifecycle actions return localized UI/API feedback.

## Expected implementation outputs

- Audit coverage for HR-sensitive employee changes.
- Archive/reactivate lifecycle services and query behavior.
- Tests for append-only audit evidence and archive-safe behavior.
- A compliance-ready employee lifecycle foundation for later planning, field, and finance consumption.

## Non-goals

- Do not build final audit dashboards or reports here.
- Do not redesign the global audit-event schema.
- Do not implement payroll or planning side effects beyond safe lifecycle hooks and visibility changes.
- Do not create destructive cleanup jobs for archived employee data.

## Verification checklist

- HR-sensitive changes emit audit events with actor attribution.
- Archive/reactivate flows update status and `archived_at` consistently.
- Archived employees are excluded from default operational queries but remain historically traceable.
- Hard deletes are prevented or absent from the relevant lifecycle paths.
- Tests cover audit completeness and lifecycle edge cases.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
