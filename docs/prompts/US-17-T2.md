# Codex Prompt — US-17-T2

## Task title

**Implement planning_record with mode-specific detail tables for event/site/trade fair/patrol.**

## Objective

Build the core planning-record aggregate that turns one customer order into one or more operational planning instances, with explicit 1:1 mode-specific detail tables instead of ambiguous JSON or duplicated commercial data.

## Source context

- Updated proposal: section 5.5 planning modes and common project structure; Appendix D planning instance entity; section 6.1 workflow step 3.
- Implementation specification: section 4.6 `ops.planning_record`, `ops.event_plan_detail`, `ops.site_plan_detail`, `ops.trade_fair_plan_detail`, and `ops.patrol_plan_detail`; section 9 `PlanningAggregate`.
- Sprint reference: `docs/sprint/sprint-06-ops-planning-foundation.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-17-T1` must already provide the order root.
- `US-16-T1` and `US-16-T2` must already provide sites, venues, fairs, zones, routes, and checkpoints that the mode-specific detail tables can reference.
- `US-5-T1` should already provide `iam.user_account` for dispatcher references.
- Keep later `US-18-*` shift-plan work in mind, but do not front-load shift-plan tables into this task beyond the FKs and service seams required by the spec.

## Scope of work

- Create migrations, models, schemas, services, routers, and tests for `ops.planning_record` and the four mode-specific detail tables.
- Implement `ops.planning_record` with order FK, planning mode code, optional parent-planning-record FK, name, planning window, dispatcher user, status, release state, and optimistic-locking behavior where the repo uses `version_no`.
- Implement one explicit 1:1 detail table per planning mode: event, site, trade fair, and patrol, using PK/FK child rows as defined by the spec.
- Add service validation that a planning record can have exactly one detail row compatible with its `planning_mode_code`, with safe handling of parent event plans and multi-day planning scenarios where applicable.
- Add list/filter/detail APIs so later order/planning UIs can navigate by customer, order, planning mode, status, release state, date range, and dispatcher.
- Support note/document-package metadata only where it belongs on the planning record itself and remains compatible with the docs backbone used later by `US-17-T3`.
- Add tests for mode/detail integrity, parent-plan rules, date validation, name uniqueness per `(tenant_id, order_id, name)`, and tenant-safe access.

## Preferred file targets

- Alembic revisions for `ops.planning_record` and the mode-specific detail tables.
- `modules/planning/` models, schemas, validators, services, routers, and tests.
- Shared query helpers only where they remain planning-owned and prepare clean APIs for the later Vben UI.
- No shift UI or shift generation code in this task.

## Hard constraints

- Keep commercial scope on `ops.customer_order`; do not duplicate order pricing or invoice truth onto `ops.planning_record`.
- Use explicit mode-specific tables, not a single JSON blob with optional fields for every planning mode.
- Do not implement `ops.shift_plan`, `ops.shift_series`, or `ops.shift` here; those belong to `US-18-*`.
- Do not implement field-execution tables or finance tables here.
- Parent-plan support must remain explicit and safe; do not create recursive plan trees that can drift without validation.
- Any user-visible validation/error content introduced here must preserve German default and English secondary behavior.

## Expected implementation outputs

- A normalized planning-record root with explicit event/site/trade-fair/patrol detail tables.
- Service/API validation that enforces mode/detail integrity and parent-plan rules.
- Tests proving the planning-record aggregate is ready for later shift-plan and release workflows.

## Non-goals

- Do not implement shifts, templates, recurrence series, or shift-generation logic here; that is `US-18-*`.
- Do not build the Vben order/planning UI here; that is `US-17-T3`.
- Do not move customer commercial settings or finance calculations into the planning-record aggregate.
- Do not implement staffing, validation overrides, or assignment tables here; those land in Sprint 7.

## Verification checklist

- [ ] Planning records can be created, updated, archived, and listed under an order with one valid mode-specific detail row.
- [ ] Invalid combinations of `planning_mode_code` and detail rows are rejected deterministically.
- [ ] Parent-plan relationships and planning windows validate correctly.
- [ ] Name uniqueness per order and tenant is enforced.
- [ ] Tests cover mode/detail integrity, parent-plan rules, date validation, and tenant scoping.

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
