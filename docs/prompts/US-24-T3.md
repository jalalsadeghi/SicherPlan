# Codex Prompt — US-24-T3

## Task title

**Implement offline storage and sync-token handling for patrol workflows.**

## Objective

Make patrol control resilient to intermittent connectivity by introducing offline-capable local storage, idempotent replay, and server-side sync-token handling that preserve capture order and prevent duplicate or lossy patrol evidence.

## Source context

- Updated proposal: section 5.2 guard patrol control requires offline sync for smartphone-based rounds.
- Implementation specification: `field.patrol_round.offline_sync_token`, unique `(patrol_round_id, sequence_no)` on `field.patrol_round_event`, append-only evidence guidance in section 4.7, and raw-evidence rules in section 10.
- Sprint reference: `docs/sprint/sprint-08-employee-app-and-field-i.md`.
- Cross-cutting rules: `AGENTS.md`, especially offline-safe field workflows, append-only evidence, and narrow change sets that do not create shadow truth.
- Earlier prompt context: `US-24-T1` and `US-24-T2` provide the patrol-round lifecycle and multi-method capture flows that this task must preserve offline.

## Dependencies

- `US-24-T1` and `US-24-T2` must already provide the online patrol-round and capture behavior to be made offline-safe.
- `US-22-T1` should already provide secure mobile session/bootstrap behavior for protected local storage and authenticated sync.
- If the repo already has a shared offline queue/sync abstraction, reuse it rather than creating a patrol-only bespoke framework.
- Any server-side idempotency additions must remain compatible with the approved patrol-round schema and event uniqueness rules.

## Scope of work

- Implement local mobile persistence for in-progress patrol rounds and unsynced patrol events, including restart-safe recovery after app interruption.
- Use `offline_sync_token` and sequence information to support idempotent round/event replay and safe duplicate suppression on the server.
- Design sync behavior for partial success, retries, and reconnect scenarios so capture order and timestamps remain trustworthy.
- Keep local/offline state clearly subordinate to server truth after successful sync; offline storage is a queue/cache, not a second long-term system of record.
- Add server-side guards or validators needed to accept replay-safe patrol uploads without creating duplicate events or invalid sequence gaps.
- Add tests for offline queue persistence, replay idempotency, duplicate suppression, reconnect behavior, and partial-failure recovery.

## Preferred file targets

- Flutter local storage, repository, sync worker, and patrol feature files under the mobile app path.
- `modules/field_execution` sync endpoints/services/validators for patrol replay and idempotency.
- Shared offline/sync utilities if the repo already contains them.
- Tests for local queue behavior, sync replay, and duplicate suppression.

## Hard constraints

- Offline state must not become a second authoritative patrol database; server-side patrol events remain the durable truth after sync.
- Replay handling must be idempotent and sequence-safe.
- Do not discard failed/unsynced evidence silently; preserve recoverable queue state and explicit error handling.
- Keep patrol events append-only even when reconciling offline replays.
- Avoid brittle always-online assumptions in UI or service logic.

## Expected implementation outputs

- Offline-capable patrol local storage and sync queue.
- Server-compatible sync-token and replay/idempotency handling.
- Recovery behavior for app restart, reconnect, retries, and partial failures.
- Tests for offline persistence, idempotent replay, and duplicate suppression.

## Non-goals

- Do not redesign the entire app around a massive offline-first framework if a focused patrol sync layer is sufficient.
- Do not create a parallel shadow reporting or evaluation store from offline data.
- Do not sacrifice evidence ordering or timestamps merely to simplify retries.
- Do not mix time-capture offline logic into this task unless a tiny shared utility is already approved and reused.

## Verification checklist

- [ ] Patrol rounds and events can survive app interruption and reconnect safely.
- [ ] Replay uses sync tokens/sequence data to avoid duplicate or missing patrol evidence.
- [ ] Server-side handling preserves append-only semantics and durable ordering.
- [ ] Offline state is treated as queued transient state rather than a second source of truth.
- [ ] Tests cover restart recovery, duplicate suppression, partial-failure retry, and idempotent replay.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
