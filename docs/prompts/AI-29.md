# AI-29 — Fix Assistant Availability Loading / Hidden Widget Issue

**Task type:** Bug fix / integration hardening  
**Target repository:** `https://github.com/jalalsadeghi/SicherPlan`  
**Primary sprint reference:** `docs/sprint/AI-Assistant.md`  
**Related completed tasks:** AI-01 through AI-28, especially AI-02, AI-03, AI-04, AI-22, AI-23, AI-25, AI-26, AI-28  
**Observed issue:** On `/admin/dashboard`, the bottom-right assistant launcher first shows `Assistant availability is loading`, but after the page finishes loading the assistant launcher/chat button disappears and no chat panel can be opened.

---

## 1. Context and Problem Statement

The AI Assistant implementation has been completed through the previous AI Assistant task prompts. In the current local web UI:

```text
http://localhost:5173/admin/dashboard
```

the assistant availability indicator briefly appears in the bottom-right corner with this text:

```text
Assistant availability is loading
```

After the page finishes loading, the indicator disappears and no persistent assistant launcher/chat box remains visible.

This is not the expected behavior for an authenticated user who should have access to the AI Assistant.

According to the Sprint Definition of Done, the assistant widget must:

- appear globally after login for authorized users;
- remain available while navigating between pages;
- be hidden only when the feature is intentionally disabled or the user is not authorized;
- not silently disappear when availability loading fails;
- make capability/loading/access failures diagnosable during development.

The purpose of this task is to inspect the current implementation, identify the exact reason why the launcher disappears, and implement the smallest correct fix without weakening the security model.

## 2. Important Existing Implementation Clues

Before changing code, verify the following existing implementation points in the repository.

### 2.1 Frontend App Mounting

The global app file currently imports and mounts:

```text
SicherPlanAssistantWidget
```

inside the root application template, below `RouterView`.

Expected file:

```text
web/apps/web-antd/src/app.vue
```

This means the assistant widget is already mounted globally. The bug is likely not that the widget is missing from `app.vue`, unless the actual branch differs.

### 2.2 Widget Visibility Logic

Expected file:

```text
web/apps/web-antd/src/components/sicherplan/assistant/SicherPlanAssistantWidget.vue
```

The widget visibility is controlled by logic equivalent to:

```ts
const shouldCheckCapabilities = computed(() => isAuthenticated.value && !isPublicRoute.value);

const shouldShowLauncher = computed(() => {
  if (!shouldCheckCapabilities.value) {
    return false;
  }
  if (assistantStore.loadingCapabilities) {
    return true;
  }
  return assistantStore.enabled && assistantStore.canChat;
});
```

This explains the observed behavior:

1. while `loadingCapabilities === true`, the loading indicator is visible;
2. after loading finishes, the launcher stays visible only if:
   - `assistantStore.enabled === true`, and
   - `assistantStore.canChat === true`.

Therefore the most likely causes are:

- `/api/assistant/capabilities` returns `enabled: false`;
- `/api/assistant/capabilities` returns `can_chat: false`;
- the endpoint fails with `401`, `403`, `404`, `500`, or network error;
- the frontend caches a stale disabled capability state from local/session storage;
- the backend AI feature flags are not enabled in local environment;
- the current user does not have `assistant.chat.access`;
- assistant permissions were seeded but not assigned to the active local demo role/user;
- the backend router is mounted under a different path than the frontend expects;
- the frontend API base URL is wrong;
- `Authorization` header is missing or uses a stale token while capabilities are requested;
- the widget hides errors instead of showing a safe diagnostic state.

## 3. Main Hypotheses to Validate

Codex must not guess. Validate these hypotheses with code inspection and runtime checks.

### Hypothesis A — AI Feature Flag Is Disabled

Check whether local backend settings actually resolve to:

```text
SP_AI_ENABLED=true
SP_AI_PROVIDER=mock or openai
```

For local development, `mock` mode is acceptable and preferred for this bug fix unless a real OpenAI key is already configured.

If `SP_AI_ENABLED` defaults to false and no local env override exists, the disappearing launcher is expected. In that case, fix the local/dev configuration and documentation, not the security policy.

### Hypothesis B — Current User Lacks `assistant.chat.access`

Check whether the logged-in user’s auth context contains:

```text
assistant.chat.access
```

Also check whether the seed data/migrations assign assistant permissions to the demo/local roles used by the active login.

If the endpoint returns `enabled: true` but `can_chat: false`, the widget disappears because the frontend correctly hides it for unauthorized users. In local development, fix seed mapping for the intended demo role/user.

### Hypothesis C — Frontend Capabilities Request Fails

Use browser DevTools or Playwright/network logging to inspect:

```text
GET /api/assistant/capabilities
```

Check:

- request URL;
- status code;
- response body;
- `Authorization` header presence;
- whether `VITE_GLOB_API_URL` creates the correct URL;
- whether backend router is registered under `/api/assistant`.

### Hypothesis D — Stale Persisted Capabilities Hide the Widget

Check local/session storage used by:

```text
assistant-session
```

The store may persist:

```text
enabled=false
canChat=false
capabilities={...}
```

If `loadCapabilities(force = false)` returns cached capabilities without refreshing, the launcher may disappear due to stale state.

Fix by ensuring that after login or on app mount, capabilities are refreshed from backend rather than relying on stale persisted `enabled/canChat`.

### Hypothesis E — Widget Error Handling Is Too Silent

Currently, if capabilities loading fails, the widget may set:

```ts
enabled = false
canChat = false
```

and disappear without telling the developer/user what happened.

For production, hiding the assistant for unauthorized users is acceptable. For local development and internal admin users, capability failures should be diagnosable through:

- a safe disabled/unavailable launcher state; or
- console debug logs in dev mode only; and
- clear store error state.

Do not leak secrets or auth tokens in logs.

### Step 3 — Fix the Real Root Cause

Implement the smallest safe fix based on what you find.

Possible fixes include one or more of these:

#### Case 1 — Local Feature Flag Missing

If `SP_AI_ENABLED` is false locally:

- update local env example/documentation;
- ensure mock provider local mode works without OpenAI key;
- do not force-enable AI in production;
- do not hardcode secrets.

Expected local dev settings:

```env
SP_AI_ENABLED=true
SP_AI_PROVIDER=mock
SP_AI_STORE_RESPONSES=false
```

If the project uses another settings pattern, follow that pattern.

#### Case 2 — Permission Missing

If `assistant.chat.access` is missing from the active demo/local user:

- update assistant permission seed/migration mapping;
- assign `assistant.chat.access` to the intended local/internal roles, such as tenant admin, platform admin, dispatcher, HR admin, finance manager, reporting/controller, field supervisor, and other roles defined by the sprint;
- ensure existing user/role seed data receives the permission on migration/seed refresh;
- do not grant admin-only permissions broadly;
- do not grant `assistant.knowledge.reindex` or `assistant.admin` to normal users.

#### Case 3 — Capabilities Request Path Wrong

If the frontend calls the wrong URL:

- fix `getApiBaseUrl()` or endpoint path construction;
- ensure the final request is exactly compatible with backend routing;
- avoid duplicate `/api/api/...`;
- avoid missing `/api`.

#### Case 4 — Auth Token Timing Problem

If capabilities are requested before token is ready:

- ensure the widget waits until authenticated state is stable;
- rerun capabilities check when token becomes available;
- avoid permanent hidden state caused by an early unauthenticated failure.

#### Case 5 — Stale Persisted Capability State

If stale storage is the issue:

- do not persist `enabled` and `canChat` as authoritative long-term values; or
- force-refresh capabilities after login/app mount; or
- add a capability cache timestamp and invalidate it quickly;
- clear stale capability state on logout/token change;
- keep `isOpen`, `draftInput`, and active conversation persistence intact.

Recommended safe behavior:

```ts
loadCapabilities({ force: true })
```

or equivalent should be used on authenticated app mount / first protected route entry.

#### Case 6 — Silent Failure

Improve frontend availability state without weakening security.

Recommended behavior:

- If loading: show launcher as loading.
- If available: show normal launcher.
- If disabled by config: hide in production, but optionally show a safe dev-only disabled reason.
- If unauthorized: hide normal launcher, but store `assistant.permission_denied` internally.
- If network/provider/backend error: do not silently disappear in development; show a small disabled/unavailable launcher with safe text or at least a visible error state if the widget was previously visible.

Do not expose raw backend stack traces, tokens, API keys, or detailed permission internals to normal users.
