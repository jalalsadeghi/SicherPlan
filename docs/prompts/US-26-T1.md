# Codex Prompt — US-26-T1

## Task title

**Implement attendance_record derivation from raw time events and released assignments.**

## Objective

Transform raw capture evidence into a normalized attendance summary per actor and shift so later reconciliation can work from a stable derived record without mutating or replacing the append-only raw event trail.

## Source context

- Updated proposal: Appendix D describes time record and actual as captured start/end, pause, location, approval state, differences, allowances, expenses, and comments; section 5.6 says raw capture becomes the basis for payroll-relevant and billing-relevant actuals.
- Implementation specification: `field.attendance_record` derives from `field.time_event`, is unique per shift+actor when active, and feeds `finance.actual_record`.
- Workflow 6.1 step 6-7 and workflow 6.2 step 7 require time capture and reconciliation to create the approved actuals basis.
- Sprint reference: `docs/sprint/sprint-09-time-capture-and-actuals.md`.
- Cross-cutting rule: attendance summaries are derived artifacts, not the raw evidence source of truth.

## Dependencies

- `US-25-T2` and `US-25-T3` must already provide stable raw-event creation and validation outcomes.
- `US-19-T1` and `US-19-T2` should already provide assignments and actor-to-shift truth.
- `US-21-T1` should already provide released shift visibility/eligibility semantics so attendance derivation does not run on unreleased or invalid planning context by accident.
- If the repository already has a background job or domain-event pattern for derivations, reuse it instead of inventing a one-off manual sync path.

## Scope of work

- Implement the derivation logic that pairs and normalizes raw time events into `field.attendance_record` (or the equivalent repository artifact) per actor and shift.
- Support both employee and subcontractor-worker actors with explicit actor typing and the spec rule that only one actor reference is populated.
- Derive normalized check-in, check-out, and break-minutes fields from the raw event stream, while preserving the source-last-event reference and any validation-status awareness needed downstream.
- Handle incomplete or suspicious streams gracefully (for example missing checkout, duplicate start, or conflicting events) by producing structured discrepancy state rather than overwriting history.
- Implement tests for standard check-in/out flows, multiple events on one shift, subcontractor-worker derivation, and incomplete-event scenarios.

## Preferred file targets

- `modules/field_execution` derivation services/jobs, repositories, and read/write contracts for attendance summaries.
- Alembic migrations only if attendance support needs additional indexes or columns beyond the current schema baseline.
- Tests for event pairing and discrepancy-aware summary creation.

## Hard constraints

- Attendance summaries are derived; do not edit raw time events in place to make the pairing easier.
- Do not let attendance derivation invent assignments or shift truth that does not exist in the planning chain.
- Keep one active attendance summary per shift+actor according to the implementation specification.
- Do not broaden this task into finance actuals, approval stages, or payroll-rate logic.
- Any discrepancy markers introduced must be explicit and traceable, not hidden in opaque status strings without tests.

## Expected implementation outputs

- Attendance derivation service/job from raw events to normalized summaries.
- Stable actor/shift pairing logic with discrepancy-aware outputs.
- Tests proving employee and subcontractor-worker derivation behavior.

## Non-goals

- Do not implement payroll/billing minute calculations beyond what is needed for attendance summary truth.
- Do not implement final finance approval or manual adjustment UX here.
- Do not delete or merge raw events to force a clean attendance result.

## Verification checklist

- [ ] A valid raw event stream produces one active attendance summary per shift+actor.
- [ ] Incomplete or conflicting event streams remain traceable and discrepancy-aware.
- [ ] Raw time events remain unchanged after derivation.
- [ ] Both employee and subcontractor-worker flows are covered.
- [ ] Tests cover standard, duplicate, incomplete, and mixed-source scenarios.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
