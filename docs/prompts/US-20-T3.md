# Codex Prompt — US-20-T3

## Task title

**Implement rules for rest periods, max hours, earnings thresholds, and minimum staffing.**

## Objective

Implement the policy-driven threshold validations that protect staffing sustainability and release quality, while staying honest about which threshold sources already exist in the repository and which arrive in later finance waves.

## Source context

- Updated proposal: section 5.5 validation rules and Appendix E entries for rest periods, maximum hours, earnings limits, and minimum staffing.
- Implementation specification: section 6 validation mapping for rest/max-hours/earnings thresholds and minimum staffing; section 7 reporting guidance that keeps threshold reporting view-driven.
- Sprint reference: `docs/sprint/sprint-07-staffing-engine.md`.
- Cross-cutting rules: `AGENTS.md`, especially the finance bridge and no-shadow-reporting rules.

## Dependencies

- `US-20-T1` and `US-20-T2` should already provide the shared validation framework.
- `US-19-T1` and `US-19-T3` must already provide demand-group, assignment, and subcontractor-release truth for minimum-staffing checks.
- `US-12-T2` and `US-12-T3` should already provide availability, time-account, allowance, and advance-related seams used by threshold checks where already present.
- Future finance source tables such as `finance.actual_record`, pay profiles, and time-account transactions may not all be on the current branch yet; this task must handle that sequencing explicitly instead of inventing incompatible shadow tables.

## Scope of work

- Implement minimum-staffing validation so a plan or shift cannot move into released/customer-visible states when required `min_qty` coverage is not met across assignments and subcontractor releases.
- Implement rest-period and maximum-hours validation using current planning/assignment windows and any already-available historical sources, with policy-driven warn/block outcomes.
- Implement an explicit provider/seam for earnings-threshold evaluation that consumes real finance/pay-profile/time-account sources where they already exist and degrades honestly where later finance waves have not landed yet.
- Return structured validation results through the shared Sprint 7 validation contract, including enough metadata for later UI alerts and release gating.
- Keep threshold reporting/view concerns out of write-side persistence; if a reporting or summary view is needed later, it should remain derived rather than stored as new planning truth.
- Add backend tests for understaffing, rest/max-hour breaches, and any current-branch earnings-threshold cases or fallback behavior.

## Preferred file targets

- `modules/planning/` validation services and tests, plus small adapter/provider interfaces where finance-backed threshold sources are not yet fully available.
- Potential integration points with HR/finance read services, but no ownership transfer out of planning validation.
- No substantial UI work in this task.

## Hard constraints

- Do not invent shadow finance or reporting tables just to make threshold rules compile early.
- Minimum staffing must act as a real release/publication gate, not as an informational-only badge.
- Rest/max-hour and earnings-threshold logic must remain policy-driven and auditable; avoid hardcoded magic numbers in random services.
- Threshold reports remain view-driven; do not turn this task into a reporting implementation.
- If later finance sources are not present yet, preserve an explicit seam or temporarily disabled policy state rather than guessing a new write-side model.
- Any new user-facing rule labels or message keys must preserve German default and English secondary behavior.

## Expected implementation outputs

- Release-gating minimum-staffing validation backed by demand groups, assignments, and subcontractor releases.
- Rest/max-hours validation and an honest seam for earnings-threshold evaluation.
- Tests covering understaffing, threshold breaches, and sequencing-aware fallback behavior where later finance sources are absent.

## Non-goals

- Do not implement payroll calculations, actual reconciliation, or finance exports here.
- Do not build reporting dashboards or threshold report views here.
- Do not implement override persistence or UI alerts here; that belongs to `US-20-T4`.
- Do not broaden this task into release-state orchestration itself; that belongs to `US-21-T1`.

## Verification checklist

- [ ] Understaffed demand groups prevent release/customer-visible publication as required.
- [ ] Rest/max-hours checks return structured, policy-driven validation results using current available data sources.
- [ ] Earnings-threshold logic either uses real current-branch finance sources or fails over through an explicit seam/disabled state without inventing incompatible tables.
- [ ] No duplicate reporting or finance write-side schema is introduced.
- [ ] Tests cover understaffing, threshold breaches, and the main sequencing-aware fallback path.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
