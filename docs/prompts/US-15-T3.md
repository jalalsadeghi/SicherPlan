# Codex Prompt — US-15-T3

## Task title

**Enable subcontractor self-allocation of workers to released shifts with validation feedback**

## Objective

Implement the partner-facing allocation workflow so a subcontractor can place its own workers onto released positions and receive immediate validation feedback, without bypassing planning ownership or compliance rules.

## Source context

- Updated proposal: section 5.4 subcontractor collaboration portal and section 6.3 step 4, where the subcontractor assigns its staff to released shifts and confirms participation.
- Implementation specification: the subcontractor chain and validation mapping in sections 5 and 6, especially the rule sources for qualification match, document completeness, and other planning validations.
- Uploaded Dienstplanmacher subcontractor source material: once shifts are assigned to the subcontractor, it can confirm them and place its employees directly on the positions, with planning-qualification checks before placement.
- Sprint reference: `docs/sprint/sprint-05-subcontractor-management.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-15-T1` must already provide portal auth and route scope.
- `US-15-T2` should already provide released-work read contracts for the portal.
- Where the current branch already contains planning release/assignment contracts from later sprints, integrate with those directly. If they are not present yet, create only the narrow command/adapter seam and UI shell needed to preserve the workflow contract without inventing premature planning persistence.
- `US-14-T3` should already provide worker readiness outputs that the allocation workflow can surface.

## Scope of work

- Implement the subcontractor-portal allocation command flow for selecting an eligible worker and applying that selection to a released shift/position through the approved planning service or adapter boundary.
- Surface validation feedback for readiness, qualification mismatch, missing mandatory documents, duplicate booking/conflict checks, and other planning rules where the upstream validation layer already exists; when it does not, keep the response structure compatible and explicit about unavailable validations.
- Support confirmation timestamps, acting user attribution, and any allowed unassign/reassign flow within the current release rules.
- Build the portal UI flow for viewing candidate workers, choosing a worker, seeing validation feedback, and confirming the allocation or participation action.
- Add tests for own-worker-only allocation, denied cross-subcontractor allocation, validation-feedback rendering, and command behavior against real planning contracts or the approved adapter seam.

## Preferred file targets

- Portal pages/components, API clients, and locale resources for allocation UX under the web application path.
- Command DTOs, service adapters, and tests under `modules/subcontractors/` and/or `modules/planning/` only where the repo already exposes the correct boundary.
- No new write tables unless real planning contracts already require them and they belong in planning, not partner.

## Hard constraints

- Planning owns releases, assignments, and validation override truth. The subcontractor portal may call planning through approved contracts, but it must not mutate planning master tables through hidden shortcuts.
- A subcontractor may allocate only its own workers to positions released to that subcontractor; cross-partner and internal-employee allocation is out of scope and must be blocked.
- Do not invent fake planning persistence if Sprint 6/7 planning tables are not on the current branch yet. Use an interface/adapter seam and disabled or empty-state UX where necessary.
- Validation feedback must be explicit enough to explain why a worker cannot be assigned; do not reduce everything to a generic failure toast.
- All new user-facing strings and workflow states must ship in German and English.

## Expected implementation outputs

- A portal allocation flow and command contract for assigning own workers to released positions.
- Validation-feedback handling aligned with readiness/planning rules or explicit future-compatible seams where those rules are not yet implemented.
- Tests proving scope safety and validation behavior.

## Non-goals

- Do not create or edit released positions themselves here; planning release creation belongs to later ops sprints.
- Do not implement invoice-check or actuals reconciliation here.
- Do not redesign the global validation engine here; this task consumes it or preserves a seam for it.

## Verification checklist

- [ ] A subcontractor can allocate only its own workers to its own released work.
- [ ] Validation feedback clearly identifies common blocking issues such as missing documents or non-ready workers where available.
- [ ] The workflow integrates through a planning boundary or approved adapter seam rather than hidden partner-side writes.
- [ ] Portal UI states behave predictably when upstream planning contracts are present versus still pending.
- [ ] German default and English secondary strings exist for the new allocation workflow.

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
