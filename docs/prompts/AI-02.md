Also read the result of **AI-01 — Repository Inspection and Implementation Plan** if it exists.

This task is **AI-02 — AI Assistant Configuration and Feature Flags**.

## Goal

Add the backend configuration baseline for the SicherPlan AI Assistant with safe defaults, feature flags, provider mode selection, model names, rate/size limits, privacy flags, and testable environment behavior.

This task must not implement the assistant API, assistant module, OpenAI calls, tool registry, migrations, frontend widget, or permissions. It only prepares configuration in the existing backend configuration style.

## Hard Rules

1. Follow the existing backend settings/config pattern.
2. Do not invent a second configuration system.
3. Do not call OpenAI.
4. Do not add frontend code.
5. Do not add database migrations.
6. Do not create the assistant backend module unless existing project convention requires a tiny constants/settings helper. Prefer config-only.
7. Do not require `SP_OPENAI_API_KEY` when AI is disabled.
8. Do not require `SP_OPENAI_API_KEY` when provider is `mock`.
9. Require `SP_OPENAI_API_KEY` only when AI is enabled and provider is `openai`, unless existing environment policy validates provider configuration later.
10. Do not log API keys, secrets, JWTs, reset tokens, or credentials.
11. Keep defaults safe and CI-friendly.
12. Add tests for config behavior if the repository has backend tests.

## Configuration Values to Add

Add or align these settings with repository naming conventions:

```text
SP_AI_ENABLED=false
SP_AI_PROVIDER=mock
SP_OPENAI_API_KEY=

SP_AI_STORE_RESPONSES=false

SP_AI_RATE_LIMIT_PER_TENANT_PER_MINUTE=100
SP_AI_REDACTION_ENABLED=true
SP_AI_AUDIT_ENABLED=true
```

If the project has different naming conventions for environment variables, keep the `SP_` names unless there is a strong existing convention that requires adaptation. Report any adaptation.

## Expected Runtime Behavior

### Case 1 — AI Disabled

```text
SP_AI_ENABLED=false
```

Expected:

- application starts normally;
- no OpenAI key is required;
- provider validation is skipped;
- do not implement `/api/assistant/capabilities` yet.

### Case 2 — AI Enabled with Mock Provider

```text
SP_AI_ENABLED=true
SP_AI_PROVIDER=mock
```

Expected:

- application starts normally;
- no OpenAI key is required;
- deterministic tests can use this mode later.

### Case 3 — AI Enabled with OpenAI Provider

```text
SP_AI_ENABLED=true
SP_AI_PROVIDER=openai
SP_OPENAI_API_KEY=<value>
```

Expected:

- application starts normally;
- API key is backend-only;
- key is never printed or serialized.

### Case 4 — AI Enabled with OpenAI Provider but Missing Key

```text
SP_AI_ENABLED=true
SP_AI_PROVIDER=openai
SP_OPENAI_API_KEY=
```

Expected:

- fail fast according to existing environment policy, or
- raise a clear configuration error when the future provider is initialized.

Choose the behavior that matches existing project conventions. Explain the choice.

### Case 5 — Invalid Provider

```text
SP_AI_ENABLED=true
SP_AI_PROVIDER=unknown
```

Expected:

- validation fails with a clear error or rejects the value according to existing settings style.

- tests cover configuration behavior or a clear explanation is given if no test framework exists;
