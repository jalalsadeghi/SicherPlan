# Codex Prompt — US-20-T2

## Task title

**Implement blocking/warning checks for mandatory docs, customer blocks, and double booking.**

## Objective

Extend the planning-validation engine with the high-risk checks that protect compliance completeness, customer-specific staffing restrictions, and overlapping assignment conflicts before staffing is saved or released.

## Source context

- Updated proposal: section 5.5 validation rules and Appendix E entries for mandatory documents complete, customer block, and double booking.
- Implementation specification: section 6 validation mapping for mandatory documents, customer blocks, and double booking; section 5 compliance, employee-planning, and subcontractor relationship chains.
- Sprint reference: `docs/sprint/sprint-07-staffing-engine.md`.
- Cross-cutting rules: `AGENTS.md`, especially document-centered compliance and explicit overlap checks.

## Dependencies

- `US-20-T1` should already provide the shared validation-service/result framework.
- `US-6-T1` must already provide centralized document-link truth for compliance documents and generated outputs.
- `US-9-T3` or the equivalent customer-control work must already provide customer-specific employee blocking data.
- `US-19-T1` and `US-19-T2` must already provide assignments and the staffing command seams where overlap validation is enforced.
- `US-12-*` and `US-14-*` must already provide employee and subcontractor-worker compliance/document projections.

## Scope of work

- Implement mandatory-document completeness checks using `compliance.requirement_rule` (or the repo equivalent) plus document links on employee, subcontractor worker, subcontractor, and related qualification records as needed.
- Implement customer-block checks so an employee cannot be assigned where an active customer-specific block exists; default behavior should be hard prevent unless current policy explicitly says otherwise.
- Implement double-booking detection for overlapping assignments across shift windows for the same employee or subcontractor worker, using explicit overlap queries and any supporting indexes the repository needs.
- Surface structured validation results through the same result contract introduced in `US-20-T1`, with clear rule codes and message keys.
- Add or refine database indexes/constraints where they materially improve overlap checks, but keep the core logic in service-level validation rather than relying on brittle database-only behavior.
- Add backend tests for missing-doc scenarios, customer-block denial, overlap detection across create/update/substitute paths, and the main warn-vs-block policy cases.

## Preferred file targets

- `modules/planning/` validation services, repositories/query helpers, and tests.
- Alembic migrations for index/constraint support only where the repository needs them for overlap or rule performance.
- No substantial UI work in this task.

## Hard constraints

- Document completeness must be derived from the centralized docs/compliance backbone; do not duplicate compliance flags into planning tables for convenience.
- Customer blocks should hard-prevent assignment by default unless the approved policy explicitly marks the rule as warn/overrideable.
- Double-booking logic must work for both employee and subcontractor-worker actors and must remain understandable and testable.
- Do not broaden this task into release-state or override persistence work; this prompt is about rule evaluation, not final override handling.
- Do not introduce cross-tenant scans or unrestricted joins while checking overlaps or customer scope.
- Any message keys or user-facing rule labels introduced here must preserve DE default and EN secondary behavior.

## Expected implementation outputs

- Structured planning validations for mandatory docs, customer blocks, and overlap conflicts.
- Any supporting query/index refinements needed for reliable overlap checks.
- Tests proving compliance completeness, customer-block enforcement, and double-booking detection.

## Non-goals

- Do not implement rest/max-hours/earnings/min-staffing checks here; that belongs to `US-20-T3`.
- Do not implement override persistence/UI here; that belongs to `US-20-T4`.
- Do not build staffing-board or coverage UI here.
- Do not implement finance/attendance reconciliation here.

## Verification checklist

- [ ] Assignments fail or warn correctly when required documents are missing for the relevant actor/context.
- [ ] Customer-specific employee blocks are enforced reliably and tenant-safely.
- [ ] Overlapping assignments for the same actor are detected across the main command paths.
- [ ] Validation results stay structured and reusable rather than scattered across endpoint-specific code.
- [ ] Tests cover missing docs, customer blocks, overlap conflicts, and policy behavior.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
