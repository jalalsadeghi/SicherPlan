# AI-53 — Fix Responses API Tool-Call Continuation for Operational Diagnostics

## Purpose

The assistant works for simple how-to questions such as:

```text
Wie kann ich einen Mitarbeiter auf der Plattform registrieren?
```

but fails for an operational diagnostic question such as:

```text
Ich habe einem Mitarbeiter eine Arbeitsschicht zugewiesen, diese wird jedoch in der mobilen App nicht angezeigt. Woran könnte das liegen?
```

The backend logs show that the first OpenAI call succeeds and returns a tool call, but the second call fails:

```text
HTTP/1.1 400 Bad Request
No tool call found for function call output with call_id call_UyiGQPNDJj2u9Aaujj19LkV9.
```

This means the assistant tool-call continuation is invalid for the OpenAI Responses API. The backend is sending a `function_call_output` item back to OpenAI without preserving the matching previous tool call context.

This must be fixed at the Responses API orchestration layer.

---

## Current Evidence

For the failing question:

```text
Ich habe einem Mitarbeiter eine Arbeitsschicht zugewiesen, diese wird jedoch in der mobilen App nicht angezeigt. Woran könnte das liegen?
```

The log sequence is:

```text
assistant_provider_call_started
POST https://api.openai.com/v1/responses -> 200 OK
assistant_provider_call_finished provider_returned_successfully=true output_tokens=22
assistant_provider_call_started
POST https://api.openai.com/v1/responses -> 400 Bad Request
assistant_provider_error:
  No tool call found for function call output with call_id ...
POST /api/assistant/conversations/{id}/messages -> 502 Bad Gateway
```

For the successful employee-registration question, only one provider call is needed and no tool continuation occurs. That is why it succeeds.

---

## Root Cause to Verify

The current implementation likely does this:

```text
1. First OpenAI call:
   - sends user message + grounding + tools
   - receives function_call with call_id

2. Backend executes the tool

3. Second OpenAI call:
   - sends function_call_output
   - but does not link it to the previous response/tool call correctly
```

In the current code, inspect:

```text
backend/app/modules/assistant/openai_client.py
backend/app/modules/assistant/service.py
backend/app/modules/assistant/provider.py
backend/app/modules/assistant/schemas.py
backend/app/modules/assistant/tool_name_adapter.py
backend/app/modules/assistant/tools/*
```

Known risky behavior to inspect:
- `_build_input()` adds `function_call_output` items from `request.tool_results`.
- The provider request may not include `previous_response_id`.
- The provider result may not retain `response.id`.
- The continuation request may be rebuilt as a fresh request with system/user messages plus `function_call_output`, which the Responses API cannot associate with the previous function call.
- `SP_AI_STORE_RESPONSES=false` may affect whether `previous_response_id` can be used safely. Codex must verify the correct approach with the installed OpenAI SDK/API behavior.

### Step 3 — Implement a Valid Responses API Continuation Mode

Codex must verify the installed OpenAI SDK behavior and implement a valid continuation strategy.

Preferred mode when provider-side response state is available:

```python
client.responses.parse(
    model=...,
    previous_response_id=first_response.id,
    input=[
        {
            "type": "function_call_output",
            "call_id": tool_call.call_id,
            "output": json.dumps(tool_result_summary)
        }
    ],
    text_format=AssistantProviderStructuredOutput,
    store=...
)
```

Important:
- Do not include a new user message in the continuation request.
- Do not include unrelated prior messages as if it were a fresh request.
- Do not include `function_call_output` without a matching previous response state.
- Pass the same or appropriate instructions if the SDK/API requires them, but keep the function output associated with the previous response.

Fallback mode if `SP_AI_STORE_RESPONSES=false` makes `previous_response_id` unusable:
- Codex must implement the documented/stateless alternative supported by the Responses API and SDK.
- The fallback must include enough previous response output items for the API to find the matching `function_call` before the `function_call_output`.
- If no safe stateless method is supported by the installed SDK, add a configuration flag to permit `previous_response_id` for tool-call continuations and document the privacy tradeoff.

Codex must not guess. It must validate with fake-client tests and, if possible, one manual smoke test.

### Step 5 — Fix Service Tool Loop

Update the service/orchestrator loop so that:

1. First provider call returns tool calls plus response state.
2. Backend maps provider-safe tool names to internal tool names.
3. Backend executes allowed tools through the tool registry.
4. Tool output summaries are attached to the exact `call_id`.
5. The next provider call is a continuation call, not a brand-new full prompt.
6. Multiple/parallel tool calls are handled:
   - execute all returned calls;
   - send one output per `call_id`;
   - do not reuse a call_id;
   - do not drop any required output.
7. If a tool fails, send a safe JSON tool output explaining the safe failure, still tied to the correct `call_id`.
8. If the model requests unknown tool names, reject safely and continue with a safe tool output.
