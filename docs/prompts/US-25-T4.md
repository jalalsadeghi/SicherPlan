# Codex Prompt — US-25-T4

## Task title

**Support QR/barcode/RFID/NFC credential capture for employees and subcontractor workers.**

## Objective

Add credential-scan support so time capture can identify the correct employee or subcontractor worker from app-presented badges, printed IDs, or approved token media, while keeping identity lookup secure, auditable, and aligned with the employee/subcontractor master-data model.

## Source context

- Updated proposal: employee IDs and work badges support barcode/QR for admission control, time capture, and operational verification; subcontractor management includes order/security badges and time evidence via RFID/NFC/barcode/QR.
- Section 5.6 time capture and section 5.3 employee app/self-service establish the QR/barcode-enabled mobile and badge path.
- Implementation specification: raw time events store `raw_token`, actor references, and device context; field services may reference HR and partner worker truth read-only.
- Dienstplanmacher material explicitly describes QR/barcode from employee badges or the mobile app, RFID/NFC support, and terminal/device scan workflows.
- Sprint reference: `docs/sprint/sprint-09-time-capture-and-actuals.md`.

## Dependencies

- `US-12-T4` and related employee-file tasks should already provide employee identity/badge context or at least the seams where an employee-facing code can be resolved.
- `US-14-*` should already provide subcontractor-worker identity records and scoped worker truth.
- `US-22-T3` should already provide mobile QR/barcode display in the employee app, or an equivalent employee credential surface.
- `US-25-T2` must already provide raw event ingest so scan resolution can flow into captured actor references.

## Scope of work

- Implement or finalize the credential-resolution service that maps scanned QR/barcode/RFID/NFC values to the correct employee or subcontractor-worker record inside the current tenant.
- Support scans originating from printed badges/ID cards and from app-displayed credentials where the repository already has those issuance/display seams.
- Ensure the capture pipeline can distinguish employee and subcontractor-worker actors while preserving only one authoritative actor link on each raw event.
- Where token issuance/lookup needs hardening, prefer hashed or indirect lookup-safe token storage patterns that fit the repository security baseline instead of storing sensitive identity values in plain reusable form.
- Add tests for employee badge scans, app-presented QR/barcode scans, subcontractor-worker scans, invalid/expired/unknown tokens, and tenant-isolation boundaries.

## Preferred file targets

- `modules/field_execution` credential-resolution services and ingest adapters.
- `modules/hr` / `modules/partner` only where narrow read-model or lookup support is necessary.
- Mobile app files only if app-presented credential payload formatting must be aligned with the scan resolver.
- Tests for token lookup, actor mapping, and invalid-scan handling.

## Hard constraints

- Do not duplicate employee or subcontractor-worker master data into field-capture tables just to simplify scanning.
- Every captured event must resolve to at most one actor reference; avoid ambiguous multi-actor matching.
- Do not expose HR-private fields, tax data, or internal identifiers in scan payloads or client-visible credential formats.
- Keep credential support extensible across QR, barcode, RFID, and NFC without making one channel the hidden master format.
- Do not collapse all failed scans into generic success paths; unresolved credentials must remain visible as failed/flagged attempts where policy requires.

## Expected implementation outputs

- Credential-resolution service for employee and subcontractor-worker scan inputs.
- Ingest integration that maps valid tokens to actor references safely.
- Any narrow app/ID-format alignment needed for repository compatibility.
- Tests proving secure actor lookup and tenant-isolated invalid-token behavior.

## Non-goals

- Do not build a full access-control or event-admission subsystem here.
- Do not redesign badge issuance UX beyond what is needed to support reliable time capture scans.
- Do not implement attendance derivation or approval logic here.

## Verification checklist

- [ ] Employee and subcontractor-worker scan tokens resolve through approved services without data duplication.
- [ ] Each raw event links to one valid actor or remains flagged/rejected when lookup fails.
- [ ] Credential formats do not leak HR-private or cross-tenant identity data.
- [ ] App-presented and printed/terminal scan paths remain compatible.
- [ ] Tests cover valid, invalid, ambiguous, and cross-tenant credential cases.
