# Codex Prompt — US-33-T3

## Task title

**Execute backup/restore, rate-limit, and secure-document permission drills.**

## Objective

Exercise the operational hardening package by turning backup/restore, rate-limiting, and secure-document access control into tested drills and documented runbooks rather than assumed infrastructure guarantees.

## Source context

- Updated proposal: the security baseline includes rate limiting, secure document permissions, backup/restore, and structured monitoring.
- Implementation specification: documents are centralized through `docs.document`, versions, and links, and generated outputs must remain durable and linked; append-only evidence and centralized docs services increase the importance of tested restore and access behavior.
- Sprint reference: `docs/sprint/sprint-11-reporting-and-hardening.md`.
- Cross-cutting rules: `AGENTS.md`, especially centralized documents, generated output tracking, outbox discipline, and least-privilege visibility.

## Dependencies

- `US-6-T1` must already provide centralized document/version/link infrastructure and document download behavior.
- `US-2-T2`/`US-2-T3` or repository-equivalent infrastructure work should already provide environment config, secrets, and deployment/test automation seams.
- `US-5-*` and related auth work must already provide permission and session behavior for secure-document access control.
- `US-33-T1` and `US-33-T2` may already have introduced performance and RLS changes that these drills should respect or at least not break.

## Scope of work

- Create or refine in-repo runbooks/scripts/tests for database backup and restore drills, including representative validation that core transactional data, generated documents, and audit evidence survive the restore path.
- Implement or verify rate-limiting behavior for high-risk endpoints such as authentication, document download, or export-heavy routes according to existing repository patterns and configuration seams.
- Create integration tests or drill scripts for secure-document permissions, proving that document access honors tenant scope, role scope, visibility scope, and business-entity linkage.
- Where object storage or file-version restoration is part of the platform contract, include a realistic validation path for version metadata and stored-file linkage rather than restoring database rows only.
- Capture drill results, assumptions, and remediation notes in repository docs so future operators can rerun the hardening checks before go-live or after significant infrastructure changes.
- Add CI-friendly smoke validations where practical, while keeping destructive or environment-heavy drills clearly separated from routine local developer workflows.

## Preferred file targets

- Infrastructure or ops docs/scripts under `docs/`, `ops/`, `infra/`, or the repository-equivalent location.
- Security/rate-limit middleware/config/tests in backend packages where existing patterns already live.
- Document-download authorization tests under platform-services or reporting/docs packages.
- No end-user UI work unless a tiny internal admin diagnostic seam already exists and is clearly helpful.

## Hard constraints

- Do not treat backup existence as equivalent to backup restore readiness; prove restore and validation behavior.
- Do not weaken document access rules for convenience during drill implementation.
- Rate limiting must be configurable and environment-safe; do not hardcode production-only values or break developer/test flows without an explicit seam.
- Keep drills and smoke tests safe for non-production environments; destructive or heavy steps must be documented clearly and isolated.
- Do not broaden this task into a full disaster-recovery program beyond the scoped hardening drills.

## Expected implementation outputs

- Backup/restore runbooks or automated drill scripts with validation steps.
- Verified/configurable rate-limiting behavior for high-risk endpoints.
- Secure-document permission drill coverage and tests proving access-control correctness.
- Hardening notes/remediation items captured in-repo.

## Non-goals

- Do not redesign the docs service or object-storage model here unless a narrow hardening fix is required.
- Do not add an unrelated security product integration in this task.
- Do not assume object storage is safe if only DB rows were validated in the restore drill.
- Do not turn hardening scripts into undocumented one-off commands.

## Verification checklist

- [ ] Backup and restore are both exercised with evidence that key data and document links survive.
- [ ] High-risk endpoints enforce rate limits according to explicit configuration.
- [ ] Secure-document access is proven against tenant/role/visibility rules.
- [ ] Runbooks/tests make the drills repeatable for UAT and go-live readiness.
- [ ] Remediation notes are captured when the drills reveal weaknesses.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
