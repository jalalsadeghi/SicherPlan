# AI-34 — Mandatory OpenAI Runtime, Provider Verification, and API-Usage Observability

## Purpose

Fix the current assistant behavior where user-facing chat can still run in `mock` mode and returns `[MOCK RAG]` responses without sending any request to the OpenAI API.

The current production-like user experience must not behave like an FAQ/mock bot. For every in-scope SicherPlan question, the backend must use the configured AI provider. In normal development/staging/production operation this means `SP_AI_PROVIDER=openai` and a real OpenAI Responses API call. Mock mode may remain only for automated tests and explicit local debugging.

This task must be implemented before deeper RAG quality work, because if OpenAI is never called, the assistant can never be an AI assistant.

---

## Context from the Current Repository

Codex must first inspect the latest repository state. In the current codebase, the following risk areas are known and must be verified against the current branch:

- `backend/app/config.py` currently defaults `SP_AI_PROVIDER` to `mock` unless the environment explicitly overrides it.
- `backend/app/modules/assistant/provider.py` contains `MockAssistantProvider`, which returns `[MOCK RAG]` style responses.
- `backend/app/modules/assistant/service.py` builds `grounding_context`, but the final runtime behavior still depends on the selected provider.
- The user has verified in the OpenAI Platform Usage dashboard that no API requests have been recorded during chat testing.

Codex must validate these facts against the current repository before changing code.

---

## Non-Negotiable Product Requirement

The user-facing assistant must not silently answer through the mock provider.

For any authenticated user asking an in-scope SicherPlan question:

```text
assistant message flow
  -> build route/auth/language context
  -> build retrieval/grounding context
  -> call configured AI provider
  -> if provider=openai, send a real OpenAI Responses API request
  -> return model-generated structured answer
```

If OpenAI is not configured, the assistant must show a clear provider-configuration error instead of producing a fake answer.

### Step 2 — Runtime Rules

Implement these runtime rules:

1. `SP_AI_PROVIDER=mock` is allowed only when one of these is true:
   - `SP_ENV=test`
   - `SP_ENV=ci`
   - explicit local override `SP_AI_ALLOW_MOCK_PROVIDER=true`
2. If `SP_AI_ENABLED=true` and the app is not in test/CI and `SP_AI_PROVIDER=mock`, the backend must not produce chat answers. It must return a safe configuration error:
   - code: `assistant.provider.mock_not_allowed`
   - message key: `errors.assistant.provider.mock_not_allowed`
3. If `SP_AI_ENABLED=true` and `SP_AI_PROVIDER=openai`, `SP_OPENAI_API_KEY` must be present.
4. If `SP_AI_PROVIDER=openai`, `SP_AI_RESPONSE_MODEL` must not be a placeholder such as `gpt-5.5-or-configured-model`.
5. Mock provider must never display `[MOCK RAG]` to a normal user in a browser session unless `SP_AI_ALLOW_MOCK_PROVIDER=true`.

### Step 3 — Capabilities Must Expose Provider Reality

Extend `GET /api/assistant/capabilities` to include safe diagnostics:

If provider mode is `mock`, capabilities must make that explicit.

## Acceptance Criteria

This task is done only when:

- User-facing chat cannot silently use mock provider.
- OpenAI provider mode requires a real API key and real model.
- In OpenAI mode, an in-scope message results in a call to the OpenAI provider wrapper.
- Capabilities clearly report provider mode.
- The UI never shows `[MOCK RAG]` unless explicit local mock mode is enabled.
- Tests prove that OpenAI is called in provider mode.
- Sensitive values are never logged.
- Existing CI tests still pass.
