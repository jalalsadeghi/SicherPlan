# Codex Prompt — US-31-T3

## Task title

**Build Vben dashboards and downloadable report endpoints with role-based scope filters.**

## Objective

Deliver the first internal reporting UI and download surfaces so tenant administrators, accounting users, controllers/QM, and other authorized internal roles can consume the Sprint 11 reporting views through role-scoped Vben dashboards and export endpoints.

## Source context

- Updated proposal: target roles include tenant administrators, accounting, controller/QM, and operational staff with scoped access to reports; section 5.7 requires controlling and compliance-oriented reporting surfaces.
- Implementation specification: section 7 says reporting definitions can be exposed through FastAPI services; the sprint doc explicitly calls for Vben dashboards and downloadable endpoints with role-based scope filters.
- AGENTS rules: web work must follow Vben Admin patterns, role guards, DE-default / EN-secondary localization, and least-privilege visibility.
- Reference admin shell: Vben Admin is the agreed web admin reference for the project.

## Dependencies

- `US-31-T1` and `US-31-T2` must already provide stable reporting views, filter contracts, and read endpoints or service seams.
- `US-3-T1`, `US-3-T3`, and `US-3-T4` must already provide the Vben shell, theme tokens, and DE/EN localization structure.
- `US-5-T1` through `US-5-T3` must already provide user accounts, permissions, role scopes, and route/API guard behavior.
- If the repository already has shared export helpers, report-table components, or role-scope composables, reuse them instead of creating a second dashboard pattern.

## Scope of work

- Create Vben Admin reporting pages/cards/tables for operational, commercial, payroll-basis, and subcontractor-control reporting using the reporting views implemented in `US-31-T1` and `US-31-T2`.
- Implement role-scoped filter bars, saved query-state handling where the repository already supports it, and reusable list/detail or table-card patterns consistent with the existing Vben workspace.
- Expose downloadable report endpoints or handlers for authorized users, reusing repository-standard CSV/XLSX/PDF or export conventions rather than inventing a one-off download format.
- Keep dashboards thin: they should consume reporting endpoints and view contracts, not reimplement heavy business joins or calculations in browser code.
- Add localized DE/EN labels, filter captions, empty states, and export action labels for every new reporting surface.
- Add tests for route guards, scope-filter enforcement, endpoint authorization, and at least representative UI rendering or component-state behavior for the reporting pages.

## Preferred file targets

- Web-admin reporting pages/components under the repository’s Vben application structure.
- Reporting routers/controllers/export endpoints under `modules/reporting/` or the repository-equivalent backend package.
- Shared filter composables, route guards, and locale resources only where they fit existing Vben patterns.
- Frontend and backend tests covering authorization, scope filtering, and export behavior.

## Hard constraints

- Dashboards must consume the reporting layer built in `US-31-T1` and `US-31-T2`; do not bypass that layer with ad hoc transactional queries from controller endpoints or UI code.
- Preserve tenant isolation and role scoping. Customer- and subcontractor-facing scopes remain out of bounds here unless a dependency explicitly requires a shared internal component.
- Use Vben Admin conventions for route guards, page layout, list/detail patterns, loading states, and reusable CRUD/reporting composition.
- Any export action must enforce the same scope and visibility rules as the dashboard view that triggered it.
- All user-facing labels, buttons, filter names, and export labels must follow German default and English secondary localization rules.
- Do not broaden this task into scheduled export delivery, compliance/QM screen packages, or hardening work.

## Expected implementation outputs

- Vben dashboard pages and reusable reporting UI components.
- Authorized download endpoints or handlers aligned with existing export conventions.
- DE/EN resources and tests proving route-guard, scope-filter, and export correctness.

## Non-goals

- Do not implement scheduled report delivery or controller/QM-specific compliance export workflows here; that belongs to `US-32-T4`.
- Do not add a generic charting/analytics framework unrelated to the scoped reporting stories.
- Do not reimplement reporting business logic in the browser.
- Do not expose these screens to customer, subcontractor, or employee portal users.

## Verification checklist

- [ ] Dashboard routes are guarded and only available to authorized internal roles.
- [ ] Dashboard filters and downloadable outputs honor tenant and role scope exactly.
- [ ] The UI consumes stable reporting contracts instead of transactional ad hoc APIs.
- [ ] DE and EN resources exist for all newly introduced UI and export labels.
- [ ] Tests cover key authorization, scope-filter, and rendering/export scenarios.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
