# Codex Prompt — US-33-T2

## Task title

**Implement optional PostgreSQL RLS policies on high-risk operational datasets.**

## Objective

Add an optional database-enforced tenant/isolation layer for selected high-risk tables so the repository can validate row-level security as a defense-in-depth measure without replacing the application-layer authorization model.

## Source context

- Updated proposal: the security baseline explicitly calls out tenant scoping and structured monitoring as foundational controls for the platform.
- Implementation specification: early modeling notes mark PostgreSQL RLS as optional but recommended, and Phase H names optional RLS policies as part of the hardening wave.
- Sprint reference: `docs/sprint/sprint-11-reporting-and-hardening.md`.
- Official PostgreSQL guidance aligns with enabling row security per table and defining explicit policies per command/role. Keep the SQL implementation faithful to current PostgreSQL policy semantics.
- Cross-cutting rules: `AGENTS.md`, especially strict tenant isolation, least-privilege visibility, and the warning that portals/users are scoped actors within a tenant rather than separate tenants.

## Dependencies

- `US-5-T1` through `US-5-T3` must already provide the user/role/scope model that the application layer uses today.
- `US-33-T1` should already have tuned the heavy paths so RLS is evaluated with awareness of performance-sensitive queries.
- High-risk tenant-owned operational tables must already be stable enough that adding optional RLS policies will not mask unfinished schema churn.
- If the repository already has DB session-context or connection-management helpers, reuse them instead of inventing inconsistent tenant-context plumbing.

## Scope of work

- Select a narrow, high-value set of tenant-owned high-risk tables for the first RLS pass, such as operational/evidence or finance-adjacent tables where defense in depth meaningfully reduces cross-tenant leakage risk.
- Implement Alembic or repository-equivalent SQL migrations that enable row-level security and create explicit policies for those tables, using a trusted session-context or database-role pattern consistent with the repository’s connection model.
- Keep RLS optional/feature-controlled so the repository can run with or without DB-enforced row filtering during rollout, testing, or operational troubleshooting.
- Define and test the privileged/bypass behavior explicitly for migration, background jobs, admin maintenance, and trusted service roles instead of assuming superuser-only operation.
- Document how RLS interacts with application-layer authorization, connection pooling, background jobs, and test fixtures so developers do not treat it as magic.
- Add integration tests proving allowed access, blocked cross-tenant access, privileged maintenance paths, and failure modes when tenant/session context is missing or wrong.

## Preferred file targets

- Alembic revisions or repository-equivalent SQL files for `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`, `CREATE POLICY`, and any policy teardown/feature-flag support.
- Database/session-context helpers, backend config, and integration tests under the repository’s IAM/core/infrastructure packages.
- Hardening documentation explaining the opt-in RLS model and operational caveats.
- No broad UI changes in this task.

## Hard constraints

- RLS is defense in depth, not a replacement for application-layer authorization. Keep existing auth/scope checks intact.
- Do not apply RLS to every table indiscriminately in the first pass. Choose a narrow set of high-value datasets and keep the rollout understandable.
- Use trusted tenant/session context or trusted service roles. Do not rely on user-supplied filter parameters to stand in for DB-enforced policy context.
- Background jobs, migrations, and generated-output flows must have an explicit and safe policy for privileged access or bypass where appropriate.
- Do not broaden this task into backup/restore drills or performance/resilience test cycles beyond what is needed to prove RLS correctness.

## Expected implementation outputs

- Optional RLS migrations/policies for selected high-risk tables.
- Connection/session-context and privileged-path support needed to operate those policies safely.
- Integration tests and docs proving allowed, denied, and privileged/bypass scenarios.

## Non-goals

- Do not remove or weaken application-layer tenant/role checks because RLS exists.
- Do not treat this as a full zero-trust redesign of the entire DB schema.
- Do not force RLS onto tables where the repository has not yet established stable tenant ownership semantics.
- Do not silently let background jobs fail because policy context was never defined.

## Verification checklist

- [ ] Selected high-risk tables enforce optional DB-level row filtering correctly.
- [ ] Cross-tenant access is denied when policy context is wrong or missing.
- [ ] Privileged maintenance/background-job flows are explicit and test-covered.
- [ ] Application-layer auth still exists and remains consistent with the DB policy model.
- [ ] Docs make the rollout and operational caveats understandable for future developers/operators.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
