# AI-57 — Fix Structured Output Truncation and JSON Parse Failures

## Purpose

The assistant still shows the generic error:

```text
Die Assistant-Anfrage konnte gerade nicht abgeschlossen werden.
```

The latest backend log proves that OpenAI returned `200 OK`, but the backend failed while parsing the structured response:

```text
AssistantProviderInvalidRequestError
provider_error_type: ValidationError
safe_message:
1 validation error for AssistantProviderStructuredOutput
Invalid JSON: EOF while parsing a string ...
input_value='{"answer":"Es gibt mehre...tions- und Freigabeproz'
```

The same log shows:

```text
max_output_tokens: 200
```

This strongly indicates that the JSON structured response was truncated because the assistant configured an output token limit that is too small for the required structured JSON response.

This task must fix structured output generation so the assistant never fails just because the model output is truncated.

---

## Current Evidence

Failing request:

```text
POST /api/assistant/conversations/{conversation_id}/messages -> 502 Bad Gateway
```

Relevant log:

```json
  "estimated_grounding_tokens": 1647,
  "estimated_tool_result_tokens": 9213,

  "max_output_tokens": 200

```

Then:

```json
{
  "event": "assistant_provider_error",
  "exception_class": "AssistantProviderInvalidRequestError",
  "provider_error_type": "ValidationError",
  "safe_message": "Invalid JSON: EOF while parsing a string..."
}
```

Root cause to verify:
- `max_output_tokens=200` is too low for a structured JSON response containing answer, diagnosis, links, next_steps, source_basis, etc.
- The model begins producing valid JSON but is cut off before closing the string/object.
- The backend maps the parse failure to 502 and the frontend shows a generic error.

### Step 2 — Raise Safe Minimum Output Limits

Add safe minimums for structured output.

Suggested settings:

```env
SP_AI_MIN_STRUCTURED_OUTPUT_TOKENS=800
SP_AI_MAX_OUTPUT_TOKENS=1200
SP_AI_CONTINUATION_MAX_OUTPUT_TOKENS=900
SP_AI_DEGRADED_MAX_OUTPUT_TOKENS=700
```

Rules:

1. Never call OpenAI with `max_output_tokens < SP_AI_MIN_STRUCTURED_OUTPUT_TOKENS` when expecting `AssistantProviderStructuredOutput`.
2. `max_output_tokens=200` is not acceptable for normal structured answers.
3. If token budget is tight, reduce input context first, not output below the structured minimum.
4. Keep smoke-test output small only if the schema can safely fit.

### Step 3 — Compute Output Budget Separately From Input Budget

Do not shrink output tokens merely because input is large.

Instead:

```text
input too large -> trim grounding/history/tool results
output too small -> increase output limit or use compact schema
```

Add logic:

```python
effective_max_output_tokens = max(configured_max_output_tokens, min_structured_output_tokens)
```

For continuation/fallback:

```python
effective_continuation_output_tokens = max(configured_continuation_output_tokens, min_structured_output_tokens)
```

### Step 4 — Add Truncation Detection

If OpenAI returns a response that cannot be parsed and looks truncated:

- invalid JSON;
- EOF while parsing;
- unterminated string;
- missing closing brace;
- finish reason indicates length/max_output_tokens, if available;
- output_text length near configured output limit.

Then do not immediately return 502.

Instead:
1. Retry once with:
   - higher output token limit;
   - smaller grounding context;
   - no optional source_basis verbosity if needed;
   - explicit instruction: “Return compact valid JSON only.”
2. If retry succeeds, return normal 201 response.
3. If retry fails again, use the graceful fallback from AI-56 if grounding facts exist.

### Step 5 — Compact Structured Output Mode

If token budget is tight, allow a compact mode:

```text
- answer: concise but complete
- diagnosis: max 3 items
- next_steps: max 4 items
- links: max 4 items
- source_basis: max 4 items
```

But still require valid JSON.

Do not omit required schema fields.

### Step 6 — Provider Error Mapping

Do not map Pydantic/JSON parse failure as `AssistantProviderInvalidRequestError` if the HTTP call returned 200 and the issue is model output truncation.

Map to a clearer error:

```text
assistant.provider.structured_output_truncated
```

or:

```text
AssistantProviderStructuredOutputError
provider_error_type="StructuredOutputTruncated"
```

This error should be eligible for retry and graceful fallback.
