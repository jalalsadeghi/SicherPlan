# Codex Prompt — US-29-T4

## Task title

**Expose released timesheets/invoices to customer portal according to permissions.**

## Objective

Publish customer-facing finance read models for released timesheets and invoices so each customer account can see only its own authorized billing outputs, documents, and status information without exposing internal finance notes, unreleased results, or personal names by default.

## Source context

- Updated proposal: section 5.2 customer portal includes released staffing data, watchbooks, timesheets, reports, and admin login history; section 5.6 explicitly requires released timesheets for the customer portal; workflow 6.1 ends with released report package, timesheets, and invoice delivery.
- The proposal’s operational note says customer-facing result views and reports should omit personal names by default unless explicitly released by the prime company.
- Implementation specification: the portal visibility chain in section 5 keeps customer portal access inside tenant scope plus local visibility scope; `finance.timesheet` carries `customer_visible_flag`; invoices and generated PDFs remain linked through finance and docs tables.
- Earlier uploaded customer-management material emphasized read-only personalized customer logins and customer-safe visibility boundaries.
- Sprint reference: `docs/sprint/sprint-10-payroll-billing-and-settlement.md`.
- Cross-cutting rules: `AGENTS.md`, especially least-privilege customer visibility, document linkage, and customer-facing name hiding.

## Dependencies

- `US-9-T1` through `US-9-T4` should already provide customer portal authentication, scope filters, portal read patterns, and name-hiding/privacy baselines.
- `US-29-T1`, `US-29-T2`, and `US-29-T3` should already provide released timesheets, invoices, document artifacts, and delivery-context fields.
- `US-21-T4` should already provide released schedule read-model patterns that this finance portal exposure can align with.
- If the repository already has customer portal download helpers or document controllers, reuse them rather than creating a second download/security path.

## Scope of work

- Implement portal-scoped read services and endpoints for released timesheet lists/details, invoice lists/details, downloadable PDFs/documents, and limited status metadata such as issue date, due date, and release state where appropriate.
- Apply tenant + customer-account + order/customer visibility filters to every query so portal users see only their own permitted finance outputs.
- Respect release and visibility flags, hiding draft/unreleased finance results and masking personal names or other restricted data by default unless an explicit customer-release rule already permits them.
- Return stable DTOs shaped for the existing customer portal, including document references, period windows, commercial totals where allowed, and safe status labels without leaking internal controller notes, reconciliation detail, or unrelated tenant data.
- Where the repository already includes customer-portal views, add narrow Vben-portal or repository-equivalent UI pages/components for finance documents with localized DE/EN strings and permission checks.
- Add tests for portal scope filtering, release gating, name masking, document download authorization, and DTO stability.

## Preferred file targets

- Customer-portal read services, DTOs, routers, and tests under `modules/customers/`, `modules/finance/`, or the repository-equivalent portal-read package.
- Portal document-download controllers or adapters that already enforce scope and visibility.
- Optional customer-portal views/components for released finance outputs if the repo already has a portal UI layer.
- Tests for visibility, masking, authorization, and download behavior.

## Hard constraints

- Do not expose draft, unreleased, or internal-only finance data in the customer portal.
- Personal names must remain hidden by default unless an explicit released-visibility rule already allows them for the customer.
- Do not leak internal finance notes, reconciliation reasons, provider metadata, or unrelated tenant/customer data.
- Portal queries must always enforce tenant scope and local customer visibility scope together.
- Do not add customer-side write actions such as dispute workflows or invoice edits unless the repository already has an explicitly approved requirement.
- Any added UI strings or portal labels must follow German-default and English-secondary localization rules.

## Expected implementation outputs

- Portal-scoped timesheet/invoice read models and endpoints.
- Authorized document-download access to released finance artifacts.
- Optional localized portal UI surfaces for finance-document visibility.
- Tests proving scope safety, release gating, and customer-safe masking.

## Non-goals

- Do not implement payment collection, customer disputes, or customer-side invoice editing in this task.
- Do not expose internal operational discrepancies or finance reconciliation notes unless explicitly intended for customer visibility.
- Do not create shadow copies of finance write-side data solely for portal rendering.
- Do not broaden this task into the full reporting/dashboard layer planned for later sprints.

## Verification checklist

- [ ] Customer portal users see only their own released timesheets and invoices.
- [ ] Document downloads are authorized, scoped, and linked to the correct released finance outputs.
- [ ] Personal names remain hidden by default unless an explicit release rule allows them.
- [ ] Draft or internal-only finance data remains invisible to customer portal users.
- [ ] Tests cover scope filtering, masking, release gating, and document-download authorization.
