# AI-16 — User Capability and Page Context Tools

## Task Objective

Implement the first safe, read-only assistant tools that allow the SicherPlan AI Assistant to understand what the current authenticated user may do and which page context the user is working in.

This task does **not** implement business diagnostics yet. It creates the assistant tool layer needed for later tasks, especially `AI-17` safe navigation links and `AI-21` shift visibility diagnostics.

The tools must use the backend authorization context as the source of truth. Frontend route context may be used as a hint, but it must never be trusted as an authorization source.

### 2. Implement `assistant.get_current_user_capabilities`

Register a read-only tool named:

```text
assistant.get_current_user_capabilities
```

Purpose:

Return a redacted summary of the current authenticated user’s assistant-relevant capabilities.

Input schema:

```json
{}
```

The tool must ignore any user-supplied tenant, role, or permission data.

Output schema should include:

```json
{
  "user_id": "uuid-or-redacted-string",
  "tenant_id": "uuid-or-null",
  "scope_kind": "platform|tenant|branch|mandate|customer|subcontractor|employee_self_service|unknown",
  "role_keys": ["string"],
  "permission_keys": ["string"],
  "assistant_capabilities": {
    "can_chat": true,
    "can_run_diagnostics": true,
    "can_use_knowledge": true,
    "can_reindex_knowledge": false,
    "can_receive_navigation_links": true
  },
  "scope_summary": {
    "branch_ids": [],
    "mandate_ids": [],
    "customer_ids": [],
    "subcontractor_ids": []
  },
  "redactions_applied": true
}
```

Adapt field names to existing schemas if needed.

Security requirements:

- `tenant_id`, `user_id`, roles, permissions, and scopes must come from backend auth context only.
- Do not expose secrets, password hashes, tokens, session tokens, reset tokens, or raw auth headers.
- If user IDs or tenant IDs are considered sensitive in this codebase, return stable safe references or omit them from provider-facing output.
- The tool output sent to the model must be redacted and minimal.

### 3. Implement `assistant.get_current_page_context`

Register a read-only tool named:

```text
assistant.get_current_page_context
```

Purpose:

Normalize and validate the current frontend route context as a non-authoritative hint.

Input schema:

```json
{
  "path": "string|null",
  "route_name": "string|null",
  "page_id": "string|null",
  "query": {},
  "ui_locale": "string|null",
  "timezone": "string|null",
  "visible_page_title": "string|null"
}
```

Output schema should include:

```json
{
  "path": "string|null",
  "route_name": "string|null",
  "page_id": "string|null",
  "module_hint": "string|null",
  "ui_locale": "string|null",
  "timezone": "string|null",
  "safe_query": {},
  "is_known_page": true,
  "is_authoritative": false,
  "notes": ["Frontend route context is a hint only. Backend permissions still apply."]
}
```

Rules:

- Never treat frontend-provided `page_id`, `route_name`, or `path` as proof of access.
- Sanitize query parameters before storing or passing to the model.
- Remove or redact suspicious query keys such as `token`, `jwt`, `password`, `secret`, `reset`, `code`, `auth`, `apikey`, `api_key`.
- Keep allowed query values small and bounded.
- Validate path length and reject or truncate extreme values.

### 4. Implement `assistant.search_accessible_pages` as a Temporary Capability-Aware Stub

Register a read-only tool named:

```text
assistant.search_accessible_pages
```

In this task, this tool may be a lightweight stub that returns accessible page hints from an in-code minimal catalog or from existing routes if no page catalog table is ready yet.

Full catalog and link generation belong to `AI-17`.

Input schema:

```json
{
  "query": "string|null",
  "module_key": "string|null",
  "page_id": "string|null",
  "limit": 10
}
```

Output schema:

```json
{
  "pages": [
    {
      "page_id": "P-04",
      "label": "Staffing Board & Coverage",
      "module_key": "planning",
      "can_access": true,
      "reason": "User has required planning/staffing read permission or matching role scope."
    }
  ],
  "truncated": false
}
```

Rules:

- Only return pages the current user is allowed to know about.
- If permission mapping is not complete yet, use conservative behavior and return fewer pages, not more.
- Mark this as a bridge for `AI-17` in code comments or documentation.
- Do not return direct clickable links here. Direct safe links belong to `navigation.build_allowed_link` in `AI-17`.
