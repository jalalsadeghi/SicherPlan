# Codex Prompt — US-27-T3

## Task title

**Support allowances, expenses, flat rates, and comments on actuals.**

## Objective

Extend reconciled actuals with finance-side line items and notes so allowances, reimbursable expenses, flat-rate handling, and explanatory comments are attached to the actual bridge in a way that later payroll, invoicing, and partner settlement can consume cleanly.

## Source context

- Updated proposal: section 5.6 references expenses, flat rates, customer-side adjustments, and payroll logic with allowances and surcharges; section 5.7 later reporting depends on these details being explicit.
- Implementation specification: `finance.actual_allowance` and `finance.actual_expense` hang from `finance.actual_record`; actuals are the parent of approvals, allowances, expenses, timesheets, and payroll export items.
- Dienstplanmacher payroll/reconciliation materials describe employee/qualification/function-based allowances, advances, travel expense handling, and visible comments/notes around reconciled shifts.
- Sprint reference: `docs/sprint/sprint-09-time-capture-and-actuals.md`.

## Dependencies

- `US-26-T2` must already provide `actual_record`.
- `US-27-T2` should already provide the finance reconciliation seam where exception-driven adjustments are attached.
- `US-12-T2` or equivalent HR/pay-profile work may already provide employee allowance/source data that finance can reference but not own.
- If the repository already has a finance note/comment audit pattern, reuse it rather than introducing a free-text shadow log.

## Scope of work

- Implement allowance and expense line support attached to actual records via `finance.actual_allowance` and `finance.actual_expense` or the approved equivalents.
- Support explicit line typing, quantity/amount/currency semantics, and source/reason metadata so later export/invoice logic can interpret the lines safely.
- Add finance-safe comment/note support on actual reconciliation where the repository needs human explanation for allowances, expenses, or flat-rate treatment.
- Support flat-rate or customer-specific commercial adjustments only through explicit fields/line items rather than hidden arithmetic baked into minute totals.
- Add tests for allowance creation, expense creation, flat-rate/customer-side annotation, and comment/audit behavior.

## Preferred file targets

- `modules/finance` models, services, schemas, and tests for actual adjustment lines.
- Vben finance edit/review screens only if the repo already has an actual reconciliation UI area.
- No major mobile work in this task.

## Hard constraints

- Allowance and expense lines must hang from `actual_record`; do not create disconnected finance adjustment tables with no actual lineage.
- Do not silently fold allowances, expenses, or flat-rate logic into minute fields.
- Comments/notes must remain auditable and permission-scoped.
- Do not implement final payroll tariff computation, customer invoice rendering, or export batching here.
- Keep DE default / EN secondary behavior for any added UI labels or messages.

## Expected implementation outputs

- Allowance/expense/flat-rate support attached to actual records.
- Audit-safe notes/comments where reconciliation needs human explanation.
- Tests proving line-item semantics and actual linkage.

## Non-goals

- Do not implement full payroll engine math or invoice PDF generation here.
- Do not create customer-facing timesheet release logic in this task.
- Do not treat comments as an unstructured substitute for missing typed fields.

## Verification checklist

- [ ] Allowance and expense lines are explicitly modeled and linked to actual records.
- [ ] Flat-rate or customer-side commercial handling remains explicit and auditable.
- [ ] Notes/comments stay permission-scoped and traceable.
- [ ] Minute totals are not used as a hidden container for commercial adjustments.
- [ ] Tests cover typed lines, linkage, and note/audit behavior.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
