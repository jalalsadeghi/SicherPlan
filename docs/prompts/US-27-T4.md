# Codex Prompt — US-27-T4

## Task title

**Build approval UI and audit trail for three-stage signoff.**

## Objective

Complete the actual-approval workflow by delivering the web-admin signoff experience and full audit trail around the three-stage chain so operational and finance users can see status, evidence, approvals, notes, and transitions in one traceable review flow.

## Source context

- Updated proposal: section 5.6 requires a three-stage approval flow from employee/field lead to object/event lead to administration; audit and traceability requirements apply to financially relevant updates.
- Role model and permission principle require explicit action rights such as Read, Create, Edit, Export/Print, and Approve, while finance-sensitive data remains separated from general operational data.
- Implementation specification: `finance.actual_approval`, audit tables, and cross-cutting audit guidance in sections 4.1, 4.8, and 2.
- Sprint reference: `docs/sprint/sprint-09-time-capture-and-actuals.md`.
- Vben Admin is the approved web-admin baseline for route guards, list/detail forms, and localized operator workflows.

## Dependencies

- `US-27-T1` must already provide stage-one and stage-two approval behavior.
- `US-27-T2` and `US-27-T3` should already provide finance reconciliation context, adjustments, and comments.
- `US-5-T2` and `US-5-T3` should already provide audit and permission scaffolding.
- `US-3-T1` and `US-3-T3` should already provide the Vben shell and DE/EN localization baseline.

## Scope of work

- Build a Vben-based actual-approval review flow with list/detail pages, stage indicators, evidence summaries, adjustment visibility, and action buttons for each approval stage according to role scope.
- Expose approval history, approver identity, timestamps, notes, discrepancy markers, and current status in a way that preserves traceability without forcing users to inspect raw tables.
- Integrate approval actions with the real finance approval commands and audit trail, including final administration/finance signoff where permitted.
- Add route guards, action-level permission checks, and localized DE/EN UI strings for the approval workspace.
- Ensure audit history is visible enough for QM/security/finance review without leaking raw HR-private data or overexposing names to the wrong role.
- Add frontend and backend tests for stage visibility, permission gating, status transitions, and audit-view completeness.

## Preferred file targets

- Vben routes/views/components for the finance actuals approval workspace.
- `modules/finance` approval routers/services only where final signoff and audit retrieval need backend work.
- Locale resources for DE and EN.
- Tests for UI behavior, permission checks, and approval/audit contracts.

## Hard constraints

- Follow Vben Admin conventions; do not create a standalone one-off workflow shell outside the established admin app.
- Stage ordering and authorization must be enforced in backend services, not just hidden in the UI.
- Audit history must remain complete and readable after final signoff.
- Do not expose HR-private data, unrestricted finance data, or hidden names outside the intended review scope.
- German must be the default locale and English must be supplied for every new user-facing string.

## Expected implementation outputs

- Localized Vben approval UI for actual review and three-stage signoff.
- Final-stage approval integration with backend audit-safe services.
- Tests covering permission behavior, status visibility, and audit completeness.

## Non-goals

- Do not implement payroll export batches, invoice rendering, or report dashboards here.
- Do not replace backend authorization with frontend-only route guards.
- Do not build a mobile approval experience in this task unless the repository already has an explicit approved requirement for it.

## Verification checklist

- [ ] The Vben actual-approval workspace shows three-stage status, evidence context, and audit history clearly.
- [ ] Permission and route guards align with backend authorization and stage ordering.
- [ ] DE is default and EN is present for every new approval UI string.
- [ ] Final signoff actions and audit history remain traceable and test-covered.
- [ ] No restricted HR/private data is leaked through the approval UI.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
