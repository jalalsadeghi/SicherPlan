# Codex Prompt — US-16-T1

## Task title

**Implement requirement types, equipment catalog, sites, venues, trade fairs, and patrol routes.**

## Objective

Build the reusable operational master-data backbone inside the planning bounded context so dispatchers can create customer-linked sites, event venues, trade fairs, patrol routes, requirement catalogs, and equipment catalogs before concrete plans or shifts are created.

## Source context

- Updated proposal: section 5.5 planning and operational execution, especially the four planning modes (event, object/site, trade fair, patrol), the project-details block, and Appendix D entities for customer order/project and planning instance.
- Implementation specification: section 4.6 `ops.requirement_type`, `ops.equipment_item`, `ops.site`, `ops.event_venue`, `ops.trade_fair`, and `ops.patrol_route`; section 8 Phase E; section 9 `PlanningAggregate`.
- Sprint reference: `docs/sprint/sprint-06-ops-planning-foundation.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-7-T1` must already provide stable customer identifiers and customer ownership because every major ops master record here is customer-linked.
- `US-4-T1` must already provide `common.address`, lookup domains, tenant/branch/mandate scoping, and reusable status conventions.
- `US-5-T3` should already provide role/tenant scoping patterns for admin APIs.
- This task should stay ahead of `US-16-T2`; leave clean child seams for patrol checkpoints and trade-fair zones without front-loading them here.

## Scope of work

- Create Alembic migrations, SQLAlchemy models, Pydantic Create/Update/Read/List schemas, services, routers, and tests for the operational-master tables listed in the implementation spec.
- Support tenant-scoped business identifiers such as `site_no`, `venue_no`, `fair_no`, and `route_no`, along with status, archive-safe lifecycle handling, and optimistic-locking support where the repo already uses `version_no`.
- Implement `ops.requirement_type` with `code`, `label`, `default_planning_mode_code`, and status so customer orders can reference a normalized requirement catalog instead of free-text requirement types.
- Implement `ops.equipment_item` as the reusable equipment catalog with code, label, unit of measure, and status; keep it normalized because orders will reference it in `ops.order_equipment`.
- Implement `ops.site`, `ops.event_venue`, and `ops.trade_fair` with explicit customer FKs, address linkage through `common.address`, geo coordinates where the operational entity needs its own map point, and watchbook/timezone fields where defined by the spec.
- Implement `ops.patrol_route` with customer FK, optional site FK, route number, start/end point text, travel-policy code, and status so later route planning and patrol execution can attach safely.
- Add service validation for unique codes/numbers per tenant, valid date windows on trade fairs, safe customer ownership, and optional site linkage rules on patrol routes.
- Add backend tests for CRUD, uniqueness, customer scoping, archive-safe listing, and address/customer foreign-key behavior.

## Preferred file targets

- Alembic revisions for the new `ops.*` master-data tables and indexes.
- `modules/planning/` or the repo equivalent for models, schemas, services, routers, validators, and tests.
- Shared lookup/address helpers only where they preserve the planning bounded-context boundary and do not move ownership out of `modules/planning`.
- No major UI work in this task beyond API contracts or fixtures needed by later Vben screens.

## Hard constraints

- Reuse `common.address`; do not duplicate postal-address columns across sites, venues, fairs, or routes unless the repository already has an approved immutable-snapshot pattern.
- Every record here remains tenant-owned and customer-linked; do not create global, cross-tenant ops masters.
- Do not implement `ops.trade_fair_zone` or `ops.patrol_checkpoint` here; those belong to `US-16-T2`.
- Do not create watchbook, patrol-round, time-capture, or finance tables in this task. This sprint is still in Phase E of the planning backbone.
- Planning may reference CRM truth, but it must not mutate customer commercial settings or customer master data.
- Any user-visible validation text or enum labels introduced here must preserve German default and English secondary behavior.

## Expected implementation outputs

- A normalized ops master-data schema for requirement types, equipment, sites, venues, trade fairs, and patrol routes.
- Stable admin-facing APIs/services that later order, planning-record, watchbook, patrol, and finance flows can reuse.
- Tests covering uniqueness, tenant/customer scoping, date validation, and address linkage.

## Non-goals

- Do not build patrol checkpoints or trade-fair zones here; that is `US-16-T2`.
- Do not build the Vben admin UI or import tooling here; that is `US-16-T3`.
- Do not implement watchbook/patrol cross-link helper behavior beyond the schema seams required now; that is `US-16-T4`.
- Do not implement orders, planning records, or shifts here; those belong to `US-17-*` and `US-18-*`.

## Verification checklist

- [ ] The new ops master-data migrations apply cleanly on a fresh database and a repo with prior Sprint 1–5 schema waves.
- [ ] Sites, venues, fairs, routes, requirement types, and equipment items can be created, updated, archived, and listed within tenant scope.
- [ ] Trade fairs enforce valid date windows and patrol routes enforce safe optional site linkage.
- [ ] Address reuse and customer ownership are enforced through explicit foreign keys rather than loose JSON or text duplication.
- [ ] Tests cover uniqueness, customer scoping, archive handling, and the most important validation failures.
