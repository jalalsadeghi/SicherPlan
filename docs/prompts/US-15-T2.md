# Codex Prompt — US-15-T2

## Task title

**Expose released positions, schedules, actuals, and invoice-check status in portal views**

## Objective

Provide the backend portal-read layer that exposes only approved released work and downstream status information to subcontractors, while remaining safe if some planning or finance modules are not yet present on the branch.

## Source context

- Updated proposal: section 5.4 subcontractor portal visibility, section 5.6 subcontractor settlement, and section 6.3 steps 2, 5, 6, and 7 of the subcontractor collaboration workflow.
- Implementation specification: subcontractor chain, field-evidence chain, finance result chain, and portal visibility chain in section 5; read-model guidance in section 7.
- Uploaded Dienstplanmacher subcontractor source material: subcontractor planning, online attendance reports, controlling of subcontractor shifts, and invoice-check visibility all imply partner-safe released-work views.
- Sprint reference: `docs/sprint/sprint-05-subcontractor-management.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-15-T1` must already provide portal authentication and effective subcontractor scope resolution.
- `US-14-T4` should already provide partner-side read contracts for workers, readiness, and identity data.
- Later planning/field/finance tables may not all exist yet in strict sprint order. This task must therefore support a forward-compatible contract strategy instead of inventing premature write-side tables.

## Scope of work

- Implement portal-scoped read services and endpoints for released positions, schedules, actual-status summaries, attendance-related visibility, and invoice-check status where upstream source tables already exist.
- When planning/field/finance modules are not yet available on the current branch, create stable endpoint contracts and safe empty-state behavior or adapter seams rather than fabricating substitute persistence.
- Filter every read by tenant, subcontractor, and release/visibility rules so portal users see only their own company and only work explicitly released or confirmed for partner visibility.
- Return portal-ready list/detail DTOs with status, time-window, location/context, readiness and confirmation hints, actuals summary, and invoice-check summary where data exists, while avoiding unnecessary personal-data exposure.
- Add tests for scope filtering, empty-state contract stability, and visibility rules around released versus non-released work.

## Preferred file targets

- Portal read/query services, DTOs, and endpoints under `modules/subcontractors/` or a portal-read package in the repo.
- Optional lightweight portal UI adapters only if the repo keeps the view/client model close to the backend contract for this task.
- Tests for visibility and empty-state compatibility.

## Hard constraints

- Do not create fake planning, field, or finance write tables simply to make the portal screens look populated in Sprint 5.
- Portal views may expose only released or otherwise explicitly visible work. Hidden, draft, stealth, or unrelated tenant data must stay invisible.
- Do not leak unrelated internal employee names, full finance internals, or cross-subcontractor information through convenience joins.
- Keep contract shape stable so later Sprint 6/7/9/10 implementations can plug into it without a breaking redesign.
- This task is primarily about backend read APIs/contracts, not about fully polishing every portal screen.

## Expected implementation outputs

- Portal-safe released-work read endpoints and DTOs.
- Forward-compatible empty-state or adapter behavior when upstream ops/field/finance modules are not yet present.
- Tests that prove visibility and scope correctness.

## Non-goals

- Do not implement self-allocation write commands here; that is `US-15-T3`.
- Do not build the full portal maintenance UI here; that is `US-15-T4`.
- Do not create planning release or invoice-check processing logic here.

## Verification checklist

- [ ] Portal users can read only their own subcontractor's released or otherwise permitted work/status data.
- [ ] Non-released or hidden operational data is excluded from portal responses.
- [ ] Contracts remain stable and predictable even when some upstream source modules are not implemented yet.
- [ ] Tests cover both populated-path and empty-state-path behavior without resorting to fake persistence.
- [ ] Returned data is sufficient for portal UX while remaining least-privilege by default.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
