# AI-13 — Knowledge Retrieval Adapter

## Task Objective

Implement the **Knowledge Retrieval Adapter** for the SicherPlan AI Assistant.

This task builds on the already planned knowledge source and chunking infrastructure from `AI-12`. The goal is to make indexed SicherPlan documentation searchable at runtime so the Assistant can answer platform “how-to” and workflow questions without rereading the full source documents on every request.

The implementation must support:

- a stable retrieval interface;
- deterministic fallback lexical retrieval for local development and CI;
- optional vector/embedding retrieval only if the current repository/dependency setup supports it;
- safe metadata filtering by language, module, page, role hints, and active source status;
- capped result counts and safe chunk sizes;
- tests that do not require an OpenAI API key.

The Assistant must use this retrieval layer for documentation knowledge. It must **not** index operational data such as employees, shifts, assignments, invoices, actuals, watchbooks, or portal records.

## Implementation Requirements

### 1. Retrieval Interface

Create or complete a retrieval interface in the assistant knowledge package.

Expected files, adjusted to actual repository conventions:

```text
backend/app/modules/assistant/knowledge/retriever.py
backend/app/modules/assistant/knowledge/repository.py
backend/app/modules/assistant/knowledge/embeddings.py
backend/app/modules/assistant/schemas.py
backend/tests/modules/assistant/test_knowledge_retrieval.py
```

The retrieval layer should expose a service/repository method similar to:

```python
async def retrieve_knowledge_chunks(
    *,
    query: str,
    language_code: str | None = None,
    module_key: str | None = None,
    page_id: str | None = None,
    role_keys: list[str] | None = None,
    permission_keys: list[str] | None = None,
    limit: int = 8,
) -> list[KnowledgeChunkResult]:
    ...
```

Adapt the exact signature to the existing sync/async conventions in the project.

### 2. Result Contract

Return a stable typed result object. It may be a Pydantic schema or dataclass, depending on project style.

Suggested fields:

```text
chunk_id
source_id
source_name
source_type
title
content
language_code
module_key
page_id
role_keys
permission_keys
score
rank
matched_by
```

`matched_by` should indicate the retrieval mode, for example:

```text
lexical
vector
hybrid
metadata
```

Do not expose raw embeddings in API responses.

### 3. Lexical Fallback Retrieval

Implement deterministic lexical retrieval that works in CI without external services.

Minimum behavior:

- normalize query and chunk text case-insensitively;
- tokenize basic words;
- score chunks by term overlap;
- boost exact phrase matches;
- boost title matches;
- boost matching `module_key`, `page_id`, and language when filters are supplied;
- return only active knowledge sources/chunks;
- cap results by configured max chunk count;
- order by score descending and then stable chunk/source order.

This fallback must be good enough for questions like:

```text
How do I release a shift to the employee app?
Wie erstelle ich einen neuen Kunden?
چطور برای یک کارمند self-service access بسازم؟
Where can I check subcontractor invoice status?
```

### 4. Optional Vector Retrieval Adapter

If vector support already exists in the repository, implement a vector adapter behind an interface.

If vector support does not exist, do **not** introduce heavy dependencies in this task unless already approved by project conventions. Instead:

- keep an `EmbeddingRetriever` or `VectorRetriever` interface;
- implement lexical retrieval as the default;
- document the future vector hook in code comments and tests.

The retrieval adapter must not fail when embeddings are missing.

### 5. Metadata Filtering

Support optional filters:

```text
language_code
module_key
page_id
role_keys
permission_keys
source_type
active source status
```

Filtering rules:

- If `language_code` is supplied, prefer same-language chunks but do not necessarily exclude English documentation if no same-language docs exist.
- If `module_key` or `page_id` is supplied, boost exact matches.
- Role and permission hints may be used for boosting or safe filtering, but do not treat them as authorization. Authorization is handled by tool/navigation policy, not by knowledge retrieval alone.
- Never use frontend-provided role/permission claims as authoritative.

### 6. Integration with Assistant Service

Wire retrieval into the assistant service only as a backend helper for future prompt assembly.

At this task stage:

- do not implement full prompt orchestration if it belongs to `AI-14`;
- provide a clean callable method that later tasks can use;
- ensure existing assistant endpoints do not break.

### 7. Runtime Limits

Respect settings from `AI-02`, for example:

```text
SP_AI_MAX_CONTEXT_CHUNKS
SP_AI_MAX_INPUT_CHARS
```

If exact setting names differ, align with existing names.

The retriever must:

- cap result count;
- truncate oversized chunk content safely if needed;
- avoid loading all huge documents into memory when a database query can narrow candidates;
- never return secrets or operational records.

### 8. Tests

Add or update tests for:

```text
backend/tests/modules/assistant/test_knowledge_retrieval.py
```

Required scenarios:

1. retrieves relevant chunks for an English how-to query;
2. retrieves relevant chunks for a German how-to query;
3. retrieves relevant chunks for a Persian how-to query using fallback lexical matching where possible;
4. boosts page/module metadata when provided;
5. skips inactive or failed knowledge sources;
6. skips unchanged source behavior remains covered by AI-12 tests and is not broken;
7. returns stable ordering for equal scores;
8. caps results at configured limit;
9. works when embeddings are missing;
10. does not require OpenAI API key.

Use fixtures/factories consistent with the repository style.
