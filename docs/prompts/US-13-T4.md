# Codex Prompt — US-13-T4

## Task title

**Enforce tenant-safe access and audit trails for subcontractor master data**

## Objective

Harden subcontractor master-data access so internal users and future portal-linked actors can only see or mutate allowed records, while sensitive edits leave durable audit evidence.

## Source context

- Updated proposal: section 4 role model for tenant administrators, dispatchers, accounting, controller/QM, customer users, and subcontractor users; section 5.4 portal access and finance-sensitive subcontractor data.
- Implementation specification: `iam.user_role_scope` with optional `subcontractor_id`, `audit.audit_event`, the portal visibility chain in section 5, and shared tenant-safety conventions in section 2.
- Uploaded Dienstplanmacher subcontractor source material: own-login access for subcontractors and a history trail of planner/administrator changes reinforce the need for role-safe access and auditability.
- Sprint reference: `docs/sprint/sprint-05-subcontractor-management.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-5-T1`, `US-5-T3`, and `US-5-T4` must already provide the IAM, scope, and audit foundation.
- `US-13-T1` and `US-13-T3` should already provide the partner master tables and lifecycle/history seams that this task must protect.

## Scope of work

- Define and enforce permission checks for subcontractor read/create/update/archive, finance-profile visibility, contact/portal-link management, and history/attachment access using the repo's IAM pattern.
- Apply tenant scoping and, where relevant, subcontractor-specific or branch/mandate-specific scope filters in service and router layers so broad tenant access is never the default.
- Introduce or refine field-level restrictions for bank, tax, payment, and portal-account linkage data so only authorized roles can read or change those fields.
- Emit `audit.audit_event` entries or the repo equivalent for sensitive actions such as create, edit, archive/reactivate, portal enable/disable, finance-profile changes, and linked-user changes.
- Add automated tests for cross-tenant denial, insufficient-role denial, field-level exposure rules, and audit-event emission for the most sensitive changes.
- Where admin UI or route metadata must change to stay aligned with backend enforcement, make the smallest corresponding adjustments without turning this into a full frontend task.

## Preferred file targets

- Authorization middleware/policy helpers, partner service guards, routers, and tests under `modules/subcontractors/` and `modules/iam/` or the repo equivalent.
- Audit-event emission helpers or adapters where the repo centralizes audit creation.
- Minimal frontend route/action-guard metadata only if needed for consistency with server-side enforcement.

## Hard constraints

- Server-side enforcement is mandatory; do not rely on client-side hiding or route guards as the source of truth.
- External or portal-linked actors must never gain broad tenant visibility through convenience joins or weak default scopes.
- Do not introduce PostgreSQL RLS as a surprise architectural shift unless the repo already uses it; this task should fit the current app-layer authorization strategy while remaining RLS-compatible later.
- Audit events must capture enough entity and before/after context to support later QM/security reporting without leaking secret values unnecessarily.
- Do not weaken usability by duplicating the same authorization logic inconsistently across services and routers; centralize policy decisions wherever possible.

## Expected implementation outputs

- A hardened permission model for subcontractor master-data APIs and sensitive fields.
- Durable audit evidence for important partner-master mutations.
- Tests that prove tenant isolation, role scoping, and audit coverage for the most critical actions.

## Non-goals

- Do not build portal authentication or portal navigation here; that is `US-15-T1`.
- Do not redesign the global IAM architecture beyond the narrow changes needed for subcontractor scope enforcement.
- Do not implement finance settlement behavior or invoice-check processing here.

## Verification checklist

- [ ] Cross-tenant access to subcontractor master data is denied consistently.
- [ ] Unauthorized roles cannot read or change finance-sensitive partner fields.
- [ ] Sensitive partner-master mutations emit audit events with the expected entity and actor metadata.
- [ ] Any corresponding UI guard changes align with backend truth instead of replacing it.
- [ ] Tests clearly distinguish allowed internal roles, denied internal roles, and portal-linked scope behavior where applicable.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
