# Codex Prompt — US-21-T2

## Task title

**Generate deployment plans, protocols, and print/PDF outputs from released planning data.**

## Objective

Implement the released-planning document outputs so deployment plans, protocols, and related print/PDF artifacts are generated from approved planning truth and stored through the centralized document backbone.

## Source context

- Updated proposal: section 5.5 released outputs, the operational note about customer-facing name hiding by default, and section 9 deliverables calling for revision-safe PDF outputs for deployment plans and related operational documents.
- Implementation specification: docs-service ownership rules, generated-output expectations, release/visibility controls in section 4.6, and reporting/output guidance in sections 5 and 7.
- Sprint reference: `docs/sprint/sprint-07-staffing-engine.md`.
- Cross-cutting rules: `AGENTS.md`, especially document-centered compliance and generated-output tracking.

## Dependencies

- `US-21-T1` must already provide release gating and visibility semantics for planning data.
- `US-6-T1` must already provide the centralized document/document_version/document_link backbone.
- `US-17-T3` should already provide order/planning document-package seams where applicable.
- `US-19-*` and `US-20-*` must already provide the released staffing truth and the validation-controlled state that output generation depends on.

## Scope of work

- Implement backend services and any necessary templates/adapters for generating deployment plans, protocols, and related print/PDF outputs from released planning data only.
- Represent generated outputs as centralized document rows and versions linked back to the owning planning entity, rather than unmanaged files or ad-hoc storage references.
- Support localized labels/metadata and stable output generation rules so the same released planning state yields reproducible operational documents.
- Respect visibility and name-disclosure rules: customer-facing variants should hide personal names by default unless the prime company explicitly releases them.
- Provide API endpoints or command handlers for requesting/regenerating outputs under the repository’s approved docs-generation pattern.
- Add backend tests for released-only generation, docs-link creation, reproducibility of key metadata, and the name-visibility rule where applicable.

## Preferred file targets

- `modules/planning/` output services and controllers where planning owns the business meaning.
- `modules/platform_services` or the repo-equivalent docs/output package only for shared generation plumbing that already belongs there.
- Tests around document links, generation permissions, and released-only behavior.

## Hard constraints

- Generated outputs must be based on released planning data only.
- Do not write files outside the centralized docs/output backbone or bypass document metadata/versioning.
- Customer-facing output variants must hide personal names by default unless explicit release rules say otherwise.
- Do not mix invoice, timesheet, or finance output generation into this task; keep scope on planning deployment/protocol outputs.
- Generated files are revision-sensitive artifacts; keep metadata, ownership links, and generation provenance durable.
- Any user-visible output labels or generation status texts must preserve German default and English secondary behavior.

## Expected implementation outputs

- Released-planning PDF/print generation for deployment plans and protocols.
- Document-backed storage and linkage for generated operational outputs.
- Tests proving released-only generation, docs linkage, and visibility-safe variants.

## Non-goals

- Do not implement customer invoices, timesheets, or payroll outputs here.
- Do not implement portal/mobile exposure of released schedules here; that belongs to `US-21-T4`.
- Do not turn this task into a generic reporting export engine.
- Do not bypass the centralized docs service for convenience.

## Verification checklist

- [ ] Deployment plans/protocols are generated only from released planning data.
- [ ] Every generated artifact is represented as a centralized document/version linked to the owning business entity.
- [ ] Customer-facing variants hide personal names by default unless explicitly released.
- [ ] Generation and regeneration remain reproducible and permission-aware.
- [ ] Tests cover docs linkage, released-only behavior, and the core visibility rule.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
