# Codex Prompt — US-32-T4

## Task title

**Build controller/QM export screens and scheduled report delivery hooks.**

## Objective

Deliver the controller/QM reporting workspace so authorized internal users can run compliance and security exports on demand and schedule report delivery through the existing communication/integration backbone without breaking tenant scope, role scope, or evidence traceability.

## Source context

- Updated proposal: section 5.7 requires audit-ready download of confirmation evidence plus internal communication and compliance reporting; acceptance requires key reports and compliance exports to be downloadable and reproducible.
- Implementation specification: the information/QM chain links `info.notice`, audience, and read confirmations into compliance/QM report views; reporting stays view-driven and should be exposed through FastAPI services.
- Sprint reference: `docs/sprint/sprint-11-reporting-and-hardening.md`.
- Cross-cutting rules: `AGENTS.md`, especially Vben Admin patterns, DE-default / EN-secondary localization, outbox/integration discipline for external side effects, and role-scoped report delivery.
- Reference admin shell: Vben Admin remains the agreed web admin reference.

## Dependencies

- `US-32-T1` through `US-32-T3` must already provide stable compliance/QM/security reporting views and filter contracts.
- `US-6-T2` and `US-6-T4` must already provide communication templates, outbound messages, delivery attempts, import/export jobs, and outbox/job-worker patterns.
- `US-3-T1` and `US-3-T4` must already provide the Vben shell and DE/EN localization structure.
- `US-5-T1` through `US-5-T3` must already provide the permission and route-guard model used to restrict controller/QM exports.

## Scope of work

- Build Vben Admin controller/QM reporting screens for compliance and security report browsing, filtering, export initiation, and export history visibility where the repository already supports generated-output history.
- Expose on-demand export endpoints or handlers for authorized users, reusing repository-standard export formats and generated-document conventions rather than inventing a parallel reporting export path.
- Add scheduled report delivery hooks that reuse the existing job/outbox/communication backbone so scheduled exports are generated and delivered outside the main transaction flow.
- Allow saved delivery configuration only if the repository already has a stable settings or scheduled-job pattern; otherwise keep the first version as a narrow scheduled-hook implementation documented for later extension.
- Ensure generated compliance/security export files are represented as document rows or repository-equivalent generated-output records when the platform already tracks generated files centrally.
- Add DE/EN localization for page labels, filters, scheduling labels, delivery status messages, and export names, plus tests for authorization, scheduling hook behavior, and generated-output traceability.

## Preferred file targets

- Vben reporting/export pages/components under the web admin application structure.
- Reporting/export controllers, generated-output services, and scheduling/job hook code under `modules/reporting/` and/or `modules/platform_services/`.
- Locale resources, route-guard configuration, and tests for export and scheduling behavior.
- Docs/runbook notes if the repository already keeps scheduler or generated-output usage guidance in-repo.

## Hard constraints

- Do not send external messages or files directly inside the same transaction that writes scheduling intent or export generation state; use the existing outbox/job-worker pattern.
- Every on-demand or scheduled export must enforce the same tenant and role scope as the corresponding report surface.
- Generated export artifacts must remain traceable and auditable; use the central document/generated-output model if the repository already has it.
- Use Vben Admin patterns for internal reporting screens and keep the UI thin over the reporting/export APIs.
- All new UI labels, export labels, and delivery/scheduling text must follow German default and English secondary localization behavior.
- Do not broaden this task into the general business-report dashboard package handled in `US-31-T3` or into hardening work handled in `US-33-*`.

## Expected implementation outputs

- Controller/QM Vben screens for browsing and exporting compliance/security reports.
- Authorized on-demand export endpoints/handlers and scheduled-delivery hooks using the existing outbox/job infrastructure.
- Generated-output traceability and DE/EN resources plus tests for scope, scheduling, and export behavior.

## Non-goals

- Do not create a second scheduler or ad hoc email sender just for reports.
- Do not expose compliance/security exports to customer, subcontractor, or employee portal users.
- Do not redesign the whole communication framework in this task.
- Do not treat this as a BI-subscription system beyond the scoped scheduled hooks required by Sprint 11.

## Verification checklist

- [ ] Authorized controller/QM users can run exports on demand and the outputs are traceable.
- [ ] Scheduled-delivery hooks reuse the repository’s existing job/outbox patterns.
- [ ] Generated export files respect tenant and role scope and remain auditable.
- [ ] DE/EN localization is present for the new screens and export actions.
- [ ] Tests cover authorization, scheduling hook behavior, and generated-output traceability.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
