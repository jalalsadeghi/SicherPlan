This task is **AI-05 — Assistant Database Models and Migrations**.

## Goal

Add the assistant persistence data model and database migration baseline required for the AI Assistant feature.

This task creates the database tables and SQLAlchemy/model definitions for:

- conversations;
- messages;
- tool-call audit rows;
- user feedback;
- knowledge sources;
- knowledge chunks;
- page route catalog;
- prompt/policy version tracking.

This task must not implement business logic beyond model definitions and migration wiring. Conversation endpoints and persistence behavior are implemented in AI-06.

## Hard Rules

1. Inspect the existing migration and model conventions before writing migrations.
2. Use the existing SQLAlchemy/base model style, timestamp style, UUID style, naming style, enum/check constraint style, and JSON column style.
3. Do not invent a second ORM pattern.
4. Do not implement service methods for conversation persistence in this task. That is AI-06.
5. Do not implement OpenAI/provider logic.
6. Do not implement knowledge ingestion logic.
7. Do not implement tool registry execution.
8. Do not implement frontend code.
9. Do not add pgvector as a hard dependency unless the repository already supports it. Use a nullable fallback column for embeddings if needed.
10. Do not duplicate permission seeds if AI-03 already implemented them. Only add missing permission-related migration content if the existing project pattern requires permissions to be seeded in the same migration and this was not done in AI-03.
11. Do not store secrets, raw provider responses, API keys, JWTs, reset tokens, or credentials.
12. Keep migrations safe when AI is disabled.
13. Maintain tenant isolation assumptions in the table design.

## Required Models and Fields

### 1. Assistant Conversation

Purpose: stores one assistant conversation per user/session context.

Required logical fields:

```text
id UUID PK
tenant_id UUID nullable only for platform-wide/system context
user_id UUID not null
title text nullable
locale text nullable
status text not null default 'active'
last_route_name text nullable
last_route_path text nullable
created_at timestamptz not null
updated_at timestamptz not null
archived_at timestamptz nullable
```

Constraints:

```text
status in ('active', 'archived')
```

Indexes:

```text
idx_assistant_conversation_tenant_user_updated
idx_assistant_conversation_user_status
```

Foreign keys:

- `tenant_id` should reference the tenant/core table if that is how other tenant-owned tables do it;
- `user_id` should reference the user/account table if existing modules use real FKs;
- if existing modules intentionally avoid cross-module FKs for some tables, follow that convention and document the reason.

### 2. Assistant Message

Purpose: stores user, assistant, tool, and system-summary messages.

Required logical fields:

```text
id UUID PK
conversation_id UUID not null
tenant_id UUID nullable
user_id UUID nullable
role text not null
content text not null
structured_payload json/jsonb nullable
detected_language text nullable
response_language text nullable
token_input integer nullable
token_output integer nullable
created_at timestamptz not null
```

Constraints:

```text
role in ('user', 'assistant', 'tool', 'system_summary')
```

Indexes:

```text
idx_assistant_message_conversation_created
```

Foreign keys:

```text
conversation_id -> assistant_conversation.id
```

### 3. Assistant Tool Call

Purpose: audits backend tool calls initiated by the assistant.

Required logical fields:

```text
id UUID PK
conversation_id UUID not null
message_id UUID nullable
tenant_id UUID nullable
user_id UUID not null
tool_name text not null
input_json json/jsonb not null
output_json_summary json/jsonb nullable
required_permissions json/jsonb nullable
permission_decision text not null
scope_kind text nullable
entity_refs json/jsonb nullable
duration_ms integer nullable
error_code text nullable
created_at timestamptz not null
```

Constraints:

```text
permission_decision in ('allowed', 'denied', 'failed')
```

Indexes:

```text
idx_assistant_tool_call_conversation_created
idx_assistant_tool_call_tenant_user_created
idx_assistant_tool_call_tool_name_created
```

Important:

- `input_json` and `output_json_summary` must be suitable for redacted summaries only.
- Do not store raw HR-private details, payroll-sensitive fields, provider secrets, JWTs, reset tokens, or raw credentials.

```text

user_id UUID not null
rating text not null

created_at timestamptz not null
```

Constraints:

```text

```

Indexes:

```text
idx_assistant_feedback_conversation_message
```

Indexes:


### 6. Assistant Knowledge Chunk

Purpose: stores searchable chunks of platform documentation.

Required logical fields:

```text
id UUID PK
source_id UUID not null
chunk_index integer not null
title text nullable
content text not null
language_code text nullable
module_key text nullable
page_id text nullable
role_keys json/jsonb nullable
permission_keys json/jsonb nullable
embedding nullable json/jsonb/bytea/vector depending on available support
token_count integer nullable
created_at timestamptz not null
```

Constraints:

```text
unique(source_id, chunk_index)
```

Indexes:

```text
idx_assistant_knowledge_chunk_source_index
idx_assistant_knowledge_chunk_module_page
idx_assistant_knowledge_chunk_language
```

Embedding decision:

- If pgvector already exists in the repository, use the existing vector convention.
- If pgvector does not exist, do **not** add it as a hard dependency in this task. Use a nullable JSON/bytea/text fallback or omit vector indexing while keeping the field/interface ready for AI-13.

```text
updated_at timestamptz not null
```

Constraints:

```text

```

Indexes:

```text
idx_assistant_page_route_catalog_page_active
idx_assistant_page_route_catalog_module_active
```

Important:

- Do not seed the full catalog in this task unless the project’s migration seed convention requires it.
- Seeding the canonical page metadata is mainly AI-17.
- It is acceptable to create the table now and leave it empty.

### 8. Assistant Prompt Policy Version

Purpose: tracks prompt/policy versions used by the assistant without storing secrets.

Required logical fields:

```text
id UUID PK
policy_key text not null
version_label text not null
content_hash text not null
active boolean not null default true
created_at timestamptz not null
created_by_user_id UUID nullable
notes text nullable
```

Constraints:

```text
unique(policy_key, version_label)
```

Indexes:

```text
idx_assistant_prompt_policy_key_active
```

Important:

- Do not store full system prompts here if the project security preference is to keep prompts in code.
- Store a hash/version reference, not secrets.
