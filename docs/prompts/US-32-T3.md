# Codex Prompt — US-32-T3

## Task title

**Implement security activity views for login failures, role changes, and sensitive edits.**

## Objective

Build the security-reporting read model so administrators, controllers, and QM/security stakeholders can inspect failed logins, role changes, sensitive master-data edits, watchbook closure events, and other high-value activity through immutable audit/login evidence.

## Source context

- Updated proposal: section 5.7 includes login activity in compliance reporting; Appendix E lists system and security outputs such as login history, role changes, failed access attempts, and major master-data change history.
- Implementation specification: section 7 recommends `rpt.security_activity_v` built from `audit.audit_event`, `audit.login_event`, `iam.user_role_scope`, and `field.watchbook`; section 4.1 defines audit/login tables as append-only sources for security reporting.
- Sprint reference: `docs/sprint/sprint-11-reporting-and-hardening.md`.
- Cross-cutting rules: `AGENTS.md`, especially append-only audit evidence, tenant isolation, least-privilege access, and auditability for sensitive master-data and finance actions.

## Dependencies

- `US-5-T4` must already provide stable audit-event and login-event capture patterns.
- `US-5-T1` and related IAM work must already provide `iam.user_role_scope` and role metadata needed for role-change reporting.
- `US-23-T2`/`US-23-T3` and field-execution work should already provide watchbook closure or equivalent audited operational events where the repository records them.
- `US-11-T4`, `US-29-*`, and other sensitive-edit tasks should already emit business-audit events if the repository follows the earlier audit pattern.

## Scope of work

- Create `rpt.security_activity_v` or a repository-equivalent view that surfaces failed logins, role changes, sensitive master-data/finance edits, watchbook closure events, and related security-relevant activity categories from append-only audit sources.
- Design an event categorization layer or SQL classification logic that keeps raw audit evidence intact while exposing stable, queryable categories and severity levels for management/security review.
- Expose read-only endpoints and filter contracts for actor, tenant, branch/mandate, event category, result status, entity table, correlation window, and date range.
- Ensure role-change reporting can explain which user changed which scope or role assignment, using normalized IAM and audit truth without flattening away the scope semantics.
- Include representative sensitive-edit categories drawn from HR/private, finance, and other auditable modules already emitting events, but keep the classification extensible rather than hardcoding every table forever.
- Add tests for failed-login reporting, role-change visibility, sensitive-edit categorization, watchbook closure inclusion where present, and strict scope safety.

## Preferred file targets

- `modules/reporting/` security view definitions, query services, schemas, routers, and tests.
- Shared audit classification helpers or enums if the repository already has an audit taxonomy pattern.
- No RLS or deep infrastructure hardening in this task; keep it to reporting surfaces over existing append-only evidence.

## Hard constraints

- Do not mutate or clean up audit/login evidence from the reporting layer. Reporting must consume append-only truth.
- Keep raw evidence accessible for internal investigations through trace fields or drill-down references; do not reduce everything to lossy counts if the story expects explainability.
- Preserve least-privilege visibility. These outputs are for authorized internal security/QM/controller/admin roles only.
- Do not bypass the repository’s existing audit-event taxonomy or emit new audit events solely from the reporting layer unless a narrow prerequisite fix is required.
- Do not broaden this task into optional PostgreSQL RLS or other hardening work planned under `US-33-*`.

## Expected implementation outputs

- A security-activity view/service package grounded in append-only audit and login evidence.
- Read-only endpoints and filter/result schemas for failed logins, role changes, and sensitive edits.
- Tests proving categorization correctness, explainability, and strict scope safety.

## Non-goals

- Do not implement export screens or scheduled delivery here; that belongs to `US-32-T4`.
- Do not redesign the audit-event write path here unless an unavoidable, narrow fix is required.
- Do not treat this task as a SIEM integration project.
- Do not apply database-level RLS here; that belongs to `US-33-T2`.

## Verification checklist

- [ ] Failed logins, role changes, and sensitive edits are queryable from immutable source evidence.
- [ ] Event categorization remains explainable and traceable back to raw audit/login rows.
- [ ] Authorization and scope rules keep these reports internal and least-privilege.
- [ ] Tests cover representative failed-login, role-change, sensitive-edit, and watchbook-related scenarios.
- [ ] No audit evidence mutation or lossy aggregation hides investigation-critical detail.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
