## Goal

Implement backend persistence behavior for assistant conversations and messages.

After this task, an authenticated and authorized user must be able to:

- create a new assistant conversation;
- load their own assistant conversation;
- add a user message to their own conversation;
- receive a safe placeholder assistant message or minimal deterministic response if the full AI response pipeline is not implemented yet;
- keep conversation ownership tenant-safe and user-safe.

This task must not implement OpenAI calls, knowledge retrieval, tool registry, diagnostics, frontend widget, full structured response rendering, or real AI reasoning. Those are later tasks.

## Hard Rules

1. Follow existing repository patterns for repositories, services, routers, schemas, transactions, dependencies, and errors.
2. Use the models and tables from AI-05.
3. Enforce assistant feature flag and `assistant.chat.access` permission or project-equivalent permission.
4. Enforce conversation ownership by current authenticated user.
5. Enforce tenant scope using backend auth context, not client-provided tenant IDs.
6. Do not allow a user to load or write to another user’s conversation.
7. Do not allow cross-tenant conversation access.
8. Do not implement OpenAI/provider calls in this task.
9. Do not implement knowledge retrieval in this task.
10. Do not implement tool calls in this task.
11. Do not implement diagnostic logic in this task.
12. Do not add frontend widget code in this task.
13. Do not store secrets, JWTs, reset tokens, credentials, or raw sensitive payloads in messages.
14. Validate message length using configured `SP_AI_MAX_INPUT_CHARS` or the project-equivalent setting.
15. Do not create write actions for business records. The only writes allowed are assistant conversation/message records.

## Expected Behavior

### Create Conversation

When the user calls `POST /api/assistant/conversations`:

1. verify AI is enabled;
2. verify current user has `assistant.chat.access`;
3. read current authenticated user ID and tenant ID from backend auth context;
4. create a new active conversation;
5. store locale and safe initial route fields if provided;
6. return the conversation.

If AI is disabled:

- return the project’s standard disabled/not-available error, or a safe `403/404` depending on existing feature-flag conventions.

If user lacks permission:

- return the project’s standard authorization error.

### Load Conversation

When the user calls `GET /api/assistant/conversations/{conversation_id}`:

1. load conversation by ID;
2. verify it belongs to the current user;
3. verify tenant scope matches current auth context;
4. reject archived conversations if project policy treats archived as unavailable, or return read-only if that is the convention;
5. return conversation and messages ordered by creation time.

Safe error rule:

- For inaccessible conversation IDs, return not found or the project’s safe access-denied pattern without revealing whether the conversation exists.

### Send Message

When the user calls `POST /api/assistant/conversations/{conversation_id}/messages`:

1. verify AI is enabled;
2. verify current user has `assistant.chat.access`;
3. load and authorize the conversation;
4. validate message length and content;
5. store a `user` message;
6. update conversation last route path/name if safe route context is provided;
7. create a deterministic placeholder `assistant` message only if needed for endpoint completeness.

Because full AI response generation is not part of this task, the placeholder assistant message should be safe and clearly temporary.

Suggested English placeholder:

```text
Assistant response generation is not implemented in this step yet. Your message was saved successfully.
```

If the project or tests prefer returning only the stored user message for now, that is acceptable, but document the decision. The full structured assistant answer is AI-07 and later.

Important:

- Do not try to answer operational questions yet.
- Do not call OpenAI.
- Do not call mock provider yet unless AI-10 was already implemented earlier by project order. In the intended order, AI-10 is later.
- Do not execute tools.
- Do not retrieve knowledge.

## Tenant and User Scope Rules

### Internal tenant users

Conversation must be scoped to:

```text
user_id = current user
tenant_id = current tenant
```

### Platform admin / system-wide context

If the current auth context has no tenant because it is a system/platform context:

- allow `tenant_id` to be null only if this matches existing platform-admin conventions;
- otherwise use the active/selected tenant from backend context if the platform already has that concept;
- do not accept tenant ID from request body as authoritative.

### Portal/self-service users

Conversation must remain scoped to the portal/self-service user and its backend scope.

Do not allow portal users to load internal admin conversations.
