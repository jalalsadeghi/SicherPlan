## Goal

Create the first backend module skeleton for the SicherPlan AI Assistant and register it with the FastAPI application in the same style as the existing project modules.

This task must expose a minimal, authenticated assistant capability endpoint and establish the module structure for later tasks. It must **not** implement conversations, database models, migrations, OpenAI calls, knowledge ingestion, diagnostic tools, frontend code, or real AI reasoning.

The main deliverable is a safe backend module foundation that later tasks can extend.

5. Do not implement conversation persistence. That is AI-06.

## Capability Rules

The service must compute capabilities using backend settings and backend authorization context.

### When AI is disabled

If `SP_AI_ENABLED=false`:

- endpoint should return `enabled: false`;
- `can_chat` must be false;
- `can_run_diagnostics` must be false;
- `can_reindex_knowledge` must be false;
- do not call any provider;
- do not attempt database persistence.

### When AI is enabled

If `SP_AI_ENABLED=true`:

- `enabled` should be true only if the assistant is globally enabled and the endpoint is available for the current authenticated context;
- `provider_mode` should reflect configured provider mode, for example `mock` or `openai`;
- `can_chat` should be true only if the current user has `assistant.chat.access` or the equivalent permission according to the repository’s permission system;
- `can_run_diagnostics` should be true only if the current user has `assistant.diagnostics.read` or equivalent;
- `can_reindex_knowledge` should be true only if the current user has `assistant.knowledge.reindex` or equivalent;
- if permission checking helpers differ from the sprint names, use existing project helpers.

### Authentication behavior

Follow existing project behavior for authenticated API endpoints.

Preferred behavior:

- unauthenticated requests receive the same auth error pattern as other secured APIs;
- authenticated users without `assistant.chat.access` may still receive capabilities with `can_chat: false`, if that is consistent with existing capability endpoints;
- do not return raw role or permission internals unless other project endpoints already do so.
