# Codex Prompt — US-25-T2

## Task title

**Build browser/mobile/shared-terminal ingest flows for raw time events.**

## Objective

Implement the append-only raw event ingest pipeline so employees, subcontractor workers, and approved shared terminals can submit time-capture evidence through browser, mobile, and terminal flows without bypassing assignment, tenant, and context discipline.

## Source context

- Updated proposal: section 5.6 time capture includes browser-based start/stop capture, mobile app capture with geolocation, assigned shared terminals, stationary terminal integration, and scanning via QR/barcode/RFID/NFC.
- Implementation specification: `field.time_event` and related field tables in section 4.7; sprint-wide note that raw time evidence stays append-only except for controlled validation-status updates.
- Proposal workflows 6.1 and 6.2 place time capture before reconciliation and payroll basis creation.
- Dienstplanmacher material describes homepage/browser capture, app-based capture, Terminal C / X, stationary terminal support, and shared terminal assignment to object/event/fair/patrol contexts.
- Sprint reference: `docs/sprint/sprint-09-time-capture-and-actuals.md`.

## Dependencies

- `US-25-T1` must provide stable device and policy records before ingest flows can resolve active context restrictions.
- `US-22-T1` and `US-22-T2` should already provide the authenticated mobile shell and employee app interaction patterns.
- `US-21-T4` should already provide released shift visibility/read models needed for employee-facing capture context.
- `US-19-T1` and `US-19-T2` should already provide assignment truth so raw events can point at the correct shift/actor context without inventing local assignment state.
- If the repository distinguishes employee and subcontractor-worker authentication differently, reuse those seams instead of creating an isolated capture-login model.

## Scope of work

- Implement raw time-event create commands/endpoints for browser start/stop capture, mobile capture, and shared-terminal capture against `field.time_event` or the approved equivalent.
- Ensure ingest can resolve the actor type (`employee` or `subcontractor_worker`), shift context, source channel, optional device, and raw credential token while preserving append-only evidence.
- Support geolocation payload capture where mobile or browser flows provide it, but treat location as evidence data, not as direct permission proof by itself.
- Add idempotency or replay-safe handling for duplicate client submissions where the repository already has an approved pattern (for example correlation IDs or raw tokens).
- Where UI is in scope, add the narrow browser/mobile/terminal interaction needed to submit capture events through the real backend contracts using Vben/Prokit conventions rather than mock data.
- Add tests for employee/browser capture, mobile capture with location payload, terminal capture with device context, subcontractor-worker capture where applicable, and duplicate/replay handling.

## Preferred file targets

- `modules/field_execution` command handlers, routers, schemas, and repositories for raw time-event ingest.
- Mobile app feature files for employee capture actions if this repository already includes the employee app.
- Any Vben admin/browser capture surface only if the repo already supports browser-side employee capture under authenticated screens.
- Tests around raw-event creation, idempotency/replay, actor scoping, and source-channel behavior.

## Hard constraints

- Raw time events are evidence: never overwrite or hard-edit them to 'fix' attendance or payroll outcomes in this task.
- Do not let browser, mobile, or terminal clients write directly to attendance summaries, actuals, or approval records.
- All ingest must remain tenant-scoped and tied to authenticated or explicitly validated actor/device context.
- Do not implement final validation/warn-vs-block logic here beyond what is necessary to persist a raw attempt; detailed policy enforcement belongs in `US-25-T3`.
- Do not leak HR-private or finance-sensitive data into mobile/browser capture surfaces.

## Expected implementation outputs

- Raw time-event ingest contracts for browser, mobile, and shared-terminal channels.
- Append-only persistence with actor/shift/device/source-channel context.
- Optional UI wiring for the real repository capture surfaces.
- Tests covering the main ingest paths and replay-safe behavior.

## Non-goals

- Do not finalize attendance summaries or finance actuals here.
- Do not implement full discrepancy review or approval chains here.
- Do not replace offline synchronization strategy with ad hoc local persistence outside the repository conventions.

## Verification checklist

- [ ] Browser, mobile, and terminal flows can create raw time events through approved contracts.
- [ ] Every stored event preserves actor type, shift context, source channel, and device context where available.
- [ ] The implementation remains append-only and replay-safe enough for downstream derivation.
- [ ] No direct writes are made to attendance or finance truth from ingest handlers.
- [ ] Tests cover employee and subcontractor-worker capture paths where both exist.

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
