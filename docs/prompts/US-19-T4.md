# Codex Prompt — US-19-T4

## Task title

**Provide coverage views for min/target staffing and confirmed/unconfirmed fill rates.**

## Objective

Create the dispatcher-facing coverage views that make staffing completeness visible at shift and demand-group level, using traffic-light style summaries and localized Vben Admin UI without introducing a shadow staffing model.

## Source context

- Updated proposal: section 5.5 operational tools explicitly calls for real-time staffing overview and traffic-light coverage; Appendix E highlights minimum staffing and planning performance outputs.
- Implementation specification: demand groups, assignments, subcontractor releases, and reporting guidance in sections 4.6 and 7; minimum staffing enforcement in section 6.
- Sprint reference: `docs/sprint/sprint-07-staffing-engine.md`.
- Cross-cutting rules: `AGENTS.md`, especially Vben patterns, DE/EN localization, and query-discipline rules.

## Dependencies

- `US-19-T1` must already provide demand-group, team, and assignment persistence.
- `US-19-T2` should already expose staffing-board API/query contracts.
- `US-19-T3` should already expose subcontractor-release quantities so coverage can account for released partner capacity.
- `US-18-T4` should already optimize the planning query paths used by board/list views.
- `US-3-T1` and `US-3-T4` must already provide the Vben shell, theme tokens, and DE/EN localization foundations.

## Scope of work

- Implement the query/view layer needed to calculate and display min quantity, target quantity, assigned count, confirmed count, released partner capacity, and derived coverage state for shifts and demand groups.
- Build Vben Admin pages/components/widgets for dispatcher coverage monitoring, following the repository’s existing list/detail/board conventions rather than inventing a new mini-framework.
- Support filters and grouping that are operationally useful for dispatchers: planning record, shift window, planning mode, site/venue, function, qualification, workforce scope, and release/confirmation status where available.
- Represent coverage with explicit, localized traffic-light or status-badge semantics derived from normalized tables instead of cached write-side summaries.
- If the repository already exposes planning query DTOs, extend them carefully for coverage fields. If not, create dedicated read DTOs that stay query-only and easy to evolve.
- Add frontend tests and, where needed, backend query tests for calculation accuracy, filtering behavior, and permission-aware rendering.

## Preferred file targets

- Vben pages, reusable board/list widgets, route configuration, permission guards, and locale files under the web admin application.
- Thin backend query endpoints or DTO extensions under `modules/planning/` if the UI needs dedicated coverage reads.
- Shared UI status helpers only where they align with the repo’s existing web architecture.

## Hard constraints

- Do not create or maintain shadow write-side coverage tables. Coverage must be derived from demand groups, assignments, and releases.
- Keep the UI inside Vben Admin patterns for routing, permissions, forms, and page composition.
- Coverage screens must remain tenant-safe and role-scoped; do not surface HR-private data or unrestricted partner detail just to fill a grid.
- German is the default UI language and English must ship alongside every new user-facing string.
- Respect the existing theme-token baseline; do not hardcode one-off colors that diverge from the agreed light/dark primary-color scheme.
- This task is about visualization and read contracts, not about redefining validation or release workflows.

## Expected implementation outputs

- Coverage-focused query contracts and Vben dispatcher views for min/target staffing and fill-rate status.
- Localized traffic-light or equivalent status rendering that is traceable to normalized planning data.
- Tests covering calculation accuracy, filter behavior, and permission-safe rendering.

## Non-goals

- Do not implement assignment commands here; those belong to `US-19-T2`.
- Do not implement the validation engine or override workflow here; those belong to `US-20-*`.
- Do not build customer, subcontractor, or employee-facing portal/mobile coverage screens here.
- Do not turn this task into analytics/reporting beyond operational staffing coverage reads.

## Verification checklist

- [ ] Coverage calculations match demand-group min/target quantities and the current assignment/release state.
- [ ] Dispatcher users can filter and inspect coverage using localized Vben screens without seeing out-of-scope tenant data.
- [ ] Traffic-light/status semantics remain derived from normalized planning data rather than cached write tables.
- [ ] German default and English secondary strings exist for all new UI elements.
- [ ] Tests cover the most important coverage states, filters, and rendering/security behaviors.
