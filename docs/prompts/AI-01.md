# Task: Repository Inspection and Implementation Plan

## Implementation Prompt for Codex

You are working on the SicherPlan repository.

Before writing or changing any code, read the sprint file:

```text
docs/sprint/AI-Assistant.md
```

This task is **AI-01 — Repository Inspection and Implementation Plan**.

## Goal

Inspect the current repository and produce a precise implementation plan for the AI Assistant feature. This is a discovery/planning task only. The purpose is to understand the actual backend, frontend, IAM, routing, config, migration, and test conventions before any implementation tasks begin.

## Hard Rules

1. Do **not** change production code in this task.
2. Do **not** create the assistant module yet.
3. Do **not** add migrations yet.
4. Do **not** add OpenAI dependencies yet.
5. Do **not** implement frontend components yet.
6. Do **not** implement permissions yet.
7. Do **not** modify unrelated files.
8. Remove any temporary scratch files before finishing.
9. The final worktree must remain clean unless you explicitly create a short planning document and explain why. Prefer no file changes.
10. If the actual repository structure differs from the sprint assumptions, report the difference clearly and propose how to adapt.

## Repository Areas to Inspect

### Backend Entry Point and App Setup

Inspect:

```text
backend/app/main.py
backend/app/config.py
backend/app/*
```

Determine:

- how FastAPI routers are registered;
- how settings/configuration are loaded;
- whether settings are Pydantic-based, dataclass-based, or custom;
- how feature flags are currently handled, if any;
- where request IDs, logging, error handling, and middleware are configured;
- where dependency injection patterns live.

### IAM / Authorization

Inspect:

```text
backend/app/modules/iam/*
backend/app/modules/core/*
backend/app/modules/auth/*
```

If some paths do not exist, search the repository for:

```text

role

```

Determine:

- how authenticated user context is represented;
- how tenant_id, user_id, roles, permissions, and scopes are accessed;
- how route-level permission checks are performed;
- how permission seed data is defined;
- how role keys and permission keys are named;
- how cross-tenant access is prevented;
- how portal/self-service users differ from internal users.
