# Codex Prompt — US-24-T1

## Task title

**Implement patrol round start/stop flows and checkpoint capture events.**

## Objective

Create the patrol-control backbone so a field user can start a patrol round against the correct released route/shift context, capture checkpoint events in order, and stop or complete the round with durable, append-only evidence that later feeds watchbook, evaluation, and compliance flows.

## Source context

- Updated proposal: section 5.2 guard patrol control system with smartphone-based rounds, fixed start/end, checkpoint sequence, offline sync, and transfer into evaluations/watchbook; section 5.5 patrol/route planning; appendix D control-round model.
- Implementation specification: planning route/checkpoint tables in section 4.6 (`ops.patrol_route`, `ops.patrol_checkpoint`, patrol plan detail); field tables in section 4.7 (`field.patrol_round`, `field.patrol_round_event`); field-evidence chain in section 5.
- Sprint reference: `docs/sprint/sprint-08-employee-app-and-field-i.md`.
- Cross-cutting rules: `AGENTS.md`, especially append-only raw evidence, Prokit mobile baseline, and field module write discipline.
- Earlier prompt context: Sprint 6 route/checkpoint work and Sprint 7 release flows provide the planning-side context that patrol control consumes.

## Dependencies

- `US-16` route, checkpoint, and patrol-planning groundwork must already exist.
- `US-22-T1` should already provide the authenticated mobile shell and employee context.
- `US-21-T1` should already define release semantics so field flows operate only on released route/shift context.
- If route or shift read contracts are still incomplete, add only the narrow field-facing query seam needed; do not broaden field writes into planning.

## Scope of work

- Implement patrol-round lifecycle handling: load eligible route context, start a round, keep live status, and stop/complete it safely.
- Persist checkpoint or lifecycle events as `field.patrol_round_event` rows with sequence ordering and capture metadata tied back to `field.patrol_round`.
- Validate the acting user/employee against the released patrol context before allowing round start or checkpoint capture.
- Surface the route/checkpoint sequence in the mobile UI with clear progress state and enough data for later offline synchronization.
- Keep the patrol-round service boundary inside `modules/field_execution`; it may reference released route/shift data but must not mutate route master data.
- Add tests for round start authorization, sequence handling, round completion, and append-only checkpoint-event persistence.

## Preferred file targets

- `modules/field_execution` patrol models, services, routers, and validators.
- Flutter patrol feature screens/state under the actual mobile app path.
- Read-model or query helpers only where field flows need narrow route/shift context views.
- Tests for patrol-round lifecycle, authorization, and event ordering.

## Hard constraints

- Patrol events are append-only evidence; do not rewrite or delete event history to simulate corrections.
- The field module may read released route/shift context, but it must not mutate patrol-route master data or planning demand.
- Checkpoint event ordering must remain durable and replay-safe.
- Only authorized field actors in the correct tenant and released context may start or operate a patrol round.
- Keep the implementation ready for later offline-sync handling instead of hardcoding always-online assumptions.

## Expected implementation outputs

- Patrol-round lifecycle services and APIs.
- Checkpoint event persistence tied to the approved patrol-round schema.
- Mobile patrol UI for round start/stop and in-round progress.
- Tests covering authorization, lifecycle handling, and append-only event ordering.

## Non-goals

- Do not implement all scan-method variants, exception evidence, or offline sync details here; those belong to later tasks in this story.
- Do not generate evaluation reports or watchbook linkage here.
- Do not alter route master data or checkpoint definitions from the field module.
- Do not create a second patrol-history schema outside the approved field tables.

## Verification checklist

- [ ] Authorized field users can start and stop patrol rounds only in valid released context.
- [ ] Checkpoint captures are stored as ordered append-only patrol-round events.
- [ ] The patrol UI reflects route/checkpoint progress without mutating route master data.
- [ ] Round completion state is durable and auditable.
- [ ] Tests cover authorization, event ordering, and lifecycle transitions.

- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
