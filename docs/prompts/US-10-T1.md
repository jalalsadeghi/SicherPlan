# Codex Prompt — US-10-T1

## Task title

**Build web/iframe applicant form with configurable fields, uploads, and GDPR consent capture**

## Objective

Deliver the public applicant-intake entry point as an iframe-friendly, tenant-scoped web flow that captures configurable applicant data, file uploads, and explicit GDPR consent in a way that cleanly feeds the Recruiting aggregate without bypassing the approved document and audit backbone.

## Source context

- Updated proposal: section 5.3 recruitment and applicant workflow; section 6.2 step 1 of the applicant-to-payroll-ready-employee chain; technical baseline in sections 7 and 8 for secure public input, object storage, and phased delivery.
- Implementation specification: section 4.4 `hr.applicant`; docs-link usage through `docs.document_link`; aggregate/package guidance for `modules/recruiting`; Phase D sequencing for applicant and employee tables.
- Uploaded Dienstplanmacher source material: Bewerberverwaltung sheet showing iframe deployment on the company website, configurable or custom mandatory fields, file attachments, GDPR consent before submit, and direct hand-off into applicant management.
- Sprint reference: `docs/sprint/sprint-04-recruiting-and-employee-core.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-4-T2` should already provide the tenant/setting administration surface or equivalent config seam needed to host applicant-form settings without inventing a second settings system.
- `US-6-T1` must already provide document upload/storage and secure retrieval patterns for applicant attachments.
- `US-3-T4` should already provide DE-default / EN-secondary localization structure for public-form labels, validation messages, and submit confirmations.
- Use the current public-web or frontend shell pattern from Sprint 1 instead of building an unrelated mini app for applicant intake.

## Scope of work

- Implement a tenant-aware public applicant form endpoint or page that can be embedded by iframe and can also run as a standalone route where the repo architecture prefers it.
- Support configurable visible/required fields using the smallest approved configuration mechanism available in the repo, preferably tenant settings and lookup-backed metadata rather than a new general-purpose form builder.
- Persist applicant submissions into `hr.applicant` with a deterministic initial status, business-facing application number, source channel, and any approved desired-role or channel metadata.
- Accept file uploads such as CVs, certificates, and supporting proofs through the centralized docs service, then link them back to the applicant aggregate through document links instead of storing binaries inline.
- Capture explicit GDPR consent with consent timestamp and the privacy-policy reference/version that was presented at submission time.
- Provide server-side validation, localized error messages, duplicate-safe submission handling, and a confirmation response suitable for iframe embedding.
- Add minimal anti-abuse handling consistent with the existing repo baseline for public endpoints, such as rate-limiting hooks, input-size validation, and attachment-type restrictions.
- Write tests for consent-required submission, field-config enforcement, upload/link behavior, localization defaults, and tenant-safe iframe/public access.

## Preferred file targets

- Recruiting module files such as `modules/recruiting/models/*`, `schemas/*`, `services/*`, and public submission routers/controllers adapted to the actual repo layout.
- Public-web or web-app applicant form pages/components under the repo's public frontend package.
- Config helpers reusing `modules/core` or platform setting packages where form-field configuration is stored.
- Tests for public submission, docs integration, validation, and localized responses.

## Hard constraints

- Do not accept a submission unless GDPR consent is explicitly confirmed.
- Do not build a separate attachment system; all applicant files must travel through the shared docs/document-link backbone.
- Do not create employee records, IAM users, or recruiter decisions in this task.
- Keep iframe support tenant-safe and origin-safe; embedded delivery must not bypass tenant scoping or secure-document rules.
- Prefer a narrow configuration model over an open-ended dynamic form-builder platform.
- German must be the default UI/content language and English must exist alongside it for all new user-facing text.

## Expected implementation outputs

- Public applicant-form route/page(s) suitable for direct use and iframe embedding.
- Backend submission APIs/services, validation rules, and applicant persistence for the intake step.
- Attachment upload/link integration through the docs backbone.
- Consent evidence capture and tests proving submit-time enforcement.

## Non-goals

- Do not implement recruiter status transitions here; that is `US-10-T2` and `US-10-T3`.
- Do not implement applicant-to-employee conversion here; that is `US-10-T4`.
- Do not build a marketing/CMS site or a generic website builder.
- Do not open unrestricted public document retrieval paths for uploaded applicant files.

## Verification checklist

- Applicants cannot submit without explicit GDPR consent.
- Configured required fields are enforced consistently in backend validation.
- Uploaded files are stored via the docs service and linked back to the applicant record.
- German is the default rendered locale and English is available as the secondary locale.
- Iframe embedding works without weakening tenant scoping, validation, or secure file handling.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
