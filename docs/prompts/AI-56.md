# AI-56 — Graceful Degradation: Never Show Generic Assistant Failure When Grounded Facts Exist

## Purpose

The UI currently shows generic errors such as:

```text
Die Assistant-Anfrage konnte gerade nicht abgeschlossen werden.
```

This happens even when the backend has already retrieved grounding facts, run tools, or completed the first OpenAI call.

The user experience must be improved: if the assistant cannot complete a provider call due to rate limits, invalid continuation, timeout, or temporary provider errors, it should return a helpful, source-grounded fallback response instead of a generic failure box whenever safe facts are available.

This is not a replacement for OpenAI. It is a resilience layer for provider failures.

---

## Non-Negotiable Rules

1. Primary path remains RAG + OpenAI.
2. Fallback path is used only after provider failure.
3. Fallback must use retrieved grounding/tool facts only.
4. Fallback must not invent records, routes, labels, or facts.
5. Fallback must be clearly marked internally as degraded, but the user-facing text should still be helpful.
6. No secrets or sensitive data may be exposed.
7. Do not return 502/503 to the frontend when a safe fallback answer can be produced.

### Step 2 — Define Fallback Eligibility

Fallback is allowed when:

```text
- question is in-scope;
- user is authenticated and authorized;
- grounding context exists;
- provider failed due to rate limit, timeout, invalid continuation, or temporary unavailable;
- at least one content-bearing source or diagnostic fact exists.
```

Fallback is not allowed when:
- user lacks permission;
- assistant disabled;
- out-of-scope/unsafe request;
- no grounding facts exist;
- data access would be misleading or unsafe.

### Step 3 — Build Fallback Answer From Facts

Create a fallback builder that uses:

```text
grounding_context.sources
source_basis
tool_results
allowed_links
missing_permissions
diagnostic facts
```

It must produce:

```json
{
  "answer": "string",
  "confidence": "low|medium",
  "provider_degraded": true,
  "diagnosis": [],
  "links": [],
  "missing_permissions": [],
  "next_steps": [],
  "source_basis": []
}
```

For German diagnostic app-visibility question, a fallback answer can safely say:

```text
Die Anfrage konnte nicht vollständig mit dem KI-Modell abgeschlossen werden. Auf Basis der bereits geprüften SicherPlan-Kontexte sollten Sie die folgenden Punkte kontrollieren: Mitarbeiterakte und App-Zugriff, aktive Zuweisung, Freigabezustand der Schicht, Sichtbarkeit für Employee Self-Service/App und ggf. Validierungen wie Abwesenheit, Qualifikation oder doppelte Besetzung.
```

But the answer must be generated from actual available grounding/tool facts. Do not add facts that were not in the context.

### Step 4 — Return 201 With Degraded Structured Response

If fallback is eligible:

- store user message;
- store assistant fallback message;
- return `201 Created`;
- include `provider_degraded=true` or `diagnosis` item explaining provider limitation;
- include source basis;
- include allowed links;
- do not show generic error box.

If fallback is not eligible, existing safe error behavior may remain.

### Step 5 — Frontend Degraded State

Update frontend to show a subtle warning inside the assistant message, not a blocking red error box, when:

```json
"provider_degraded": true
```

Suggested German text:

```text
Diese Antwort wurde aus bereits geprüften SicherPlan-Kontexten erstellt, weil der KI-Anbieter die Anfrage gerade nicht vollständig abschließen konnte.
```

Use existing i18n patterns.
