# Codex Prompt — US-27-T2

## Task title

**Implement finance reconciliation for sickness, cancellations, replacements, and customer adjustments.**

## Objective

Add the finance reconciliation layer that transforms approved operational actuals into billing- and payroll-ready finance actuals when real-world exceptions occur, such as sickness, no-shows, replacements, cancellations, and customer-side minute adjustments.

## Source context

- Updated proposal: section 5.6 actual-vs-plan reconciliation explicitly includes cancellations, sickness, replacement staff, breaks, expenses, flat rates, and separate customer-side adjustments.
- Workflow 6.1 step 7 and workflow 6.2 step 7 require reconciliation to review evidence and create the approved actual and payroll basis.
- Implementation specification: `finance.actual_record` includes `customer_adjustment_minutes` and is the parent of later finance lines; finance service owns reconciliation outputs while preserving field evidence.
- Dienstplanmacher reconciliation materials describe differentiated employee-vs-customer treatment, remarks for delayed relief, and adjustment of payroll/customer-side outcomes from the same operational event basis.
- Sprint reference: `docs/sprint/sprint-09-time-capture-and-actuals.md`.

## Dependencies

- `US-26-T2` must already provide the finance actual bridge.
- `US-27-T1` should already provide preliminary and operational approval stages where required.
- `US-11-*` and `US-12-*` should already provide absence/sickness data or the seams where sickness/cancellation context can be read.
- `US-19-T2` should already provide assignment and replacement-person truth.

## Scope of work

- Implement finance reconciliation commands/services that can record and apply sickness, cancellation, replacement, and customer-side minute adjustments to actual records without mutating raw field evidence.
- Support separate payroll-side and customer-side interpretation where the repository already models them distinctly, especially when customer adjustments differ from employee payable logic.
- Require clear provenance for each reconciliation decision, including actor, timestamp, source evidence, and reason/note.
- Ensure replacement logic stays traceable to assignment/attendance reality rather than erasing the originally planned person from history.
- Add tests for sickness replacement, late cancellation, customer-minute adjustment, and replacement-worker reconciliation scenarios.

## Preferred file targets

- `modules/finance` reconciliation services, schemas, and tests.
- Read-model helpers only where the repository needs explicit reconciliation review projections.
- No significant mobile work in this task.

## Hard constraints

- Do not rewrite raw field evidence, attendance summaries, or assignment history to express finance reconciliation outcomes.
- Customer-side adjustments must stay explicit and traceable; do not hide them inside generic minute overrides with no reason code.
- Replacement handling must preserve both planned and actual actor history where relevant.
- Do not implement payroll tariff/rate evaluation here beyond what is needed to persist reconciliation-ready actual fields.
- Keep authorization strict; finance reconciliation actions must not leak into employee or general dispatcher scopes by default.

## Expected implementation outputs

- Finance reconciliation commands/services for sickness, cancellations, replacements, and customer adjustments.
- Audit-safe persistence of reconciliation reasons and adjustment lineage.
- Tests proving exception handling and provenance retention.

## Non-goals

- Do not implement tariff table math, surcharge calculation, or export batching here.
- Do not build the full finance dashboard/reporting set in this task.
- Do not hide operational discrepancies by prematurely finalizing records without explanation.

## Verification checklist

- [ ] Finance reconciliation can model sickness, cancellations, replacements, and customer adjustments explicitly.
- [ ] Reconciliation outcomes remain traceable to actual records and source evidence.
- [ ] Planned-vs-actual actor history is preserved where replacements occur.
- [ ] Authorization boundaries prevent non-finance roles from using reconciliation actions unless explicitly allowed.
- [ ] Tests cover the main exception scenarios and lineage expectations.

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
