# AI-32 — Implement RAG-Based Grounded Answer Generation for SicherPlan Assistant

## Purpose

Convert the SicherPlan AI Assistant into a real **RAG — Retrieval-Augmented Generation** assistant.

The assistant must not behave like a FAQ bot or a collection of canned answers. For every in-scope SicherPlan question, the system must retrieve the relevant private SicherPlan context at runtime and pass that context to the OpenAI-backed generation step. The final answer must be generated by the model, grounded in retrieved facts, user permissions, route context, and safe backend tools.

This task builds on the previous AI Assistant tasks, including the work already done for backend module setup, knowledge ingestion, tool registry, page help catalog, OpenAI provider, UI widget, and the previous refactor attempt to remove canned-answer paths.

---

## Critical Product Requirement

The assistant must feel like a knowledgeable human product specialist, not a hardcoded FAQ system.

That means:

- The backend may retrieve facts, permissions, UI labels, page metadata, workflow facts, knowledge chunks, and diagnostic results.
- The backend may enforce security, tenant scope, permissions, redaction, tool policy, and structured output validation.
- The backend must not generate final in-scope answers using deterministic templates.
- OpenAI must generate the final natural-language answer using the retrieved grounding context.
- The answer must be in the same language as the user’s question.
- If evidence is incomplete, the assistant must say that the verified context is incomplete instead of inventing missing UI labels or workflow steps.

## Current Problem to Fix

After previous implementation steps, the assistant is visible and can answer, but current behavior shows that the assistant is still not a proper RAG assistant.

Examples observed:

1. User asks in Persian:

   ```text
   چطور میتونم یک کارمند جدید بسازم؟
   ```

   The answer is good because it uses a verified Employee UI action.

2. User then asks:

   ```text
   برای اینکه یک کارمند جدید بتونم در یک شیفت Assign بدم چکار باید بکنم؟
   ```

   The assistant incorrectly repeats the employee-create answer, because the intent detection is too greedy and the final answer is still generated from a deterministic UI-how-to path.

3. User asks:

   ```text
   الان میخوام یک order جدید درست کنم، دقیقا باید چکار کنم؟
   ```

   The assistant incorrectly treats this as out-of-scope, even though orders are a core SicherPlan workflow under `P-02 — Orders & Planning Records`.

4. User asks in English:

   ```text
   I want to create a new order. how can I do?
   ```

   The assistant returns a Mock Provider placeholder instead of a grounded answer.

These issues indicate that current code likely still contains one or more of the following anti-patterns:

- deterministic final answer rendering in backend;
- hardcoded UI-how-to answer paths;
- overly simplistic intent detection;
- incomplete RAG grounding context;
- provider not used for all in-scope answers;
- mock provider appearing as user-facing production answer;
- OpenAI tool-call orchestration not fully implemented or not used for these cases.

## Target Architecture

Implement the assistant message flow like this:

```text
User message
  ↓
Load AuthContext / tenant / user / roles / permissions / current route
  ↓
Language detection
  ↓
Safety and out-of-scope check
  ↓
Retrieval planning
  ↓
Retrieve private SicherPlan context:
    - documentation chunks
    - API/page/workflow knowledge
    - Page Route Catalog facts
    - UI Action Catalog / Page Help Manifest facts
    - current route facts
    - current user capability facts
    - allowed navigation facts
    - operational diagnostic facts when relevant
  ↓
Build Grounding Context Package
  ↓
Send grounding context + user question + available tools to OpenAI
  ↓
If OpenAI requests tools:
    - validate tool name
    - validate input
    - enforce read-only policy
    - enforce tenant/user/permission/scope
    - execute tool
    - audit tool call
    - redact output
    - send tool output back to OpenAI
  ↓
OpenAI generates final structured answer
  ↓
Backend validates structured output and link safety
  ↓
Persist messages and return response to frontend
```

---

## Core Rule

For in-scope SicherPlan questions:

```text
Backend retrieves facts.
OpenAI generates the final answer.
```

The backend may produce final text only for:

- assistant disabled;
- unauthorized user;
- rate limit;
- provider unavailable;
- invalid request;
- clearly unsafe prompt injection;
- clearly out-of-scope questions if the existing safety policy requires local refusal.

For normal SicherPlan questions, including how-to, workflow, navigation, and operational troubleshooting questions, final answer text must come from the provider/generation path, not from backend templates.

## Implementation Requirements

## 1. Introduce a Grounding Context Package

Create a typed grounding context object that can be passed to the provider.

Suggested files:

```text
backend/app/modules/assistant/grounding.py
backend/app/modules/assistant/retrieval_planner.py
```

Suggested schema:

```python
class AssistantGroundingSource(BaseModel):
    source_type: str  # knowledge_chunk | page_route | ui_action | workflow | capability | diagnostic | current_route
    source_name: str | None = None
    page_id: str | None = None
    module_key: str | None = None
    title: str | None = None
    content: str | None = None
    facts: dict[str, Any] = Field(default_factory=dict)
    relevance_score: float | None = None
    verified: bool = False
    permission_checked: bool = False

class AssistantGroundingContext(BaseModel):
    detected_language: str
    response_language: str
    route_context: dict[str, Any] | None = None
    auth_summary: dict[str, Any]
    retrieval_plan: dict[str, Any]
    sources: list[AssistantGroundingSource]
    missing_context: list[str] = Field(default_factory=list)
    missing_permissions: list[dict[str, str]] = Field(default_factory=list)
```

The exact classes may be adapted to project conventions, but the final design must keep a clear separation between:

- retrieved facts;
- final answer;
- security policy;
- model generation.

## 2. Retrieval Planner

Implement a retrieval planner that does not answer the user.

It should only decide what context sources are useful.

Suggested intent categories:

```text
platform_how_to
workflow_how_to
ui_action_question
navigation_question
operational_diagnostic
configuration_question
permission_question
out_of_scope
unsafe
unknown_platform_related
```

The planner must support multilingual and mixed-language user messages, including common typos:

```text
employee:
  employee, Employee, Mitarbeiter, کارمند, پرسنل, نگهبان

shift:
  shift, Shift, Schicht, Dienst, شیفت, شفت, صیفت

assign:
  assign, assigned, assignment, Assgin, تخصیص, اختصاص, نسبت, نصبت, اضافه کردن, zuweisen

order:
  order, Order, Auftrag, سفارش, پروژه, عملیات

create:
  create, new, add, make, erstellen, anlegen, ثبت, ایجاد, ساخت, درست کردن
```

Important routing example:

If a message contains employee + shift + assign semantics, it must not be classified as simple `employee.create`. It should be treated as a workflow question such as:

```text
employee_to_shift_assignment_workflow
```

## 3. Multi-Source Retrieval

For each in-scope question, retrieve context from the appropriate sources.

### Documentation / Knowledge Chunks

Use existing `AssistantKnowledgeRetriever` and improve if needed.

Retrieve relevant chunks by:

- user message;
- detected language;
- current `page_id`;
- role keys;
- permission keys;
- likely module keys;
- likely page IDs.

### Page Route Catalog

Retrieve page facts for likely related pages.

Examples:

```text
E-01 — Employees Workspace
P-02 — Orders & Planning Records
P-03 — Shift Planning
P-04 — Staffing Board & Coverage
P-05 — Dispatch, Outputs & Subcontractor Releases
FI-01 — Actuals / Actual-Freigaben Workspace
ES-01 — Employee Self-Service Portal
```

### UI Action Catalog / Page Help Manifest

Retrieve verified UI action facts, but never render them directly.

Correct tool output:

```json
{
  "source_type": "ui_action",
  "page_id": "E-01",
  "page_title": "Employees Workspace",
  "action_key": "employee.create.open",
  "label": "Create employee file",
  "location": "top-right toolbar",
  "required_permissions": ["..."],
  "opens": "Identity and personnel number",
  "verified": true
}
```

Incorrect behavior:

```json
{
  "answer": "برای ایجاد Employee جدید، وارد..."
}
```

### Workflow Facts

Add or retrieve workflow-level facts for common cross-page processes, especially:

```text
employee_to_shift_assignment
customer_order_to_planning_record
order_to_shift_planning
shift_to_assignment
shift_release_to_employee_app
actuals_to_billing
```

This can initially be implemented as structured seed data or derived from docs/page catalog. It must be used as grounding context, not canned answer text.

### Operational Diagnostic Facts

For operational troubleshooting questions, retrieve diagnostic facts through read-only tools.

Example:

```text
“Markus has a shift but cannot see it in the app”
```

The diagnostic tools may retrieve facts such as:

- employee exists / not visible;
- assignment exists;
- shift release state;
- shift visibility;
- employee mobile access;
- validation blockers.

The final diagnostic answer must still be generated by OpenAI using those facts. Do not return a deterministic diagnostic paragraph from backend.

Suggested change:

```python
@dataclass(frozen=True)
class AssistantProviderRequest:
    ...
    grounding_context: dict[str, Any] | None = None
```

The provider prompt must clearly instruct the model:


## 6. Implement Tool-Calling Loop If Missing

Codex must verify whether `OpenAIResponsesProvider` currently handles tool calls only by extracting them, or whether it actually executes and returns tool outputs to the model.

If missing, implement the tool-calling loop in the service/provider boundary according to current project conventions.

Required flow:

```text
1. Send request to OpenAI with tools and grounding context.
2. If the model returns function_call items:
   - validate tool name against AssistantToolRegistry;
   - parse arguments safely;
   - execute tool server-side;
   - audit tool call;
   - redact output;
   - append tool output back into model conversation;
   - call OpenAI again.
3. Repeat until final structured response or max_tool_calls.
4. Enforce max_tool_calls.
5. If tool execution fails safely, pass a redacted failure fact back to the model or return a safe provider error.
```

Do not allow the model to execute arbitrary SQL or write operations.


But in production/openai mode, if provider fails, return a clear provider-unavailable error instead of falling back to mock.

## 10. Security Requirements

- no direct SQL generated by model;
- no write actions;
- no cross-tenant access;
- no unauthorized links;
- no HR-private leakage;
- no payroll-sensitive leakage without permission;

- no secrets in logs, prompts, provider metadata, or frontend;
- all executed tools are audited;
