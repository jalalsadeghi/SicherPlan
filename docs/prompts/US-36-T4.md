# Codex Prompt — US-36-T4

## Task title

**Hold sprint retrospective and release lessons-learned workshop.**

## Objective

Close Sprint 12 and the initial release by capturing what worked, what failed, what risks remain, and which follow-up actions belong in the stabilization or next-release backlog, with durable traceability to stories, modules, runbooks, and evidence.

## Source context

- Sprint reference: `docs/sprint/sprint-12-migration-uat-and-go-live.md`.
- Updated proposal: Phase 6 ends with stabilization, and the platform is meant to remain extensible for later modules without reworking the core architecture.
- Cross-cutting rules: `AGENTS.md`, especially stable story/task IDs, repository-backed traceability, honest evidence handling, and carry-forward discipline.
- This task turns UAT, cutover, hypercare, training, localization, and monitoring outcomes into durable release-learning assets.

## Dependencies

- `US-36-T1` through `US-36-T3` must already provide release, hypercare, and support-handover evidence.
- `US-35-*` and `US-34-*` should already provide the migration/UAT/training/review context that the retrospective needs to assess honestly.
- Existing backlog or issue-tracking conventions in the repository should be reused so follow-up items remain connected to prior work.
- Any runbooks, evidence indexes, or release notes created during Sprint 12 should be referenced rather than paraphrased vaguely.

## Scope of work

- Create a retrospective template and a finalized Sprint 12 / release lessons-learned report covering migration, seeded configuration, UAT, training, localization/accessibility, cutover, hypercare, monitoring, and support handover.
- Record action items with owner, priority, due date or follow-up horizon, affected story/task/module, and whether the work belongs to immediate stabilization or a later release.
- Capture architecture, delivery, and operational lessons around multi-tenant scoping, centralized documents, field/offline behavior, `finance.actual_record`, reporting, release governance, and support readiness where those topics surfaced during the release.
- Publish an indexed summary that future delivery cycles can reference without re-reading every issue or meeting note.
- Link key findings back to evidence, incidents, UAT defects, hardening gaps, or support handover notes so the retrospective remains grounded.
- Keep the workshop output action-oriented rather than purely narrative.

## Preferred file targets

- `docs/retrospectives/` for the release retrospective and action-item register.
- Backlog index or follow-up tracking docs already used by the repository.
- Optional cross-links from release notes, support handover docs, or sprint indexes if the repo already maintains them.

## Hard constraints

- Findings must be honest, evidence-backed, and linked to durable artifacts.
- Every follow-up action must have an owner and backlog destination.
- Do not turn the retrospective into a blame document or a vague celebration deck with no actions.
- Keep customer, employee, and tenant-sensitive data out of the published retrospective artifacts.
- Do not silently rewrite prior decisions without recording the lesson or follow-up.

## Expected implementation outputs

- A Sprint 12 / release retrospective report.
- A follow-up action register with ownership and backlog destination.
- An indexed lessons-learned summary linked to release evidence and support artifacts.
- Cross-links that make future release planning easier.

## Non-goals

- Do not execute all follow-up actions in this task.
- Do not reopen already accepted work without evidence.
- Do not hide failed assumptions or incidents to make the release look cleaner.
- Do not replace the stabilization backlog with prose-only retrospective notes.

## Verification checklist

- [ ] The retrospective covers migration, UAT, training, localization/accessibility, cutover, hypercare, monitoring, and support handover.
- [ ] Follow-up items have owners, priorities, and backlog destinations.
- [ ] Key lessons are linked to evidence or tracked incidents/issues.
- [ ] Artifacts are privacy-safe and reusable for future releases.
- [ ] The release closes with actionable learning rather than unstructured commentary.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
