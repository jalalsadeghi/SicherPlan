# Codex Prompt — US-28-T4

## Task title

**Support optional payslip import/archiving and payroll reconciliation reports.**

## Objective

Close the payroll cycle with document-safe payslip archiving and reproducible payroll reconciliation reporting so HR/payroll teams can compare approved actuals, allowances, advances, exports, and archived provider results from one auditable source.

## Source context

- Updated proposal: section 5.6 includes optional payslip import and section 5.7 requires reporting across earnings thresholds, under/over time, sickness, vacation, and time records; section 6.2 ends with payroll export/import and optional payslip archiving.
- Implementation specification: section 4.8 links payroll export batches to `docs.document`; section 7 recommends `rpt.payroll_basis_v` over actuals, pay profiles, tariffs, allowances, and advances; section 10 says generated or imported files should be represented as document rows linked to their owning business entity.
- The proposal’s HR data protection note requires payroll-sensitive artifacts to remain visible only to authorized HR/payroll roles.
- Sprint reference: `docs/sprint/sprint-10-payroll-billing-and-settlement.md`.
- Cross-cutting rules: `AGENTS.md`, especially document-centered compliance, reproducible reporting, and least-privilege privacy rules.

## Dependencies

- `US-28-T3` should already provide payroll export batches/items and document linkage for generated outputs.
- `US-6-T1` should already provide document storage/versioning/linking behavior.
- `US-27-T3` and `US-28-T2` should already provide the allowance/advance/pay-profile context needed for reconciliation views.
- If the repository already exposes SQL-view-based reporting helpers, reuse that approach instead of introducing ad hoc write-side report tables.

## Scope of work

- Implement the optional archive/import flow for payslip files or provider-return artifacts, linking them to the relevant export batch, employee, and period through `docs.document` or an approved repository-equivalent document association.
- Support secure manual upload and/or adapter-fed archive ingestion with metadata such as period, employee, provider, archive status, and source document reference.
- Build payroll reconciliation queries or SQL views that compare approved actuals, surcharge/allowance lines, advances/time-account hooks, export items, and archived payslip/document availability for a selected period or employee scope.
- Expose narrow APIs and, where repository conventions allow it, Vben finance-report screens for payroll operators to inspect missing exports, missing payslips, export mismatches, or unresolved payroll-basis gaps.
- Keep report generation reproducible from transactional data and document links rather than introducing a second finance write-side reporting schema.
- Add tests for document linkage, authorization, reconciliation accuracy, empty-state/report completeness, and any optional UI/report endpoints introduced.

## Preferred file targets

- `modules/finance/` payroll archive/report services, schemas, routers, and tests.
- SQL view definitions or reporting query files if the repository keeps report logic under `modules/reporting/` or migrations.
- Document-link integration under the existing docs service or repository-equivalent file-link layer.
- Optional Vben payroll reconciliation views/components if the current branch already has a finance-admin UI area.

## Hard constraints

- Do not expose payslips or payroll reconciliation details to non-HR/non-payroll roles.
- Treat payslips and provider-return artifacts as document records with durable linkage and audit metadata; do not scatter files in opaque storage without business references.
- Reconciliation reports must be reproducible from transactional finance/HR data and document links, not hand-maintained summary tables.
- Do not rely on destructive overwrites when re-archiving or replacing a payslip; preserve archive history according to repo conventions.
- Do not broaden this task into employee self-service payroll access unless that requirement already exists and is explicitly scoped on the current branch.
- Any user-facing maintenance or report surface must follow DE default / EN secondary localization rules.

## Expected implementation outputs

- Payslip archive/import contracts and document-link behavior.
- Payroll reconciliation queries or report endpoints grounded in transactional data.
- Optional localized Vben payroll reconciliation UI surfaces where the repo supports them.
- Tests proving permissions, document linkage, and report reproducibility.

## Non-goals

- Do not implement OCR or deep parsing of payslip contents unless the current repository already has an approved structured-import requirement.
- Do not build a full BI/dashboard suite here; keep the scope to payroll reconciliation and archive visibility.
- Do not reopen payroll batch generation or provider adapters unless a narrow fix is required for archive/report linkage.
- Do not expose tax/social details through generic employee or management views.

## Verification checklist

- [ ] Archived payslip/provider documents are linked to the correct employee, period, and export-batch context.
- [ ] Payroll reconciliation outputs can explain missing exports, missing archive artifacts, or quantity/amount mismatches reproducibly.
- [ ] Permissions prevent non-payroll roles from accessing payroll archive and reconciliation surfaces.
- [ ] Any added report/UI text is localized with German default and English secondary.
- [ ] Tests cover document linkage, report accuracy, authorization, and empty-state behavior.

- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
