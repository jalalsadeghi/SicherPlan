# Codex Prompt — US-31-T2

## Task title

**Implement planning performance, payroll basis, and customer/order profitability views.**

## Objective

Complete the core reporting read-model layer for planning performance, payroll basis, and profitability so controllers and finance users can analyze coverage, hours, surcharges, allowances, advances, and customer/order margin using reproducible read models instead of spreadsheet-only logic.

## Source context

- Updated proposal: section 5.7 and Appendix E require planning-performance, payroll-and-finance, and profitability-oriented outputs across planning modes, payroll basis, surcharges, advances, revenue, and contribution margin.
- Implementation specification: section 7 recommends `rpt.planning_performance_v` and `rpt.payroll_basis_v`, and Phase H says reporting should be created last so it reflects the final write model; the same section allows view or materialized-view decisions based on workload.
- Sprint reference: `docs/sprint/sprint-11-reporting-and-hardening.md`.
- Cross-cutting rules: `AGENTS.md`, especially the rule that finance truth must flow through `finance.actual_record`, and reporting must remain reproducible from the transactional backbone.

## Dependencies

- `US-31-T1` should already provide the first reporting-package conventions, filter shapes, and read-only reporting service seams.
- `US-28-T1` through `US-28-T4` must already provide payroll tariff, pay-profile, allowance, advance, export, and optional payslip-archiving truth.
- `US-29-T1` through `US-29-T3` must already provide invoice and invoice-line truth used for revenue and profitability calculations.
- `US-19-T1` through `US-21-T4` and `US-26-T2` through `US-27-T4` should already provide planning, assignment, validation, and actual-record context required for hours, coverage, and profitability metrics.

## Scope of work

- Create SQL view/materialized-view definitions for `rpt.planning_performance_v` and `rpt.payroll_basis_v`, plus a repository-equivalent profitability read model if profitability is not already represented as fields inside `rpt.customer_revenue_v` or another stable reporting surface.
- Model planning performance with hours, coverage gaps, fill rates, released vs fulfilled demand, planning-mode and parent-plan comparisons, and profitability-support fields that can be justified from planning and finance truth.
- Model payroll basis with gross wage basis, surcharges, allowances, advances, export completeness/state, and other read-only payroll readiness fields expected by the proposal and report catalogue.
- Where customer/order profitability needs a distinct read model, derive it from approved actuals, invoice lines, and commercial context rather than adding manual profit tables; if the repository already prefers extending `rpt.customer_revenue_v`, keep that extension narrow and documented.
- Expose FastAPI query endpoints and filter contracts for the new views, including date spans, planning mode, customer, order, employee, export state, and profitability-centric grouping options where supported by the repository.
- Add tests for reproducibility, profitability consistency, export-state correctness, and prevention of drift between payroll-basis outputs and upstream payroll/finance truth.

## Preferred file targets

- `modules/reporting/` view definitions, query services, schemas, routers, and tests.
- Alembic revisions or repository-equivalent database migration files for new reporting views/materialized views and any required supporting indexes.
- Shared reporting constants or grouping/filter helpers only if they remain reusable and do not spread business logic into the UI layer.
- No Vben dashboard implementation in this task.

## Hard constraints

- Do not recalculate payroll or finance truth outside the reporting layer. All payroll-basis reporting must derive from normalized finance/payroll tables and `finance.actual_record`.
- Do not introduce manual profitability override fields or spreadsheet-imported totals into the reporting write path.
- Keep planning-performance metrics grounded in released planning and approved actuals; avoid mixing draft planning states into management truth unless the story explicitly requires draft visibility and the contract names it.
- If materialized views are chosen for workload reasons, keep refresh strategy explicit and safe; do not hide stale-data behavior.
- Do not broaden this task into dashboard screens, exports, scheduled delivery, or hardening work.

## Expected implementation outputs

- Planning-performance and payroll-basis views plus any profitability read model or field extension needed by the repository.
- Read-only reporting services/endpoints and explicit filter/result schemas for those views.
- Tests proving reproducibility and consistency against upstream finance, payroll, planning, and invoice truth.

## Non-goals

- Do not build UI dashboards or downloadable exports here; that belongs to `US-31-T3`.
- Do not implement compliance/QM/security reporting here; that belongs to `US-32-*`.
- Do not use this task to redesign payroll formulas, invoice calculation rules, or planning validation behavior.
- Do not force a distinct profitability table if a stable view-based extension is enough.

## Verification checklist

- [ ] Planning-performance and payroll-basis outputs are reproducible from transactional tables and finance truth.
- [ ] Profitability values are deterministic and traceable to approved actuals plus invoice/commercial data.
- [ ] No direct write-side duplication or manual reconciliation tables were introduced.
- [ ] Endpoints/filter schemas remain tenant-safe, role-safe, and suitable for later dashboards/exports.
- [ ] Tests cover representative performance, payroll, and profitability scenarios including empty, mixed-mode, and edge-state periods.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
