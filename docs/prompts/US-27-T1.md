# Codex Prompt — US-27-T1

## Task title

**Implement preliminary actual submission by employee/field lead and operational confirmation.**

## Objective

Deliver the first two stages of the actual-approval chain so employees or field leads can submit preliminary actual information, object/event leads can confirm it operationally, and the finance stage later receives a traceable, pre-validated actual basis.

## Source context

- Updated proposal: section 5.6 three-stage approval flow says preliminary actuals are entered by employee or field lead, then operationally confirmed by object/event lead, before final finance/payroll reconciliation by administration.
- Role model: field supervisor / object or event lead confirms attendance and preliminary actuals for assigned plans/objects/events; employee/guard can capture time and access released actuals/pay data where allowed.
- Implementation specification: `finance.actual_approval` models staged approval rows; field and finance service boundaries remain separate.
- Dienstplanmacher reconciliation materials describe employee-entered preliminary times plus second-step confirmation by object/event lead.
- Sprint reference: `docs/sprint/sprint-09-time-capture-and-actuals.md`.

## Dependencies

- `US-26-T1` and `US-26-T2` must already provide attendance summaries and finance actual bridge records.
- `US-22-T2` and `US-22-T4` may already provide mobile employee interactions and released actual/read-model surfaces if employee submission is exposed in the app.
- `US-5-T3` must already provide role and route-guard scaffolding for employee, field lead, and object/event lead scopes.
- If the repository already has actor/audit command patterns for staged approvals, reuse them instead of inventing a task-specific approval transport.

## Scope of work

- Implement the command flow for preliminary actual submission by an employee or field lead against the correct actual/attendance context.
- Implement the operational confirmation stage for field supervisors / object-event leads, including authorization limited to assigned operational scope.
- Persist stage state, approver identity, timestamps, and notes in `finance.actual_approval` or the approved equivalent without skipping ahead to final finance approval.
- Expose the required backend contracts and, where appropriate, the narrow web/mobile screens/actions for submission and operational confirmation using existing Vben/Prokit shells.
- Add tests for employee self-scope, field-lead scope, operational lead confirmation, forbidden cross-scope actions, and audit trace creation.

## Preferred file targets

- `modules/finance` approval services, schemas, and routers.
- Vben review/confirmation screens for operational leads if the repo already has an actuals review area.
- Mobile employee action surfaces only if the repository already exposes actual review in the app.
- Tests for stage transitions, authorization, and audit trails.

## Hard constraints

- Do not implement final administration/finance signoff in this task; stop at preliminary + operational stages.
- Employee submissions must stay within own-record scope; field-lead confirmations must stay within assigned operational scope.
- Do not allow approval-stage writes to mutate raw evidence timestamps or hide discrepancies.
- Keep audit trails explicit for who submitted/confirmed, when, and with what note.
- Any new user-facing actions must preserve DE default / EN secondary behavior.

## Expected implementation outputs

- Stage-one and stage-two actual approval commands and persistence.
- Role-scoped submission/confirmation surfaces or APIs.
- Tests proving authorization, stage ordering, and audit trace creation.

## Non-goals

- Do not implement final finance reconciliation approval here.
- Do not add payroll export or invoice generation in this task.
- Do not redesign attendance capture UX; this task starts from existing actual/attendance context.

## Verification checklist

- [ ] Employees or field leads can submit preliminary actuals only within allowed scope.
- [ ] Object/event leads can operationally confirm only the records they are allowed to manage.
- [ ] Approval rows persist stage, actor, time, and note data cleanly.
- [ ] Raw evidence and discrepancy provenance remain visible after stage transitions.
- [ ] Tests cover scope, ordering, auditability, and forbidden actions.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
