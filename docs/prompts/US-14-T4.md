# Codex Prompt — US-14-T4

## Task title

**Create partner-side read models consumed by planning, field, and finance flows**

## Objective

Package stable partner-facing read contracts so later planning, field execution, and finance modules can consume subcontractor company/worker/compliance facts without crossing write boundaries or inventing duplicate truth.

## Source context

- Updated proposal: section 5.4 subcontractor collaboration and section 6.3 steps 3 through 7, where staff data, assignment readiness, attendance evidence, and commercial status must flow across planning, field, and finance without losing ownership boundaries.
- Implementation specification: section 5 cross-module relationship map, especially the subcontractor chain; section 7 reporting/read-model guidance; section 9 `SubcontractorAggregate` ownership and downstream consumers.
- Uploaded Dienstplanmacher subcontractor source material: partner staff readiness, attendance reporting, controlling of subcontractor shifts, and invoice-check visibility imply stable partner read contracts for downstream modules.
- Sprint reference: `docs/sprint/sprint-05-subcontractor-management.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-13-T1` through `US-14-T3` should already provide subcontractor, worker, and readiness truth.
- Later planning, field, and finance tables may not all exist yet in strict sprint order. This task must therefore create stable contracts/read models without inventing premature write-side tables.
- Where the repo already has shared contract or projection packages, follow them rather than exposing brittle cross-module direct model imports.

## Scope of work

- Create reusable read/query contracts that expose the partner facts later modules need: active subcontractor master data, active workers, valid qualifications, readiness summaries, active credentials, scope rows, portal-ready contacts, and relevant commercial settings.
- Provide internal service methods, DTOs, or read APIs that planning can use for candidate search and release validation, field execution can use for actor lookup/scan resolution, and finance can use for partner identity and commercial-control joins.
- Where useful, add SQL views or optimized query paths that preserve module ownership and are clearly read-only.
- Document and test the contract boundaries so downstream modules know which partner data they may read and which writes must remain in `modules/subcontractors`.
- If future ops/field/finance tables are not yet present, keep those contracts empty-state-capable and interface-first; do not fabricate placeholder planning or finance persistence just to satisfy a hypothetical join.

## Preferred file targets

- `modules/subcontractors/read_models/`, `modules/subcontractors/contracts/`, or the repo equivalent for partner-facing projections and DTOs.
- Shared tests that prove downstream callers can consume the contracts without mutating partner truth.
- Lightweight architecture notes or inline docs if the repo keeps boundary documentation near the code.

## Hard constraints

- These outputs are read models/contracts, not a second write-side source of truth. Downstream modules may read them, but must not mutate partner master tables through hidden paths.
- Do not create fake ops, field, or finance persistence in Sprint 5 just because later consumers are known.
- Keep partner ownership explicit: worker readiness, credentials, and contact/portal truth remain in the partner module.
- Be careful with personally identifying data in downstream contracts; expose only what planning/field/finance actually need.
- Prefer stable, typed contracts over ad-hoc dictionary payloads or hidden ORM cross-imports.

## Expected implementation outputs

- Stable partner read/projection contracts for downstream modules.
- Optimized or well-structured query paths for partner identity, worker readiness, and credential lookup.
- Tests and lightweight documentation that make the boundary explicit for later sprint tasks.

## Non-goals

- Do not implement planning releases, assignments, time events, actuals, or invoice checks here.
- Do not build the portal pages here; those belong to `US-15-*`.
- Do not add a reporting warehouse or second write schema here.

## Verification checklist

- [ ] Downstream callers can obtain active partner, worker, readiness, and credential data through stable read contracts.
- [ ] The read layer remains tenant-safe and does not expose unnecessary fields.
- [ ] No hidden write path from planning/field/finance into partner master tables is introduced.
- [ ] If future modules are absent, the interfaces still compile and behave predictably with empty-state outputs or no-op adapters.
- [ ] Tests/documentation make the module boundary clear enough for later sprint implementers.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
