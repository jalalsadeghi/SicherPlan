# Codex Prompt — US-33-T1

## Task title

**Apply optimization indexes, partition candidates, and query tuning on heavy tables/views.**

## Objective

Execute the first hardening pass on database/query performance so reporting, audit-heavy, time-capture, and finance-heavy paths meet Sprint 11 readiness expectations through evidence-based indexing, targeted query tuning, and explicit partition-candidate preparation.

## Source context

- Updated proposal: technical quality requirements include structured monitoring, rate limiting, backup/restore, and an architecture that supports reporting and finance workloads without rework.
- Implementation specification: Phase H is the right time for optimization indexes and partitioning if needed; earlier sections already identify append-only heavy tables such as audit/login events and raw capture tables, plus reporting views that may become materialized depending on workload.
- Sprint reference: `docs/sprint/sprint-11-reporting-and-hardening.md`.
- Cross-cutting rules: `AGENTS.md`, especially reporting as views/materialized views, append-only evidence for audit/time/patrol rows, and explicit relationship/loading discipline in SQLAlchemy.

## Dependencies

- `US-31-*` and `US-32-*` must already provide the reporting views and heavy read paths whose performance now needs tuning.
- `US-25-*`, `US-26-*`, `US-27-*`, and earlier audit/IAM work must already provide the high-volume raw/event and actual-record tables that are likely optimization candidates.
- `US-2-T4` or repository-equivalent observability work should already provide query logging/metrics or at least a pattern for capturing performance evidence.
- If the repository already has explain-plan helpers, benchmark fixtures, or DB-performance docs, extend them instead of creating parallel tooling.

## Scope of work

- Capture baseline query evidence for the heaviest reporting, audit, login, time-capture, and finance queries using repository-appropriate profiling or explain-plan workflows before changing indexes.
- Add targeted indexes, partial indexes, covering indexes, or support indexes for heavy views/materialized views and their transactional source tables where the evidence shows clear value.
- Tune reporting and service-layer queries to avoid unnecessary wide joins, N+1 loading, or poor filter paths; if SQLAlchemy mappings are part of the problem, fix loading strategies explicitly.
- Identify append-only or very large tables that are credible partition candidates (for example raw event or audit tables) and implement partition preparation only where it is justified and safe; otherwise document the candidate plan and prerequisites clearly.
- If materialized views are already used or become necessary, implement explicit refresh strategy and index support so performance gains do not come from hidden stale behavior.
- Add tests or benchmark scripts proving the tuned paths remain correct and materially improved, and document any partition-candidate decisions that are deferred rather than implemented.

## Preferred file targets

- Alembic revisions or repository-equivalent DB migration files for indexes, materialized-view support indexes, and any safe partition-preparation changes.
- `modules/reporting/`, `modules/finance/`, `modules/field_execution/`, and `modules/iam/` query/service files where targeted tuning is required.
- Docs or benchmark scripts under a repository-appropriate hardening/performance area.
- Tests/benchmarks proving correctness after tuning.

## Hard constraints

- Do not add speculative indexes or partition logic without evidence from the actual heavy paths in this repository.
- Performance changes must not alter business semantics or weaken scope enforcement.
- If partitioning is not yet safe or justified, document the candidate and leave a stable seam rather than forcing premature physical redesign.
- Materialized-view choices, if any, must have explicit refresh behavior and correctness guarantees.
- Do not broaden this task into RLS, backup/restore drills, or penetration/resilience testing yet.

## Expected implementation outputs

- Evidence-based index and query-tuning changes for the heavy reporting and evidence tables.
- Optional partition-preparation changes or documented partition-candidate plan where justified.
- Benchmarks/tests and hardening notes proving correctness and measurable improvement.

## Non-goals

- Do not solve performance by duplicating transactional truth into write-side cache tables.
- Do not redesign the whole schema or module boundaries in this task.
- Do not hide correctness regressions behind benchmark-only improvements.
- Do not apply RLS or backup/restore changes here; those belong to later `US-33` tasks.

## Verification checklist

- [ ] The tuned queries and indexes are evidence-based and measurably improve the targeted heavy paths.
- [ ] Business semantics, tenant scope, and role scope remain unchanged.
- [ ] Any materialized-view or partition decisions are explicit and operationally safe.
- [ ] Tests/benchmarks/documentation exist for the tuned paths.
- [ ] The platform is more ready for UAT/go-live loads after the changes.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
