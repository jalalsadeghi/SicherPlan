# Codex Prompt — US-34-T1

## Task title

**Define migration templates for customers, employees, subcontractors, orders, and documents.**

## Objective

Turn the stabilized SicherPlan write model into an executable migration-template package so trial and production migrations can load customers, employees, subcontractors, customer orders, and linked documents without schema guesswork, hidden cross-module writes, or tenant-scope drift.

## Source context

- Updated proposal: Phase 6 covers migration support, user training, UAT, production go-live, and stabilization, and the assumptions explicitly say source master data, document classifications, and migration formats will be clarified and provided for migration planning.
- Implementation specification: the document is the backend source of truth for SQLAlchemy models, Pydantic schemas, Alembic migrations, and aggregate boundaries; migration sequencing places seed/reference completion after the write model is stable, and documents remain centrally tracked through the docs service.
- Sprint reference: `docs/sprint/sprint-12-migration-uat-and-go-live.md`.
- Cross-cutting rules: `AGENTS.md`, especially tenant-safe business numbers, centralized documents, explicit foreign keys, service ownership, and append-only evidence handling.

## Dependencies

- `US-6-T1` and `US-6-T4` must already provide the document backbone plus import/export job and outbox patterns.
- `US-7-*`, `US-11-*`, `US-13-*`, and `US-17-*` must already provide the target customer, employee, subcontractor, and order aggregates.
- `US-33-*` should already have stabilized the platform enough that migration work can target the final write model rather than a moving schema.
- Any existing bootstrap, seed, or import-helper conventions from earlier sprints should be reused instead of introducing a second migration/import style.

## Scope of work

- Create versioned migration templates and field dictionaries for customer, employee, subcontractor, order, and document-import packages, preferably in machine-readable formats the repository can validate and humans can review.
- Define required fields, optional fields, business keys, tenant-scoped identifier rules, lookup resolution rules, cross-sheet foreign-key references, and load ordering across the template set.
- Implement preflight import schemas and validation helpers that detect missing required fields, unknown lookup values, invalid cross-aggregate references, duplicate business keys, and unsafe status combinations before any write is attempted.
- Define explicit create/update matching rules for migrated records, including when a row is treated as a create, an update, or a conflict that must stop the import.
- Provide representative example rows and import-error report shapes so data-preparation teams can correct files without reverse-engineering backend code.
- Keep document migration as an explicit manifest-based flow that links files and metadata through the docs service rather than attaching opaque blobs directly to business rows.

## Preferred file targets

- `docs/migration/` for the template package, field dictionaries, and operator notes.
- Import/preflight code under repository-appropriate service areas such as `modules/platform_services/` plus aggregate-local validators where needed.
- Pydantic schemas or import DTOs under `modules/customers/`, `modules/employees/`, `modules/subcontractors/`, `modules/planning/`, and the docs module when the repository splits import contracts by bounded context.
- Tests and sample fixtures that prove template validation and matching behavior.

## Hard constraints

- Do not bulk-write directly across multiple aggregates in hidden SQL scripts; route writes through owning service boundaries or clearly defined import services.
- Preserve business-facing numbers such as `customer_no`, `personnel_no`, and `order_no`; do not replace them with ad hoc migration-only identifiers.
- Do not use fuzzy matching on company names or personal names. Matching rules must be explicit, deterministic, and auditable.
- Document files must be represented through centralized docs metadata, versions, and links, not repository-local one-off attachment tables.
- If template labels or admin-facing import UX are exposed to users, German remains default and English secondary.

## Expected implementation outputs

- A versioned migration-template package for customers, employees, subcontractors, orders, and documents.
- Preflight schemas, validators, and import-error report contracts.
- Operator-facing mapping notes and example template rows.
- Tests proving required-field, key-matching, lookup-resolution, and document-manifest behavior.

## Non-goals

- Do not run a full production migration in this task.
- Do not invent automatic data cleansing or fuzzy enrichment logic that hides poor source quality.
- Do not redesign stabilized aggregates just to fit a legacy export shape.
- Do not execute cutover or acceptance signoff here.

## Verification checklist

- [ ] Template coverage exists for customers, employees, subcontractors, orders, and documents with explicit cross-file relationships.
- [ ] Preflight validation catches required-field, duplicate-key, lookup, and foreign-key issues before writes.
- [ ] Create/update/conflict rules are explicit, deterministic, and tested.
- [ ] Document-import manifests route through centralized docs/version/link handling.
- [ ] Templates and validators align with tenant-scoped business numbers and owning service boundaries.

- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
