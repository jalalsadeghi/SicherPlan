# Codex Prompt — US-19-T2

## Task title

**Build staffing board APIs for assign, unassign, substitute, and mixed workforce scenarios.**

## Objective

Deliver the planning-service command and query contracts that power a dispatcher staffing board, so internal employees and subcontractor workers can be assigned, replaced, or removed without breaking planning ownership or tenant-safe visibility.

## Source context

- Updated proposal: section 5.5 operational tools calls for real-time staffing overview, qualification/function filters, traffic-light coverage, planning communication, and copy functions on top of the shift/demand/assignment model.
- Implementation specification: section 4.6 `ops.assignment`; section 5 customer, employee, and subcontractor relationship chains; section 6 validation enforcement points before assignment save and release.
- Sprint reference: `docs/sprint/sprint-07-staffing-engine.md`.
- Cross-cutting rules: `AGENTS.md`, especially planning ownership, tenant isolation, and SQLAlchemy query discipline.

## Dependencies

- `US-19-T1` must already provide stable demand-group, team, and assignment persistence.
- `US-18-T4` should already optimize the planning query paths that board-style staffing screens depend on.
- `US-12-T1` and `US-14-T4` should already provide read-safe internal and subcontractor candidate inputs.
- `US-5-T3` must already enforce role scope on dispatcher-facing APIs.
- Keep `US-20-*` in mind: this task should call into shared validation seams where they exist, but it should not re-implement the validation engine inline.

## Scope of work

- Implement planning-service command endpoints and DTOs for assign, unassign, substitute/reassign, and mixed-workforce scenarios over existing shifts and demand groups.
- Add board/query endpoints that return the minimum data required for dispatcher staffing slices: shifts, demand groups, current assignments, fill gaps, and candidate-selection context.
- Support internal employees and subcontractor workers through one consistent contract, while preserving the exactly-one-actor rule at the assignment level.
- Record assignment source/status transitions, offered timestamps, confirmed timestamps, and user attribution where the repository already carries those fields.
- Handle common operational conflict paths explicitly: same-person duplication on the same shift, stale-version edits, substitution of one actor with another, and re-opening a previously removed slot.
- Where the repository already has validation-service hooks, call them and return structured results. Where full Sprint 7 validation depth is not present yet, preserve a compatible command/result seam instead of inventing ad-hoc rule logic here.
- Add backend tests for command correctness, mixed-workforce behavior, substitution semantics, scope enforcement, and the most important conflict/error paths.

## Preferred file targets

- `modules/planning/` services, repositories, routers/controllers, command/query DTOs, and tests.
- Small read-model helpers or list-item DTOs dedicated to staffing-board API responses.
- Optional contract helpers for later Vben pages, but no full board UI in this task.

## Hard constraints

- Planning owns assignment writes. Do not create hidden shortcut writes into HR or partner modules.
- Keep commands explicit and auditable; avoid generic 'save board state' payloads that blur business intent.
- Do not silently override validation or release rules. Structured failures/warnings must be returned in a reviewable form.
- Do not create shadow board tables or denormalized staffing snapshots just to serve one screen.
- Mixed workforce means one board can work with employees and subcontractor workers; it does not permit cross-tenant or cross-partner leakage.
- Any user-facing message keys or API validation texts introduced here must preserve DE default and EN secondary localization behavior.

## Expected implementation outputs

- Dispatcher-oriented staffing command/query APIs for assign, unassign, substitute, and mixed-workforce flows.
- Structured command results that can carry validation, conflict, and audit-relevant metadata into later UI work.
- Tests proving planning ownership, scope safety, and the core staffing lifecycle behaviors.

## Non-goals

- Do not build the Vben staffing board UI here; that belongs primarily to `US-19-T4` and later UI prompts.
- Do not implement subcontractor release creation here; that belongs to `US-19-T3`.
- Do not implement the full validation engine here; that belongs to `US-20-*`.
- Do not implement release-state publication or portal/mobile exposure here; that belongs to `US-21-*`.

## Verification checklist

- [ ] Dispatchers can assign, unassign, and substitute staff through explicit commands without breaking tenant or role scope.
- [ ] The same API contracts support internal, subcontractor, and mixed-workforce scenarios.
- [ ] Command responses make conflicts and validation outcomes explicit rather than collapsing them into generic failures.
- [ ] No staffing-board API path performs hidden writes into HR, partner, field, or finance ownership.
- [ ] Tests cover substitution, stale/conflict handling, and mixed-workforce assignment behavior.

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
