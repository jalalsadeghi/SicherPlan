# Codex Prompt — US-31-T4

## Task title

**Validate report reproducibility against transactional tables and released evidence.**

## Objective

Prove that the Sprint 11 reporting package is reproducible from transactional and document evidence by adding reconciliation tests, report-validation utilities, and a clear traceability path from views back to approved operational and finance truth.

## Source context

- Updated proposal: deliverables and acceptance require a document/reporting layer with reproducible key reports and compliance exports; workflow 6.1 ends with the released report package, timesheets, invoice, and archived audit trail.
- Implementation specification: reporting must remain derived from transactional tables through views/materialized views, and Phase H explicitly treats reporting as the final reflection of the write model rather than a second source of truth.
- Appendix E of the proposal lists concrete report families whose outputs must match the underlying normalized chains for planning, finance, compliance, and security.
- Sprint reference: `docs/sprint/sprint-11-reporting-and-hardening.md`.

## Dependencies

- `US-31-T1` through `US-31-T3` must already provide reporting views, endpoints, and UI/export surfaces to validate.
- `US-29-T1` through `US-29-T4` and `US-30-T1` through `US-30-T4` must already provide released timesheet, invoice, and subcontractor-control evidence the reports summarize.
- `US-23-T3` and related document-generation tasks should already provide revision-safe outputs and document-link patterns that can be used for evidence traceability.
- The repository should already have an integration-test or QA harness from earlier sprints; extend it rather than inventing a disconnected reporting test rig.

## Scope of work

- Add integration tests or repository-equivalent reproducibility checks that compare view outputs and exported report payloads against authoritative transactional aggregates for representative periods and scopes.
- Create reporting-validation utilities or fixtures that can prove totals, counts, statuses, and evidence links remain explainable from assignments, actual records, invoices, payroll/export state, notice reads, and audit/login events as applicable.
- Where the repository already supports generated documents or downloadable outputs, add traceability assertions that the released file or export package matches the underlying reporting query result at the time of generation.
- Expose an internal diagnostic or QA-only report reconciliation surface if the repository already has an admin diagnostics pattern; otherwise keep the validation harness in tests/docs rather than adding a production endpoint unnecessarily.
- Document the reproducibility assumptions, refresh behavior for materialized views if any exist, and the exact authoritative source chain for each report family.
- Add failure-focused tests for empty periods, partial workflow completion, archived records, reruns after refresh, and representative edge cases where evidence and released outputs can drift if the implementation is wrong.

## Preferred file targets

- Integration/QA test suites under `modules/reporting/tests/` or the repository-equivalent QA package.
- Optional internal diagnostics helpers or admin-only verification routes only if they fit existing repo patterns.
- Reporting documentation under `docs/` or a repository-equivalent QA/hardening area explaining reproducibility assumptions.
- No new customer-facing or broad operational UI beyond what is needed for verification.

## Hard constraints

- Reproducibility checks must compare against the true transactional backbone, not against duplicated snapshots or manually curated sample files.
- Where materialized views exist, refresh behavior must be explicit in tests and docs; do not let stale data masquerade as reproducibility.
- Keep the validation layer audit-safe and read-only. Do not mutate business state just to make a report match.
- Do not weaken scope rules while validating reports; internal QA or diagnostic tools must still respect tenant isolation and appropriate access control.
- Do not broaden this task into a generic data-warehouse reconciliation program beyond the Sprint 11 reporting scope.

## Expected implementation outputs

- A report reproducibility test harness or equivalent validation suite.
- Traceability documentation or diagnostics showing how report outputs map back to transactional evidence.
- Tests covering refresh/re-run consistency, scope safety, and representative edge cases.

## Non-goals

- Do not add new business reports in this task; validate the ones already implemented.
- Do not redesign invoice, payroll, planning, or watchbook generation logic here unless a discovered defect is a narrow prerequisite fix.
- Do not use golden files that bypass comparison with transactional truth.
- Do not turn internal diagnostics into public or portal-facing endpoints.

## Verification checklist

- [ ] Each implemented reporting family has a reproducibility path back to authoritative transactional/evidence tables.
- [ ] Tests prove that exported or API-served report values match the source-of-truth aggregates.
- [ ] Materialized-view refresh or staleness assumptions are explicit where relevant.
- [ ] Tenant and role scope remain enforced even in validation or diagnostic tooling.
- [ ] Documentation and tests are strong enough to support UAT and go-live confidence.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
