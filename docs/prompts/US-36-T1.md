# Codex Prompt — US-36-T1

## Task title

**Execute production cutover, configuration freeze, and monitored release steps.**

## Objective

Codify and, where the connected environment allows, run the production cutover sequence with configuration freeze, migrations, deployment ordering, smoke checks, and monitored release gates so the first live release follows one controlled path.

## Source context

- Updated proposal: Phase 6 includes production go-live and stabilization, while the technical baseline requires CI/CD, observability, secure permissions, backup/restore, and structured monitoring.
- Implementation specification: the write model, generated outputs, and service boundaries are already stable by this phase, so release logic should target the final integrated platform rather than temporary migration shortcuts.
- Sprint reference: `docs/sprint/sprint-12-migration-uat-and-go-live.md`.
- Cross-cutting rules: `AGENTS.md`, especially repository-backed operational discipline, tenant safety, centralized docs, and truthful evidence handling.

## Dependencies

- `US-35-T4` must already provide the finalized go-live checklist, rehearsal runbooks, and acceptance package.
- `US-33-T3` and `US-33-T4` should already provide backup/restore, hardening, and readiness evidence that the release path depends on.
- `US-2-T3` and `US-2-T4` or repository-equivalent work must already provide deployment automation, health checks, and observability seams.
- `US-34-*` must already provide the migration and configuration assets needed at cutover.

## Scope of work

- Implement the authoritative release runbook and automation for configuration freeze, pre-cutover backup verification, migration execution, application deployment ordering, smoke checks, and monitored rollout gates.
- Wire environment-specific commands, scripts, or pipeline jobs where the repository already supports them, and keep manual operator steps explicit where full automation is not possible.
- Capture release evidence such as timestamps, executed steps, smoke-test results, operator notes, and rollback checkpoints in durable repository artifacts or linked runbook outputs.
- Include high-value post-deploy smoke checks for authentication, tenant scoping, document access, portals, planning, actuals, and generated-output behavior.
- If direct production execution is not available from the repository, produce the exact operator-driven cutover package and dry-run evidence rather than pretending execution occurred.
- Keep the release path narrow, explicit, and reversible.

## Preferred file targets

- CI/CD or release-automation files already used by the repository.
- `docs/runbooks/production-cutover.md` or a repository-equivalent live-release guide.
- Smoke-test scripts and health-check helpers under QA/ops areas already present in the repo.
- Release evidence templates or operator logs kept in-repo where appropriate.

## Hard constraints

- Do not leak secrets, tokens, or environment credentials into versioned artifacts.
- Do not rely on untracked manual hotfixes or shell history as the release record.
- Database migration order, configuration freeze, and rollback checkpoints must be explicit.
- Release verification must include tenant-scope, authorization, and document/output integrity checks, not only HTTP liveness.
- Be honest about whether the repository executed real release steps or only prepared the operator path.

## Expected implementation outputs

- An authoritative production cutover runbook and/or automation path.
- Smoke-test and monitored-release gate assets.
- Release evidence templates or captured outputs.
- Explicit rollback and freeze instructions.

## Non-goals

- Do not turn this into a general platform redesign.
- Do not perform post-release stabilization triage here beyond what is needed to hand off to hypercare.
- Do not introduce emergency scripts that bypass versioned automation and runbooks.
- Do not claim production execution happened if repository access does not support it.

## Verification checklist

- [ ] One authoritative cutover path exists for freeze, backup check, migration, deploy, smoke verification, monitoring, and rollback.
- [ ] Automation and manual operator steps are clearly distinguished.
- [ ] Smoke checks cover more than liveness and include key secured workflows.
- [ ] Release evidence is durable and reviewable.
- [ ] Rollback and freeze steps are explicit and actionable.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
