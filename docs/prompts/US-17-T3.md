# Codex Prompt — US-17-T3

## Task title

**Build order/planning APIs and UI for release states, notes, and document packages.**

## Objective

Deliver the dispatcher-facing order and planning administration experience in Vben so users can manage release-ready order/planning data, notes, and document packages through localized, permission-aware screens.

## Source context

- Updated proposal: section 5.5 project details, event documents, object documents, stand protocol printouts, route-map context, and release-state expectations; section 6.1 workflow steps 2–5.
- Implementation specification: section 4.6 order and planning-record tables plus docs-backed generated-output/document ownership conventions; package map for `PlanningAggregate` and `InformationAggregate`.
- Sprint reference: `docs/sprint/sprint-06-ops-planning-foundation.md`.
- Cross-cutting rules: `AGENTS.md`, especially Vben, docs ownership, and DE/EN rules.

## Dependencies

- `US-17-T1` and `US-17-T2` must already provide stable backend APIs for orders, planning records, and mode-specific details.
- `US-6-T1` must already provide the docs backbone for document packages and attachments.
- `US-3-T1` and `US-3-T4` should already provide the Vben shell and localization baseline.
- `US-5-T3` should already provide route/action permission scaffolding for dispatcher/admin roles.

## Scope of work

- Build Vben routes, list/detail pages, forms, API bindings, and localized UI resources for customer orders and planning records.
- Support release-state display and transition actions where the backend contract allows them, including safe status badges, confirmation dialogs, and audit-friendly UX language.
- Provide note editing and document-package management on the order/planning screens, reusing the centralized docs service rather than inventing a local file store.
- Make it easy for dispatchers to navigate between the order root and one or more planning records, with clear planning-mode-specific detail sections for event, site, trade fair, and patrol data.
- Use Vben CRUD patterns for tables, filters, drawers/forms, breadcrumbs, permission metadata, and archive-state handling.
- Add frontend tests and, where available, end-to-end coverage for release-state actions, document-package flows, note editing, and permission behavior.
- If a small backend contract gap prevents the UI from working cleanly, close it narrowly here without expanding the task into the next stories.

## Preferred file targets

- Vben views/components/routes under the planning or dispatch section of the web-admin workspace.
- Web API client bindings, composables/stores, and i18n resources.
- Small backend API additions only where necessary to support a coherent order/planning UI.
- Frontend tests for CRUD, release-state handling, notes, docs package flows, and permissions.

## Hard constraints

- Follow Vben Admin conventions instead of inventing a custom workflow shell.
- Keep document handling on the docs backbone; no direct blob/file shortcuts from the UI.
- Do not expose unreleased results to customer/subcontractor views from this admin UI.
- Do not turn this task into shift editing, staffing board, or finance layout work.
- German must be the default locale and English must be added alongside it for all new UI strings.
- Release-state UX must stay permission-aware and audit-friendly; do not hide dangerous state transitions behind silent one-click actions.

## Expected implementation outputs

- Localized Vben order/planning administration screens and API bindings.
- Document-package and note-management flows grounded in the docs backbone.
- Frontend tests for release-state actions, note editing, docs package flows, and permissions.

## Non-goals

- Do not implement shift maintenance UI here; that is `US-18-T3`.
- Do not implement staffing, validation overrides, or assignment boards here; that comes in Sprint 7.
- Do not implement finance calculations, invoice previews, or customer/subcontractor portal exposure here.
- Do not invent a second note/document store outside the approved planning + docs model.

## Verification checklist

- [ ] Authorized users can create, edit, archive, and inspect orders and planning records through localized Vben screens.
- [ ] Release-state controls are permission-aware, clear, and aligned with backend validation.
- [ ] Notes and document packages resolve through the approved docs contracts.
- [ ] German is the default locale and English is available for all new order/planning UI strings.
- [ ] Frontend tests cover permission behavior and the core release-state/note/document flows.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
