# Codex Prompt — US-32-T1

## Task title

**Implement compliance status views for qualification expiry and missing mandatory documents.**

## Objective

Build the first compliance/QM reporting slice so expiring qualifications and missing mandatory documents are visible through a reproducible reporting view and service layer, without duplicating or mutating the compliance truth owned by HR, partner, docs, and planning release flows.

## Source context

- Updated proposal: section 5.7 requires compliance reporting for qualification expiry, inactivity, and related operational/QM evidence; Appendix E lists qualification register with expiry and mandatory-document related evidence under QM and compliance.
- Implementation specification: section 7 recommends `rpt.compliance_status_v`, built from compliance rules, qualification tables, document links, notice-read data, HR absence tables, and IAM status; section 6 maps certificate validity and mandatory-document completeness to normalized source tables and release enforcement points.
- Sprint reference: `docs/sprint/sprint-11-reporting-and-hardening.md`.
- Cross-cutting rules: `AGENTS.md`, especially document-centered compliance, tenant-safe role scoping, and read-only reporting discipline.

## Dependencies

- `US-12-T1` must already provide employee qualifications and proof/document truth.
- `US-14-T1` and `US-14-T3` must already provide subcontractor-worker qualifications, document proofs, and compliance completeness seams.
- `US-20-T1` and `US-20-T2` must already provide validation rules and requirement-rule interpretation for qualification validity and mandatory-document completeness.
- `US-6-T1` and related document-link work must already provide centralized document metadata and linking behavior.

## Scope of work

- Implement the compliance reporting view package for qualification expiry and mandatory-document gaps, preferably as the first slice of `rpt.compliance_status_v` or a repository-equivalent view that can be extended in later tasks without contract breakage.
- Model at least actor type, actor reference, tenant scope, qualification/document rule context, expiry horizon, missing/expired status, severity, and any release-relevant status fields that controllers/QM need to act on.
- Derive missing-document and expiry states from the normalized source tables named in the implementation spec rather than duplicating compliance truth into a new table.
- Expose read-only FastAPI services/endpoints and filter schemas so internal users can query by actor type, branch/mandate, rule code, document type, expiry window, severity, and status.
- Support both internal employees and subcontractor workers where the underlying requirement rules and source tables already make that distinction available.
- Add tests for expiry-window logic, missing-document detection, mixed employee/subcontractor cases, and tenant-safe filtering.

## Preferred file targets

- `modules/reporting/` compliance view definitions, query services, schemas, routers, and tests.
- Alembic revisions or repository-equivalent SQL files for the compliance-status view and any supporting indexes required immediately.
- Shared reporting/compliance enums or DTO helpers only if they fit later `US-32-T2` extension work.
- No Vben screen implementation in this task.

## Hard constraints

- Do not create a second compliance master table for expiry or completeness status. The report must derive from normalized qualification, requirement-rule, and document-link truth.
- Do not re-run planning release side effects or mutate assignment state from the reporting query layer.
- Keep employee vs subcontractor-worker differences explicit and auditable; do not collapse them into an opaque 'person' abstraction if that would hide source semantics.
- Preserve tenant isolation and least-privilege visibility. Compliance reports are internal management/QM outputs, not portal data.
- Do not broaden this task into read-confirmation, absence, inactivity, or security-activity reporting yet.

## Expected implementation outputs

- Compliance reporting view/service support for qualification expiry and missing mandatory documents.
- Read-only reporting endpoints and explicit filter/result schemas.
- Tests covering expiry logic, missing-document logic, actor-type differences, and scope safety.

## Non-goals

- Do not implement notice-read stats, free Sundays, sickness/vacation, or inactivity here; that belongs to `US-32-T2`.
- Do not implement security-activity reporting here; that belongs to `US-32-T3`.
- Do not build Vben export screens or scheduled delivery here; that belongs to `US-32-T4`.
- Do not change the enforcement behavior of planning release validations in this reporting task.

## Verification checklist

- [ ] Expiring and missing-compliance cases are derived correctly from normalized tables and configurable rule context.
- [ ] Employee and subcontractor-worker cases are both covered where the underlying rules apply.
- [ ] No compliance write-side duplication or release-state mutation occurs.
- [ ] Endpoints and schemas are stable for later UI/export work.
- [ ] Tests cover representative expiry windows, missing proofs, and scope boundaries.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
