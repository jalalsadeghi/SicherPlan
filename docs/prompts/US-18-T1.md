# Codex Prompt — US-18-T1

## Task title

**Implement shift plans, templates, recurrence series, and concrete shift generation.**

## Objective

Build the transaction-heavy shift-planning backbone so one planning record can own reusable shift plans, templates, recurring series, and generated concrete shifts across event, site, trade-fair, and patrol scenarios.

## Source context

- Updated proposal: section 5.5 shift and staffing model; planning-mode copy/reuse expectations; Appendix D `Shift plan`, `Shift`, and `Templates and series`.
- Implementation specification: section 4.6 `ops.shift_plan`, `ops.shift_template`, `ops.shift_series`, and `ops.shift`; section 9 `PlanningAggregate`; Phase E note about board-style planning indexes.
- Sprint reference: `docs/sprint/sprint-06-ops-planning-foundation.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-17-T2` must already provide `ops.planning_record`.
- `US-5-T1` / `US-5-T3` should already provide user/account scope and dispatcher role patterns for API access.
- Keep Sprint 7 demand-group and assignment work in mind because `ops.shift` becomes their parent, but do not implement those downstream tables here.

## Scope of work

- Create migrations, models, schemas, services, routers, and tests for `ops.shift_plan`, `ops.shift_template`, `ops.shift_series`, and `ops.shift`.
- Implement shift plans as the planning slice under one planning record, including workforce scope, planning period, status, and remarks.
- Implement shift templates as reusable tenant-scoped patterns with code, label, local start/end time, default break minutes, shift type, and meeting point.
- Implement recurrence-series headers that connect one shift plan to a template and recurrence rule/timezone/date window.
- Implement service logic that generates concrete `ops.shift` rows from a shift series/template combination, including `starts_at`, `ends_at`, break minutes, shift type, location text, meeting point, release state, status, and visibility flags (`customer_visible_flag`, `subcontractor_visible_flag`, `stealth_mode_flag`).
- Add validation for time ranges, series windows, template uniqueness, safe timezone handling, and generation idempotency or duplicate-prevention behavior that matches repo conventions.
- Add tests for generation correctness, timezone/date math, and visibility/default-state behavior.

## Preferred file targets

- Alembic revisions for shift-plan, template, series, and shift tables plus supporting indexes.
- `modules/planning/` models, schemas, services, routers, generation helpers, and tests.
- Shared recurrence helpers only where they remain inside the planning bounded context.
- No major UI work in this task; the admin screens land in `US-18-T3`.

## Hard constraints

- Do not implement demand groups, assignments, teams, or subcontractor releases here; those belong to Sprint 7 even though the shift table later becomes their parent.
- Keep recurrence logic explicit and testable; do not hide generation semantics in opaque JSON blobs or hardcoded frontend behavior.
- Timezone handling must be deliberate; do not treat local template times as UTC instants without conversion.
- Generated shifts remain planning truth; do not introduce finance or field-execution writes here.
- Visibility flags must be stored explicitly on the shift record and must not imply public release by themselves.
- Any user-visible validation content introduced here must preserve German default and English secondary behavior.

## Expected implementation outputs

- A normalized shift-planning schema with plan, template, series, and concrete shift tables.
- Reliable concrete-shift generation services with test-covered recurrence/timezone behavior.
- Stable APIs that later UI, staffing, field, and finance tasks can consume.

## Non-goals

- Do not implement copy/day-week reuse or recurring-exception flows here beyond the base recurrence/generation model; that is `US-18-T2`.
- Do not build the shift editor UI here; that is `US-18-T3`.
- Do not optimize every planning-board query path here; that is `US-18-T4`.
- Do not implement demand groups, assignments, teams, or release workflows here; those belong to Sprint 7.

## Verification checklist

- [ ] Shift plans, templates, series, and concrete shifts can be created and queried under one planning record.
- [ ] Concrete shift generation is deterministic, timezone-aware, and resistant to duplicate creation in the supported execution path.
- [ ] Shift records carry the expected visibility flags, status, release-state, meeting-point, and timing fields.
- [ ] The implementation leaves clean parent seams for demand groups and assignments without prematurely creating them.
- [ ] Tests cover recurrence windows, timezone/date handling, duplicate-prevention behavior, and invalid time ranges.

- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
