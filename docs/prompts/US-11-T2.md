# Codex Prompt — US-11-T2

## Task title

**Build employee file UI for operational data, reminders, notes, photo, and group assignments**

## Objective

Deliver the admin-facing employee file experience in Vben patterns so authorized internal users can maintain operational employee data, reminders/notes, profile photo, and group assignments without exposing HR-private fields to the wrong roles.

## Source context

- Updated proposal: section 5.3 digital employee file with master data, profile photo, document archive, groups, reminders, positive activity notes, and controlled self-service context; Appendix C operational controls.
- Implementation specification: `EmployeeAggregate` and the related `hr.employee_group`, `hr.employee_group_member`, and `hr.employee_note` support rows inside the employee bounded context.
- Uploaded Dienstplanmacher Mitarbeiterakte source material: electronic employee file, role-sensitive visibility, positive activity notes, profile photo, V-Card/Sedcard context, customer blocks, and group/reminder-friendly operations.
- Sprint reference: `docs/sprint/sprint-04-recruiting-and-employee-core.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-11-T1` must already provide employee master/private/address/group foundations.
- `US-6-T1` must already provide document/photo storage and retrieval patterns.
- `US-5-T3` must already enforce route/API role scoping so the UI can hide or deny restricted sections.
- `US-3-T1` and `US-3-T4` should already provide Vben shell patterns and DE/EN localization resources.

## Scope of work

- Build Vben-aligned employee list and detail pages for operational employee data, including status, master data, contact data, branch/group context, and current address visibility as allowed by role.
- Implement reminder/note handling for operational notes and positive activity notes using the repo's approved employee-note service/model pattern.
- Provide profile-photo upload, preview, replace, and archive-safe retrieval using the centralized docs service rather than inline file handling.
- Enable employee-group assignment and maintenance through searchable selectors or assignment tables aligned with `hr.employee_group` and `hr.employee_group_member`.
- Separate operational sections from any HR-private sections clearly; if a restricted HR subsection exists in the same screen, it must be route-/permission-guarded and isolated in both API calls and UI state.
- Localize all new UI strings in German and English, and follow Vben route/list/detail/form conventions.
- Add frontend tests and, where supported, end-to-end coverage for permissions, note/reminder actions, group assignments, and photo handling.

## Preferred file targets

- Vben routes, list/detail pages, composables/stores, API clients, and DE/EN locale resources under the web-admin package.
- Employee-note/group/photo-related backend endpoints if the existing backend contract needs completion for this UI task.
- Frontend tests for employee-file navigation, permissions, photo upload, and note/group actions.

## Hard constraints

- Do not expose HR-private profile fields through operational pages or shared DTOs.
- Use the docs service for photo/document handling; do not create a second asset upload path.
- Do not turn this task into a full print/export module for vCard, Sedcard, or badges unless the repo already has a minimal no-drift hook.
- Respect Vben patterns for navigation, CRUD layout, and permission guards.
- German must be the default locale and English must exist for all new UI strings.

## Expected implementation outputs

- Employee list/detail admin screens with operational data maintenance.
- Reminder/note and group-assignment UI flows with permission-aware behavior.
- Profile-photo upload/preview wired through the docs backbone.
- Localized UI resources and frontend tests for critical flows.

## Non-goals

- Do not implement import/export here; that is `US-11-T3`.
- Do not implement archival lifecycle/audit rules here; that is `US-11-T4`.
- Do not implement full qualifications/absence/balance UI here; those belong to `US-12` tasks.
- Do not expose unrestricted employee self-service editing from the admin UI here.

## Verification checklist

- Authorized users can list and edit operational employee data without seeing HR-private data they should not access.
- Notes/reminders and group assignments work through stable backend contracts.
- Profile photos upload and render through the secure docs path.
- German is the default locale and English is available for all new employee-file UI strings.
- Frontend tests cover permission behavior and core note/photo/group flows.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
