# Codex Prompt — US-24-T4

## Task title

**Link patrol outputs into watchbook entries, evaluations, and compliance evidence.**

## Objective

Close the patrol-control story by wiring completed patrol evidence into the rest of the operational chain so rounds and checkpoint events can enrich watchbooks, feed patrol evaluations, and produce traceable compliance evidence without creating a second patrol truth outside the approved field and reporting architecture.

## Source context

- Updated proposal: section 5.2 guard patrol control says patrol evidence transfers directly into evaluations and the watchbook; section 5.5 released outputs; deliverables/acceptance require revision-safe patrol evidence and consistent audit outputs.
- Implementation specification: field-evidence chain in section 5, reporting/read-model guidance in section 7, patrol and watchbook tables in section 4.7, docs-generated-output rules in section 10, and aggregate/package ownership for `FieldAggregate` / `InformationAggregate` in section 9.
- Sprint reference: `docs/sprint/sprint-08-employee-app-and-field-i.md`.
- Cross-cutting rules: `AGENTS.md`, especially end-to-end traceability, append-only raw evidence, docs-linked generated outputs, and no second write-side reporting schema.
- Earlier prompt context: `US-23-T2`, `US-23-T3`, and `US-24-T1` to `US-24-T3` provide the watchbook and patrol evidence that this task must connect.

## Dependencies

- `US-24-T1`, `US-24-T2`, and `US-24-T3` must already provide stable patrol-round and event evidence.
- `US-23-T2` must already provide watchbook entry creation and review flows.
- `US-23-T3` should already provide docs-linked watchbook outputs and released read-model patterns.
- If the repository does not yet have a dedicated patrol-evaluation projection, create a derived read model or service seam rather than inventing a second patrol write-side aggregate.
- Any cross-module side effects that reach external providers or async processors must follow the outbox/integration pattern.

## Scope of work

- Link completed patrol-round outputs into watchbook-entry creation or association so field supervisors and later customer-facing outputs can see patrol context alongside daily log evidence.
- Build or refine patrol-evaluation read models/projections that summarize checkpoint completion, exceptions, aborts, and route outcomes without duplicating raw evidence into a new write schema.
- Expose compliance-ready evidence chains from patrol rounds/events to downstream reportable artifacts or status views.
- Where generated patrol summaries or exports are needed, register them through the docs backbone rather than leaving detached files or ad hoc JSON blobs.
- Keep the linkage explicit and traceable: patrol_round -> patrol_round_event -> watchbook/evaluation/compliance read models.
- Add tests for linkage integrity, derived-evaluation correctness, watchbook association, and evidence traceability.

## Preferred file targets

- `modules/field_execution` services/read models for patrol-to-watchbook/evaluation linkage.
- `modules/platform_services` or docs helpers only where generated patrol outputs need docs registration.
- Reporting/read-model SQL or query files if the repo already keeps derived patrol/compliance views there.
- Tests for traceability across patrol, watchbook, and derived evaluation/compliance views.

## Hard constraints

- Do not create a second patrol write-side truth just to make evaluations easier; use derived read models or approved service seams.
- Raw patrol rounds/events remain append-only and authoritative.
- Any generated patrol outputs or summaries must be docs-backed if they become durable artifacts.
- Watchbook and evaluation linkage must stay tenant-safe, scope-safe, and traceable to the source patrol evidence.
- Cross-module notifications or exports must use the approved outbox/integration pattern rather than inline provider calls.

## Expected implementation outputs

- Patrol-to-watchbook linkage or association flow.
- Derived patrol-evaluation/compliance read models or service projections.
- Docs-backed patrol output registration where durable artifacts are generated.
- Tests proving end-to-end traceability from patrol evidence into watchbook/evaluation/compliance views.

## Non-goals

- Do not build the full later reporting dashboard set here.
- Do not rewrite raw patrol history into summarized tables as a new source of truth.
- Do not bypass watchbook or docs ownership rules to attach patrol artifacts quickly.
- Do not mix finance actuals reconciliation into this task.

## Verification checklist

- [ ] Completed patrol evidence can be traced into watchbook association and derived evaluation/compliance views.
- [ ] No second patrol write-side schema is introduced for evaluations.
- [ ] Any durable patrol summaries/exports are docs-backed and linked to their source evidence.
- [ ] Tenant/scope safety remains intact across the linkage chain.
- [ ] Tests cover patrol-to-watchbook linkage, derived-evaluation correctness, and evidence traceability.

- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
