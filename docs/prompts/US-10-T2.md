# Codex Prompt — US-10-T2

## Task title

**Implement applicant status workflow and recruiter activity trail**

## Objective

Create a deterministic, auditable applicant workflow so recruiters can move applicants through intake, review, interview, decision, and ready-for-conversion states with append-only status history and actor attribution.

## Source context

- Updated proposal: section 5.3 recruitment workflow and section 6.2 step 2, where the recruiter processes applicant status, records interview steps, and decides accept or reject.
- Implementation specification: section 4.4 `hr.applicant` and append-only `hr.applicant_status_event`; audit guidance and aggregate boundary for `ApplicantAggregate`.
- Uploaded Dienstplanmacher Bewerberverwaltung material: documented four-step applicant process with accept/reject decisions, interview appointment, final decision, and hiring date, all tracked inside the applicant administration flow.
- Sprint reference: `docs/sprint/sprint-04-recruiting-and-employee-core.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-10-T1` must already provide stable applicant intake and persistence.
- `US-5-T4` should already provide the audit/event baseline for sensitive recruiter actions and actor attribution.
- If communication hooks already exist from `US-6-T2`, keep the workflow capable of later notification hooks without making messaging mandatory in this task.

## Scope of work

- Define the approved applicant status state machine, including the minimal intake, review, interview, accepted, rejected, and converted-ready semantics the repository needs for Sprint 4.
- Implement transition services that validate allowed status moves and create append-only `hr.applicant_status_event` rows for every meaningful change.
- Capture recruiter notes, interview comments, and decision rationale in the append-only workflow trail or in the narrow recruiting activity structure already aligned with repo patterns.
- Store actor identity, timestamps, and optional scheduling metadata for interview-related transitions without inventing a heavyweight recruiting CRM.
- Expose query/read contracts so later recruiter UI work can show current status, previous states, and the ordered activity timeline.
- Emit audit-safe business events for sensitive transitions such as reject, accept, reopen, and convert-ready.
- Write tests for allowed transitions, blocked transitions, append-only history, actor attribution, and ordering of status/activity events.

## Preferred file targets

- Recruiting workflow services and validators under `modules/recruiting/services/*` or the repo equivalent.
- SQLAlchemy models, enums/lookups, and Pydantic read models for applicant workflow state and history.
- Tests for transition guards, history creation, and audit emission.

## Hard constraints

- Workflow history must be append-only; do not rewrite or delete prior status events to 'clean up' history.
- Do not couple workflow transitions to UI-only state; the backend must own the transition rules.
- Keep status semantics explicit and machine-readable rather than depending on free-text notes alone.
- Do not convert applicants into employees in this task.
- Do not hide recruiter decisions from audit or actor attribution.

## Expected implementation outputs

- A validated applicant status state machine.
- Append-only applicant workflow/history persistence and read contracts.
- Recruiter activity and decision trail suitable for UI rendering and audit use.
- Tests for transition correctness and history immutability.

## Non-goals

- Do not build recruiter screens here; that is `US-10-T3`.
- Do not build applicant intake/public form logic here; that is `US-10-T1`.
- Do not implement applicant conversion to the employee aggregate here; that is `US-10-T4`.
- Do not create a broad CRM note/task subsystem beyond what the recruiting workflow actually needs.

## Verification checklist

- Invalid workflow transitions are blocked by backend rules.
- Every valid transition writes an append-only history row with actor and timestamp.
- Interview and decision notes are durable and visible in ordered history reads.
- Accept/reject/reopen paths remain deterministic and test-covered.
- Audit hooks fire for sensitive recruiter actions.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
