# Codex Prompt — US-35-T2

## Task title

**Prepare admin/user training materials and role-specific onboarding sessions.**

## Objective

Translate the implemented SicherPlan platform into role-specific training assets and onboarding plans so internal admins, operational users, portal users, and employee-app users can start using the product without reverse-engineering the repository or relying on tribal knowledge.

## Source context

- Updated proposal: project deliverables include user training material and go-live support, and the role model defines distinct permissions and visibility for platform admins, tenant admins, dispatchers, accounting, controller/QM, customer users, subcontractor users, and employees.
- Sprint reference: `docs/sprint/sprint-12-migration-uat-and-go-live.md`.
- Cross-cutting rules: `AGENTS.md`, especially role-scoped behavior, DE-default / EN-secondary user-facing materials, Vben Admin as the web reference shell, and the Prokit/Flutter-based mobile baseline.
- This task should reflect the actual UAT-ready product, not hypothetical future phases.

## Dependencies

- `US-35-T1` should already provide the main UAT scenarios and stable role/account context that training can build on.
- `US-3-*` and `US-22-*` must already provide the shared web/mobile shell, theme, and localization rules reflected in training materials.
- Customer, subcontractor, reporting, planning, finance, and field-execution features from earlier sprints must already be integrated enough for the role guides to be truthful.
- Any repository docs conventions, screenshot tooling, or docs-site structure should be reused instead of introducing a disconnected training format.

## Scope of work

- Create role-based quick-start guides and task-oriented training materials for platform admin, tenant admin, dispatcher, accounting, controller/QM, customer user, subcontractor administrator/user, and employee app user personas that actually exist in the repository.
- Prepare onboarding session outlines, agendas, prerequisites, exercises, expected outcomes, and follow-up knowledge checks for each major role cluster.
- Document navigation and workflow guidance for the Vben-based web experience and the Flutter/Prokit-inspired mobile experience where those surfaces are part of the approved product scope.
- Include privacy and permission reminders so training reinforces least-privilege behavior instead of accidentally teaching over-broad access assumptions.
- Provide sanitized example data or route/screen references suitable for repeatable onboarding sessions.
- Use German as default and English as secondary in user-facing training content, or provide translation-ready paired resources if the repository favors that pattern.

## Preferred file targets

- `docs/training/` for role guides, exercises, and quick starts.
- `docs/runbooks/onboarding/` or a repository-equivalent docs area for session plans and facilitator notes.
- Any docs-site navigation or index pages already used in the repository for operator documentation.
- Optional lightweight media or screenshot placeholders only if the repository already supports them.

## Hard constraints

- Training content must describe implemented behavior, not a roadmap or wish list.
- All example data must be sanitized and privacy-safe.
- German remains the default language and English secondary for user-facing training assets.
- Do not embed unstable or private environment details that would make the materials unsafe to reuse.
- Keep permission boundaries explicit so training does not normalize unauthorized data access.

## Expected implementation outputs

- Role-based training guides and quick-start materials.
- Session plans, onboarding checklists, and role-specific exercises.
- Sanitized examples or navigation references grounded in the real product.
- DE/EN or translation-ready training content structure.

## Non-goals

- Do not produce marketing collateral or sales decks.
- Do not write legal-policy manuals beyond what is necessary for user operation and privacy reminders.
- Do not promise features that are not actually available in the repository.
- Do not replace UAT evidence with training artifacts.

## Verification checklist

- [ ] Each major user role or role cluster has a truthful training path.
- [ ] Onboarding sessions include prerequisites, exercises, and expected outcomes.
- [ ] Training materials reflect real permissions, workflows, and screen structures.
- [ ] User-facing training assets follow German default and English secondary rules.
- [ ] Example data and references are sanitized and safe to reuse.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
