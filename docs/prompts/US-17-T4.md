# Codex Prompt — US-17-T4

## Task title

**Ensure commercial scope stays linked to customer billing rules and downstream finance flows.**

## Objective

Stabilize the contract between CRM-owned commercial truth and planning-owned operational structures so orders and planning records stay finance-ready without duplicating pricing logic or weakening the end-to-end order-to-invoice chain.

## Source context

- Updated proposal: section 5.2 customer commercial settings, section 5.5 common project structure, section 5.6 customer billing, and section 6.1 workflow steps 1, 2, 7, and 8.
- Implementation specification: CRM owns customer commercial settings; planning references customer/order truth; finance consumes released operational evidence through `finance.actual_record`; section 3.2 cross-service write discipline; package map for `CustomerAggregate`, `PlanningAggregate`, and `FinanceAggregate`.
- Sprint reference: `docs/sprint/sprint-06-ops-planning-foundation.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-8-T1`, `US-8-T2`, and `US-8-T4` must already provide the customer-commercial backbone or at least the approved contracts for billing profile, invoice parties, pricing rules, and dispatch settings.
- `US-17-T1` and `US-17-T2` must already provide the order and planning-record aggregates.
- Later finance tasks in Sprint 10 will consume this linkage. Leave those tasks a clear read contract instead of burying finance logic in planning.

## Scope of work

- Add or refine planning-side service contracts, projections, and validation hooks so orders/planning records remain linked to the correct CRM commercial profile and finance-facing references.
- Ensure the order root exposes the stable identifiers and reference fields that downstream actuals, timesheets, invoices, and invoice checks will need, without copying rate-card or invoice-layout data into ops tables.
- Where the repository already has release-state hooks, add release-readiness checks or warnings for missing mandatory commercial prerequisites such as missing billing profile, invoice party, or required dispatch settings.
- Provide explicit query/service interfaces that finance tasks can reuse later to resolve customer billing context from an order or planning record.
- Add tests proving the planning module reads CRM commercial truth safely, preserves tenant boundaries, and does not create a second commercial source of truth.
- If the repo already uses typed integration/read DTOs between planning and finance, update them narrowly and document the assumptions in code comments or module docs where appropriate.

## Preferred file targets

- `modules/planning/` services, validators, read DTOs, and tests.
- Small `modules/customers/` contract additions only if a missing CRM read service blocks a clean planning/finance handoff.
- No major UI work beyond minimal status indicators if the current repository requires them for release checks.
- No finance-output generation code in this task.

## Hard constraints

- CRM remains the single source of commercial truth for customer billing rules.
- Do not duplicate rate cards, surcharge logic, invoice layouts, or invoice-party data inside planning tables unless the repository already has an explicitly approved snapshot pattern.
- Do not bypass `finance.actual_record`; this task is about linkage, not billing calculation.
- Planning must not mutate CRM commercial data directly.
- Release-readiness checks must be deterministic and auditable if they influence status transitions.
- Any user-visible validation text introduced here must preserve German default and English secondary behavior.

## Expected implementation outputs

- A stable planning-to-CRM/finance commercial-link contract grounded in customer and order identifiers.
- Release-readiness checks or warnings for missing mandatory billing prerequisites where the repo contract already supports them.
- Tests proving tenant-safe commercial linkage without pricing duplication.

## Non-goals

- Do not implement actual billing, timesheet generation, invoice rendering, or payment/dunning logic here.
- Do not move customer pricing calculation into planning.
- Do not implement customer portal billing visibility here; that comes later with finance and portal work.
- Do not expand this task into a full reporting or reconciliation feature.

## Verification checklist

- [ ] Orders and planning records keep a clean, test-covered link to customer-commercial settings without duplicating pricing truth.
- [ ] Missing mandatory billing prerequisites are detected or surfaced where the approved release flow requires them.
- [ ] Tenant scope is preserved when resolving customer billing context from planning.
- [ ] The implementation leaves Sprint 10 finance tasks a clean read contract instead of hidden planning-side assumptions.
- [ ] Tests cover linkage behavior, missing-prerequisite handling, and no-duplication guarantees.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
