# Codex Prompt — US-31-T1

## Task title

**Implement reporting views for employee activity, customer revenue, and subcontractor control.**

## Objective

Build the first reporting read-model package so employee activity, customer revenue, and subcontractor control can be queried through stable SQL views or materialized views and exposed through narrow reporting APIs without creating a second write-side schema.

## Source context

- Updated proposal: section 5.7 requires controlling reports for employee activity, customer revenues, subcontractor assignments, event performance, plan hours, revenue, contribution margin, and profitability; Appendix E lists employee operations, customer and sales, and subcontractor control outputs.
- Implementation specification: section 7 explicitly recommends `rpt.employee_activity_v`, `rpt.customer_revenue_v`, and `rpt.subcontractor_control_v` as reporting views built from HR, ops, finance, audit, CRM, partner, and field tables.
- Sprint reference: `docs/sprint/sprint-11-reporting-and-hardening.md`.
- Cross-cutting rules: `AGENTS.md`, especially reporting as SQL views/materialized views, tenant-safe role scoping, finance-bridge discipline, and least-privilege external visibility.

## Dependencies

- `US-28-T1` through `US-28-T4` should already provide payroll basis source data, allowances, advances, and payroll export status signals that employee activity reporting may reference indirectly.
- `US-29-T1` through `US-29-T4` should already provide timesheet/invoice truth and customer-release context for revenue reporting.
- `US-30-T1` through `US-30-T4` should already provide subcontractor invoice-check truth and portal-safe settlement status seams.
- `US-26-T2` and `US-27-T1` through `US-27-T4` should already provide approved `finance.actual_record` truth and staged reconciliation outcomes.

## Scope of work

- Create SQL view or materialized-view definitions for `rpt.employee_activity_v`, `rpt.customer_revenue_v`, and `rpt.subcontractor_control_v`, or repository-equivalent naming if the reporting package already uses a stable naming convention.
- Implement read-only SQLAlchemy mappings or repository query objects for those views, plus Pydantic Read/ListItem/Filter schemas that keep filter and result contracts explicit.
- Model employee activity with at least confirmed vs unconfirmed shifts, login frequency, applications, absences, under/overtime, and other fields the proposal/report catalogue already expects from normalized upstream data.
- Model customer revenue with at least tenant-scoped customer/order references, planning mode or operational context, invoice status, due dates, revenue totals, and any safe profitability precursor columns that do not duplicate later write-side finance truth.
- Model subcontractor control with at least released/assigned shift counts, actual hours, attendance-evidence markers, invoice-check variance signals, and compliance-completeness indicators sourced from partner/field/finance truth.
- Expose narrow reporting service/query endpoints with deterministic tenant and role filters, pagination, sorting, and date-range handling suitable for later dashboard and export work.
- Add tests that prove the views are reproducible from transactional tables, respect tenant isolation, and return stable filter/result contracts for later UI consumers.

## Preferred file targets

- `modules/reporting/` SQL definitions, view mappings, query services, schemas, routers, and tests.
- Alembic revisions or repository-equivalent database migration files for creating views/materialized views and any supporting indexes required immediately for reasonable performance.
- Shared reporting filter enums/value objects only where they are reusable across later reporting tasks.
- No substantial Vben UI work in this task.

## Hard constraints

- Do not introduce a second write-side reporting schema or denormalized shadow tables for these reports.
- Keep report definitions rooted in transactional truth. When aggregation is needed, do it in SQL views/materialized views or query-layer projections rather than manual ad hoc service joins scattered across endpoints.
- Preserve tenant isolation and role scoping. Internal reporting consumers may see more than portal users, but the contracts must still remain least-privilege by role.
- Do not expose personal names or partner/customer data more broadly than the underlying visibility model allows; keep customer-facing masking rules compatible with the proposal even if the first consumers are internal.
- Do not broaden this task into dashboard rendering, scheduled delivery, or hardening work planned in later Sprint 11 tasks.
- Any user-facing labels or error messages introduced for filter values or API-visible enums must follow German default and English secondary localization behavior.

## Expected implementation outputs

- View/materialized-view definitions for employee activity, customer revenue, and subcontractor control.
- Read-only reporting services/endpoints and explicit filter/result schemas.
- Tests proving reproducibility, scope safety, and stable contracts for later UI/export consumers.

## Non-goals

- Do not implement planning-performance, payroll-basis, or profitability-focused reporting here; that belongs to `US-31-T2`.
- Do not build Vben dashboards or downloadable export screens here; that belongs to `US-31-T3`.
- Do not implement scheduled export delivery, compliance/QM reports, or security-activity views here; those belong to `US-32-*` tasks.
- Do not treat this as a generic analytics or BI warehouse project.

## Verification checklist

- [ ] The three reporting views compile cleanly and can be queried with deterministic filters.
- [ ] Results are derived from transactional truth and do not require manual shadow-state maintenance.
- [ ] Tenant and role scope are enforced on every endpoint/query surface that exposes reporting data.
- [ ] Schemas and query services are stable enough for later dashboard and export layers.
- [ ] Tests cover reproducibility, scope safety, and representative edge cases for empty and populated periods.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
