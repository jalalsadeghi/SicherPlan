# Codex Prompt — US-18-T4

## Task title

**Optimize indexes and query paths for board-style planning performance.**

## Objective

Prepare the planning backbone for high-volume dispatcher usage by adding the right indexes, loading strategies, and query paths for board-style schedule views before staffing and field modules amplify query pressure.

## Source context

- Updated proposal: section 5.5 operational tools including real-time staffing overview, traffic-light coverage, planning communication, and copy functions.
- Implementation specification: Phase E requires indexes for board-style planning queries before UAT; `ops.shift` includes an index by `shift_plan_id + starts_at`; package/aggregate notes stress explicit loading strategies and no second write-side reporting schema.
- Sprint reference: `docs/sprint/sprint-06-ops-planning-foundation.md`.
- Cross-cutting rules: `AGENTS.md`, especially reporting and SQLAlchemy/Pydantic guidance.

## Dependencies

- `US-17-T1`, `US-17-T2`, `US-18-T1`, and `US-18-T2` must already provide the order/planning/shift tables and core query contracts.
- Keep Sprint 7 and later in mind because demand groups, assignments, and released portal/mobile reads will add more joins soon.
- Use the current repo’s PostgreSQL and SQLAlchemy patterns; do not assume a different storage engine or ORM style.

## Scope of work

- Review and optimize the key planning query paths for dispatcher-facing board/list/calendar views, especially queries filtered by tenant, planning record, order, date/time range, status, planning mode, and visibility/workforce scope.
- Add or adjust database indexes that materially improve the expected planning-board access patterns while staying consistent with the normalized schema and avoiding speculative over-indexing.
- Refine SQLAlchemy loading strategies and repository/service query composition so list views and board slices fetch only what they actually need.
- Where the repository already has query-service projections or list-item DTOs, tune them for board usage instead of eager-loading large nested aggregates.
- Add tests or lightweight performance assertions for the critical query paths where the repo supports them, and document the rationale for the key indexes/joins that were introduced.
- If the repo has explain-plan tooling, use it. If not, still leave the code and tests in a state that makes index intent obvious to future maintainers.

## Preferred file targets

- Alembic migration(s) for performance indexes and constraint refinements.
- `modules/planning/` repositories/query services, SQLAlchemy loading options, and tests.
- Small read-DTO or query-helper additions where they reduce over-fetching for planning-board views.
- No major UI work in this task.

## Hard constraints

- Do not create a second write-side reporting schema or planning shadow tables just to speed up one screen.
- Do not denormalize commercial, HR, partner, or field truth into planning tables for convenience.
- Preserve tenant and role scope in every optimized query path.
- Favor explainable, maintainable indexes and loading strategies over premature micro-optimization.
- Do not bypass future `finance.actual_record` or reporting-view guidance; this task is about operational planning reads, not finance/reporting denormalization.
- Do not turn optimization into a silent behavior change that alters release, visibility, or archive semantics.

## Expected implementation outputs

- A set of board-oriented indexes and tuned query paths for the planning module.
- Reduced over-fetching and clearer query contracts for dispatcher-facing planning views.
- Tests or documented evidence showing the important planning queries are ready for heavier usage in later sprints.

## Non-goals

- Do not implement full staffing-board coverage metrics or demand/assignment query logic here; that comes in Sprint 7.
- Do not build analytics/reporting views here.
- Do not invent caching layers, search engines, or separate read stores unless the existing repo already has an approved pattern that this task explicitly needs.
- Do not broaden this task into frontend polish or drag-and-drop board behavior.

## Verification checklist

- [ ] The main planning-board query paths are identifiable, scoped, and supported by intentional indexes.
- [ ] ORM loading strategies avoid unnecessary deep eager loading for list/board views.
- [ ] Tenant and permission filters are still applied correctly after optimization changes.
- [ ] The planning module remains normalized and does not introduce shadow write-side copies.
- [ ] Tests or documented query evidence cover the most important optimized paths.

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
