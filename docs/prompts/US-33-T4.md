# Codex Prompt — US-33-T4

## Task title

**Run performance, penetration-ready, and resilience test cycles with remediation backlog.**

## Objective

Execute the final Sprint 11 hardening cycle by running performance, security-ready, and resilience-oriented validation suites and turning the findings into a prioritized remediation backlog that prepares the platform for UAT, migration rehearsal, and production cutover.

## Source context

- Updated proposal: Phase 6 is reporting, rollout, and hardening; technical quality requirements emphasize tenant isolation, auditability, secure monitoring, backup/restore, and extensibility; acceptance requires key reports and compliance exports to be reproducible.
- Implementation specification: Phase H is the final reporting/hardening wave after the write model is complete, which makes it the right time to validate the actual integrated system rather than isolated module assumptions.
- Sprint reference: `docs/sprint/sprint-11-reporting-and-hardening.md`.
- Cross-cutting rules: `AGENTS.md`, especially audit-safe evidence, least-privilege visibility, append-only raw evidence, and readiness for UAT/go-live without architectural surprises.

## Dependencies

- `US-33-T1` through `US-33-T3` must already provide tuned queries, optional RLS where chosen, and hardening drill/runbook coverage.
- `US-31-*` and `US-32-*` must already provide the reporting, compliance, and security surfaces that now need integrated validation.
- `US-2-*` and repository-equivalent CI/observability work must already provide baseline test/deployment/monitoring patterns.
- Any benchmark, smoke-test, or security-check helpers added earlier should be reused here so Sprint 11 closes with one coherent readiness package rather than scattered scripts.

## Scope of work

- Assemble and run performance-test scenarios that cover representative integrated workflows, especially heavy reporting, export generation, audit/login queries, document access, and selected operational paths.
- Run penetration-ready or security-readiness checks appropriate for the repository, such as auth/permission misuse cases, scope-escape attempts, document/download abuse paths, RLS validation where enabled, and insecure default detection.
- Run resilience-oriented validations for queue/outbox retries, export generation failure/retry paths, document/version access during degraded conditions, and other representative failure modes already modeled in the repository.
- Capture findings in a structured remediation backlog inside the repository, with severity, affected area, reproduction notes, owner suggestions, and whether the issue blocks Sprint 12 migration/UAT/go-live work.
- Fix only narrow, low-risk blockers inside this task if the repository workflow allows it; otherwise keep the focus on truthful discovery, prioritization, and traceable follow-up backlog items.
- Document the final hardening status so Sprint 12 starts with a grounded view of outstanding risk instead of an implied all-green posture.

## Preferred file targets

- Benchmark/security/resilience test configs or scripts under a repository-appropriate QA/hardening area.
- A remediation backlog document or machine-readable issue file kept in-repo.
- Any tiny prerequisite fixes that are clearly contained and directly tied to the executed validation cycle.
- Docs summarizing Sprint 11 hardening outcomes and carry-forward risks.

## Hard constraints

- Do not hide or downplay real findings to make Sprint 11 look complete; the remediation backlog is part of the deliverable.
- Keep performance and security test scope representative of the implemented platform, not a synthetic microbenchmark disconnected from real usage.
- Where optional controls such as RLS are disabled, say so clearly and test the actual deployed posture rather than the hoped-for future posture.
- Do not silently add broad product changes under the label of hardening if they belong in a later planned backlog item.
- Maintain tenant isolation, role scoping, and auditability even in test harnesses and failure simulations.

## Expected implementation outputs

- Performance/security/resilience validation artifacts or scripts.
- A prioritized remediation backlog with clear severity and follow-up ownership hints.
- Sprint-hardening summary docs that make UAT/go-live readiness explicit and honest.
- Any narrow blocker fixes that were truly necessary and contained.

## Non-goals

- Do not attempt to finish every remediation item inside this task.
- Do not invent synthetic pass criteria unrelated to the proposal/spec acceptance intent.
- Do not turn this into a full external penetration-test engagement replacement.
- Do not leave findings in chat-only form; persist them in the repository.

## Verification checklist

- [ ] Integrated performance, security-ready, and resilience checks have actually been run or automated in-repo.
- [ ] Findings are captured in a remediation backlog with severity and follow-up guidance.
- [ ] Any blocker risks for Sprint 12 are made explicit.
- [ ] Validation artifacts and docs are sufficient for handoff into migration/UAT/go-live work.
- [ ] The task ends with a truthful hardening posture rather than an assumed green status.

- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
