# Codex Prompt — US-25-T1

## Task title

**Implement time capture devices, device types, and context policies.**

## Objective

Establish the configuration backbone for time capture so the repository has a tenant-safe, migration-ready model for shared terminals, scanners, and context policies that can later govern browser, mobile, and station-bound capture flows across sites, shifts, routes, and planning contexts.

## Source context

- Updated proposal: section 5.6 time capture covers browser, mobile, shared terminals, stationary terminals, and QR/barcode/RFID/NFC workflows; Appendix E adds location/IP/terminal rule behavior.
- Implementation specification: `field.time_capture_device` and `field.time_capture_policy` in section 4.7; cross-cutting modeling rules in section 2; service-boundary rules in section 5 saying field capture may reference released ops context but must not rewrite planning/commercial truth.
- Dienstplanmacher product sheets: Terminal C / X, homepage/browser capture, fixed-IP terminal restrictions, app geolocation, and context-bound terminal assignment reinforce the need for device and policy configuration before ingest logic.
- Sprint reference: `docs/sprint/sprint-09-time-capture-and-actuals.md`.
- Cross-cutting rules: `AGENTS.md`, especially tenant safety, append-only evidence principles, DE default / EN secondary UI behavior, and explicit bounded-context ownership.

## Dependencies

- `US-16-T1` and related ops master-data tasks should already provide sites, route context, and the location backbone that time-capture devices may attach to.
- `US-18-T1` and `US-19-T1` should already provide stable planning/shift/assignment identifiers that capture policies can point at indirectly through approved context references.
- `US-3-T1`, `US-3-T3`, and `US-5-T3` should already provide the base web shell, localization, and permission scaffolding if an admin UI is introduced here.
- If the repository already uses lookup-backed technical codes, reuse that pattern for device types and enforce-mode values rather than inventing ad hoc string constants in multiple places.

## Scope of work

- Implement the persistent model, migrations, SQLAlchemy entities, and Pydantic schemas for `field.time_capture_device` and `field.time_capture_policy` or the repository-equivalent module boundary.
- Support device registration for shared terminals/scanners with explicit device codes, device types, optional site linkage, activation state, and optional fixed IP/CIDR restrictions.
- Support policy records that can apply to a shift, site, planning context, patrol route, or other approved capture scope via the intentional generic context pattern from the implementation specification.
- Define clear service-level validation rules for policy uniqueness, active-flag behavior, device-to-site compatibility, and context ownership inside the current tenant.
- Where repository conventions allow it, add admin-facing backend endpoints and narrow Vben CRUD screens for configuring devices and policies, including DE/EN labels and permission checks.
- Add automated tests for uniqueness, tenant scoping, policy-context validation, activation/archive behavior, and device/policy retrieval by active operational context.

## Preferred file targets

- `modules/field_execution` or the repository-equivalent field/time-capture package for models, schemas, services, and routers.
- Alembic migrations for the new device/policy tables and required indexes/constraints.
- Admin-side Vben screens only if the repo already has a field-ops configuration area; otherwise limit this task to backend contracts plus minimal API documentation.
- Tests for migration, service validation, and any admin API endpoints added.

## Hard constraints

- Keep `field.time_capture_device` and `field.time_capture_policy` in the field-capture bounded context; do not move policy truth into planning or HR tables.
- Policies may reference operational context, but they must not mutate shift demand, release state, route design, or commercial rules.
- Preserve tenant isolation on every query path and never allow device or policy records to reference foreign-tenant context objects.
- Do not broaden this task into raw time-event ingest, attendance derivation, or approval flow work; those belong to later tasks in this sprint.
- If UI work is included, German must remain the default language and English must be supplied for every new user-facing string.

## Expected implementation outputs

- Migrations and domain models for registered time-capture devices and context policies.
- Create/Update/Read/List schemas and service-layer validation for device/policy management.
- Any narrow admin/API surface needed to maintain these records safely.
- Tests proving tenant safety, uniqueness rules, context validation, and active/inactive behavior.

## Non-goals

- Do not implement actual event ingestion, pairing logic, or payroll/finance derivation in this task.
- Do not implement terminal dashboards, live attendance boards, or supervisor confirmation UX here.
- Do not hardcode one single planning mode if the schema is intended to support event, object/site, fair, and patrol contexts.

## Verification checklist

- [ ] Device and policy tables exist with the intended constraints, indexes, and tenant-safe relationships.
- [ ] Active devices can be scoped to sites and optional network restrictions without leaking across tenants.
- [ ] Policies can target approved operational contexts and reject invalid or cross-tenant references.
- [ ] Any added UI/API paths are localized and permission-guarded.
- [ ] Tests cover CRUD, uniqueness, tenant safety, and context resolution behavior.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
