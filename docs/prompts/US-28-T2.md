# Codex Prompt — US-28-T2

## Task title

**Implement employee pay profiles, overrides, allowances, advances, and time-account hooks.**

## Objective

Add the employee-specific payroll layer so approved actuals can be enriched by effective pay profiles, override rates, recurring allowance/advance rules, and time-account references without leaking HR-private data or creating duplicate payroll truth across finance and HR.

## Source context

- Updated proposal: section 5.6 payroll logic includes employee, function, and qualification-based allowances, advances, and time accounts; Appendix C employee/payroll fields include hourly wage, employment type, target hours, allowances, advances, and time-value accounts.
- Implementation specification: section 4.8 defines `finance.employee_pay_profile` and notes that `finance.actual_allowance` may originate from `hr.employee_allowance`; section 7 recommends `rpt.payroll_basis_v` as a read model over actuals, pay profiles, tariffs, and advances.
- The updated proposal also requires HR-sensitive payroll data to remain restricted to authorized HR/payroll roles, while dispatch and field roles receive only operationally necessary data.
- Sprint reference: `docs/sprint/sprint-10-payroll-billing-and-settlement.md`.
- Cross-cutting rules: `AGENTS.md`, especially finance bridge discipline, least-privilege portal/privacy rules, and centralized overlap validation for temporal records.

## Dependencies

- `US-28-T1` should already provide tariff tables and resolution primitives used by employee pay profiles.
- `US-11-T1` through `US-12-T3` should already provide employee identity, employment context, allowances, advances, and time-account records or equivalent HR-owned tables.
- `US-26-T2` and `US-27-T3` should already provide approved actuals, allowance/expense lines, and reconciliation context that payroll hooks must enrich rather than replace.
- If the repository already has a role/permission split for HR-private fields, reuse it for payroll-profile access instead of adding a parallel authorization concept.

## Scope of work

- Implement migrations, models, Pydantic schemas, services, and APIs for `finance.employee_pay_profile`, including effective dating, pay cycle, cost center, export code, and optional employee-specific base-rate override behavior.
- Add deterministic overlap prevention so one employee does not have ambiguous active payroll profiles for the same period.
- Implement finance-side hook/resolution logic that can combine employee pay profile data with tariff tables, approved actuals, reconciliation allowances, HR allowances, employee advances, and time-account rows through read-side contracts or adapter seams.
- Keep the authoritative ownership of allowances, advances, and time-account transactions where it already belongs. Finance should resolve and consume them, not duplicate or silently rewrite upstream HR truth.
- Where repository conventions allow it, expose narrow backend endpoints and optionally Vben admin pages for payroll profile maintenance, including localized DE/EN labels and role-based permission checks for HR/payroll operators.
- Add automated tests for overlap handling, resolution precedence between tariff rates and employee overrides, allowance/advance/time-account hook behavior, tenant safety, and authorization boundaries around payroll-sensitive data.

## Preferred file targets

- `modules/finance/` payroll profile services, schemas, routers, and tests.
- Read-adapter or query helper files that consume HR-owned allowance/advance/time-account data without violating bounded-context ownership.
- Alembic migrations for employee pay profile persistence and supporting indexes/constraints.
- Optional Vben payroll-profile maintenance screens only if the repository already has an approved finance-admin area.

## Hard constraints

- Do not create duplicate master tables for allowances, advances, or time accounts if HR already owns them.
- Do not expose payroll-sensitive or HR-private fields to dispatcher, customer, subcontractor, or general portal roles.
- Employee pay profile windows must be overlap-safe and deterministic.
- Keep payroll enrichment read-oriented with respect to HR tables; finance may consume upstream truth but must not mutate HR-private master records directly.
- Do not finalize export batches or provider payloads in this task.
- If user-facing maintenance is included, German must remain the default locale and English must be supplied alongside it.

## Expected implementation outputs

- Finance-owned employee pay-profile persistence and resolution services.
- Documented or implemented hook contracts for allowances, advances, and time-account consumption.
- Authorization-safe payroll maintenance APIs and optional admin UI seams.
- Tests proving overlap safety, precedence logic, and privacy/authorization boundaries.

## Non-goals

- Do not implement provider-specific payroll export files or APIs here; that belongs to `US-28-T3`.
- Do not implement payslip import/archive or payroll reporting UI here; that belongs to `US-28-T4` or later reporting work.
- Do not create net-pay or statutory tax engines in this task.
- Do not flatten HR-private payroll data into general employee or planning DTOs.

## Verification checklist

- [ ] Each employee can resolve to one effective payroll profile per period according to deterministic overlap rules.
- [ ] Override and fallback behavior between employee pay profiles and tariff tables is predictable and test-covered.
- [ ] Allowance, advance, and time-account hooks consume upstream data without duplicating or mutating HR truth.
- [ ] Payroll-sensitive maintenance and read paths are permission-scoped and least-privilege by default.
- [ ] Tests cover overlap handling, precedence rules, authorization, and tenant safety.

- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
