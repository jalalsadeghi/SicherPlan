# Codex Prompt — US-23-T4

## Task title

**Support customer and subcontractor participation in watchbook flows where enabled.**

## Objective

Extend the watchbook safely to external actors so customer and subcontractor users can participate only when the prime tenant explicitly enables it, with tight tenant/portal scoping, durable audit evidence, and no accidental exposure of broader operational or HR/finance data.

## Source context

- Updated proposal: section 5.2 online watchbook allows optional entries by customer and subcontractor users; customer and subcontractor portals are read-scoped elsewhere in sections 5.2, 5.4, and 5.5; role model in section 4 keeps external actors inside the tenant security model.
- Implementation specification: watchbook tables in section 4.7; portal visibility chain in section 5; user-role scoping in section 4.1; docs linkage in section 4.2. `field.watchbook_entry` already supports `author_user_id`, which should represent external portal actors instead of inventing extra author columns.
- Sprint reference: `docs/sprint/sprint-08-employee-app-and-field-i.md`.
- Cross-cutting rules: `AGENTS.md`, especially portal privacy rules, customer-facing name hiding by default, and strict local scope filters for customer/subcontractor actors.
- Earlier prompt context: `US-9-T1` / `US-9-T2` and `US-15-T1` / `US-15-T2` define customer/subcontractor portal auth and read boundaries; `US-23-T2` and `US-23-T3` provide the base watchbook flows and outputs.

## Dependencies

- `US-23-T2` must already provide the base watchbook aggregate and entry flows.
- `US-23-T3` should already provide released watchbook outputs and read models.
- `US-9-T1` and `US-9-T2` should already provide customer-portal authentication and released read patterns.
- `US-15-T1` and `US-15-T2` should already provide subcontractor-portal authentication and scope boundaries.
- Any write enablement must remain explicit and configuration-driven; if no approved feature flag or visibility setting exists yet, add the narrowest one consistent with the repo’s patterns.

## Scope of work

- Implement controlled participation rules that allow customer or subcontractor users to add watchbook entries only when explicitly enabled for the relevant context.
- Use the existing user-account and portal-scope model to authorize writes; external authors should be represented via `author_user_id` rather than custom contact/partner author columns.
- Restrict each external actor to their own released customer/subcontractor context and only the watchbook functions that are explicitly granted.
- Support attachments or comments only through the approved docs-backed and audit-safe pathways.
- Ensure internal review/supervisor flows remain intact and can distinguish internal from external contribution history.
- Add tests for disabled-vs-enabled participation, customer/subcontractor scope isolation, author attribution, and forbidden access to unrelated watchbooks.

## Preferred file targets

- `modules/field_execution` authorization/service files for external watchbook participation.
- Customer and subcontractor portal API/UI files only where the repo includes the relevant surfaces.
- Feature-flag or visibility-config wiring where a narrow enablement seam is needed.
- Tests for external-author scope enforcement and audit attribution.

## Hard constraints

- External participation must be opt-in and scope-bound; it is never globally on by default.
- Customer and subcontractor users remain actors inside the tenant security model and must never browse unrelated tenant data.
- External authorship must use the approved user-account pathway; do not add ad hoc author columns to the watchbook schema.
- External write participation must not expose HR-private, finance-sensitive, or unrelated operational details.
- All external contributions remain auditable and compatible with later revision-safe output generation.

## Expected implementation outputs

- Opt-in customer/subcontractor participation rules and service-layer enforcement for watchbook entries.
- Portal-facing write/read adjustments where the relevant surfaces exist.
- Audit-safe author attribution for external contributions.
- Tests proving disabled/allowed participation boundaries and actor isolation.

## Non-goals

- Do not turn customer or subcontractor portals into full operational editing workspaces.
- Do not loosen default read/name-hiding/privacy rules for external actors.
- Do not duplicate watchbook data into portal-owned tables.
- Do not bypass supervisor review or docs/audit handling for external submissions.

## Verification checklist

- [ ] Customer/subcontractor watchbook participation is disabled by default and enabled only where configured.
- [ ] External actors can interact only with their own released watchbook context.
- [ ] External entries are attributed through the approved user-account pathway and remain auditable.
- [ ] Privacy-safe output rules remain intact after external participation is enabled.
- [ ] Tests cover disabled/allowed modes, scope isolation, and external author attribution.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
