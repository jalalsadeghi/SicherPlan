# Codex Prompt — US-22-T1

## Task title

**Adapt the Prokit-based mobile shell for authenticated employee login, role guard, and profile context.**

## Objective

Extend the Sprint 1 Flutter shell into a real employee-app entry point that authenticates the user, resolves exactly one employee context, applies role-aware navigation guards, and prepares all later field workflows to run only inside the correct tenant-scoped own-record session.

## Source context

- Updated proposal: section 5.3 employee app and self-service scope; section 7 technical architecture for Flutter mobile delivery; role model in section 4 where Employee / Guard has access only to own profile, own shifts, and explicitly shared information.
- Implementation specification: IAM/session foundation in section 4.1; `hr.employee.user_id` linkage and employee master ownership in section 4.4; portal-visibility chain in section 5; aggregate/package map pointing mobile/employee work to `modules/employees`, `modules/iam`, and mobile app packages.
- Sprint reference: `docs/sprint/sprint-08-employee-app-and-field-i.md`.
- Cross-cutting rules: `AGENTS.md`, especially Prokit as interaction reference, DE default / EN secondary, exact theme tokens, scoped operational data, and no HR-private leakage into mobile surfaces.
- Uploaded Dienstplanmacher employee-app material from prior context: authenticated mobile entry, schedule context, self-service access, and QR/barcode-based field usage expectations.

## Dependencies

- `US-3-T2` must already provide the base Flutter shell and Prokit-style navigation structure.
- `US-5-T1`, `US-5-T2`, and `US-5-T3` should already provide user identities, sessions, and tenant/role-aware authorization semantics.
- `US-11-T3` must already link employee records to IAM users for app access.
- `US-3-T4` should already provide the DE/EN localization baseline that the login and shell screens will reuse.
- Keep the shell forward-compatible with `US-21-T4` released-schedule contracts and later field modules even if those screens are not fully wired in this task.

## Scope of work

- Implement mobile login/session bootstrap that uses the approved backend auth flow and secure token storage, including logout, refresh, and inactive/blocked session handling.
- Resolve the authenticated user into exactly one employee-app context and surface only the operational profile fields needed for later self-service and field flows.
- Add role guards and navigation guards so only approved employee/guard-oriented actors enter the employee shell, while the structure remains extensible for field-supervisor reuse if the repo later supports it.
- Create a stable app-shell state object for tenant, employee identity, branch/mandate hints where relevant, release-safe profile metadata, and device/session status.
- Provide placeholder but real guarded entry points for schedule, information feed, documents, watchbook, patrol, and settings/profile screens so later Sprint 8 tasks can attach without reworking the shell.
- Add tests for unauthenticated access, wrong-role access, inactive employee linkage, tenant mismatch, and successful own-context bootstrap.

## Preferred file targets

- Flutter mobile auth/shell files under the actual app path, especially feature folders for auth, employee shell, profile context, and guarded navigation.
- IAM or employee API/service files only where a minimal mobile-context endpoint or DTO adjustment is required.
- Secure-storage/session helpers and route guards under the mobile app structure.
- Tests for auth bootstrap, route guarding, and employee-context resolution.

## Hard constraints

- Do not expose HR-private profile fields such as tax, bank, religion, health-insurance, or social-insurance data through the mobile shell context.
- The mobile shell must remain own-record scoped; no list endpoints or broad tenant datasets should become reachable through this login flow.
- Use Prokit as the interaction baseline, but do not dump a template into production unchanged or let template naming leak into product terminology.
- Keep auth/session handling aligned with the repository IAM truth rather than inventing a parallel mobile-only identity layer.
- Preserve DE default / EN secondary behavior and the shared theme-token contract even if full styling work is completed in `US-22-T4`.

## Expected implementation outputs

- Authenticated employee mobile shell with guarded navigation and secure session lifecycle handling.
- Employee-context resolution contract that exposes only allowed operational profile data.
- Route guards and shell state that later schedule, watchbook, patrol, and document features can reuse.
- Tests for session bootstrap, own-context enforcement, and role/tenant guard behavior.

## Non-goals

- Do not build the full schedule, confirm/decline, event-application, or patrol flows here.
- Do not expose payroll-sensitive or HR-private data in order to make the shell feel “complete.”
- Do not implement push notifications, document downloads, or QR/barcode rendering in this task.
- Do not create a second mobile-only employee table or cached shadow identity store.

## Verification checklist

- [ ] Unauthenticated users cannot enter guarded employee-app routes.
- [ ] Authenticated users without a valid employee linkage are rejected or handled safely.
- [ ] The resolved mobile profile context is tenant-scoped and contains only allowed operational fields.
- [ ] The shell exposes stable guarded entry points for later Sprint 8 features without leaking unfinished data.
- [ ] Tests cover wrong-role, inactive-linkage, tenant-mismatch, and happy-path bootstrap behavior.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
