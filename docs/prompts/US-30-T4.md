# Codex Prompt — US-30-T4

## Task title

**Expose invoice-check status and approved basis to the subcontractor portal.**

## Objective

Provide subcontractor-facing visibility into invoice-check status and approved settlement basis so partner users can see what the prime company accepted for their own work without exposing internal controller notes, unreleased finance detail, or other partners’ data.

## Source context

- Updated proposal: subcontractor portal users may view invoice or reconciliation status according to permission; workflow 6.3 ends with subcontractor invoice status and performance reports visible in the control layer.
- Implementation specification: the portal visibility chain constrains external users by tenant scope plus local role scope; `finance.subcontractor_invoice_check` and `_line` contain the approved-basis data, while partner master and release context remain upstream sources.
- Earlier uploaded subcontractor-management materials emphasized partner portal visibility, invoice verification status, and scoped access rather than unrestricted finance detail.
- Sprint reference: `docs/sprint/sprint-10-payroll-billing-and-settlement.md`.
- Cross-cutting rules: `AGENTS.md`, especially least-privilege portal visibility, subcontractor scope boundaries, DE default / EN secondary behavior, and audit-safe finance outputs.

## Dependencies

- `US-15-T1` and `US-15-T2` should already provide subcontractor portal authentication, scope resolution, and released-work read patterns.
- `US-30-T1` through `US-30-T3` should already provide invoice-check data, comparison results, and review-state handling.
- `US-14-T4` should already provide partner-side worker/read contracts that can inform safe display of approved basis detail.
- If the repository already has subcontractor-portal document/download helpers or status widgets, reuse them instead of creating a second portal pattern.

## Scope of work

- Implement portal-scoped read services and endpoints for subcontractor invoice-check list/detail views, including period, status, submitted invoice reference, approved hours/cost basis, and safe line-level variance summaries where appropriate.
- Filter every read by tenant, subcontractor identity, and release/visibility rules so portal users see only their own company’s invoice checks.
- Expose the approved settlement basis and status in a partner-understandable form while hiding internal controller notes, private review metadata, or other finance details not intended for external visibility.
- Return stable DTOs suitable for the existing subcontractor portal, including downloadable summaries or document references if the repository already produces approved invoice-check PDFs or attachments.
- Where the repository already includes subcontractor portal pages, add narrow UI components with localized DE/EN strings for invoice-check status and approved-basis visibility.
- Add tests for portal scope filtering, visibility gating, hidden internal-note behavior, DTO stability, and document/download authorization where applicable.

## Preferred file targets

- Subcontractor-portal read services, DTOs, routers, and tests under `modules/subcontractors/`, `modules/finance/`, or the repository-equivalent portal-read package.
- Portal document-download controllers or helpers that already enforce subcontractor scope.
- Optional subcontractor-portal UI pages/components for invoice-check status visibility if the repo already has a portal front end.
- Tests covering authorization, masking, and safe external visibility.

## Hard constraints

- Portal users must see only their own subcontractor company’s invoice checks and only the released/approved visibility that the platform intends to expose.
- Do not expose internal controller notes, private reconciliation metadata, or other subcontractors’ data in the portal.
- Do not create partner-facing write actions such as approval overrides or finance-note edits unless explicitly approved elsewhere in the repository.
- Keep the portal contract least-privilege and stable; expose only the approved basis and status needed for collaboration.
- Any added portal UI strings must follow German-default and English-secondary localization rules.
- Do not introduce shadow write-side copies of finance data solely for portal rendering.

## Expected implementation outputs

- Portal-scoped invoice-check status and approved-basis read models/endpoints.
- Optional localized subcontractor-portal UI surfaces for finance-control visibility.
- Authorized access to any approved invoice-check documents or summaries already supported by the repository.
- Tests proving scope safety, hidden internal metadata, and stable partner-facing DTOs.

## Non-goals

- Do not implement invoice submission, dispute workflows, or portal-side finance edits in this task unless the repository already has an explicitly approved requirement.
- Do not expose full internal variance analysis or controller notes if they are not intended for partner visibility.
- Do not broaden the task into overall subcontractor reporting dashboards planned for later reporting sprints.
- Do not recreate invoice-check calculations in the portal layer.

## Verification checklist

- [ ] Subcontractor portal users can see only their own company’s released invoice-check status and approved basis.
- [ ] Internal controller notes and unrelated finance metadata remain hidden.
- [ ] Any document/download access is authorized and scoped to the correct subcontractor.
- [ ] Portal DTOs are stable and suitable for later portal/reporting expansion.
- [ ] Tests cover scope filtering, hidden-field behavior, and document/download authorization.
