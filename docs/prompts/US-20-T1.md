# Codex Prompt — US-20-T1

## Task title

**Implement validation services for qualification/function match and certificate validity.**

## Objective

Build the first planning-validation services that check whether a proposed assignment satisfies function/qualification requirements and whether the relevant certificates remain valid at shift time, using policy-driven block/warn outcomes.

## Source context

- Updated proposal: section 5.5 validation rules and Appendix E entries for qualification/function match and certificate validity.
- Implementation specification: section 6 validation-to-table mapping for qualification/function match and certificate validity; section 5 employee/subcontractor relationship chains; section 3 cross-service write discipline.
- Sprint reference: `docs/sprint/sprint-07-staffing-engine.md`.
- Cross-cutting rules: `AGENTS.md`, especially planning->HR and planning->partner read-only rules.

## Dependencies

- `US-19-T1` and `US-19-T2` must already provide demand groups, assignments, and assignment-command seams.
- `US-12-T1` must already provide employee qualification and proof structures.
- `US-14-T1` and `US-14-T4` must already provide subcontractor-worker qualification truth and read models.
- `US-4-T1` should already provide `compliance.requirement_rule` or the equivalent rule source used to decide block vs warn behavior.

## Scope of work

- Create a reusable planning-validation service layer and result DTO shape that can be invoked before assignment save and before release.
- Implement qualification/function matching by comparing demand-group requirements against employee or subcontractor-worker qualification/function truth using read-only access to HR and partner contexts.
- Implement certificate-validity checks against qualification/document expiry fields using shift timing as the enforcement reference.
- Return explicit structured validation results with rule code, severity, message key, source references, and actor context so later UI/release workflows can present them consistently.
- Where policy settings already exist, drive block-vs-warn outcomes from those settings. Where the repository has only default policy today, implement that default behind a clear seam rather than hardcoding it everywhere.
- Add backend tests for positive/negative cases across both employee and subcontractor-worker actors, including certificate-expiry edge cases on shift boundaries.

## Preferred file targets

- `modules/planning/` validation services, DTOs, service hooks, and tests.
- Small shared policy/enum helpers if the repository already has a compliance-rule package.
- No substantial UI work in this task.

## Hard constraints

- Planning may read HR and partner qualification/compliance truth, but it must not mutate those source tables.
- Do not scatter validation logic across controllers and UI layers; keep the primary logic close to planning command handlers/services.
- Use structured validation results; do not reduce business rules to generic HTTP errors or stringly typed booleans.
- Do not implement mandatory-doc, customer-block, double-booking, or threshold logic in this task; keep the rule set narrow and traceable.
- Do not persist override evidence here; that belongs to `US-20-T4`.
- Any new message keys or labels must preserve German default and English secondary behavior.

## Expected implementation outputs

- A reusable validation-service foundation for qualification/function and certificate-validity checks.
- Structured validation results suitable for later board, release, and override flows.
- Tests covering both internal and subcontractor-worker validation scenarios.

## Non-goals

- Do not implement mandatory documents, customer blocks, or double booking here; that belongs to `US-20-T2`.
- Do not implement rest/max-hours/earnings/min-staffing checks here; that belongs to `US-20-T3`.
- Do not implement override persistence or UI alerts here; that belongs to `US-20-T4`.
- Do not broaden this task into release-state transitions or staffing-board UI work.

## Verification checklist

- [ ] Assignments can be validated against required function/qualification truth for both employees and subcontractor workers.
- [ ] Certificate-validity checks use shift timing correctly and produce policy-driven block/warn results.
- [ ] Validation outputs are structured, traceable, and reusable by later command/release/UI tasks.
- [ ] No write operations occur against HR or partner master/compliance tables.
- [ ] Tests cover the key valid, invalid, and expiry-edge scenarios.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
