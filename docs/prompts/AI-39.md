# AI-39 — Fix OpenAI Responses Tool Name Compatibility and Tool-Call Mapping

## Purpose

The assistant now reaches the OpenAI API, but the OpenAI Responses request fails with:

```text
HTTP/1.1 400 Bad Request
Invalid 'tools[0].name': string does not match pattern.
Expected a string that matches the pattern '^[a-zA-Z0-9_-]+$'.
```

This means the backend is sending at least one tool name to OpenAI with an invalid character, most likely a dot (`.`), for example:

```text
assistant.get_current_user_capabilities
navigation.build_allowed_link
planning.find_shifts
employees.search_employee_by_name
```

OpenAI tool/function names may contain only letters, numbers, underscore, and hyphen. They must not contain dots.

The internal SicherPlan tool registry may keep readable internal names with dots, but the provider payload sent to OpenAI must use provider-safe names and must map them back to the internal tool names when OpenAI requests a tool call.

---

## Current Evidence

The API call now reaches OpenAI:

```text
POST https://api.openai.com/v1/responses -> 400 Bad Request
```

OpenAI returns:

```text
Invalid 'tools[0].name': string does not match pattern '^[a-zA-Z0-9_-]+$'
```

The current assistant tool registry exposes tool definitions based on `definition.name`. In the current repository, `AssistantToolRegistry.list_available_tools()` returns each registered tool name directly from the internal tool definition. This is risky because internal names are likely dotted names and therefore invalid for OpenAI tool names.

## Step 2 — Add Provider-Safe Tool Name Adapter

Create a small adapter module, for example:

```text
backend/app/modules/assistant/tool_name_adapter.py
```

or place it in the provider layer if that better matches the current structure.

It must provide:

```python
def to_provider_tool_name(internal_name: str) -> str:
    ...

def build_provider_tool_name_map(internal_tool_names: list[str]) -> dict[str, str]:
    # returns {provider_safe_name: internal_name}
```

Rules:

1. Provider tool name must match:

```text
^[a-zA-Z0-9_-]+$
```

2. Replace invalid characters with `_`.
3. Ensure uniqueness.
4. Keep names stable across requests.
5. If two internal names collide after sanitization, append a deterministic suffix.
6. Preserve a mapping from provider-safe name back to internal name.
7. Internal tool audit must still use the internal tool name, not the provider-safe alias.

Example mapping:

```text
assistant.get_current_user_capabilities -> assistant_get_current_user_capabilities
navigation.build_allowed_link -> navigation_build_allowed_link
planning.find_shifts -> planning_find_shifts
employees.search_employee_by_name -> employees_search_employee_by_name
```

## Step 3 — Transform Tool Payload Before Sending to OpenAI

Before calling OpenAI, transform every tool definition so that the sent tool name is provider-safe.

If the current internal format is:

```json
{
  "type": "function",
  "function": {
    "name": "assistant.get_current_user_capabilities",
    "description": "...",
    "parameters": {}
  }
}
```

then either convert it to the exact Responses API-supported format or at minimum ensure the name OpenAI receives is safe.

Expected provider payload must contain no dotted tool names.

Codex must verify the exact format required by the installed OpenAI SDK and adjust accordingly.

## Step 5 — Translate Tool Calls Back Before Execution

When OpenAI returns a tool call such as:

```json
{
  "name": "planning_find_shifts",
  "arguments": "{...}"
}
```

the assistant service must translate it back before execution:

```python
internal_name = tool_name_map["planning_find_shifts"]
tool_registry.execute_tool(tool_name=internal_name, input_data=arguments, ...)
```

If OpenAI returns an unknown provider tool name:
- do not execute it;
- record a safe failed tool result;
- ask the model to answer with available context or return a safe error;
- do not expose stack traces.
