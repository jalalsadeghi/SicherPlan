# AI-25 — Backend Security and Integration Tests

## Purpose

Add comprehensive backend security and integration tests for the SicherPlan AI Assistant baseline.

This task validates the backend behavior delivered by tasks `AI-04` through `AI-21`, including assistant capabilities, conversation persistence, same-language responses, out-of-scope handling, provider mock mode, knowledge retrieval, tool registry policy, permission enforcement, navigation-link safety, prompt-injection resistance, cross-tenant isolation, and the primary Markus shift visibility diagnostic.

This task is test-focused. It must not add new assistant capabilities beyond the scope already defined in `docs/sprint/AI-Assistant.md`.

## Scope

Add backend tests for the AI Assistant feature.

The tests must run in local and CI environments without a real OpenAI API key. Use `SP_AI_PROVIDER=mock` or the repository's equivalent test configuration.

The tests must verify security boundaries independently of model behavior. Do not rely on a real LLM response to prove authorization, redaction, or tenant isolation.

## Implementation Rules

1. Do not call the real OpenAI API in tests.
2. Do not require a real OpenAI API key.
3. Do not weaken production authorization code to make tests pass.
4. Do not bypass IAM/authz patterns.
5. Do not add write actions to the assistant.
6. Do not add direct SQL execution for the model.
7. Do not expose secrets, JWTs, reset tokens, or raw credentials in fixtures or expected outputs.
8. Do not use snapshots that hide security assertions.
9. Prefer explicit assertions for tenant, permission, redaction, and tool-audit behavior.
10. If a required fixture is too large, create minimal domain rows only for the test scenario.
11. If an existing module does not expose the needed repository/service function, add a narrow read-only test-safe query path only if necessary and explain why.
