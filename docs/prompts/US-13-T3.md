# Codex Prompt — US-13-T3

## Task title

**Add subcontractor history, attachments, and status lifecycle controls**

## Objective

Extend the subcontractor aggregate with append-only history, centralized attachments, and explicit lifecycle controls so internal users can track processing, invoice discussions, and state changes without losing auditability.

## Source context

- Updated proposal: section 5.4 history, control, and invoice verification for subcontractors; section 6.3 step 1 and step 6/7, which require a durable commercial and process trail around collaboration and validation.
- Implementation specification: section 4.5 `partner.subcontractor_history_entry`; section 2 lifecycle and archive conventions; shared docs-link pattern in section 4.2.
- Uploaded Dienstplanmacher subcontractor source material: the Subunternehmerverwaltung sheet explicitly describes subcontractor history containing processing and invoicing information, user-change traceability, and custom entries with attachments.
- Sprint reference: `docs/sprint/sprint-05-subcontractor-management.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-13-T1` must already provide the subcontractor master tables and API base.
- `US-6-T1` must already provide `docs.document`, `docs.document_version`, and `docs.document_link` patterns for attachment reuse.
- `US-5-T4` should already provide the audit-event baseline for sensitive lifecycle changes and actor attribution.

## Scope of work

- Create migrations/models/schemas/services/endpoints for `partner.subcontractor_history_entry` and its attachment-link behavior via `docs.document_link`.
- Support append-only timeline entries for processing notes, invoice discussions, manual commentary, portal enablement notes, and lifecycle-relevant events, including title/body, entry type, occurred-at timestamp, and author attribution.
- Implement archive/reactivate and other approved lifecycle transitions for the subcontractor master using explicit status semantics, optimistic locking, and clear state-transition rules.
- Ensure lifecycle actions can write companion history entries and companion audit events when the repo's audit seam expects both business history and security/compliance audit.
- Add UI support inside the admin detail page for viewing the ordered history timeline, uploading/attaching documents through the docs service, and executing approved lifecycle controls.
- Add tests for append-only history behavior, attachment linking, ordered retrieval, archive/reactivate transitions, and actor attribution.

## Preferred file targets

- Partner migrations, models, schemas, services, and tests under `modules/subcontractors/` or the repo equivalent.
- Docs-link integration helpers only where necessary for attachment reuse.
- Vben detail-page tabs/components for history timeline and lifecycle actions.

## Hard constraints

- History entries are append-only; do not edit or hard-delete prior timeline rows to 'correct' the past.
- Use the centralized docs service for attachments; do not add a second file-storage path inside the partner module.
- Keep lifecycle states machine-readable and service-enforced, not purely UI-driven.
- Do not turn this task into invoice-check accounting logic; commercial verification rows belong to later finance work even if history references invoice discussions.
- Any new end-user strings, activity labels, or transition messages must preserve German default and English secondary behavior.

## Expected implementation outputs

- A subcontractor timeline/history capability with document attachments.
- Explicit, tested lifecycle transitions for archive/reactivate or similar approved states.
- Admin UI support for timeline review, attachment management, and lifecycle actions.

## Non-goals

- Do not redesign the full permission matrix here; that is `US-13-T4`.
- Do not implement subcontractor workforce history here; this task is about the company-level master timeline.
- Do not build finance invoice-check calculations or line-item variance analysis here; those belong to Sprint 10.

## Verification checklist

- [ ] History rows are append-only, ordered correctly, and attributable to a known user where applicable.
- [ ] Attachments are linked through the docs backbone and remain accessible according to scope rules.
- [ ] Archive/reactivate or equivalent lifecycle actions enforce allowed transitions and optimistic-lock safety.
- [ ] UI timeline and status actions reflect backend truth instead of inventing client-only state.
- [ ] Sensitive lifecycle changes also produce audit-safe evidence where the audit layer expects it.
