# Codex Prompt — US-30-T3

## Task title

**Build controller UI for review, notes, approval state, and exception handling.**

## Objective

Deliver the Vben-based controller workspace so finance/control users can review subcontractor invoice checks, inspect variance lines, record notes, manage approval or exception status, and keep the full control flow auditable and role-scoped.

## Source context

- Updated proposal: controller/QM and accounting roles are part of the web platform target audience; subcontractor history, control, and invoice verification require visible, auditable review workflows.
- Implementation specification: `finance.subcontractor_invoice_check` and `_line` provide the workflow backbone; audit and permission rules from core/IAM still apply to finance-sensitive review actions.
- The Vben Admin web-admin shell is the approved baseline for route guards, list/detail pages, reusable CRUD patterns, and localized admin workflows.
- Sprint reference: `docs/sprint/sprint-10-payroll-billing-and-settlement.md`.
- Cross-cutting rules: `AGENTS.md`, especially Vben conventions, DE default / EN secondary localization, least-privilege visibility, and audit-safe finance operations.

## Dependencies

- `US-30-T1` and `US-30-T2` should already provide invoice-check headers/lines and comparison logic for the UI to consume.
- `US-5-T1` through `US-5-T4` should already provide RBAC, session scope, and audit logging patterns needed for finance-control actions.
- `US-3-T1`, `US-3-T3`, and `US-3-T4` should already provide the Vben shell, theme baseline, and localization structure.
- If the repository already has finance review-workspace patterns from actual approvals or billing flows, reuse them rather than inventing a divergent controller shell.

## Scope of work

- Build Vben-based list/detail pages, filters, and action panels for subcontractor invoice checks, including period/subcontractor/status filters, variance totals, and drill-down into line-level discrepancies.
- Expose controller-safe note entry, approval-state changes, and exception-handling actions with backend-authoritative validation and audit logging.
- Display submitted invoice references, expected vs approved totals, variance reasons, and status history in a way that is explainable to controllers without requiring direct table inspection.
- Add route guards, action-level permissions, localized DE/EN UI strings, and any small backend endpoints required for notes/status mutations or audit retrieval.
- Keep the UI aligned with Vben Admin patterns for forms, tables, detail panes, status badges, and reusable review actions.
- Add frontend and backend tests for permission gating, status-transition behavior, note persistence/auditability, and high-risk controller interaction flows.

## Preferred file targets

- Vben routes/views/components for finance controller workspace under the repository’s admin app.
- `modules/finance/` routers/services for note entry, status transitions, and audit retrieval only where backend work is needed.
- Locale resources for DE and EN.
- Frontend/backend tests for review flow, permissions, and status changes.

## Hard constraints

- Follow Vben Admin conventions; do not create a one-off admin workflow outside the established shell.
- Approval-state and exception transitions must be enforced in backend services, not just hidden in the UI.
- Do not expose unrelated HR-private data, other subcontractors, or internal-only finance metadata beyond the intended controller scope.
- Keep notes and status changes audit-safe and attributable to the acting user.
- German must remain the default locale and English must be supplied for every new user-facing string.
- Do not broaden this task into portal exposure or payment execution.

## Expected implementation outputs

- Localized Vben controller workspace for subcontractor invoice-check review.
- Backend note/status mutation support where needed, with audit-safe handling.
- Tests proving route/action permissions, status behavior, and review-flow completeness.
- A reusable review pattern aligned with other finance-control surfaces.

## Non-goals

- Do not build the subcontractor portal visibility layer here; that belongs to `US-30-T4`.
- Do not implement payment runs, booking exports, or bank integrations here.
- Do not move commercial-comparison logic into UI-only code; consume the backend comparison results from earlier tasks.
- Do not redesign the overall admin shell or theme system.

## Verification checklist

- [ ] Controllers can review invoice-check headers and line-level variance data through a Vben-aligned workspace.
- [ ] Permission checks and backend validation enforce who may view, note, approve, or exception an invoice check.
- [ ] Notes and status transitions are audit-safe and attributable.
- [ ] DE is default and EN is present for all newly added UI strings.
- [ ] Tests cover permission gating, transition rules, and controller review interactions.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
