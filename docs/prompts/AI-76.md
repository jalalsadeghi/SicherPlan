# AI-76 — Collision-Aware Assistant Placement and Safe-Area Avoidance

## Purpose

Even after draggable movement, the Assistant launcher should avoid overlapping critical UI elements by default.

The current screen shows the launcher overlapping page actions near the bottom-right area, such as `Apply demand groups`. This is common in SicherPlan because many workspaces have sticky bottom action bars, wizard footers, and right-aligned primary buttons.

The Assistant launcher should be collision-aware and choose a safer position automatically, while still allowing the user to override by dragging.

## Non-Negotiable Requirements

1. Do not hide the Assistant completely.
2. Do not cover critical page actions by default.
3. Respect user-dragged position unless it becomes invalid or off-screen.
4. If user manually moves the launcher, automatic collision avoidance should not fight the user.
5. On route changes, if no custom position exists, choose a safe default.
6. On viewport resize, keep launcher inside viewport.
7. Keep implementation performant.

### Step 2 — Add Safe-Area Contract for Page Actions

Add a lightweight convention that page authors can use:

```html
data-assistant-avoid
```

or class:

```css
.sp-assistant-avoid
```

Elements marked with this should be treated as collision/avoid zones.

Apply this attribute/class to key bottom action containers, especially:
- customer order workspace footer/actions;
- wizard footer with Previous / Apply demand groups;
- other obvious sticky/floating action rows if easy.

Do not over-mark the whole page.

### Step 3 — Collision Detection

Implement a small collision avoidance utility in frontend.

Suggested function:

```ts
interface RectLike {
  top: number;
  right: number;
  bottom: number;
  left: number;
  width: number;
  height: number;
}

function findAssistantSafePosition(options: {
  launcherSize: { width: number; height: number };
  viewport: { width: number; height: number };
  avoidRects: RectLike[];
  preferredDock: 'bottom-right' | 'bottom-left' | 'top-right' | 'top-left';
  margin: number;
}): { x: number; y: number; dockSide: string }
```

Candidate positions:
- bottom-right;
- bottom-left;
- right-middle;
- left-middle;
- top-right;
- top-left.

Pick the first candidate that does not intersect avoid rects.

### Step 4 — Only Auto-Move When Appropriate

Rules:
- If the user has never manually positioned the launcher, auto-place it using collision avoidance.
- If the user manually dragged it, do not auto-move unless:
  - stored position is off-screen;
  - it overlaps a marked avoid rect after a route change and the user has not moved it recently;
  - or the overlap is severe.
- If auto-moved, keep it subtle and predictable.

Add metadata:

```ts
{
  userPinned: boolean;
  lastMovedBy: 'system' | 'user';
  updatedAt: string;
}
```

### Step 5 — Observe Route and Layout Changes

Recompute safe position on:
- route change;
- viewport resize;
- Assistant launcher mount;
- when panel opens/closes;
- optionally after DOM settles with `nextTick`.

Use:
- `ResizeObserver`;
- throttled `requestAnimationFrame`;
- `window.resize`;
- route watcher.

Avoid excessive reflows.

### Step 6 — Visual Hint When Collision Avoidance Moves It

If the system moves the launcher to avoid a page action:
- show a subtle one-time hint:
  ```text

  ```
- German:

This should be small and dismiss automatically.

### Step 7 — CSS Safe Area

Respect mobile/browser safe areas:

```css
bottom: calc(1rem + env(safe-area-inset-bottom));
right: calc(1rem + env(safe-area-inset-right));
left: calc(1rem + env(safe-area-inset-left));
```

When using transform/inline position, incorporate safe area margins in calculations.
