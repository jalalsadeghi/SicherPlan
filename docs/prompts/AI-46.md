# AI-46 — End-to-End RAG Failure Audit and Quality Gate

## Purpose

The AI Assistant still returns weak answers even after AI-42 to AI-45. The current behavior proves that the system can call OpenAI, but it does not yet prove that the answer is generated from a rich SicherPlan-specific RAG context.

Observed failures:

1. German product overview question is incorrectly treated as out-of-scope:
   - User: `Bitte erklären Sie mir kurz und bündig, was diese Software genau macht.`
   - Assistant returns out-of-scope refusal.

2. German customer creation question receives only shallow page/path context:
   - User: `Bitte erklären Sie mir genau, wie ich einen Kunden registrieren kann.`
   - Assistant says exact UI details are unavailable and only links `/admin/customers`.

3. Employee creation answer is better because `E-01` apparently has page-help/UI facts, but this quality is not available for other core pages.

This task must audit the complete runtime path, identify the exact failure points, and add quality gates so Codex cannot mark RAG as complete unless the required end-to-end questions pass.

---

## Non-Negotiable Rule

Do **not** add canned final answers.

Expected answer outlines in this task are QA/evaluation criteria only. Runtime answers must still be generated by OpenAI from retrieved context.

You are working on the SicherPlan repository.

Implement **AI-46 — End-to-End RAG Failure Audit and Quality Gate**.

### Step 1 — Reproduce the Failures

Run the assistant backend with:

```env

SP_AI_RESPONSE_MODEL=gpt-4o
```

Reproduce these questions through the assistant API or UI:

```text
Bitte erklären Sie mir kurz und bündig, was diese Software genau macht.
Bitte erklären Sie mir genau, wie ich einen Kunden registrieren kann.

```

For each question, capture:

```text

```

### Step 2 — Determine the Root Cause

Inspect and report root causes in these categories:

1. **Classifier failure**
   - Did the product overview question become out-of-scope?
   - Why?
   - Which classifier rule caused it?

2. **Corpus failure**
   - Are product overview chunks indexed?
   - Are `C-01` customer chunks indexed?
   - Are workflow chunks for customer creation indexed?

3. **Retrieval failure**
   - Did retrieval choose shallow route/path sources instead of handbook/manual/workflow/page-help content?
   - Did it retrieve `C-01` content or only `/admin/customers`?

4. **Prompt failure**
   - Was the model given enough context to answer?
   - Was the model instructed to act as a SicherPlan expert?

5. **UI/page-help coverage failure**
   - Does `C-01` have page-help actions/form sections like `E-01`?
   - If not, this must be reported and fixed by later tasks.

6. **Source-basis failure**
   - Does final response expose the sources used?
   - Are sources content-bearing?

### Step 4 — Implement Quality Gate Evaluation

Add a backend quality gate function for in-scope answers:

```python
evaluate_rag_answer_quality(...)
```

It must check:

1. provider was called;
2. retrieval executed;
3. grounding context attached;
4. at least one content-bearing source exists for how-to questions;
5. source_basis exists for non-trivial answers;
6. overview questions are not out-of-scope;
7. exact UI labels are only used if verified UI/page-help source exists;
8. if source context is insufficient, answer must be low confidence and ask for clarification or state what is missing.

This quality gate may run in tests and debug mode. It should not block production responses yet unless a feature flag enables blocking.

Suggested flag:

```env
SP_AI_RAG_QUALITY_GATE_MODE=log_only
```

Supported values:
- `off`
- `log_only`
- `enforce_in_tests`
