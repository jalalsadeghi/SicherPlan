# Codex Prompt — US-34-T3

## Task title

**Pilot historical document import and barcode/QR output generation.**

## Objective

Prove that the final documents/output backbone can absorb migrated historical files and generate barcode/QR-bearing operational outputs without losing versioning, ownership links, authorization, or audit traceability.

## Source context

- Updated proposal: documents and media are centralized with versioned attachments and revision-safe PDF output; employee IDs and work badges support barcode/QR; Phase 6 includes migration support and rollout readiness.
- Implementation specification: documents live in object storage with database metadata, versions, ownership links, visibility, and generated-output references; the docs service stores metadata and links but does not decide business meaning.
- Sprint reference: `docs/sprint/sprint-12-migration-uat-and-go-live.md`.
- Cross-cutting rules: `AGENTS.md`, especially centralized documents, generated-output tracking, tenant-safe visibility, and append-only evidence handling.

## Dependencies

- `US-6-T1` must already provide `docs.document`, `docs.document_version`, and `docs.document_link` behavior.
- `US-12-T3`, `US-23-T3`, `US-24-*`, `US-25-*`, and related output/capture work should already define the barcode/QR and generated-output contexts this pilot must respect.
- `US-34-T1` and `US-34-T2` should already provide the import manifest structure plus document-type/template metadata.
- Any existing object-storage, checksum, or generated-document helper conventions should be reused instead of creating a parallel storage path.

## Scope of work

- Implement a pilot import flow for historical documents using explicit manifests that describe source file, checksum or integrity metadata, owner linkage, document type, visibility, and import provenance.
- Support representative historical imports across customer, employee, subcontractor, order, and finance-adjacent document groups already in approved scope.
- Implement or finalize barcode/QR output generation for representative go-live artifacts such as employee/company badges, work/order badges, or other approved operational outputs already in the repository.
- Ensure both imported files and newly generated barcode/QR outputs become centrally tracked document/version rows with correct links, download permissions, and generated-output traceability.
- Add validation for missing files, duplicate manifests, malformed payload definitions, and orphaned document links.
- Provide pilot fixtures and tests proving that imported historical documents and generated outputs can coexist safely.

## Preferred file targets

- Document-import and generated-output code under repository-appropriate docs/platform-service modules.
- `docs/migration/` for historical-document import notes and pilot instructions.
- Template/output helpers reused by badge, watchbook, patrol, timesheet, or invoice generation where already present.
- Tests for manifest integrity, authorization, and barcode/QR output reproducibility.

## Hard constraints

- Do not import files through a filesystem-only shortcut that bypasses docs metadata/version tracking.
- Barcode/QR payload formats must be deterministic, documented, and linked to real business entities or credentials already modeled in the platform.
- Do not let the docs service infer business semantics from filenames or folders; use explicit owner linkage from manifests or owning services.
- Preserve source/provenance data explicitly for historical imports rather than falsifying application audit timestamps.
- Do not broaden this pilot into a full mass-archive migration or a redesign of all document templates.

## Expected implementation outputs

- A pilot historical-document import flow with manifest validation and provenance handling.
- Representative barcode/QR generated-output flows tracked through centralized docs records.
- Authorization-safe download/link behavior for imported and generated artifacts.
- Tests and pilot fixtures for integrity, traceability, and coexistence.

## Non-goals

- Do not archive every legacy document set in this task.
- Do not redesign scanner/mobile capture flows beyond payload compatibility and output generation.
- Do not do a full visual redesign of all printed templates here.
- Do not perform production cutover or bulk historical import execution.

## Verification checklist

- [ ] Imported historical documents are stored as tracked document/version records with correct owner links.
- [ ] Import provenance is preserved explicitly and honestly.
- [ ] Barcode/QR outputs are reproducible, centrally tracked, and access-controlled.
- [ ] Duplicate, missing-file, malformed-payload, and orphan-link cases are tested.
- [ ] Imported historical docs and generated outputs can coexist without breaking the docs service model.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
