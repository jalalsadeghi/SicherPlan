# AI-67 — Portable Field/Lookup Corpus Artifact for Stage Runtime

## Purpose

Fix the Local vs Stage mismatch for short field-label questions.

Observed behavior:

- Local answers correctly to:
  `was bedeutet Vertragsreferenz`
- Stage returns the out-of-scope refusal:
  `Ich bin nur dafür vorgesehen, Fragen und Probleme zur SicherPlan-Plattform zu beantworten...`

Stage verification has already shown:

```text
/app/backend
WEB_LOCALE_MISSING
```

This means the Stage backend container does not contain frontend source files such as:

```text
web/apps/web-antd/src/locales/langs/de-DE/sicherplan.json
web/apps/web-antd/src/locales/langs/en-US/sicherplan.json
web/apps/web-antd/src/sicherplan-legacy/views/*.vue
web/apps/web-antd/src/views/sicherplan/*.vue
```

But the assistant field/lookup dictionary currently reads frontend labels and Vue form bindings to recognize field names like `Vertragsreferenz`, `Rechtlicher Name`, `Kundennummer`, `Freigabestatus`, etc.

The fix must make Stage independent from live frontend source files at runtime.

---

## Non-Negotiable Rules

1. Do not solve this by adding only one hardcoded keyword.
2. Do not create canned final answers.
3. Do not require frontend source files in the Stage backend container at runtime.
4. Local and Stage must classify known field-label questions the same way.
5. The classifier must still reject unrelated generic terms such as `Apfelkuchen`.
6. The solution must be source-grounded and testable inside the backend Docker image.

### Step 2 — Add Generated Artifact

Create a generated artifact inside the backend package, for example:

```text
backend/app/modules/assistant/generated/__init__.py
backend/app/modules/assistant/generated/field_lookup_corpus.json
```

The artifact must contain all extracted field/lookup definitions needed by the classifier and RAG layer.

Minimum artifact structure:

```json
{
  "schema_version": 1,
  "generated_from": ["frontend_i18n", "frontend_vue", "typescript_api", "backend_schema", "page_help"],
  "fields": [],
  "lookups": [],
  "source_hashes": {}
}
```

Each field entry must include:

```json
{
  "field_key": "customer.legal_name",
  "canonical_name": "legal_name",
  "labels": {"de": ["Rechtlicher Name"], "en": ["Legal name"]},
  "aliases": ["rechtlicher name", "legal name", "customerAdmin.fields.legalName"],
  "module_key": "customers",
  "page_id": "C-01",
  "entity_type": "Customer",
  "definition_de": "...",
  "definition_en": "...",
  "source_basis": []
}
```

### Step 3 — Add Export Command

Add a deterministic generation command, for example:

```bash
cd backend
python -m app.modules.assistant.field_dictionary_export   --repo-root ..   --output app/modules/assistant/generated/field_lookup_corpus.json
```

The command must:

1. Read the full repository checkout.
2. Extract fields/lookups from frontend i18n, Vue bindings, TypeScript API interfaces, backend schemas/models, page-help, and workflows.
3. Write stable sorted JSON.
4. Fail clearly if required source paths are missing during generation.
5. Print a summary: field count, lookup count, counts by module, warnings.

### Step 4 — Refactor Runtime Loader

Update `field_dictionary.py` so runtime loading works like this:

1. Load `generated/field_lookup_corpus.json` first when it exists.
2. In local/dev, optionally augment or regenerate from live sources if present.
3. In Stage/backend-only runtime, use the generated artifact without frontend source files.
4. If neither artifact nor sources exist, log a clear warning and expose an empty/minimal corpus.
5. Do not silently diverge between Local and Stage.

Use `importlib.resources` or another package-safe method so installed packages and Docker images can read the artifact.
