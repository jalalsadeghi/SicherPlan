# Codex Prompt — US-26-T2

## Task title

**Implement finance.actual_record bridge with planned/actual times, breaks, and payable/billable minutes.**

## Objective

Create the finance-owned actual bridge so released planning data and normalized field evidence can produce one auditable actual result per assignment, with clear planned-vs-actual timing and minute fields ready for approvals, payroll export, billing, and partner settlement.

## Source context

- Updated proposal: section 5.6 actual-vs-plan reconciliation, customer billing, payroll logic, and subcontractor settlement all depend on a stable actual result set.
- Implementation specification: `finance.actual_record` is the finance-owned bridge between assignments/shifts/attendance and downstream approvals, allowances, expenses, timesheets, and payroll export items.
- Workflow 6.1 step 7-8 and workflow 6.2 step 7-8 show actuals as the handoff into customer-facing packages, payroll basis, and invoice/export generation.
- Sprint reference: `docs/sprint/sprint-09-time-capture-and-actuals.md`.

## Dependencies

- `US-26-T1` must already produce normalized attendance summaries.
- `US-19-T1` and `US-19-T2` must already provide the released assignment/shift chain.
- `US-21-T1` should already provide release semantics so finance actuals derive from approved upstream planning truth.
- If the repository already has finance service packages or aggregate boundaries, keep `actual_record` inside that finance-owned boundary rather than inside field services.

## Scope of work

- Implement the creation/update derivation logic for `finance.actual_record` from assignment, shift, and attendance inputs.
- Populate planned start/end, actual start/end, break minutes, payable minutes, billable minutes, and default customer-adjustment minutes according to the repository’s approved business calculation seams.
- Enforce the spec rule that there is one final actual per assignment while allowing draft/intermediate derivation states where the repository needs them.
- Make the bridge explicit in services, models, schemas, and tests so later tasks can attach approvals, allowances, expenses, timesheets, invoices, and payroll export items without schema drift.
- Add tests for standard derivation, break handling, assignment/attendance mismatch handling, and one-final-actual-per-assignment behavior.

## Preferred file targets

- `modules/finance` or the repository-equivalent finance/billing support package for actual models, services, and schemas.
- Alembic migrations for `finance.actual_record` and any supporting indexes or status columns needed by the current repository state.
- Tests for derivation correctness, uniqueness, and downstream relationship readiness.

## Hard constraints

- `actual_record` is finance-owned output; do not leave it in the field module as a convenience projection.
- Do not bypass assignments or shifts when creating actuals; the record must stay traceable to planning truth.
- Do not mutate raw field evidence or attendance summaries to make minute calculations work.
- Keep planned/billable/payable/customer-adjustment fields explicit rather than stuffing them into opaque JSON blobs.
- Do not implement tariff, allowance, or expense details here beyond the base bridge fields; those belong to later tasks in this sprint.

## Expected implementation outputs

- Finance-owned `actual_record` model/service/schema implementation.
- Derivation logic from assignment + shift + attendance into explicit minute fields.
- Tests proving traceability, uniqueness, and calculation readiness.

## Non-goals

- Do not implement the three-stage approval rows yet.
- Do not implement tariff/rate lookup, allowances, expenses, or export batching yet.
- Do not generate customer-facing documents or invoices in this task.

## Verification checklist

- [ ] Actual records derive from released assignments/shifts and normalized attendance summaries.
- [ ] Planned, actual, payable, and billable minute fields are explicit and test-covered.
- [ ] The one-final-actual-per-assignment rule is enforced safely.
- [ ] Finance owns the actual bridge and downstream relations can attach cleanly.
- [ ] Tests cover the main standard and mismatch scenarios.

- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
