# Codex Prompt — US-25-T3

## Task title

**Enforce device/IP/geolocation/terminal validation and retain flagged raw attempts.**

## Objective

Apply policy-aware validation to raw capture attempts so invalid or suspicious time events are rejected or flagged according to configured rules while the underlying raw attempt remains preserved for audit, reconciliation, and later investigations.

## Source context

- Updated proposal Appendix E explicitly calls out location/IP/terminal rules that should reject or flag the time event.
- Section 5.6 time capture and the field-execution workflow require reliable, auditable evidence rather than blind acceptance of every scan.
- Implementation specification: `field.time_capture_policy`, `field.time_event.validation_status_code`, and field-service ownership rules in sections 4.7 and 5.
- Dienstplanmacher product sheets describe fixed-IP restrictions, context-bound terminal assignment, double-confirm terminal usage, and geolocation-rich app capture.
- Sprint reference: `docs/sprint/sprint-09-time-capture-and-actuals.md`.

## Dependencies

- `US-25-T1` must already provide the device and policy backbone.
- `US-25-T2` must already provide raw ingest handlers that can pass through policy evaluation outcomes.
- `US-16-*` should already provide sites/routes/ops context that policy checks may resolve.
- `US-20-T1` may already provide reusable structured validation-result patterns that can be reused here rather than introducing a second validation contract.

## Scope of work

- Implement policy evaluation for allowed device, allowed device type, allowed CIDR/IP range, geolocation radius, and approved context applicability on incoming raw time events.
- Support both hard reject and soft flag behavior via the configured enforce-mode/status mapping while preserving the attempted raw evidence and its validation outcome.
- Ensure terminal and device checks can be resolved against shift/site/route/planning context in service logic without baking brittle polymorphic logic into every endpoint.
- Persist enough structured outcome detail for later review, discrepancy handling, and audits, whether through event status, linked audit events, or approved metadata patterns already used by the repository.
- Add tests for valid capture, blocked capture, flagged capture, wrong device, wrong IP/CIDR, geofence violation, and context mismatch scenarios.

## Preferred file targets

- `modules/field_execution` validation services, policy-resolution helpers, and event-ingest hooks.
- Any shared validation result / message-key utilities if the repository already centralizes warn-vs-block output patterns.
- Tests for policy evaluation and raw-attempt retention.

## Hard constraints

- Rejected or flagged capture attempts must remain auditable; do not silently drop invalid raw attempts.
- Do not mutate planning or assignment truth when a capture is invalid; only the raw capture status/evidence path should change.
- Keep rule evaluation deterministic and testable; avoid burying authorization, networking, and policy resolution inside controller-only code.
- Do not broaden this task into attendance pairing or finance reconciliation behavior.
- Any message keys or user-facing error labels introduced here must preserve DE default and EN secondary behavior.

## Expected implementation outputs

- Policy-aware validation service for device, IP/CIDR, geolocation, and context checks.
- Structured storage of blocked/flagged/accepted validation outcomes on raw time events or adjacent audit-safe metadata.
- Tests proving raw-attempt retention and rule evaluation behavior.

## Non-goals

- Do not implement final discrepancy resolution UI or finance approval logic here.
- Do not create a second raw-event archive table just to keep rejected attempts.
- Do not change employee/subcontractor credential issuance in this task unless strictly needed to validate scans.

## Verification checklist

- [ ] Device, IP, geolocation, and context policies are enforced through a single service path.
- [ ] Blocked and flagged attempts remain stored and auditable.
- [ ] Accepted attempts pass through without violating append-only evidence handling.
- [ ] Service logic does not mutate planning or finance truth during validation.
- [ ] Tests cover accepted, flagged, and rejected outcomes for the main rule families.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
