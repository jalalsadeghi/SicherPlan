# Codex Prompt — US-12-T2

## Task title

**Implement availability rules, event applications, absences, and leave balances**

## Objective

Build the normalized HR planning-support foundation for availability, free wishes, employee event applications, absences, and leave balances so later dispatch and self-service features consume one authoritative HR source for workforce availability.

## Source context

- Updated proposal: section 5.3 absence, availability, and HR administration scope covering vacation account and requests, sickness management, availability/free wishes, event applications, and yearly leave/balance behavior; workflow 6.2 step 5.
- Implementation specification: `hr.employee_availability_rule`, `hr.employee_event_application`, `hr.employee_absence`, and `hr.employee_leave_balance`; planning-read-only discipline in section 3.2; Phase D before Phase E sequencing note.
- Uploaded Dienstplanmacher employee and app material: employees can enter availability/free wishes, apply to events, request vacation, report sickness categories, and planners are warned when absences conflict with staffing.
- Sprint reference: `docs/sprint/sprint-04-recruiting-and-employee-core.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-11-T1` must already provide the employee aggregate foundation.
- `US-5-T1` and `US-5-T4` should already provide user references and audit patterns for approvals.
- Because `hr.employee_event_application` references `ops.planning_record`, coordinate carefully with migration timing: use the real planning FK when `ops.planning_record` exists, and do not invent fake planning tables or duplicate event masters to satisfy the reference early.
- `US-12-T1` function/qualification data may already exist and can help later planning filters, but this task should not depend on planning logic being implemented.

## Scope of work

- Implement availability/free-wish rules with support for date-time windows, recurrence where approved, status, and note fields aligned with the implementation spec.
- Implement employee absences for vacation, sickness, child-care, and other approved absence types with status, approver attribution, and date-range validation.
- Implement yearly leave-balance rows and service logic that keeps entitlement, carry-over, consumed, and pending values consistent with approved absence changes.
- Implement employee event-application persistence and APIs only against the real `ops.planning_record` contract; if the planning aggregate is not yet present in the repo, keep the migration/service seam forward-compatible and avoid fake stand-in tables.
- Expose query contracts that planning and employee self-service can later consume without giving planning write ownership of HR availability/absence truth.
- Write tests for recurrence validation, overlap rules, leave-balance updates, approval attribution, and safe handling of the event-application cross-phase dependency.

## Preferred file targets

- Employee bounded-context models/services/schemas under `modules/employees/*`.
- Approval or query helpers that planning will later read from, without cross-context writes.
- Tests for absence overlap, balance updates, and migration-safe event-application handling.

## Hard constraints

- Do not invent temporary planning/event master tables to satisfy `hr.employee_event_application` before the real planning aggregate exists.
- Approved absences must be treated as authoritative HR unavailability.
- Prevent invalid date ranges and overlapping approved absences according to policy.
- Keep leave-balance math centralized and test-covered instead of spreading it across controllers.
- Do not let planning or mobile write directly into HR truth through ad hoc shortcuts.

## Expected implementation outputs

- Normalized availability, absence, leave-balance, and event-application foundations.
- Service-layer rules for absence approval, overlap handling, and balance updates.
- Read/query contracts consumable by later planning and self-service tasks.
- Tests covering date validation, overlap logic, balance updates, and cross-phase integrity.

## Non-goals

- Do not build the planning board or staffing engine here.
- Do not clear or reassign actual shifts here; later planning tasks own operational replanning behavior.
- Do not implement the employee mobile UI here.
- Do not create shadow event records or fake order tables before Sprint 6.

## Verification checklist

- Availability rules validate time windows and recurrence semantics correctly.
- Approved absences block invalid overlaps and carry approver attribution.
- Leave-balance rows update consistently when vacation-style absences change status.
- Event-application handling is either wired to the real planning-record contract or left as an explicit forward-compatible seam without fake dependencies.
- Tests cover overlap rules, balance math, and the planning-reference sequencing seam.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
