# Codex Prompt — US-14-T2

## Task title

**Build import/export and maintenance screens for subcontractor workforce records**

## Objective

Provide the admin-facing workforce maintenance experience so tenant teams can manage subcontractor workers, qualifications, credentials, and bulk data exchange efficiently without bypassing the approved backend contracts.

## Source context

- Updated proposal: section 5.4 subcontractor collaboration portal and document/qualification control, which imply ongoing maintenance of partner staff records and controlled data exchange.
- Implementation specification: partner workforce tables in section 4.5 and the shared import/export/integration backbone from section 4.2 and phase sequencing in section 8.
- Uploaded Dienstplanmacher subcontractor source material: subcontractors can pre-create workers, import employee data, maintain worker documents, and handle staff assignment preparation before the first shift.
- Sprint reference: `docs/sprint/sprint-05-subcontractor-management.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-14-T1` must already provide stable partner-worker API contracts.
- `US-6-T4` should already provide the preferred import/export job or asynchronous processing backbone for bulk operations.
- `US-3-T1`, `US-3-T3`, and `US-3-T4` should already provide Vben shell patterns, theme tokens, and DE/EN localization resources.

## Scope of work

- Build Vben Admin list/detail pages for subcontractor workers, qualifications, credentials, and proof-document maintenance on top of the backend contracts from `US-14-T1`.
- Provide bulk import and export flows for subcontractor workforce data using the repo's approved job/backbone pattern, including template download, preview/validation feedback, result reporting, and safe upsert or staged-write semantics.
- Support maintenance UX for worker base data, qualification rows, credential rows, and document attachments, with clear status indicators and validation errors.
- Allow filtering by subcontractor, status, qualification readiness, expiry state, and credential presence where supported by the backend contracts.
- Use DE-default / EN-secondary localization for menus, labels, import instructions, validation messages, and result states.
- Add frontend tests for list/detail flows and import/export happy-path/error-path behavior, plus backend tests if the task adds import/export endpoints or job orchestration code.

## Preferred file targets

- Vben routes, list/detail pages, import/export dialogs, API clients, composables, and locale resources under the actual web-admin path.
- Import/export job adapters or endpoints under `modules/subcontractors/` or `modules/platform_services/` if the repo requires them for this task.
- Frontend tests and any small backend tests needed for import/export orchestration.

## Hard constraints

- Use the approved bulk-job/import backbone where it already exists; do not build a one-off spreadsheet parser entirely in the browser for non-trivial imports.
- Do not let UI bulk actions bypass backend validation, tenant scoping, or document ownership rules.
- Keep this task focused on internal admin maintenance; partner self-service editing belongs primarily to `US-15-T4`.
- Do not bury worker readiness or compliance logic directly in UI-only calculations if a backend read model already exists or is coming from `US-14-T3`.
- All user-facing strings introduced here must ship in German and English and follow the shared Vben styling/token rules.

## Expected implementation outputs

- Subcontractor workforce admin pages with CRUD and bulk data-exchange capability.
- Import/export flows that reuse the approved integration/job foundation instead of inventing an incompatible side path.
- Frontend/backend test coverage for the core maintenance and bulk-operation flows.

## Non-goals

- Do not implement portal authentication or subcontractor-facing self-service here; that is `US-15-*`.
- Do not implement readiness scoring or compliance completeness rules here beyond surfacing the data the backend exposes; that is `US-14-T3`.
- Do not implement planning-facing read models here; that is `US-14-T4`.

## Verification checklist

- [ ] Authorized internal users can list, create, edit, and archive or deactivate subcontractor workers from the admin UI.
- [ ] Bulk import preview/validation clearly reports row-level issues before commit or job execution.
- [ ] Exports respect tenant and permission scope and include only allowed data.
- [ ] Qualification, credential, and document-maintenance flows remain aligned with backend validations.
- [ ] German is default and English is available for all new workforce-admin UI strings.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
