## 4. Backend Rules

### 4.1 Ownership and Scope

The backend must verify:

1. Current user is authenticated.
2. Assistant feature is enabled.
3. User has `assistant.feedback.write` or an equivalent permission.
4. Conversation exists.
5. Conversation belongs to the current user and current tenant/scope, unless a specific support/admin permission exists.
6. Message exists inside the conversation.
7. Feedback target is an assistant message, not a user message, unless current design explicitly allows both.
8. Portal users cannot submit feedback for internal user conversations.

### 4.2 Tenant Handling

- `tenant_id` must come from the backend auth context or the conversation record.
- Never accept `tenant_id` from the request body.
- Cross-tenant feedback submission must be blocked.

### 4.3 Comment Handling

- Comment is optional.
- Comment should be length-limited.
- Suggested max length: 1000 characters.
- Comment must be stored as plain text.
- Do not render or store it as HTML.
- Do not include raw tool outputs or sensitive backend data.

### 4.4 Duplicate Feedback Behavior

Choose the behavior that best matches repository conventions and document it.

Preferred behavior:

- one feedback row per `(conversation_id, message_id, user_id)`;
- if submitted again, update/replace rating and comment.

Alternative acceptable behavior:

- allow multiple feedback events if the existing product convention treats feedback as event history.

Whichever behavior is chosen must be tested.

### 4.5 Audit and Privacy

- Feedback storage must not copy the full assistant structured payload.
- Feedback must not copy tool output.
- Feedback may store only IDs, rating, comment, user/tenant, and timestamps.
- If the existing assistant audit system records user actions, add a minimal audit entry for feedback submission.
