# Codex Prompt — US-10-T3

## Task title

**Create recruiter UI for review, interview notes, and accept/reject decisions**

## Objective

Build the recruiter-facing administration experience in Vben-style list/detail flows so recruiting users can review applicant submissions, inspect attachments and consent evidence, maintain interview notes, and execute accept/reject decisions through the approved workflow services.

## Source context

- Updated proposal: section 5.3 recruiting workflow; section 6.2 step 2 for review/interview/decision behavior; technical architecture section 7 for Vue/Vben-based admin delivery.
- Implementation specification: `ApplicantAggregate` boundaries, `hr.applicant`, `hr.applicant_status_event`, docs-link usage for attachments, and IAM role-scoping expectations.
- Uploaded Dienstplanmacher applicant-management source material: applicant administration overview, interview scheduling/steps, decision tracking, attachment review, and the promise that all steps stay documented.
- Sprint reference: `docs/sprint/sprint-04-recruiting-and-employee-core.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-10-T1` and `US-10-T2` must already expose stable intake data, workflow history, and attachment access contracts.
- `US-5-T3` must already provide route/API authorization and role scoping for recruiter access.
- `US-3-T1` and `US-3-T4` should already provide the Vben shell and DE/EN localization baseline.
- `US-6-T1` must already provide secure document download/preview flows for applicant attachments.

## Scope of work

- Create Vben-aligned recruiter list/detail pages with status filters, source-channel filters, search, timeline visibility, and list columns that fit applicant operations.
- Expose applicant detail sections for submission data, consent evidence, uploaded attachments, and workflow history.
- Provide recruiter actions to add interview notes, record review comments, and execute accept/reject/reopen decisions through backend workflow APIs.
- Use confirmation dialogs, guardrails, and localized UI feedback for sensitive decisions such as rejection or acceptance.
- Render attachment previews/download actions through the approved secure-document path, not via raw object-storage URLs.
- Ensure the UI distinguishes current status, previous steps, and next allowed actions clearly so recruiters do not guess the process state.
- Add frontend tests and, where the repo supports it, end-to-end coverage for route permissions, form validation, decision actions, and timeline rendering.

## Preferred file targets

- Vben routes, views, API client modules, stores/composables, and localized resource files under the repo's web-admin package.
- Shared recruiter UI components for applicant history, document preview actions, and status-action buttons.
- Frontend tests and optional E2E flows for recruiter permissions and decision handling.

## Hard constraints

- Do not bypass backend workflow validation from the UI.
- Do not expose applicant attachments through insecure direct-storage links.
- Do not create employee records from the recruiter UI here; conversion belongs to `US-10-T4`.
- Keep German as the default locale and English as the secondary locale for all new recruiter UI strings.
- Respect Vben patterns for route guards, page layout, and reusable list/detail CRUD conventions.

## Expected implementation outputs

- Recruiter list and detail screens with localized labels and permission guards.
- Decision and note-entry UI connected to the approved applicant workflow APIs.
- Secure attachment/consent evidence rendering inside applicant detail flows.
- Frontend tests for critical recruiter workflows.

## Non-goals

- Do not implement the public applicant form here; that belongs to `US-10-T1`.
- Do not redesign the global admin shell or permission model.
- Do not build outbound email/SMS scheduling for interviews here unless the repo already has a minimal no-drift hook.
- Do not implement applicant conversion here; that is `US-10-T4`.

## Verification checklist

- Recruiters can list and filter applicants by meaningful workflow criteria.
- Applicant detail pages show consent evidence, attachments, and ordered workflow history.
- Accept/reject actions call backend workflow APIs and respect role-based access rules.
- German is the default locale and English is available for all new recruiter-facing strings.
- Tests cover route protection, note handling, and decision execution.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
