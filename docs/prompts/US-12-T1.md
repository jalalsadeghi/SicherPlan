# Codex Prompt — US-12-T1

## Task title

**Implement function types, qualification types, employee qualifications, and document proofs**

## Objective

Create the normalized HR qualification/function foundation so planning, compliance, and later payroll flows can rely on searchable, time-aware employee qualification facts backed by proof documents instead of ad hoc free text.

## Source context

- Updated proposal: section 5.3 qualifications and compliance, including planning qualifications, general qualifications, functions, Guard Register and certificate proofs, Scan-and-Drop uploads, expiry dates, planning filters, and mandatory-document auditability.
- Implementation specification: section 4.4 `hr.function_type`, `hr.qualification_type`, and `hr.employee_qualification`; docs/document-link proof usage; EmployeeAggregate ownership; overlap/temporal-validation guidance.
- Uploaded Dienstplanmacher employee-management source material: distinction between planning qualifications, general qualifications, and functions; qualification-driven planning filters; internally granted qualifications after training; certificate/proof expectations.
- Sprint reference: `docs/sprint/sprint-04-recruiting-and-employee-core.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-11-T1` must already provide the employee aggregate foundation.
- `US-6-T1` must already provide document storage and linking for qualification proofs.
- If function/qualification lookup seeds already exist from Sprint 2 core lookup work, reuse them carefully instead of duplicating catalogs.

## Scope of work

- Implement migrations/models/schemas/services for `hr.function_type`, `hr.qualification_type`, and `hr.employee_qualification` aligned with the implementation spec.
- Provide tenant-scoped catalogs for functions and qualifications, including code, label, category/class metadata, activation flags, and validity-policy fields where approved.
- Implement employee qualification records with certificate number, issued date, valid-until date, issuing authority, status, and linked proof documents via the docs backbone.
- Support Scan-and-Drop-friendly proof upload flows so multiple proofs can be added quickly without inventing a second attachment subsystem.
- Centralize validity/default-expiry rules from qualification types so employee qualification records validate required expiry behavior consistently.
- Expose read/query contracts that later planning and compliance logic can consume without taking write ownership of HR qualification truth.
- Write tests for catalog uniqueness, proof-link handling, required-validity behavior, and employee qualification queries.

## Preferred file targets

- Employee bounded-context models, services, and schemas under `modules/employees/*`.
- Document-link helper integrations for qualification proofs.
- Tests for catalog integrity, expiry rules, and proof attachment behavior.

## Hard constraints

- Do not store qualification truth as free-text notes when normalized catalogs and rows are required.
- Proof files must use the centralized docs service and document links.
- Do not make planning or payroll the write owner of function/qualification truth.
- Keep tenant-scoped code uniqueness explicit and test-covered.
- Do not invent a heavyweight training-management subsystem here just because qualifications may originate from training later.

## Expected implementation outputs

- Normalized function and qualification catalogs.
- Employee qualification records with proof-document linkage and validity behavior.
- Read/query contracts consumable by later planning/compliance features.
- Tests for uniqueness, expiry logic, and proof-link correctness.

## Non-goals

- Do not implement the full planning validation engine here.
- Do not implement subcontractor worker qualifications here; those belong to Sprint 5.
- Do not build the employee self-service/mobile UI here.
- Do not design a full training workflow or certification course scheduler here.

## Verification checklist

- Function and qualification catalogs are tenant-scoped and uniqueness-enforced.
- Employee qualification rows can store proof documents through docs links.
- Validity requirements from qualification types are enforced consistently.
- Queries can filter or list employee qualifications without free-text parsing.
- Tests cover catalog integrity, proof linkage, and expiry behavior.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
