# AI-17 — Page Route Catalog and Safe Navigation Links

## Task Objective

Implement the **Page Route Catalog** and safe navigation-link generation for the SicherPlan AI Assistant.

The assistant must be able to guide a user to the correct SicherPlan page, but only if the current user is allowed to access that page. Link generation must be performed by backend-controlled navigation tools, not invented by the model.

This task builds on the user capability/page context tools from `AI-16` and prepares the assistant for operational diagnostics such as the Markus shift visibility scenario.

---

This task assumes the following tasks are complete or available:

- `AI-03` Assistant Permission Seeds and Access Policy Baseline
- `AI-05` Assistant Database Models and Migrations
- `AI-07` Structured Response Contract
- `AI-15` Tool Registry Core
- `AI-16` User Capability and Page Context Tools

If the page route catalog table from `AI-05` is missing, stop and report the gap before coding. If the migration used prefix-based table names instead of schemas, follow that convention.

### 3. Permission Mapping

Build permission checks conservatively.

Rules:

- If exact required permissions exist, use them.
- If route meta authority uses role keys instead of permission keys, map conservatively from existing role/authority conventions.
- If neither exists, require `assistant.chat.access` plus a conservative module permission if available.
- Never return admin/internal links to customer portal, employee self-service, or subcontractor portal users unless those routes are explicitly in their scope.
- External users must only receive portal/self-service links inside their own scope.

Examples:

```text
E-01 Employees Workspace:
  internal HR/tenant/admin/authorized roles only

P-04 Staffing Board & Coverage:
  tenant_admin, dispatcher, field_supervisor where allowed

ES-01 Employee Self-Service Portal:
  employee_self_service user only, or internal users only if existing route/policy allows support mode

CP-01 Customer Portal:
  customer portal user only, or internal support only if existing route/policy allows it

SP-01 Subcontractor Portal:
  subcontractor portal user only, or internal support only if existing route/policy allows it
```

### 5. Implement `navigation.build_allowed_link`

Register a read-only tool named:

```text
navigation.build_allowed_link
```

Purpose:

Build a safe internal navigation link for one catalog page and optional safe query parameters.

Input schema:

```json
{
  "page_id": "string",
  "entity_context": {
    "employee_id": "uuid|null",
    "shift_id": "uuid|null",
    "assignment_id": "uuid|null",
    "planning_record_id": "uuid|null",
    "customer_id": "uuid|null",
    "subcontractor_id": "uuid|null",
    "date": "YYYY-MM-DD|null"
  },
  "reason": "string|null"
}
```

Adapt schema to existing validation style. The entity fields are optional and must be sanitized.

Output schema:

```json
{
  "allowed": true,
  "link": {
    "label": "Staffing Board & Coverage",
    "path": "/admin/planning-staffing?date=2026-05-01",
    "route_name": "SicherPlanPlanningStaffing",
    "page_id": "P-04",
    "reason": "Assignment and validation state can be inspected here."
  },
  "missing_permissions": []
}
```

Denied output:

```json
{
  "allowed": false,
  "link": null,
  "missing_permissions": [
    {
      "permission": "planning.assignments.read",
      "reason": "Current user cannot access this planning workspace."
    }
  ]
}
```

Rules:

- The backend must check permission before returning `allowed=true`.
- The model must not be allowed to fabricate links.
- Entity IDs in links must not leak inaccessible record existence.
- Do not include raw tenant IDs in query parameters unless existing routes require them and the route is already tenant-scoped safely.
- Only allow whitelisted query parameters from `entity_param_map`.
- Drop or reject query parameters not allowed for the target page.
- Encode query parameters safely.
