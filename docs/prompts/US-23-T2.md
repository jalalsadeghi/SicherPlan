# Codex Prompt — US-23-T2

## Task title

**Implement watchbook creation, entries, attachments/photos, and supervisor review.**

## Objective

Build the operational online watchbook so internal field actors can create or continue the correct daily log, append narrative and evidence entries, attach photos/documents through the docs service, and complete supervisor review/confirmation without breaking the approved field-data model.

## Source context

- Updated proposal: section 5.2 online watchbook scope with daily logs, remarks, events, photos/documents, author tracking, supervisor review, traffic-light status, and revision-safe daily PDFs; workflow 6.1 step 6 where execution produces watchbook evidence.
- Implementation specification: section 4.7 `field.watchbook` and `field.watchbook_entry`, including one-open-log-per-context-and-date logic, author-reference rules, supervisor confirmation, and attachments through `docs.document_link`; section 5 field-evidence chain.
- Appendix D of the updated proposal: watchbook entry model with object/day, author, entry type, narrative, photos/documents, supervisor confirmation, and closure state.
- Sprint reference: `docs/sprint/sprint-08-employee-app-and-field-i.md`.
- Cross-cutting rules: `AGENTS.md`, especially append-only/correction-safe evidence handling and centralized docs ownership.

## Dependencies

- `US-6-T1` must already provide document storage/linking for watchbook attachments and generated outputs.
- `US-16-T1` and the planning foundation should already provide the site/order/planning contexts that a watchbook can belong to.
- `US-22-T1` should already provide authenticated employee/field-shell entry into mobile screens.
- `US-5-T4` should already provide audit-event handling for sensitive operational updates.
- Keep external participation out of the base watchbook flow until `US-23-T4` explicitly enables it.

## Scope of work

- Implement watchbook creation/open/load flows keyed by site, order, or planning-record context plus log date, respecting the one-open-log-per-context-and-date rule.
- Implement watchbook entry creation with entry type, narrative/body, traffic-light code where applicable, related assignment linkage, and exact author attribution using the approved actor-reference model.
- Support photo and document attachments via `docs.document_link` instead of embedding files into watchbook tables.
- Implement supervisor review/confirmation behavior and safe closure-state handling using the approved watchbook fields and audit trail expectations.
- Provide Vben and/or mobile interaction surfaces as appropriate for the actors who create and review watchbook evidence in Sprint 8.
- Add tests for context uniqueness, author attribution, attachment flows, supervisor confirmation, and safe close/reopen rules if reopen is allowed.

## Preferred file targets

- `modules/field_execution` models, services, routers, and validators for watchbook and watchbook entries.
- Web/mobile feature files for watchbook entry creation and supervisor review if those clients are included in the repo.
- Docs-link integration helpers only where watchbook attachments need a narrow adapter.
- Tests for watchbook uniqueness, entry authorship, and review/closure behavior.

## Hard constraints

- Respect the approved watchbook owner contexts and the partial unique rule for one open log per context/date.
- Author attribution must use the approved actor references; do not invent extra author columns for customer/subcontractor contacts in the base flow.
- Photos and attachments must flow through the docs service and remain linked back to watchbook entities.
- Operational evidence should remain correction-safe; do not destructively rewrite or delete watchbook history to simulate edits.
- Field execution may reference planning/site/order context, but it must not mutate planning demand or commercial rules.

## Expected implementation outputs

- Watchbook aggregate implementation with create/open/load behavior by approved context.
- Watchbook entry creation with attachments/photos, author attribution, and supervisor review.
- Client interaction surfaces for the relevant actors in Sprint 8.
- Tests covering uniqueness, authorship, attachment linking, and review/closure behavior.

## Non-goals

- Do not generate the revision-safe PDF output here; that belongs to `US-23-T3`.
- Do not enable customer/subcontractor write participation in this base task; that belongs to `US-23-T4`.
- Do not mix watchbook evidence into finance actuals or invoice generation in this task.
- Do not store images or binary payloads directly in watchbook tables.

## Verification checklist

- [ ] Only one open watchbook exists per approved context/date combination.
- [ ] Entries preserve correct author attribution and link to attachments through the docs service.
- [ ] Supervisor review/confirmation works and is auditable.
- [ ] Client flows allow field users to create evidence without mutating planning/commercial truth.
- [ ] Tests cover context uniqueness, authorship, attachment flows, and review behavior.

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
