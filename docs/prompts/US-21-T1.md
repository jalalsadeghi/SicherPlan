# Codex Prompt — US-21-T1

## Task title

**Implement release workflows and visibility flags for employees, customers, and subcontractors.**

## Objective

Implement the release-state transitions and visibility controls that turn staffing plans into downstream-ready operational outputs for employees, customers, and subcontractors without exposing hidden or invalid planning data.

## Source context

- Updated proposal: section 5.5 released outputs and operational note; Phase 3 planning-engine deliverables in section 8; customer/subcontractor/employee visibility intent across the proposal.
- Implementation specification: `ops.shift.release_state_code`, `customer_visible_flag`, `subcontractor_visible_flag` in section 4.6; portal visibility chain and minimum-staffing/release enforcement in sections 5 and 6.
- Sprint reference: `docs/sprint/sprint-07-staffing-engine.md`.
- Cross-cutting rules: `AGENTS.md`, especially released outputs as contracts for portals, mobile, field, and finance.

## Dependencies

- `US-19-*` must already provide demand, assignment, and subcontractor-release truth.
- `US-20-*` must already provide structured validation outcomes, including minimum-staffing and blocking-rule results.
- `US-17-T3` should already provide earlier order/planning release-state and document-package administration seams where applicable.
- `US-5-T3` and `US-5-T4` must already provide role scope and audit patterns for sensitive state transitions.

## Scope of work

- Implement planning-service commands and APIs for moving staffing outputs through the approved release lifecycle, including release, revoke/unpublish, and visibility toggling for the relevant downstream actors.
- Use existing release-state fields and explicit visibility flags where they already exist in the schema; if employee visibility is represented by release state rather than a dedicated boolean, preserve that design instead of inventing redundant flags.
- Gate release transitions on structured validation outcomes, minimum staffing, and any other mandatory blocking conditions already delivered in Sprint 7.
- Provide query endpoints or diagnostics that explain why a given shift/plan cannot yet be released, so later UI prompts can surface blockers cleanly.
- Audit sensitive release-state transitions and the acting user through the repository’s existing audit/business-event patterns.
- Add backend tests for release success, release denial, visibility toggling, revoke/unpublish flows, and the main role-scope constraints.

## Preferred file targets

- `modules/planning/` services, routers/controllers, release DTOs, and tests.
- Potential small query/read helpers for release blocker inspection.
- No major PDF generation or communication UI work in this task.

## Hard constraints

- Released outputs are contracts for downstream consumers. Do not allow release when blocking validation or minimum-staffing gates fail.
- Do not invent redundant visibility flags when release-state semantics already cover the employee/mobile visibility contract.
- Customer and subcontractor visibility must remain least-privilege and tenant-scoped; hidden data must stay hidden even when internal release exists.
- Do not silently publish unreleased or draft planning data into downstream read paths.
- Release/revoke actions are audit-sensitive and must be attributable to the acting user.
- Any user-facing state or reason labels introduced here must preserve German default and English secondary behavior.

## Expected implementation outputs

- A release-state and visibility-control workflow for staffing outputs.
- Structured release blocker diagnostics for later Vben/portal/mobile consumers.
- Tests proving validation-gated release behavior, visibility control, and audit-safe state transitions.

## Non-goals

- Do not generate PDFs or print outputs here; that belongs to `US-21-T2`.
- Do not implement planning communication or stealth-mode messaging here; that belongs to `US-21-T3`.
- Do not expose portal/mobile read models here; that belongs to `US-21-T4`.
- Do not rewrite the validation engine here; consume it.

## Verification checklist

- [ ] A shift/plan cannot be released or customer/subcontractor-visible while blocking validation failures or understaffing remain unresolved.
- [ ] Release and revoke/unpublish flows are tenant-scoped, permission-aware, and auditable.
- [ ] Visibility flags/semantics behave predictably for employees, customers, and subcontractors without leaking hidden data.
- [ ] Release blocker diagnostics are explicit enough for later UI prompts.
- [ ] Tests cover release success, denial, revoke flows, and scope/security behavior.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
