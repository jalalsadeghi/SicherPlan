# Codex Prompt — US-11-T1

## Task title

**Implement employee master, private profile, and address history models with strict role separation**

## Objective

Establish the normalized employee aggregate in code so operational staff can work with the employee master file while HR/payroll-only data remains structurally separated, access-controlled, and ready for later self-service, planning, and finance consumption.

## Source context

- Updated proposal: section 5.3 digital employee file, HR data protection requirement, Appendix C employee data model and personnel-file groups, and the employee completion steps in workflow 6.2.
- Implementation specification: section 4.4 `hr.employee`, `hr.employee_private_profile`, `hr.employee_address_history`, and related `hr.employee_group` / `hr.employee_group_member` tables; Phase D migration guidance; aggregate/package map for `EmployeeAggregate`.
- Uploaded Dienstplanmacher Mitarbeiterakte material: electronic employee file, profile photo, document archive, positive activity notes, customer blocks, role-based access differences between dispatcher/admin/object lead, and current/past address expectations from the personnel file.
- Sprint reference: `docs/sprint/sprint-04-recruiting-and-employee-core.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-10-T4` should provide applicant conversion seams, but this task must still stand on its own as the real employee aggregate foundation.
- `US-4-T1` and `US-5-T1` must already provide tenant, branch, user, and role references.
- `US-6-T1` should already provide docs/document-link foundations for profile photos and identity proofs.

## Scope of work

- Create forward-only migrations, SQLAlchemy models, and Pydantic Create/Update/Read/List schemas for `hr.employee`, `hr.employee_private_profile`, and `hr.employee_address_history`.
- Implement the key uniqueness and relationship rules from the implementation spec, including tenant-scoped personnel numbers, optional unique `user_id`, 1:1 private-profile linkage, and prevention of overlapping active address windows.
- Apply a strict role split between operational employee fields and HR-private fields so service/query layers can expose least-privilege DTOs without mixing them into one broad schema.
- If the repo does not yet contain `hr.employee_group` and `hr.employee_group_member`, add the narrow group foundation needed for later UI group assignment without inventing an ad hoc tagging system.
- Wire profile photo and identity/proof document linkage through the docs service instead of inline blob columns wherever the repo needs a file association.
- Add tests for uniqueness, 1:1 private-profile behavior, address-history window rules, and role-scoped read/write service boundaries.

## Preferred file targets

- Employee bounded-context model, schema, repository, and service files under `modules/employees/*` or the actual repo equivalent.
- Alembic revisions for employee master/private/address/group tables.
- Tests covering model constraints, service-layer field exposure, and address-history validation.

## Hard constraints

- Do not mix HR-private fields into the operational employee table just for convenience.
- HR-private reads and writes must be separable at the service/schema layer and enforceable by role scope.
- Do not replace address history with a single mutable free-text address column.
- Do not implement payroll/tariff calculation logic here.
- Use explicit foreign keys and tenant-aware uniqueness rather than loose references or JSON blobs.

## Expected implementation outputs

- Alembic migrations and SQLAlchemy models for the employee master/private/address foundation.
- Pydantic DTO families for operational versus HR-private employee operations.
- Optional narrow group-foundation tables when absent in the repo.
- Tests for constraints, privacy split, and address-history rules.

## Non-goals

- Do not build employee-file UI here; that is `US-11-T2`.
- Do not implement import/export or IAM user linkage here; that is `US-11-T3`.
- Do not implement audit logging/archival lifecycle here; that is `US-11-T4`.
- Do not implement qualifications, absences, balances, or credentials here; those belong to `US-12` tasks.

## Verification checklist

- Operational and HR-private employee data are persisted in separate approved tables.
- Personnel-number uniqueness and optional user linkage constraints are enforced.
- Address-history windows cannot overlap incorrectly for the same employee.
- Least-privilege DTO/service separation exists for operational versus HR-private reads and writes.
- Tests cover privacy split and core relationship rules.

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
