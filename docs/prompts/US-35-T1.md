# Codex Prompt — US-35-T1

## Task title

**Execute UAT for customer-order-to-invoice, applicant-to-payroll, and subcontractor collaboration flows.**

## Objective

Convert the three proposal reference workflows into executable UAT packs with stable test data, explicit expected outcomes, and evidence capture so formal acceptance can be run from repository assets instead of ad hoc walkthroughs.

## Source context

- Updated proposal: sections 6.1 through 6.3 define the three anchor workflows and section 9 defines acceptance around end-to-end execution, tenant scope, qualification/document controls, finance correctness, and reproducible outputs.
- Sprint reference: `docs/sprint/sprint-12-migration-uat-and-go-live.md`.
- Cross-cutting rules: `AGENTS.md`, especially role scope, portal privacy, DE-default / EN-secondary behavior, and traceability from master data to finance/reporting outputs.
- This task follows the migration-validation work from `US-34-*` and prepares formal business acceptance rather than purely technical smoke testing.

## Dependencies

- `US-34-T4` must already provide validated migrated data and configuration suitable for UAT use.
- The main feature streams from Sprints 3 through 10 plus the reporting/hardening work from Sprint 11 must already be integrated enough to support end-to-end scenarios.
- Any existing UI/API automation, seeded demo data, or smoke-test harnesses should be reused to keep UAT assets aligned with repository reality.
- `US-35-T3` and `US-35-T4` will later depend on the evidence and gaps discovered here.

## Scope of work

- Create UAT scenario scripts and evidence checklists for the customer-order-to-invoice, applicant-to-payroll, and subcontractor collaboration workflows.
- Define preconditions, actor accounts, test data requirements, step-by-step actions, expected outcomes, negative cases, and evidence capture points for each workflow.
- Provide a stable UAT account matrix and environment preflight checklist that covers web admin, portals, mobile/field flows where relevant, generated outputs, and integrations needed for acceptance.
- Reuse repository-standard e2e or smoke automation where available for the highest-risk checkpoints, while keeping manual business-readable scripts for the full flow.
- Define a defect-triage and evidence-capture format so failed UAT steps become actionable backlog items rather than vague notes.
- Ensure the UAT assets reflect German as default and English as secondary for user-facing acceptance expectations.

## Preferred file targets

- `docs/uat/` for scenario packs, role matrices, evidence forms, and environment-preflight checklists.
- Seed/reset helpers or fixture generators under repository-appropriate QA or platform-service areas.
- Any existing e2e/smoke test folders used by the repository for UI/API acceptance checks.
- Defect/evidence templates stored alongside UAT assets.

## Hard constraints

- UAT must reflect the real proposal workflows and role scopes, not a synthetic list of isolated CRUD checks.
- Expected outcomes must be explicit, auditable, and linked to acceptance intent.
- Do not use real production personal data or uncontrolled credentials in UAT assets.
- Keep blocker, major, and minor defect handling distinct so business acceptance stays honest.
- Do not treat automation as a substitute for business-readable UAT instructions.

## Expected implementation outputs

- Business-readable UAT packs for all three anchor workflows.
- Stable test-data/account/preflight assets for UAT execution.
- Optional smoke/e2e automation for the highest-risk checkpoints where the repo already supports it.
- Defect and evidence templates tied to formal acceptance.

## Non-goals

- Do not implement net-new product features in this task.
- Do not do full performance testing or security-hardening work here.
- Do not write training materials here beyond what is strictly needed to execute UAT.
- Do not sign off go-live readiness yet.

## Verification checklist

- [ ] Each anchor workflow has an end-to-end UAT script with explicit preconditions, steps, expected results, and evidence points.
- [ ] A stable UAT data/account setup and reset path exists.
- [ ] At least the highest-risk checkpoints are backed by automation or documented manual evidence capture.
- [ ] Role scope, portal privacy, and DE/EN expectations are included in the UAT assets.
- [ ] Defects discovered during UAT can be routed and tracked durably.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
