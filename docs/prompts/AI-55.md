# AI-55 — Token Budget Manager and Rate-Limit Resilient Responses API Continuation

## Purpose

The assistant currently fails for an operational diagnostic question because the second OpenAI call after a tool call hits a token-per-minute rate limit.

Observed log:

```text
Rate limit reached for gpt-4o ... tokens per min (TPM): Limit 30000, Used 10636, Requested 21912.
```

This means the continuation request is too large. The system is likely resending a large prompt/grounding context during continuation instead of sending only the required tool outputs and minimal instructions.

This task adds token budgeting, continuation slimming, retry/backoff, and optional model fallback to prevent 429/503 errors.

---

## Non-Negotiable Rules

1. Do not introduce canned final answers.
2. Do not silently send huge repeated prompts.
3. Do not expose sensitive tool outputs.
4. Keep tenant/RBAC enforcement.
5. Prefer compact continuation over larger model calls.
6. Preserve `SP_AI_STORE_RESPONSES=false` if possible; document any required tradeoff.

### Step 3 — Add Approximate Token Estimation

Add a helper:

```python
estimate_tokens(text_or_payload) -> int
```

Use a conservative approximation if no tokenizer is installed:
- 1 token ≈ 4 characters for English/German;
- JSON overhead included;
- never underestimate by more than obvious margin.

Use it before provider calls to enforce budgets.

### Step 4 — Trim Grounding Context Before Provider Call

Before sending to OpenAI:

1. Sort sources by relevance/content-bearing score.
2. Keep source previews instead of full source content when needed.
3. Drop shallow route sources first.
4. Keep workflow/diagnostic/page-help facts before low-value docs.
5. Remove repeated source content.
6. Remove irrelevant recent conversation history.
7. Preserve source IDs/title/page/module for source_basis.

If the context is trimmed, add a safe note in RAG trace:

```json
"grounding_trimmed": true,
"trim_reason": "token_budget"
```

### Step 5 — Slim Continuation Requests

For a continuation after tool calls:

1. Do not resend full grounding context unless strictly required.
2. Do not resend recent conversation history.
3. Do not resend the original user message unless the API requires it.
4. Send only:
   - valid continuation state (`previous_response_id` or validated stateless equivalent);
   - function_call_output items;
   - concise instruction: "Use the tool outputs and prior response context to produce the final structured answer."
5. Set lower `max_output_tokens`.

Codex must validate with the installed OpenAI SDK and current Responses API behavior.

### Step 6 — Rate Limit Handling

If OpenAI returns 429:

1. Parse safe retry delay from provider message or response headers if available.
2. Wait once or twice within configured max retry seconds.
3. Before retry, reduce request size if possible:
   - lower max output tokens;
   - remove optional sources;
   - remove tools if final answer can be generated from collected facts.
4. If still rate-limited, do not return a generic frontend error; pass a recoverable provider-failure object to the service for graceful fallback.

### Step 7 — Optional Model Fallback

If configured:

```env
SP_AI_FALLBACK_RESPONSE_MODEL=gpt-4o-mini
```

then after rate-limit failure on `gpt-4o`, retry with fallback model using compact context.

Rules:
- only for non-sensitive answer generation;
- same structured schema;
- same grounding facts;
- log model fallback safely.
