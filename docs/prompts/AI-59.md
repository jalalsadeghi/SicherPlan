# AI-59 — Provider Failure Recovery Contract: Always Return a Safe Assistant Message for In-Scope Requests

## Purpose

Even after AI-56, the frontend still displays the generic error:

```text
Die Assistant-Anfrage konnte gerade nicht abgeschlossen werden.
```

The latest failure happened after OpenAI returned 200 OK but the backend failed parsing structured JSON due to truncation. This means fallback did not catch all provider failure paths.

For in-scope assistant requests, the backend must return a safe assistant message whenever it has enough grounding facts, even if the OpenAI provider fails after the request has started.

---

## Non-Negotiable Rule

The assistant must not show a generic error box for an in-scope question when any of the following exist:

- RAG grounding context;
- source_basis;
- diagnostic facts;
- allowed navigation links;
- tool summaries;
- provider raw text that is partially useful and safe;
- first provider call succeeded but later continuation/parsing failed.

The fallback response may be low confidence and clearly limited, but it must be helpful.

### Step 2 — Define Fallback Result Type

Add a fallback response builder that can construct a valid `AssistantStructuredResponse` from:

```text
grounding_context
rag_trace
tool_results
source_basis
allowed_links
question language
provider error type
partial provider raw_text if safe
```

It must return:

```json
{
  "answer": "...",
  "confidence": "low|medium",
  "out_of_scope": false,
  "diagnosis": [
    {
      "finding": "provider_degraded",
      "severity": "warning",
      "evidence": "The AI provider could not complete the full response, but SicherPlan grounding facts were available."
    }
  ],
  "links": [],
  "missing_permissions": [],
  "next_steps": [],
  "source_basis": [],
  "provider_degraded": true
}
```

If `provider_degraded` is not in the schema yet, add it.

### Step 3 — Fallback Content Rules

For German, if the question is about assigned shift not visible in app, and available facts are general diagnostic/workflow facts, fallback answer should be fact-grounded and say:

```text
Die vollständige KI-Antwort konnte gerade nicht abgeschlossen werden. Auf Basis der verfügbaren SicherPlan-Kontexte sollten Sie folgende Punkte prüfen: Mitarbeiterprofil und App-/Self-Service-Zugriff, aktive Zuweisung, Freigabestatus der Schicht bzw. des Schichtplans, Sichtbarkeit für die Mitarbeiter-App und Validierungen wie Abwesenheit, Qualifikation, Kundenblock oder Doppelbesetzung.
```

This text must be assembled from available diagnostic/workflow facts where possible. It is acceptable for fallback builder to use a small language-specific wrapper, but it must not invent business facts.

### Step 4 — Catch Structured Output Parse Failures

If OpenAI returns raw text/partial text but structured parsing fails:

1. Try to salvage a safe plain answer from `raw_text` if it is not empty and not malformed sensitive data.
2. Strip incomplete JSON syntax if necessary.
3. If raw_text is unsafe or too incomplete, use grounding-based fallback.
4. Do not return 502 if fallback is eligible.

### Step 5 — Catch Rate Limit and Continuation Failures

If the second provider call fails after first call/tool facts:

1. Use the collected tool facts and grounding context.
2. Return a degraded assistant message.
3. Include allowed links.
4. Store the message.
5. Return HTTP 201.

### Step 6 — Frontend Rendering

If backend returns `provider_degraded=true`, frontend should show a subtle warning inside the assistant message, not a blocking error box.

If backend still returns 502/503 with no fallback, existing error box is acceptable.

### Step 7 — Tests

Add tests:

```text
backend/tests/modules/assistant/test_provider_failure_recovery_contract.py
backend/tests/modules/assistant/test_structured_parse_failure_returns_degraded_message.py
backend/tests/modules/assistant/test_rate_limit_after_grounding_returns_degraded_message.py
backend/tests/modules/assistant/test_frontend_provider_degraded_message.ts
```

Required cases:

1. Structured JSON EOF parse error with grounding -> returns 201 degraded response.
2. Rate limit after grounding -> returns 201 degraded response.
3. Invalid continuation after tool facts -> returns 201 degraded response.
4. Provider auth error -> no fallback if provider cannot be used and no grounding exists.
5. Out-of-scope request -> no degraded fallback.
6. Permission denied -> no fallback leakage.

### Acceptance Criteria

This task is complete only when:

1. The current failing case no longer shows the generic frontend error.
2. Structured-output parse failures are recoverable when grounding exists.
3. Rate-limit failures are recoverable when grounding exists.
4. Degraded responses are stored as assistant messages and returned with HTTP 201.
5. User sees a helpful low/medium confidence answer.
6. Tests pass.
