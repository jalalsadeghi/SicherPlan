# Codex Prompt — US-36-T2

## Task title

**Run hypercare triage, issue routing, and stabilization backlog management.**

## Objective

Establish the post-go-live operating model for incident intake, severity assessment, ownership routing, hotfix escalation, and stabilization backlog curation during the first live support window.

## Source context

- Updated proposal: Phase 6 explicitly ends with stabilization after go-live.
- Sprint reference: `docs/sprint/sprint-12-migration-uat-and-go-live.md`.
- Cross-cutting rules: `AGENTS.md`, especially durable evidence, traceable backlog IDs, tenant/privacy protection, and honest operational reporting.
- This task translates release-time findings into a structured hypercare process rather than an ad hoc support chat stream.

## Dependencies

- `US-36-T1` must already provide the release path and cutover evidence that hypercare inherits.
- `US-35-T1` / `US-35-T4` should already provide UAT and acceptance issue categories that can be reused in hypercare severity/routing.
- Observability and alerting work from earlier sprints must already provide the event sources triage will consume.
- Any repository-native issue templates, label systems, or docs navigation should be reused instead of creating a disconnected support process.

## Scope of work

- Define the hypercare severity matrix, triage workflow, routing rules, owner/escalation matrix, and communication cadence for the initial stabilization window.
- Create issue/backlog templates, routing metadata, and stabilization backlog structure so incidents and release findings land in durable tracked artifacts.
- Define how issues move between immediate blocker/hotfix handling, short-term stabilization backlog, and later roadmap work.
- Document expected evidence for incidents, including logs, screenshots, user scope, tenant context, rollback/hotfix notes, and reproduction steps.
- Provide daily hypercare reporting and escalation templates that make support status visible without exposing unnecessary tenant-sensitive data.
- Keep the process tight enough for real operations rather than a heavyweight ITIL program.

## Preferred file targets

- `docs/support/` or `docs/runbooks/` for hypercare and issue-routing procedures.
- Issue templates, label configs, or repository-equivalent workflow metadata if the repo stores them in version control.
- Stabilization backlog index files or docs that the team can update during live support.
- Communication/report templates kept alongside the hypercare docs.

## Hard constraints

- All production issues and release findings must land in durable tracked artifacts, not only ephemeral chat or meeting notes.
- Severity, routing, and escalation must distinguish blockers, hotfixes, data-correction items, and backlog improvements.
- Do not hide incidents to preserve release optics.
- Keep tenant-sensitive data sanitized in shared artifacts and reports.
- Do not create a shadow backlog disconnected from the main repository workflow.

## Expected implementation outputs

- A hypercare runbook with severity and routing rules.
- Issue/backlog templates and stabilization backlog structure.
- Communication and daily-report templates for the support window.
- Owner/escalation guidance linked to real modules or service areas.

## Non-goals

- Do not redesign the permanent support organization beyond the scoped hypercare window.
- Do not build a new monitoring system here.
- Do not normalize silent hotfixes outside tracked workflow.
- Do not replace actual issue ownership with generic mailbox forwarding.

## Verification checklist

- [ ] A single triage path and severity matrix exist for hypercare.
- [ ] Issue evidence, routing, and ownership requirements are explicit.
- [ ] The stabilization backlog is durable and linked to repository tracking.
- [ ] Daily reporting and escalation cadence are documented.
- [ ] Shared artifacts remain privacy-safe while still operationally useful.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
