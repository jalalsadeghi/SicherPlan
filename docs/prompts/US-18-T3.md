# Codex Prompt — US-18-T3

## Task title

**Build shift maintenance UI with meeting point, location, break, type, and visibility flags.**

## Objective

Deliver the dispatcher-facing shift-maintenance experience in Vben so users can create and maintain reusable shifts with clear operational details, visibility controls, and localized workflows before the staffing engine lands.

## Source context

- Updated proposal: section 5.5 shift and staffing model plus operational tools; employee app expectations for shift detail with meeting point and route/map context; Appendix D `Shift`.
- Implementation specification: `ops.shift_plan`, `ops.shift_template`, `ops.shift_series`, and `ops.shift` fields including meeting point, location text, break minutes, shift type, release state, and visibility flags.
- Sprint reference: `docs/sprint/sprint-06-ops-planning-foundation.md`.
- Cross-cutting rules: `AGENTS.md`, especially Vben Admin, DE/EN, and theme/permission rules.

## Dependencies

- `US-18-T1` and `US-18-T2` must already provide backend contracts for shifts, templates, series, copy/reuse, workforce scope, and exception handling.
- `US-3-T1`, `US-3-T3`, and `US-3-T4` should already provide the Vben shell, theme tokens, and localization baseline.
- `US-5-T3` should already provide permission and route-guard scaffolding.

## Scope of work

- Build Vben list/detail/form workflows for shift plans, templates, recurring series, and concrete shift maintenance.
- Support editing of meeting point, location text, break minutes, shift type, start/end times, release state where allowed, and the visibility flags for customer, subcontractor, and stealth mode.
- Surface workforce scope, template selection, and recurrence/copy actions using Vben patterns that a dispatcher can understand without exposing unfinished staffing-engine concepts.
- Provide clear empty states and list filters by planning record, date, status, shift type, and visibility state.
- Keep the UI ready for later route/map and mobile consumers by presenting structured operational fields rather than free-form overloaded text.
- Add German and English locale resources for all new UI strings.
- Add frontend tests for edit flows, visibility-flag handling, route/permission guards, and the most important recurrence/copy actions.

## Preferred file targets

- Vben views/components/routes and API bindings under the planning/dispatch section of the web-admin workspace.
- Locale resources for DE and EN.
- Frontend tests for shift editing, recurrence/copy action visibility, and permission behavior.
- Only small backend contract additions if they are necessary to make the UI coherent.

## Hard constraints

- Follow Vben Admin conventions; do not build a custom standalone scheduling shell here.
- Do not leak unreleased or improperly scoped shift visibility to customer/subcontractor surfaces from admin workflows.
- Keep staffing-specific concepts like demand groups and assignments out of the UI for now; they land in Sprint 7.
- German must be the default locale and English must be provided for every new string.
- Visibility flags must remain explicit and understandable to operators; do not bury them in ambiguous advanced menus without labels/help text.
- Do not turn this task into full board-performance optimization; that belongs to `US-18-T4`.

## Expected implementation outputs

- Localized Vben shift-maintenance screens for plans, templates, series, and concrete shifts.
- Clear operator-facing controls for shift timing, location, break, type, and visibility flags.
- Frontend tests for the core edit and permission flows.

## Non-goals

- Do not implement staffing-board assignment UI, coverage views, or validation alerts here; those belong to Sprint 7.
- Do not implement live field, map, or patrol mobile flows here.
- Do not build reporting dashboards or finance views here.
- Do not move recurrence/copy logic into frontend-only state that bypasses backend services.

## Verification checklist

- [ ] Dispatchers can create and edit shift structures through localized Vben forms and lists.
- [ ] Meeting point, location, break, type, and visibility flags round-trip correctly between UI and backend.
- [ ] Recurrence/copy actions are exposed safely and only where the backend contract supports them.
- [ ] German is default and English is present for all new shift-maintenance UI strings.
- [ ] Frontend tests cover edit flows, permission behavior, and visibility-flag handling.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
