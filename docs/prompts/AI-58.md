# AI-58 — Reduce Diagnostic Tool Result Payloads and Prevent Huge 20k Token Prompts

## Purpose

The latest logs show that even after trimming, the assistant still sends a very large prompt:

```json
{
  "estimated_input_tokens": 20535,
  "estimated_grounding_tokens": 1647,
  "estimated_tool_result_tokens": 9213,
  "grounding_source_count": 1,
  "trimmed": true
}
```

The `tool_result_tokens` alone are estimated at 9,213 tokens. This is too large for a single assistant answer and contributes to rate limits, truncation, and unreliable structured output.

The assistant must summarize tool results aggressively before sending them to OpenAI. Tool outputs should be diagnostic facts, not raw large payloads.

---

## Product Requirement

Operational diagnostic questions should send a compact, high-signal diagnostic summary to OpenAI.

For the mobile-app visibility question, OpenAI should receive facts like:

```json
{
  "diagnostic_summary": {
    "question_type": "assigned_shift_not_visible_in_employee_app",
    "employee_identified": false,
    "date_identified": false,
    "completed_checks": ["accessible_pages", "general_workflow_context"],
    "missing_inputs": ["employee_name", "shift_date"],
    "possible_causes": [
      "assignment not active/confirmed",
      "shift or shift plan not released",
      "employee mobile/self-service account missing",
      "employee visibility flag not enabled",
      "absence/qualification/customer block/double booking validation",
      "released schedule not generated"
    ],
    "allowed_pages": ["Employees Workspace", "Shift Planning", "Staffing Board", "Dispatch & Releases"]
  }
}
```

It should not receive thousands of tokens of raw staffing board, planning, employee, document, or API payloads.

### Step 2 — Define Compact Tool Result Contract

For every assistant tool result sent to OpenAI, enforce:

```json
{
  "tool_name": "planning.find_assignments",
  "ok": true,
  "summary": {
    "result_state": "found|not_found|missing_input|permission_limited|failed",
    "facts": [],
    "counts": {},
    "missing_inputs": [],
    "missing_permissions": [],
    "entity_refs": [],
    "safe_note": ""
  }
}
```

Do not include:
- full records;
- full API payloads;
- long arrays;
- repeated source text;
- private/sensitive fields;
- more than a small number of matches.

### Step 3 — Add Tool Result Summarizers

Create summarizers for:

```text
employees.search_employee_by_name

```

Each summarizer should produce concise facts.

### Step 4 — Add Payload Limits

Add config:

```env
SP_AI_MAX_TOOL_RESULT_CHARS=1500
SP_AI_MAX_TOOL_RESULT_ITEMS=5
SP_AI_MAX_TOTAL_TOOL_RESULT_CHARS=3500
SP_AI_MAX_DIAGNOSTIC_FACTS=12
```

Before provider call:
- enforce per-tool limit;
- enforce total tool result limit;
- record truncation in RAG trace.

### Step 5 — Deduplicate Context

Do not send the same fact in multiple places.

If workflow facts are already in grounding sources, do not also include them as large tool results.

If page-help facts are already in source_basis, include only compact title/purpose/action facts.
