# AI-73 — Scope Classifier and Retrieval for Known Platform Terms

## Purpose

After AI-72 adds a Platform UI Term / Concept Corpus, the Assistant scope classifier must treat known UI terms as in-scope even when the user does not mention `SicherPlan`, `Shift coverage`, or a page name.

Failing example:

```text
was bedeutet Demand groups
```

Expected:
- classify as in-scope;
- intent: `platform_term_meaning_question`;
- retrieve platform term `planning.staffing.demand_groups`;
- answer in German because the question is German;
- mention that the term is used in Shift Coverage / Staffing Coverage.

---

## Non-Negotiable Requirements

1. Do not make every English phrase in-scope.
2. Only known platform terms/concepts from the corpus should trigger in-scope behavior.
3. Keep refusing unrelated non-platform terms.
4. Do not require explicit `im SicherPlan`.
5. Do not create canned final answers.

### Step 3 — Scope Decision Order

Update classifier order:

1. Safety/unsafe detection.
2. Field/lookup dictionary match.
3. Platform UI term/concept corpus match.
4. Current route/page context match.
5. Known platform workflow/domain terms.
6. Out-of-scope fallback.

If the query matches a platform term with sufficient confidence, classify as in-scope.

### Step 4 — Fuzzy Matching

Platform term matching must support:

- case-insensitive matching;
- singular/plural;
- camelCase key aliases;
- German/English mixed terms;
- hyphen variants:
  `Demand Group`, `Demand Groups`, `demand-group`, `demand groups`;
- partial label questions:
  `was bedeutet Demand groups`;
- i18n key aliases:
  `demandGroupsTitle`, `fieldsDemandGroup`.

Do not match very generic words alone:
- group
- type
- action
- title
- status

unless route context or stronger evidence exists.

### Step 5 — Retrieval Planner Integration
