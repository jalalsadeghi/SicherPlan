# AI-15 — Tool Registry Core

## Task Objective

Implement the **Tool Registry Core** for the SicherPlan AI Assistant.

The AI model must not access the database directly, execute SQL, or call arbitrary backend functions. It may only request tools that are explicitly registered, read-only for this sprint, permission-checked, scope-checked, redacted, audited, and validated.

This task implements the central infrastructure for assistant tools. It does **not** need to implement the full employee/planning/field diagnostic tools yet; those come in later tasks. This task must provide the base framework and enough sample/test tools to prove the policy works.

---

## Dependencies

This task assumes the following tasks are complete or available:

- `AI-02` AI Assistant Configuration and Feature Flags
- `AI-03` Assistant Permission Seeds and Access Policy Baseline
- `AI-05` Assistant Database Models and Migrations
- `AI-07` Structured Response Contract
- `AI-11` OpenAI Client Wrapper
- `AI-14` Prompt Builder and Security System Instructions

If tool-call audit tables or assistant permissions are missing, stop and report the gap before coding.

## Implementation Requirements

### 1. Tool Package Structure

Create or complete the assistant tools package.

Expected files, adjusted to actual repository conventions:

```text
backend/app/modules/assistant/tools/__init__.py
backend/app/modules/assistant/tools/base.py
backend/app/modules/assistant/tools/registry.py
backend/app/modules/assistant/tools/errors.py
backend/app/modules/assistant/tools/redaction.py
backend/app/modules/assistant/policy.py
backend/app/modules/assistant/repository.py
backend/tests/modules/assistant/test_tool_registry.py
backend/tests/modules/assistant/test_tool_permissions.py
```

Later tasks will add:

```text
backend/app/modules/assistant/tools/user_tools.py
backend/app/modules/assistant/tools/navigation_tools.py
backend/app/modules/assistant/tools/employee_tools.py
backend/app/modules/assistant/tools/planning_tools.py
backend/app/modules/assistant/tools/field_tools.py
```

You may create placeholder modules only if they are needed for imports, but do not implement domain logic prematurely.

### 2. Tool Definition Metadata

Every tool must have a typed definition with at least:

```text
name
description
input_schema
output_schema
required_permissions
scope_behavior
redaction_policy
classification
enabled
max_rows
timeout_seconds
```

Suggested enum values:

```text
classification:
  read_only
  write

scope_behavior:
  platform
  tenant
  branch
  mandate
  customer
  subcontractor
  employee_self_service
  current_user
```

For this sprint, only `read_only` tools may execute.

### 3. Base Tool Interface

Implement a base interface/class similar to:

```python
class AssistantTool(Protocol):
    definition: AssistantToolDefinition

    async def execute(
        self,
        *,
        input_data: dict[str, Any],
        context: AssistantToolExecutionContext,
    ) -> AssistantToolResult:
        ...
```

Adapt to existing sync/async patterns.

The execution context must include backend-derived data only:

```text
request auth context
tenant_id
user_id
role_keys
permission_keys
scope information
conversation_id
message_id
request_id/correlation_id if available
```

The tool must **not** trust tenant ID, user ID, role, or permissions supplied by the model.

### 4. Registry Behavior

Implement a central registry that can:

```text
register tool
list available tools for a user/context
get tool by name
validate tool input
check tool classification
check required permissions
execute tool
redact output
audit tool call
return safe result or safe error
```

Required behavior:

- unknown tool -> rejected;
- disabled tool -> rejected;
- write tool -> rejected in this sprint;
- missing permission -> rejected and recorded as denied;
- input validation error -> safe failure;
- unexpected exception -> safe failure without stack trace in user response;
- all decisions are auditable.

### 5. Permission Enforcement

Use existing IAM/authz helpers discovered in AI-01/AI-03.

Do not invent a separate authorization system if the project already has one.

The registry must check:

```text
required_permissions
assistant.chat.access if needed by caller
assistant.diagnostics.read for diagnostic tools
assistant.knowledge.read for retrieval-like tools if applicable
```

The registry should produce structured missing permission output that can be used by `AI-11` and future diagnostic answers.

### 6. Read-only Enforcement

Add a hard guard:

```text
if tool.definition.classification != "read_only":
    reject execution
```

Even if a write tool is accidentally registered, it must not execute during this sprint.

### 7. Tool Input Validation

Use Pydantic or existing validation conventions.

Validation should:

- reject unexpected required-field omissions;
- reject invalid dates/UUIDs when schemas define them;
- cap list sizes;
- cap text field lengths;
- reject raw SQL-like execution fields if a tool input tries to smuggle them.

Do not rely only on model instructions.

### 8. Tool Output Redaction

Add a redaction helper.

It must remove or mask common sensitive keys:

```text
password
password_hash
token
jwt
secret
api_key
authorization
refresh_token
reset_token
social_insurance_no
tax_id
bank
iban
bic
health_insurance
religion
private_hr
payroll_amount
salary
wage
```

Adjust to actual model fields and project naming conventions.

The redaction helper should work recursively on dictionaries/lists.

### 9. Tool Call Audit

When a tool is requested, record an audit row in `assistant_tool_call` or the existing equivalent table.

Audit should include:

```text
conversation_id
message_id if available
tenant_id
user_id
tool_name
input_json after safe redaction if needed
output_json_summary after redaction
required_permissions
permission_decision: allowed | denied | failed
scope_kind
entity_refs if available
duration_ms
error_code if any
created_at
```

Do not store raw secrets or full sensitive payloads.

If conversation/message IDs are unavailable in low-level tests, allow a safe testing context or document how the registry handles it.

### 10. Safe Error Types

Implement safe internal error types such as:

```text
AssistantToolNotFoundError
AssistantToolDisabledError
AssistantToolWriteBlockedError
AssistantToolPermissionDeniedError
AssistantToolInputValidationError
AssistantToolExecutionError
```

The API/service layer should receive a safe structured result, not raw exceptions.

### 11. Sample/Test Tools

For this task only, implement minimal test tools to prove the registry works. Examples:

```text
assistant.test_echo_read
assistant.test_write_blocked
assistant.test_permission_required
```

These tools must be test-only or clearly internal and not exposed in production unless the project has a debug/test registration mode.

Do not expose test tools to normal runtime if that conflicts with production safety.

### 12. Tests

Add or update:

```text
backend/tests/modules/assistant/test_tool_registry.py
backend/tests/modules/assistant/test_tool_permissions.py
```

Required scenarios:

1. known read-only tool executes when permission is present;
2. unknown tool is rejected;
3. disabled tool is rejected;
4. write tool is rejected even if registered;
5. missing permission is rejected;
6. invalid input is rejected;
7. redaction masks sensitive fields recursively;
8. tool call audit row is created for allowed execution;
9. tool call audit row is created for denied execution;
10. unexpected tool exception is converted to safe failure;
11. output does not include stack traces;
12. tool list for model contains only available allowed tools.

Tests must not call OpenAI.
