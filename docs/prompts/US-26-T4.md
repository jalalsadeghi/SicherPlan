# Codex Prompt — US-26-T4

## Task title

**Keep raw evidence append-only while allowing controlled derived-status transitions.**

## Objective

Formalize the lifecycle rules that let time events, attendance summaries, and actual bridges progress through derived states and correction-aware transitions without ever losing or rewriting the underlying evidence that explains how the result was produced.

## Source context

- Sprint-wide notes require raw time capture to stay append-only except for controlled validation-status updates, while attendance and finance actuals remain derived layers.
- Implementation specification section 2 prefers append-only or archive-only behavior for finance, compliance, and audit-sensitive records; section 4.7 defines raw field evidence as immutable wherever possible.
- Proposal audit and traceability requirements require complete history for compliance actions and financially relevant updates.
- Sprint reference: `docs/sprint/sprint-09-time-capture-and-actuals.md`.

## Dependencies

- `US-25-T2` and `US-25-T3` must already define raw-event write and validation-status behavior.
- `US-26-T1` and `US-26-T2` must already define attendance and actual derivation paths.
- `US-5-T2` or equivalent audit baseline should already exist so controlled status transitions can emit proper audit traces.

## Scope of work

- Implement explicit lifecycle/state-transition rules for raw time events, attendance summaries, and actual bridges where the repository still lacks them.
- Allow only narrowly approved mutable fields on raw evidence (for example validation-status or archival metadata) and make all other correction flows happen through derived or linked records instead of in-place mutation.
- Add audit-safe transition handling for discrepancy acknowledgment, derivation refresh, superseded summaries, or similar repository-approved lifecycle events.
- Document and test the allowed transition paths so future tasks do not accidentally reintroduce destructive edits to raw evidence.
- Where necessary, add service-level guards, repository helpers, or model protections that prevent unsafe update paths.

## Preferred file targets

- `modules/field_execution` and `modules/finance` state-transition services/guards.
- Audit helpers or repository-level protections where they already exist.
- Tests specifically targeting forbidden mutation paths and allowed lifecycle transitions.

## Hard constraints

- Do not add convenience update endpoints that can directly rewrite captured timestamps, raw token data, or evidence payloads.
- Derived-status transitions must be explicit, tested, and audit-safe.
- Do not turn soft supersession into hard deletion of historically relevant records.
- Keep lifecycle rules understandable to future maintainers; avoid hidden ORM magic with no tests or docs.
- Do not broaden this task into full approval workflow UX or export generation.

## Expected implementation outputs

- Explicit lifecycle/transition rules for raw events, attendance summaries, and actual records.
- Guards against unsafe in-place mutation of evidence records.
- Tests proving allowed vs forbidden state transitions and auditability.

## Non-goals

- Do not implement all finance approval stages here.
- Do not create new archival warehouses or data lakes for evidence history.
- Do not add user-facing discrepancy review dashboards beyond what is strictly needed for lifecycle integrity.

## Verification checklist

- [ ] Unsafe raw-evidence mutations are blocked by service or model-level safeguards.
- [ ] Allowed validation-status and derived-state transitions remain possible and audited.
- [ ] Superseded or refreshed derived records preserve historical traceability.
- [ ] Tests cover both forbidden writes and approved transitions.
- [ ] Lifecycle rules are readable enough for later Sprint 9 and Sprint 10 tasks to reuse safely.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
