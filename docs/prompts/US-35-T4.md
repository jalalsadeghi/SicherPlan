# Codex Prompt — US-35-T4

## Task title

**Finalize go-live checklist, cutover rehearsals, and business acceptance signoff.**

## Objective

Turn UAT, migration, and hardening outputs into a release-readiness package with explicit cutover, rollback, communication, and acceptance checkpoints so go-live becomes a controlled business event rather than an improvised deployment.

## Source context

- Updated proposal: Phase 6 covers reporting, migration support, user training, UAT, production go-live, and stabilization; deliverables include go-live support and acceptance across the core workflows and quality controls.
- Sprint reference: `docs/sprint/sprint-12-migration-uat-and-go-live.md`.
- Cross-cutting rules: `AGENTS.md`, especially traceability, tenant safety, document/output durability, and repository-backed operational discipline.
- This task bridges the evidence from migration validation, UAT, localization/accessibility review, and hardening into a final release-readiness package.

## Dependencies

- `US-35-T1` through `US-35-T3` must already provide UAT evidence, training readiness, localization review, and print-template signoff status.
- `US-34-*` must already provide migration templates, seeded configuration, document/output pilots, and migration validation evidence.
- `US-33-T3` and `US-33-T4` should already provide hardening-drill outcomes and residual-risk visibility.
- Existing deployment, backup, and observability assets from Sprint 1 / Sprint 11 should be reused in the checklist and rehearsal package.

## Scope of work

- Create a go-live checklist that covers environment readiness, backups, migrations, seed/config completion, integration status, portal/mobile readiness, generated-output readiness, monitoring, support contacts, and rollback prerequisites.
- Prepare cutover rehearsal runbooks with role assignments, timing checkpoints, rehearsal evidence, issue-capture forms, and rerun expectations.
- Define a business-acceptance signoff template mapped directly to the proposal acceptance criteria and UAT evidence rather than a generic approval form.
- Document communications plan, freeze windows, stakeholder approvals, and handover prerequisites required before production release.
- Link rehearsal outcomes and unresolved risks into the stabilization backlog so carry-forward work remains explicit.
- Keep the release-readiness package in-repo and versioned so production cutover starts from one authoritative source.

## Preferred file targets

- `docs/runbooks/` for go-live checklists and cutover rehearsal instructions.
- `docs/acceptance/` for business signoff templates and acceptance traceability.
- Any repository-equivalent ops or QA docs areas already used for release management.
- Supporting templates for risk registers, communication notes, or issue capture if the repo keeps such assets in-repo.

## Hard constraints

- The checklist and signoff package must reference real repository assets, environments, and evidence rather than generic placeholders only.
- Do not approve business acceptance without explicit disclosure of unresolved blockers or major risks.
- Rollback criteria must be concrete and testable.
- Do not execute production changes in this task.
- Keep the package traceable to proposal acceptance criteria and actual UAT/hardening evidence.

## Expected implementation outputs

- A release-readiness / go-live checklist package.
- Cutover rehearsal runbooks and issue-capture templates.
- Business-acceptance signoff templates mapped to proposal acceptance criteria.
- An explicit risk register or carry-forward list tied to stabilization.

## Non-goals

- Do not execute the production cutover yet.
- Do not replace detailed runbooks from earlier hardening tasks with hand-wavy summaries.
- Do not hide residual risks just to make the release look ready.
- Do not implement new business features while preparing signoff.

## Verification checklist

- [ ] The go-live checklist maps to real evidence, environments, and repository assets.
- [ ] Cutover rehearsals are documented in a repeatable way.
- [ ] Business signoff is traceable to proposal acceptance criteria and UAT evidence.
- [ ] Rollback, communication, and support prerequisites are explicit.
- [ ] Residual risks and carry-forward actions are documented honestly.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
