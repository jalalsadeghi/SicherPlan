# Codex Prompt — US-24-T2

## Task title

**Support QR, barcode, NFC/manual scan methods and exception/abort evidence.**

## Objective

Expand patrol control so checkpoint capture can happen through the approved scan methods or controlled manual fallback, while exception and abort cases remain fully evidenced with notes and photos in a way that stays compatible with the normalized patrol tables.

## Source context

- Updated proposal: section 5.2 guard patrol control system explicitly requires NFC, barcode, or QR checkpoints, variable checkpoint sequence, manual point capture or round abort with reason/photo, and direct transfer into evaluations/watchbook.
- Implementation specification: `ops.patrol_checkpoint` defines scan type and expected token; `field.patrol_round_event` stores `event_type_code`, `scan_method_code`, `token_value`, lat/long, note, and attachments through `docs.document_link`; `field.patrol_round` stores round-level status and `abort_reason`.
- Sprint reference: `docs/sprint/sprint-08-employee-app-and-field-i.md`.
- Cross-cutting rules: `AGENTS.md`, especially append-only event evidence and docs-backed generated/attached outputs.
- Earlier prompt context: `US-24-T1` provides the base patrol-round lifecycle and event sequencing.

## Dependencies

- `US-24-T1` must already provide the patrol-round lifecycle and event persistence baseline.
- `US-6-T1` must already provide docs attachment/link handling for photo evidence.
- The mobile shell and device integrations must already be ready enough to host scan adapters without re-architecting auth or navigation.
- If a device lacks NFC or barcode capability, the manual fallback must still preserve evidence and policy behavior rather than silently skipping validation.

## Scope of work

- Implement checkpoint capture through QR, barcode, NFC, and controlled manual capture paths, mapping each capture into the approved `scan_method_code` and event metadata fields.
- Validate captured token values against the route/checkpoint context where policy requires it, while preserving failed or exceptional attempts as auditable event evidence when appropriate.
- Implement round-abort and exception capture with reason, note, and photo/document evidence in a schema-consistent way.
- When abort needs photo evidence, model it through an exception/abort patrol-round event plus docs attachments rather than stuffing binary evidence into `field.patrol_round`.
- Surface mobile UI states for expected-token capture, mismatches, manual fallback, and abort/exception handling.
- Add tests for scan-method mapping, token validation behavior, abort evidence persistence, and docs-linked photo handling.

## Preferred file targets

- Flutter patrol capture and device-adapter files under the mobile feature path.
- `modules/field_execution` services/validators for scan-method handling and exception events.
- Docs-link helpers only where abort/exception attachments need a narrow adapter.
- Tests for scan-method capture, token validation, and abort evidence.

## Hard constraints

- Keep checkpoint and exception evidence in append-only patrol-round events; do not overwrite prior capture rows.
- Use the approved checkpoint/token fields and docs linkage rather than inventing separate evidence tables for each scan type.
- Manual fallback must be explicit and auditable; it cannot silently bypass checkpoint policy.
- Abort/photo evidence must remain durable and linked through the docs service.
- Do not conflate patrol-checkpoint tokens with employee credential tokens or unrelated scan workflows.

## Expected implementation outputs

- Multi-method checkpoint capture support across QR, barcode, NFC, and manual modes.
- Exception and abort evidence handling with reason/note/photo support.
- Mobile UI states for scan success, mismatch, fallback, and abort flows.
- Tests for scan-method behavior, evidence persistence, and docs linkage.

## Non-goals

- Do not implement offline synchronization here; that belongs to `US-24-T3`.
- Do not generate watchbook/evaluation linkage here; that belongs to `US-24-T4`.
- Do not add separate per-scan-type tables or bypass the approved patrol event model.
- Do not treat failed or exceptional events as disposable if they should remain auditable.

## Verification checklist

- [ ] Each scan method maps cleanly into the approved patrol event model.
- [ ] Manual fallback and abort flows remain explicit, auditable, and evidence-bearing.
- [ ] Photo/doc evidence for aborts or exceptions is linked through the docs backbone.
- [ ] Token validation behavior is safe and consistent with checkpoint policy.
- [ ] Tests cover scan-method mapping, mismatch/exception handling, and abort evidence.
