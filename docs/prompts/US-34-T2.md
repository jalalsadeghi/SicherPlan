# Codex Prompt — US-34-T2

## Task title

**Seed lookup/reference data, numbering rules, document types, and print templates.**

## Objective

Finalize the configuration baseline so migrated and newly created records can rely on stable reference catalogs, numbering rules, document-type definitions, and print-template metadata from day one of UAT and go-live.

## Source context

- Updated proposal: the deliverables explicitly include a configuration package for roles, document types, qualification rules, tariff rules, and print templates, and the assumptions say UI labels, print templates, localization details, and report layouts are finalized during design workshops.
- Implementation specification: `core.lookup_value` is the tenant-extensible catalog for business lists, `core.tenant_setting` is the source for numbering rules, print-template configuration, and portal toggles, and Phase H is where seed/reference data is finalized against the completed write model.
- Sprint reference: `docs/sprint/sprint-12-migration-uat-and-go-live.md`.
- Cross-cutting rules: `AGENTS.md`, especially DE-default / EN-secondary behavior, centralized generated-output tracking, and configuration ownership through core/platform services.

## Dependencies

- `US-4-T3` should already provide the earlier lookup-seeding baseline that this task extends and finalizes.
- `US-6-T1` must already provide document and generated-output tracking conventions used by document types and print outputs.
- `US-23-T3`, `US-29-T2`, and related generated-output tasks should already define the main PDF surfaces that now need centralized template metadata.
- `US-34-T1` should already define migration-template fields that rely on the seeded domains finalized here.

## Scope of work

- Define and implement idempotent seed bundles for the lookup and reference domains required at go-live, limited to domains actually used by the approved proposal/specification.
- Implement numbering-rule configuration for key tenant-scoped business numbers such as customer, personnel, order, invoice, and other approved operational/finance sequences used by the repository.
- Define document-type catalogs and visibility/usage metadata for HR, customer, subcontractor, planning, finance, compliance, and generated-output flows.
- Register print-template metadata/configuration for key outputs such as watchbook PDFs, patrol reports, timesheets, invoices, ID/work badges, and other approved generated artifacts already present in scope.
- Ensure user-facing reference labels and template names follow German default and English secondary conventions.
- Add seed loaders, bootstrap hooks, or data migrations plus tests that prove idempotency, upgrade safety, and completeness.

## Preferred file targets

- Seed or bootstrap logic under repository-appropriate config/platform-service areas such as `modules/core/` and `modules/platform_services/`.
- Alembic data revisions or repository-equivalent seed assets.
- `docs/configuration/` for numbering-rule and print-template notes if the repo keeps operational configuration docs in-repo.
- Tests for seed idempotency, numbering-rule behavior, and template/document-type registration.

## Hard constraints

- Do not scatter reference constants across backend and UI layers when `core.lookup_value` or `core.tenant_setting` should own them.
- Seed data must be idempotent, version-aware, and safe to re-run in staging/UAT/production automation.
- Numbering rules must not depend on ad hoc client-side logic or untracked manual counters.
- Print-template configuration must remain separate from generated document instances.
- Do not over-seed speculative domains that are not grounded in the approved proposal or implementation spec.

## Expected implementation outputs

- Idempotent seed bundles for required lookup/reference domains.
- Centralized numbering-rule configuration and validation.
- Document-type definitions plus print-template metadata/configuration.
- Tests and operator notes for bootstrap/reseed behavior.

## Non-goals

- Do not build a full WYSIWYG template designer.
- Do not rewrite the document-generation engines themselves unless a narrow configuration seam is missing.
- Do not add unsupported languages beyond German and English.
- Do not invent speculative report categories or template families without source justification.

## Verification checklist

- [ ] Seeds can be applied repeatedly without duplicating or corrupting data.
- [ ] Numbering rules cover the approved tenant-scoped business numbers and are tested.
- [ ] Document types and print-template metadata map to real repository outputs rather than unused placeholders.
- [ ] User-facing seed labels and template names are present in German and English.
- [ ] Bootstrap/reseed behavior remains safe across staging/UAT/prod-style environments.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
