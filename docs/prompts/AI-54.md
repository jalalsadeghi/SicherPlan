# AI-54 — Proactive Diagnostic Prefetch and Single-Call OpenAI Generation

## Purpose

The assistant fails for this operational diagnostic question:

```text
Ich habe einem Mitarbeiter eine Schicht zugewiesen, aber sie wird in der mobilen App nicht angezeigt. Woran könnte das liegen?
```

The logs show that the first OpenAI call succeeds, then OpenAI requests a backend tool call, then a second OpenAI call is required to continue the response. That second call can hit TPM/rate limits because it sends another large context.

For known SicherPlan diagnostic scenarios, the backend should not wait for OpenAI to discover obvious tools. It should proactively run read-only diagnostic tools before the OpenAI call, add compact diagnostic facts to the RAG grounding context, and ask OpenAI to generate the final answer in a single provider call.

This keeps the response expert-like and OpenAI-generated, while reducing tool-loop failures and rate-limit exposure.

---

## Non-Negotiable Rules

1. Do not create canned final answers.
2. OpenAI must still generate the final user-facing answer.
3. Backend may run read-only diagnostic tools proactively as grounding sources.
4. All tools must still enforce tenant, role, permission, and scope.
5. If diagnostic input is incomplete, OpenAI should ask a targeted clarification instead of causing backend failure.

## Prompt for Codex

You are working on the SicherPlan repository.

Implement **AI-54 — Proactive Diagnostic Prefetch and Single-Call OpenAI Generation**.

### Step 1 — Reproduce and Classify the Failing Question

Before coding, reproduce:

```text
Ich habe einem Mitarbeiter eine Schicht zugewiesen, aber sie wird in der mobilen App nicht angezeigt. Woran könnte das liegen?
```

Report:

1. Detected language.
2. Classification / intent.
3. Retrieval plan.
4. Whether the system currently waits for OpenAI tool calls.
5. Which tool OpenAI requests.
6. Whether the second provider call causes 429/502.

### Step 2 — Add a Diagnostic Prefetch Planner

Create or extend a planner that identifies known operational diagnostic scenarios before calling OpenAI.

Required initial intents:

```text
employee_shift_not_visible_in_mobile_app
employee_assignment_visibility
shift_release_visibility
employee_app_schedule_visibility
```

German examples:

```text
Mitarbeiter sieht Schicht nicht in der App
Schicht wird in der mobilen App nicht angezeigt
zugewiesen, aber nicht sichtbar
Arbeitsschicht zugewiesen, mobile App zeigt sie nicht
```

English examples:

```text
employee cannot see assigned shift in app
shift assigned but not visible in mobile app
assigned shift not showing in employee app
why does the employee not see the shift
```

### Step 3 — Extract Missing Diagnostic Inputs

The diagnostic prefetch should extract, when present:

```json
{
  "employee_name": "optional",
  "date": "optional",
  "shift_ref": "optional",
  "assignment_ref": "optional",
  "route_context": "current page context"
}
```

If employee name/date/shift are missing, that is not an error. The diagnostic context should say what is missing and which checks can still be explained generically.

### Step 4 — Proactively Run Read-Only Tools

For the failing question class, run relevant tools before OpenAI when permissions allow:

```text
assistant.get_current_user_capabilities
assistant.search_accessible_pages
employees.search_employee_by_name if employee name exists
employees.get_employee_operational_profile if employee resolved
employees.get_employee_mobile_access_status if employee resolved and allowed
employees.get_employee_readiness_summary if employee resolved/date known and allowed
planning.find_assignments if employee/date known or route context can narrow
planning.find_shifts if date/route context known
planning.inspect_assignment if assignment resolved
planning.inspect_shift_release_state if shift resolved
planning.inspect_shift_visibility if shift resolved
field.inspect_released_schedule_visibility if enough input exists
navigation.build_allowed_link for E-01, P-03, P-04, P-05, ES-01 where allowed
```

All tool outputs must be redacted and summarized.

### Step 5 — Build Compact Diagnostic Grounding Facts

Add compact diagnostic sources to `grounding_context.sources` with source types such as:

```text
diagnostic_fact
tool_result_summary
missing_diagnostic_input
allowed_navigation_link
```

Example:

```json
{
  "source_type": "diagnostic_fact",
  "source_name": "Employee App Visibility Diagnostic",
  "title": "Shift assigned but not visible in employee app",
  "content": "Possible checked causes: employee account/mobile access, assignment status, shift release state, employee visibility flag, released schedule availability, absence/qualification validation.",
  "facts": {
    "employee_resolved": false,
    "date_present": false,
    "checks_completed": ["current_user_capabilities", "accessible_pages"],
    "checks_missing_input": ["employee_name", "date"]
  },
  "verified": true,
  "permission_checked": true
}
```

### Step 6 — Single OpenAI Call for Known Diagnostic Intents

For known diagnostic intents, do not expose diagnostic tools to OpenAI unless a follow-up tool is truly needed.

Preferred behavior:

```text
user question
-> backend detects diagnostic intent
-> backend runs diagnostic prefetch tools
-> backend builds compact grounding context
-> OpenAI generates final answer once
```

The model can still say:
- what it could verify;
- what it could not verify because employee/date are missing;
- which pages to open;
- what likely causes to check.

### Step 7 — Prompt Builder Update

Tell OpenAI:

```text
If diagnostic facts are present, answer from those facts.
If employee/date/shift are missing, ask for those details and still explain the general verified check sequence.
Do not request a tool unless the available diagnostic facts are insufficient and the missing input is present.
Do not invent actual records.
```

5. Tenant/RBAC enforcement remains intact.
6. Tests pass.
