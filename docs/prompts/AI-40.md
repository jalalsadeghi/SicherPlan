# AI-40 — Prove and Enforce Real RAG: Grounding Trace, Source Attribution, and Content-Bearing Retrieval

## Purpose

The assistant currently produces answers that look like a generic model response, for example:

> "Zurzeit habe ich keine verifizierten Informationen über den genauen Prozess zur Registrierung eines neuen Vertrags in SicherPlan..."

This is not acceptable for the SicherPlan AI Assistant. The user expects a true RAG system where every in-scope answer is generated by OpenAI **from retrieved SicherPlan-specific context**.

This task must make RAG behavior observable, testable, and enforceable. Codex must determine whether the current implementation is truly doing RAG or only sending the user question plus weak page hints to OpenAI.

---

## Product Requirement

For every in-scope SicherPlan question, the backend must be able to prove:

1. OpenAI was called.
2. Retrieval was executed.
3. Retrieval returned relevant, content-bearing sources.
4. The model received those sources as grounding context.
5. The final answer was generated from the grounding context.
6. The answer includes a safe source/evidence basis in the structured response.

A page ID alone is not enough. A grounding source such as only `C-01` or `PS-01` without content is not a sufficient RAG source for a how-to answer.

---

## Current Problem to Investigate

The answer for:

```text
Wie registriere ich einen neuen Vertrag?
```

was too generic and did not explain the actual SicherPlan handling of contracts/documents/customer/order attachments.

Codex must verify whether this happened because:

- no relevant docs were ingested;
- only page catalog entries were retrieved;
- the retriever found wrong or shallow sources;
- the grounding context was not actually included in the OpenAI input;
- the prompt did not require source-grounded answers;
- the model had no source IDs/snippets to cite;
- the term `Vertrag` was not mapped to `contract`, `document`, `customer`, `order attachments`, and `Platform Services`.

### Step 2 — Define Content-Bearing Grounding Sources

Create or formalize a concept of a **content-bearing source**.

Examples of content-bearing sources:

```text
knowledge_chunk with non-empty content
page_help_manifest with action/field/section details
workflow_fact with detailed verified steps
diagnostic_result with real checked facts
API endpoint map chunk with endpoint/action details
role/page coverage chunk with permission/page scope details
```

Examples of insufficient-only sources:

```text
page_id only
route_name only
page title only
current route only
generic role name only
```

The RAG orchestrator may include insufficient-only sources, but it must not treat them as enough to answer a detailed how-to question.

### Step 3 — Add RAG Trace Data

Add a safe RAG trace object to the structured assistant response or store it server-side with an admin/debug endpoint.

Preferred structured payload field:

```json
{
  "rag_trace": {
    "provider_called": true,
    "provider_mode": "openai",
    "retrieval_executed": true,
    "grounding_attached": true,
    "grounding_source_count": 9,
    "content_bearing_source_count": 4,
    "source_type_counts": {
      "knowledge_chunk": 3,
      "page_help_manifest": 1,
      "page_route": 2,
      "current_route": 1
    },
    "top_sources": [
      {
        "source_type": "knowledge_chunk",
        "source_name": "SicherPlan Operational Handbook",
        "page_id": "PS-01",
        "module_key": "platform_services",
        "title": "Platform Services Workspace",
        "score": 42.5,
        "content_preview": "Use the documents area to create a document, add versions, link it..."
      }
    ],
    "missing_context": [],
    "retrieval_plan": {
      "intent_category": "contract_or_document_register",
      "likely_page_ids": ["PS-01", "C-01", "P-02"],
      "required_sources": ["knowledge_chunks", "page_help_manifest", "page_route_catalog"]
    }
  }
}
```

If exposing this to all users is too noisy, store it server-side and expose it only via an admin/debug endpoint, for example:

```text
GET /api/assistant/conversations/{conversation_id}/messages/{message_id}/rag-trace
```

Access must require `assistant.admin` or a safe debug permission.

Do not include raw sensitive operational data in the trace.

### Step 4 — Add Source Basis to Final Answers

Extend the structured assistant response with a safe `source_basis` field:

```json
{
  "source_basis": [
    {
      "source_type": "knowledge_chunk",
      "source_name": "SicherPlan Operational Handbook",
      "page_id": "PS-01",
      "title": "Platform Services Workspace",
      "evidence": "Documents area supports create/version/link/download workflow."
    }
  ]
}
```

Rules:

- `source_basis` must be generated from retrieved sources, not invented by the model.
- The backend may derive `source_basis` from the grounding context and/or validate model-proposed sources.
- Do not expose sensitive operational data.
- If a claim mentions a precise UI label, the source basis must include a verified UI/page-help source.

### Step 5 — Enforce RAG Minimum Context for How-To Questions

For in-scope how-to questions, require at least one content-bearing source.

If retrieval returns only shallow page route sources, then the assistant must not produce a confident generic answer. Instead, OpenAI should be given a grounding context that says:

```text
The system found only shallow page hints and no verified workflow details. Answer carefully, say what is verified, and ask a clarifying question if needed. Do not invent exact steps.
```

Backend must set confidence low or medium when content-bearing sources are missing.

### Step 6 — Update Prompt Builder

Update `prompt_builder.py` to tell OpenAI:

1. You are answering with RAG.
2. Use only grounding sources for SicherPlan-specific claims.
3. Mention exact buttons/actions only when a verified UI source exists.
4. If only page hints are available, do not produce a full step-by-step workflow.
5. If a term is ambiguous, ask a clarifying question.
6. Do not recommend "look around" generically unless no better verified context exists.
7. Answer naturally in the user's language.
8. Include useful next steps from the retrieved context.

### Step 7 — Add Tests Proving RAG Is Used

Add tests:

```text
backend/tests/modules/assistant/test_rag_trace.py
backend/tests/modules/assistant/test_rag_content_bearing_sources.py
backend/tests/modules/assistant/test_rag_source_basis.py
backend/tests/modules/assistant/test_rag_prompt_includes_grounding_context.py
```

Tests must verify:

1. In-scope messages call provider.
2. Retrieval is executed.
3. Grounding context is attached to provider request.
4. At least one content-bearing source is required for detailed how-to confidence.
5. The final response includes `source_basis` or a stored RAG trace.
6. A fake provider can inspect the grounding context and prove it was passed.
7. If only shallow page sources are retrieved, confidence is reduced and missing context is reported.

### Step 8 — Regression Example

Add a regression test for:

```text
Wie registriere ich einen neuen Vertrag?
```

Expected behavior:

- Retrieval plan should include `contract_or_document_register`.
- Likely pages should include `PS-01`, `C-01`, and/or `P-02`.
- Top grounding sources must not be only `F-02` or `E-01`.
- If contract registration is not a standalone verified workflow, the answer must say that.
- The answer must explain the verified document/customer/order areas based on sources.
- The answer must include a source basis.
- The answer must not sound like generic external model knowledge.

## Acceptance Criteria

This task is complete only when:

1. RAG behavior is visible through trace/source evidence.
2. Every in-scope response can prove provider call + retrieval + grounding context.
3. Content-bearing source count is tracked.
4. Generic answers without sources are prevented or downgraded.
5. The `Vertrag` question produces a source-grounded answer or a source-grounded clarification.
6. Tests prove grounding context is actually passed to OpenAI/fake provider.
7. No secrets or sensitive data are exposed in traces.
