# Codex Prompt — US-21-T3

## Task title

**Support planning communication, stealth mode, and group-targeted dispatch messages.**

## Objective

Implement the planning-communication controls that let dispatch users target operational messages to the right groups while honoring stealth-mode visibility restrictions and the existing communication/outbox backbone.

## Source context

- Updated proposal: section 5.5 operational tools includes planning communication and stealth mode; section 5.7 internal communication scope; section 9 deliverables include messaging integration.
- Implementation specification: `ops.shift.stealth_mode_flag` in section 4.6, portal visibility chain in section 5, and communication/docs/integration ownership rules in sections 3 and 4.
- Sprint reference: `docs/sprint/sprint-07-staffing-engine.md`.
- Cross-cutting rules: `AGENTS.md`, especially outbox/integration discipline and Vben patterns.

## Dependencies

- `US-21-T1` must already provide release and visibility-state semantics.
- `US-6-T2` must already provide communication templates, outbound message records, recipients, delivery attempts, and outbox-style provider handling.
- `US-19-T2` and `US-19-T4` should already provide staffing/coverage context that the dispatcher uses to target communication.
- `US-3-T1` and `US-3-T4` must already provide the Vben shell and DE/EN localization structure.

## Scope of work

- Implement planning-service commands and supporting UI flows for preparing and queueing dispatch messages scoped to relevant operational audiences such as assigned employees, selected teams, released subcontractor participants, or other approved role-targeted groups.
- Honor `stealth_mode_flag` and related visibility rules when building message payloads, previews, recipient lists, and message history so hidden operational details are not leaked to the wrong audience.
- Reuse the existing communication/outbox backbone for templates, recipients, and delivery attempts rather than calling providers directly from planning transactions.
- Build Vben Admin controls for composing, previewing, confirming, and reviewing planning messages in the staffing/release context.
- Add backend and frontend tests for audience scoping, stealth-mode suppression, queueing behavior, and localized UI strings.

## Preferred file targets

- `modules/planning/` for planning-owned message commands/context builders.
- `modules/platform_services` or the repo-equivalent communication package for shared template/outbox integration where ownership already exists.
- Vben pages/components, message panels, modals, and locale files under the web admin application.

## Hard constraints

- Do not call email/SMS/push providers directly from the same transaction that writes planning state; use the existing outbox/integration pattern.
- Stealth mode must materially suppress or redact data for scoped audiences where the planning rules require it.
- Audience targeting must remain tenant-safe and role-scoped; do not let broad recipient selection bypass portal/mobile visibility rules.
- UI composition and history views must follow Vben patterns and DE default / EN secondary localization requirements.
- Do not turn this task into a generic company-wide notice system; it is planning/dispatch communication rooted in released or release-bound operational context.
- Do not expose finance-sensitive or HR-private data inside dispatch messages unless a prior approved contract explicitly allows it.

## Expected implementation outputs

- Planning-context communication commands integrated with the existing outbox/message backbone.
- Stealth-mode-safe audience targeting and message preview/history behavior.
- Tests covering recipient scope, stealth suppression, queueing behavior, and localized UI flows.

## Non-goals

- Do not implement general enterprise notices here; that already belongs to the information portal backbone.
- Do not implement portal/mobile message-reader UIs here unless the repo already needs a tiny contract seam.
- Do not bypass the existing communication infrastructure with ad-hoc provider integrations.
- Do not broaden this task into release-state or PDF-generation logic.

## Verification checklist

- [ ] Dispatch users can compose and queue planning-context messages to correctly scoped audiences.
- [ ] Stealth mode suppresses/redacts disallowed details consistently across previews, recipients, and stored delivery context.
- [ ] All external delivery uses the approved communication/outbox path.
- [ ] Vben UI flows ship with German default and English secondary strings.
- [ ] Tests cover recipient scoping, stealth-mode behavior, and queueing/integration handoff.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
