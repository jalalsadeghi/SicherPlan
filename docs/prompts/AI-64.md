# AI-64 — Context-Aware Field and Lookup Questions Must Be In-Scope

## Purpose

Fix the Assistant scope classifier so short field-label questions are treated as in-scope when asked by authenticated users inside the platform.

Current failing example:

```text
User: was bedeutet Rechtlicher Name
Assistant: Ich bin nur dafür vorgesehen...
```

Expected:
The Assistant recognizes this as a field-help question because `Rechtlicher Name` is a known SicherPlan field label or matches the current page context.

---

## Non-Negotiable Rule

Do not make every generic phrase in the world in-scope. Use field/lookup dictionary matching, route context, and authenticated platform context.

---

### Step 3 — Scope Decision Order

Change scope decision order:

1. Unsafe detection.
2. Field/lookup dictionary exact or fuzzy match.
3. Current route/page context match.
4. Known platform/workflow/domain terms.
5. Out-of-scope fallback.

If a message matches a field/lookup definition, it is in-scope even if it does not contain `SicherPlan`.

### Step 4 — Use Current Route Context

If the user asks a vague field question such as:

```text
was bedeutet dieses Feld
what is this field for
```

and route context includes a known page/form, the Assistant should:
- use current page/page_id/visible title;
- if no exact field is named, ask which field the user means;
- if the frontend can provide focused field metadata in future, use that.

For now, implement named-field matching first.

### Step 5 — Fuzzy Field Label Matching

Implement robust matching:
- case-insensitive;
- accent/diacritic tolerant;
- punctuation tolerant;
- singular/plural tolerant where useful;
- German capitalization tolerant;
- supports labels with spaces, underscores, camelCase, i18n keys.

Examples:

```text
Rechtlicher Name
rechtlicher name
legal_name
LegalName
customerAdmin.fields.legalName
```

### Step 6 — Ambiguity Handling

If a label appears in multiple modules, the Assistant should not refuse. It should:
- prefer current route/page context;
- otherwise present likely candidates and ask a short clarification;
- keep confidence medium/low.

Example:
```text
Status
```

Possible fields:
- Customer status;
- Employee status;
- Order release state;
- Shift release state.

### Step 7 — Retrieval Planner Integration

For field/lookup intent:
- retrieve field dictionary matches;
- retrieve page help/form section facts;
- retrieve related workflow/page facts;
- retrieve allowed navigation link if page is known.

### Step 8 — Prompt Builder Update

Add instructions:

```text
If the user asks what a field, label, lookup, status, or option means, answer as a SicherPlan field-help assistant.
Do not require the user to write "SicherPlan" when the label matches a known platform field.
Explain meaning, where it appears, how it is used, and if relevant how it affects downstream workflows.
If ambiguous, ask a concise clarification and list likely places.
```
