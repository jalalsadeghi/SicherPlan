# Codex Prompt — US-34-T4

## Task title

**Validate migrated records against portal, planning, and finance workflows.**

## Objective

Run a repeatable trial-migration validation cycle that proves migrated master data, documents, and configuration are sufficient for real portal, planning, and finance operations before formal UAT and cutover begin.

## Source context

- Updated proposal: acceptance requires tenant isolation, end-to-end execution of the three reference workflows, enforced qualification/document rules, consistent actuals/audit evidence, correct payroll and invoice results, and reproducible reports/exports.
- Implementation specification: migration phases A–H, strict service ownership, centralized documents, and `finance.actual_record` as the finance-owned bridge between released planning/field evidence and commercial/payroll outputs.
- Sprint reference: `docs/sprint/sprint-12-migration-uat-and-go-live.md`.
- Cross-cutting rules: `AGENTS.md`, especially tenant safety, portal privacy, centralized docs, and reporting reproducibility.

## Dependencies

- `US-34-T1` through `US-34-T3` must already provide import templates, seed/configuration, and historical-document/output pilot behavior.
- `US-9-*`, `US-15-*`, and `US-22-*` through `US-30-*` must already provide the portal, planning, field, actual, payroll, invoice, and subcontractor-control surfaces that migrated data now needs to exercise.
- `US-31-*` and `US-32-*` should already provide reporting/export surfaces needed for validation of migrated outputs.
- Any environment reset or demo-data automation from earlier sprints should be reused to keep the validation cycle repeatable.

## Scope of work

- Build trial-migration validation scenarios that load migrated customers, employees, subcontractors, orders, documents, and seeded configuration through normal service/UI/read-model paths.
- Validate customer and subcontractor portal visibility, planning release and assignment validation, representative actual reconciliation, invoice/timesheet generation, and selected reporting/export flows against migrated data.
- Add consistency checks for business numbers, lookup resolution, role scopes, document permissions, generated outputs, and cross-module references after import.
- Produce durable validation evidence and a repository-backed defect or remediation list for issues discovered during the trial cycle.
- Document reset/replay instructions so the same migration validation can be rerun in staging or UAT environments with minimal drift.

## Preferred file targets

- `docs/migration/` or `docs/uat/` for validation scenarios, evidence indexes, and replay instructions.
- API/e2e/smoke-test areas already used by the repository for integrated validation.
- Import/export or QA harness code that supports repeatable trial-load verification.
- Issue/backlog documents for migration defects and carry-forward fixes.

## Hard constraints

- Validation must use normal service, UI, or read-model paths; direct database inspection alone is not sufficient.
- Do not patch migrated data manually without recording the defect, the correction path, and the rerun evidence.
- Do not bypass `finance.actual_record`, portal scope filters, or document-permission rules just to make validation pass.
- Evidence must be stored in repository artifacts or durable docs, not left only in chat or personal notes.
- Keep tenant and privacy scope intact even in trial-migration environments.

## Expected implementation outputs

- A repeatable migration-validation suite or checklist package.
- Evidence artifacts for portal, planning, finance, and reporting validation against migrated data.
- A migration-defect/remediation list with clear rerun expectations.
- Replay/reset instructions for staging/UAT validation cycles.

## Non-goals

- Do not treat this as final business UAT signoff.
- Do not execute production cutover.
- Do not redesign upstream modules to accommodate bad legacy data without explicit backlog decisions.
- Do not rely on manual one-off fixes that cannot be replayed.

## Verification checklist

- [ ] Migrated data successfully powers representative portal, planning, finance, and reporting flows.
- [ ] Scope and visibility rules still hold after import.
- [ ] Generated outputs and linked documents remain accessible only to authorized users.
- [ ] The validation cycle can be replayed repeatably in staging/UAT-like environments.
- [ ] Issues discovered during validation are captured durably with rerun expectations.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
