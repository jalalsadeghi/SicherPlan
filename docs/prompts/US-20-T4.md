# Codex Prompt — US-20-T4

## Task title

**Capture override reasons in audit-safe validation records and UI alerts.**

## Objective

Implement the durable override-evidence flow for planning validations so justified exceptions remain explicit, permission-controlled, append-only, and visible to dispatch users through localized Vben alert patterns.

## Source context

- Updated proposal: section 5.5 validation rules and Appendix E both require override logging when warnings/blocks are allowed to be bypassed.
- Implementation specification: section 4.6 `ops.assignment_validation_override`; section 5 compliance chain; section 6 validation mapping and override evidence guidance.
- Sprint reference: `docs/sprint/sprint-07-staffing-engine.md`.
- Cross-cutting rules: `AGENTS.md`, especially auditability, Vben patterns, and DE/EN localization.

## Dependencies

- `US-20-T1` through `US-20-T3` must already provide structured validation results and rule codes.
- `US-19-T1` must already provide stable assignment identifiers that override records can attach to.
- `US-5-T4` should already provide audit-event capture patterns for sensitive business actions.
- `US-3-T1` and `US-3-T4` must already provide the Vben shell and DE/EN resource structure used by planning UI alerts.

## Scope of work

- Create the migration, SQLAlchemy model, schemas, services, routers, and tests for `ops.assignment_validation_override` as an append-only audit table keyed to assignments and rule codes.
- Implement an explicit override command flow that requires acting-user attribution, reason text, timestamp capture, and permission checks before a rule can be bypassed.
- Respect policy distinctions between warnings, overrideable blocking rules, and non-overrideable hard prevents. If a rule is not overrideable, the API and UI must say so clearly.
- Add Vben Admin alert/badge/dialog flows that display validation outcomes, show whether an override is possible, collect the required reason, and surface existing override evidence where useful.
- Keep UI state aligned with the backend validation/override contract instead of re-implementing rule logic in the browser.
- Add backend and frontend tests covering required-reason behavior, permission denial, append-only evidence, and alert/dialog rendering.

## Preferred file targets

- Alembic revisions and planning-domain files under `modules/planning/` for override persistence and command handling.
- Vben pages/components, alert helpers, modals/drawers, and locale files under the web admin application.
- Audit integration hooks only where they follow the repository’s existing business-audit pattern.

## Hard constraints

- `ops.assignment_validation_override` must be append-only. Do not allow silent edits or destructive mutation of historical override evidence.
- Do not permit overrides when the current rule policy marks the failure as non-overrideable hard block.
- UI alerts must reflect backend truth; do not create frontend-only bypass behavior.
- Override reasons are compliance-relevant data and must be tenant-scoped, permission-controlled, and auditable.
- German is the default UI language and English must ship alongside every new alert, dialog label, and action message.
- Do not broaden this task into rewriting validation rules themselves or implementing release publication.

## Expected implementation outputs

- Append-only override persistence and command handling for validation exceptions.
- Localized Vben alert/dialog flows for viewing validation outcomes and recording permitted overrides.
- Tests proving reason capture, permission control, append-only evidence, and UI/backend contract alignment.

## Non-goals

- Do not implement the underlying validation rules here; those belong to `US-20-T1` through `US-20-T3`.
- Do not implement release-state publication or portal/mobile exposure here; that belongs to `US-21-*`.
- Do not turn override evidence into a generic comment system unrelated to validation rules.
- Do not allow silent auto-overrides or hidden policy downgrades.

## Verification checklist

- [ ] Override records are append-only, tenant-scoped, and linked to specific assignments and rule codes.
- [ ] A reason is mandatory whenever a permitted override is recorded.
- [ ] Non-overrideable hard prevents cannot be bypassed through API or UI tricks.
- [ ] Vben alert/dialog flows reflect backend validation truth and ship with German default and English secondary strings.
- [ ] Tests cover permission, append-only evidence, required reason, and UI rendering behavior.
