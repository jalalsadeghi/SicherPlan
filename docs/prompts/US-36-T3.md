# Codex Prompt — US-36-T3

## Task title

**Confirm monitoring/KPI dashboards and operational support handover.**

## Objective

Finalize the operational handover so support teams inherit real dashboards, alert paths, KPI definitions, runbooks, and ownership boundaries for the live SicherPlan platform instead of a vague 'ops will monitor it' assumption.

## Source context

- Updated proposal: operational delivery includes observability, and the report catalogue includes system/security, QM/compliance, planning, payroll/finance, customer, and subcontractor outputs that must remain reproducible.
- Implementation specification: reporting is view-driven, security and compliance activity have dedicated read models, and earlier phases already established the transactional and reporting backbone this handover now needs to expose safely.
- Sprint reference: `docs/sprint/sprint-12-migration-uat-and-go-live.md`.
- Cross-cutting rules: `AGENTS.md`, especially report reproducibility, role scope on dashboards/exports, centralized docs, and honest operational discipline.

## Dependencies

- `US-2-T4` must already provide the logging, tracing, and health baseline.
- `US-31-*` and `US-32-*` should already provide the key reporting/read-model surfaces that inform operational/business KPIs.
- `US-36-T1` and `US-36-T2` must already define the release path and hypercare routing that operations will hand over into.
- Any repo-managed observability dashboards, alert configs, or runbook catalogs should be reused instead of starting a second ops index.

## Scope of work

- Confirm and document the dashboards or equivalent operational views for platform health, authentication failures, job/outbox backlog, message delivery, import/export jobs, generated documents, portal availability, and selected approved business KPIs that are already backed by implemented metrics or read models.
- Document alert thresholds, ownership, response runbooks, and escalation paths for the monitored surfaces that matter during and after go-live.
- Create an operational support handover pack covering architecture snapshot, module ownership, backup/restore responsibilities, scheduler/integration duties, common failure modes, and support expectations.
- Link dashboards, alerts, and runbooks from a single ops index or catalog so on-call/support staff do not have to hunt across the repository.
- Make business KPI definitions explicit about their data source, refresh semantics, and intended audience.
- Keep support handover grounded in what the repository and live platform actually expose.

## Preferred file targets

- `docs/support/` for handover, dashboard catalog, and alert/runbook mapping.
- Observability config files or references if the repository stores them in version control.
- Runbook indexes or ops docs already present in the repository.
- Any repository-managed dashboard metadata or query definitions.

## Hard constraints

- KPI definitions must map to actual implemented metrics, dashboards, or read models, not aspirational numbers.
- Operational handover must cover both technical platform support and key business-operation support boundaries where the product demands them.
- Avoid duplicating live data in static docs when dashboards already provide it; document the source of truth instead.
- Dashboard and support access must remain role-scoped and tenant-safe where relevant.
- Do not turn this into a new analytics warehouse or BI program.

## Expected implementation outputs

- A dashboard and KPI catalog with real data-source references.
- Alert-to-runbook ownership mapping and escalation guidance.
- Operational support handover documentation covering live ownership and common failure modes.
- A single indexed path to dashboards, alerts, and runbooks.

## Non-goals

- Do not redesign the reporting layer or dashboard product here.
- Do not add unsupported KPIs that are not backed by implemented data.
- Do not replace live dashboards with static screenshots or spreadsheets as the primary source of truth.
- Do not broaden this into long-term data-warehouse planning.

## Verification checklist

- [ ] Real dashboards/queries/metrics are referenced for the defined operational and business KPIs.
- [ ] Alert thresholds, owners, and linked runbooks are explicit.
- [ ] Support handover documentation covers architecture, ownership, and common failure handling.
- [ ] There is a single discoverable index for operational support assets.
- [ ] Access control and tenant/privacy scope remain clear for dashboards and exports.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
