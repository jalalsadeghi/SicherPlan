# Codex Prompt — US-22-T2

## Task title

**Build monthly schedule, shift detail, maps, confirm/decline, and event-application flows.**

## Objective

Implement the employee-facing schedule experience in the mobile app so a logged-in employee can see released shifts, open detailed shift data, launch map routing, confirm or decline where the planning workflow allows it, and submit event applications through the approved HR/planning seams.

## Source context

- Updated proposal: section 5.3 employee app and self-service scope, including monthly schedule, detailed shift view, route via maps, confirm/decline, and event applications; workflow 6.2 where the employee uses availability/free wishes and applications after onboarding.
- Implementation specification: employee self-service tables in section 4.4, especially `hr.employee_event_application`; planning chain and released-schedule visibility guidance in section 5; cross-service rule that planning owns assignment truth and HR owns employee/application facts.
- Sprint reference: `docs/sprint/sprint-08-employee-app-and-field-i.md`.
- Cross-cutting rules: `AGENTS.md`, especially released-data discipline for mobile, Prokit-style navigation, and no shadow write-side schemas.
- Earlier prompt context: `US-12-T4` for self-service APIs and `US-21-T4` for released schedule read models are the intended upstream seams.

## Dependencies

- `US-22-T1` must already provide authenticated employee-shell access and profile context.
- `US-21-T4` should already provide planning-owned released schedule read models for downstream portal/mobile clients.
- `US-12-T4` should already provide employee self-service APIs for event applications and controlled own-record interactions.
- `US-21-T1` should already define release-state and visibility semantics for downstream clients.
- If confirm/decline command handlers do not yet exist, add only the narrow planning-owned command seam required for employee responses instead of inventing a mobile-only status store.

## Scope of work

- Implement a monthly schedule view and a detailed shift screen in Flutter using the released-schedule contracts from planning.
- Surface allowed operational details such as timing, location, meeting point, role/function context, team or assignment status, and related released instructions where already available.
- Add map/deep-link launch behavior from shift detail to the approved route/meeting-point context without baking external map-provider logic into domain models.
- Implement employee confirm/decline flows only through planning-owned commands and audit-safe status transitions; do not let the mobile client write raw assignment rows directly.
- Wire event-application submit/cancel/read flows to the real HR/planning seam from `hr.employee_event_application` or an already-approved service contract, not to fake local event entities.
- Add tests for released-only schedule visibility, own-record scoping, response command authorization, and event-application contract behavior.

## Preferred file targets

- Flutter screens/state for schedule list, schedule detail, response actions, and event-application flows under the actual mobile feature path.
- Planning or employee API/service files only where released-schedule or confirm/decline command contracts need narrow adjustments.
- Shared DTOs or mobile adapters only if the repository already has an approved contract-sharing pattern.
- Tests for schedule read, response commands, and event-application integration.

## Hard constraints

- Use released schedule truth from planning; do not create mobile-side schedule tables or cached shadow writes that become a second source of truth.
- Confirm/decline actions must remain planning-owned, authorized, and auditable.
- Event applications must use the real HR/planning contract and `hr.employee_event_application` semantics where available.
- Do not expose unreleased shifts, customer-hidden data, HR-private data, or finance-sensitive data in these screens.
- Keep mobile DTOs explicit and stable so later actuals/pay-data work can extend them without breaking the app.

## Expected implementation outputs

- Monthly schedule and shift-detail mobile screens backed by released planning data.
- Authorized confirm/decline flow integrated with planning-owned status handling.
- Event-application mobile flow connected to the approved self-service seam.
- Tests proving own-record scope, released-only gating, and safe response/application behavior.

## Non-goals

- Do not implement document download, push notifications, or QR/barcode display here.
- Do not expose released actuals/pay data yet if the finance-side contracts are not ready.
- Do not create temporary event or schedule tables just to make the UI work.
- Do not implement customer or subcontractor schedule views in this task.

## Verification checklist

- [ ] The employee sees only their own released schedule data.
- [ ] Shift detail shows allowed operational data without leaking private HR or finance fields.
- [ ] Confirm/decline actions are authorized, auditable, and flow through planning-owned commands.
- [ ] Event applications use the approved HR/planning seam rather than a fake mobile-side model.
- [ ] Tests cover released-only gating, own-record scope, response actions, and application behavior.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
