# Codex Prompt — US-28-T1

## Task title

**Implement payroll tariff tables, base rates, and surcharge rules by region/date/function.**

## Objective

Create the normalized payroll-rule backbone so SicherPlan can resolve pay basis deterministically from approved actuals, effective tariff tables, regional rules, function/qualification dimensions, and surcharge windows without embedding payroll logic in ad hoc code paths.

## Source context

- Updated proposal: section 5.6 payroll logic requires tariff tables by state/role/date, future-effective rate changes, night/Sunday/holiday surcharges, function or qualification-based allowances, advances, time accounts, and payroll export/import readiness; the applicant-to-payroll workflow in section 6.2 ends with payroll export/import and optional payslip archiving.
- Implementation specification: section 4.8 defines `finance.payroll_tariff_table`, `finance.payroll_tariff_rate`, and `finance.payroll_surcharge_rule`; section 10 says temporal overlap checks should be centralized and concrete finance outputs must stay linked to `finance.actual_record`.
- Earlier uploaded customer/employee/subcontractor source material informed the updated proposal’s emphasis on employment/payroll fields, future-effective commercial rules, and finance-safe configuration rather than hardcoded rates.
- Sprint reference: `docs/sprint/sprint-10-payroll-billing-and-settlement.md`.
- Cross-cutting rules: `AGENTS.md`, especially finance-bridge discipline, docs/outbox rules, DE default / EN secondary behavior for any UI, and tenant-safe bounded-context ownership.

## Dependencies

- `US-12-T1` should already provide stable function and qualification catalogs that tariff rates may reference.
- `US-12-T3` should already provide the upstream HR concepts for allowances, advances, and time-account behavior that later payroll tasks will hook into.
- `US-26-T2` and `US-27-T1` through `US-27-T3` should already provide the approved `finance.actual_record` bridge and reconciliation lines that payroll logic must consume.
- If the repository already has a shared temporal-validation helper from rate-card or scope-assignment work, reuse it rather than inventing a second overlap-handling pattern.

## Scope of work

- Implement migrations, SQLAlchemy models, Pydantic Create/Update/Read/List/Filter schemas, services, and APIs for payroll tariff tables, base-rate rows, and surcharge rules in the finance bounded context.
- Support effective-dated tariff tables keyed by region/state and status, with deterministic validation for overlapping validity windows and future-effective replacement behavior.
- Support rate rows dimensioned by function, qualification, employment type, pay unit, and currency, while allowing nullable dimensions only where the normalized model intentionally supports broader fallback rows.
- Support surcharge rules for night, Sunday, holiday, or custom windows using explicit weekday masks, start/end times, holiday-region scope, percentage/fixed-amount fields, and consistent validation of mutually exclusive or combined amount styles according to repo conventions.
- Provide a finance-facing rule-resolution service that can determine the applicable tariff row and surcharge set for a given actual-record segment, date/time, actor context, and region without reading raw time events directly.
- Where the repository already has an admin finance-configuration area, add narrow Vben-backed CRUD/API support for tariff tables and surcharge rules with localized DE/EN labels and permission checks; otherwise keep this task backend-first and document the contract seam.
- Add automated tests for overlap handling, dimension uniqueness, fallback resolution precedence, region/date lookup behavior, surcharge-window matching, and tenant isolation.

## Preferred file targets

- `modules/finance/` or a finance-payroll subpackage for models, schemas, services, routers, and tests.
- Alembic migrations for tariff header/rate/surcharge tables, indexes, and constraints.
- Optional Vben admin finance-configuration views/components only if the repository already has a compatible admin area.
- Tests covering migration behavior, rule resolution, and any API endpoints introduced.

## Hard constraints

- Payroll tariff truth is finance-owned. Do not duplicate tariff tables in HR, planning, or integration modules.
- Do not bypass `finance.actual_record` when resolving payroll basis for downstream outputs.
- Overlapping active tariff windows and ambiguous dimension tuples must be prevented deterministically.
- Do not hide tariff and surcharge logic inside opaque JSON blobs or provider-specific payload builders when the schema already defines normalized columns.
- Holiday, region, pay-unit, and surcharge-type codes should remain controlled values that align with existing lookup/catalog patterns.
- If UI work is included, German must remain the default language and English must be supplied for every new user-facing string.

## Expected implementation outputs

- Normalized tariff-table, tariff-rate, and surcharge-rule persistence and domain models.
- Deterministic rule-resolution and validation services for later payroll export work.
- Optional localized admin/API maintenance surface where the repo already supports finance configuration.
- Tests proving overlap safety, rule precedence, and tenant-safe finance behavior.

## Non-goals

- Do not implement employee-specific pay profiles, override handling, or allowance/advance/time-account hooks here; that belongs to `US-28-T2`.
- Do not implement export batches, provider adapters, or payslip archiving here; those belong to later tasks in this story.
- Do not implement tax/social-insurance calculations, net-pay logic, or country-specific payroll compliance engines in this task.
- Do not read raw time-capture events directly for payroll math.
