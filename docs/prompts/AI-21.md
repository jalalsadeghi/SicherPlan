# AI-21 — Shift Visibility Diagnostic Service

## Task Objective

Implement the primary end-to-end diagnostic service for the SicherPlan AI Assistant: diagnosing why an assigned shift is not visible to an employee in the Employee Self-Service / mobile app.

This is the main Markus-style diagnostic scenario from `docs/sprint/AI-Assistant.md`.

The diagnostic must combine employee tools, planning tools, field/released schedule visibility checks, permission-aware limitations, and safe navigation links to produce an evidence-based structured result that the assistant can turn into a same-language answer.

This task must remain read-only. The diagnostic may explain what the user should check or where to navigate, but it must not fix, release, update, approve, create, dispatch, or modify any record.

### 3. Diagnostic Input Schema

The diagnostic must support natural-language extraction by the assistant and direct structured tool input.

Input schema:

```json
{
  "employee_name": "string|null",
  "employee_ref": "uuid-or-safe-reference|null",
  "date": "YYYY-MM-DD|null",
  "date_from": "YYYY-MM-DD|null",
  "date_to": "YYYY-MM-DD|null",
  "assignment_ref": "uuid-or-safe-reference|null",
  "shift_ref": "uuid-or-safe-reference|null",
  "question_language": "fa|de|en|other|null",
  "route_context": {
    "path": "string|null",
    "route_name": "string|null",
    "page_id": "string|null",
    "query": {}
  }
}
```

Validation rules:

- At least one usable locator must exist:
  - `assignment_ref`, or
  - `shift_ref + employee_ref/name`, or
  - `employee_ref/name + date/date range`.
- If only a name and date are supplied, use employee search and assignment search.
- If multiple employees match, do not guess. Return clarification-needed structured output with safe visible options.
- If multiple assignments match, return ambiguity with safe options when allowed.
- Do not accept tenant ID from input.
- Do not trust route context as authority; use it only as context/hint.

### 5. Required Diagnostic Flow

Implement this flow with safe early exits.

#### Step 1 — Validate Input and Current Auth Context

- Verify assistant is enabled and user has `assistant.chat.access` through existing assistant service flow.
- Verify diagnostic permission when the current user is an internal diagnostic user.
- Do not trust tenant/user/scope from input.
- Normalize date/date range.
- Respect current timezone conventions.

#### Step 2 — Resolve Employee

Use `employees.search_employee_by_name` or equivalent employee service/tool when name is provided.

Cases:

```text
single visible match -> continue
multiple visible matches -> return ambiguous with safe visible options or ask clarification
no visible match -> return not_found_or_not_permitted
missing employee permission -> continue only if assignment_ref is enough; otherwise return insufficient_permissions
```

Do not reveal whether an employee exists outside current scope.

#### Step 3 — Inspect Employee Operational and Mobile Access Status

Use employee tools from `AI-18` when permitted.

Check:

```text
employee active
has linked self-service/mobile user
linked user active
self-service/mobile context available
can receive released schedules when known
```

If permission is missing, add `missing_permissions` and mark these rules `not_checked` or `unknown`.

#### Step 4 — Find Assignment(s)

Use planning tools from `AI-19`.

Cases:

```text
assignment_ref supplied -> inspect that assignment
employee + date supplied -> find assignments for employee/date
shift + employee supplied -> find assignment for that employee on the shift
multiple assignments -> return ambiguous or list safe options
no assignment -> likely cause: assignment_not_found
```

Check:

```text
assignment belongs to employee
assignment actor type is internal employee
assignment active/offered/confirmed according to repository rules
assignment not cancelled/archived/draft-hidden
```

#### Step 5 — Inspect Shift and Release State

Use planning tools from `AI-19`.

Check:

```text
shift exists
shift status active
shift release state
shift plan status/release state
planning record status/release state
visibility flags
stealth mode
```

Common likely causes:

```text
shift_not_released
shift_plan_not_released
planning_record_not_released
visibility_flag_disabled
stealth_mode
```

#### Step 6 — Inspect Validations and Blockers

Use planning validation tools from `AI-19` and employee readiness tools from `AI-18`.

Check, when permission and data allow:

```text
qualification mismatch
expired certificate
missing mandatory document
approved absence
customer-specific employee block
double booking
minimum staffing not met
release validation failure
```

Return safe summaries only. Do not expose HR-private details, certificate numbers, private documents, or sensitive notes.

#### Step 7 — Inspect Released Schedule / Mobile Visibility

Use `field.inspect_released_schedule_visibility` from `AI-20`.

Check:

```text
whether employee self-service/mobile released schedule query would include the assignment
whether visible/not_visible/unknown/ambiguous
blocking reasons returned by the field visibility tool
```

This is the final visibility confirmation layer.

#### Step 8 — Build Safe Links

Use only `navigation.build_allowed_link` from `AI-17`.

Candidate pages:

```text
P-03 — Shift Planning
P-04 — Staffing Board & Coverage
P-05 — Dispatch, Outputs & Subcontractor Releases
E-01 — Employees Workspace
ES-01 — Employee Self-Service Portal
FD-01 — Field Operations Workspace, if relevant and allowed
```

Rules:

- Do not construct raw links directly in the diagnostic service.
- Do not return a link if navigation tool says it is not allowed.
- Include query params only when allowed and safe.
- If no link is allowed, explain which role/permission may be needed without leaking protected data.

#### Step 9 — Rank Most Likely Causes

Rank findings as:

1. verified blocking causes;
2. verified warning causes;
3. missing-permission limitations;
4. unknown/not-checked items.

Examples of blocking causes:

```text
assignment_not_found
assignment_cancelled
shift_not_released
shift_plan_not_released
missing_access_link
inactive_linked_user
released_schedule_not_visible
approved_absence
customer_block
mandatory_document_missing
```

Do not overstate a cause if the tool result is `unknown`.
