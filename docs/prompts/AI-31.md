# AI-31 — Replace Canned Answer Paths with OpenAI-Orchestrated Grounded Assistant

## Purpose

The current Assistant is technically visible and working, but its behavior is not acceptable for the intended product experience.

The Assistant must not behave like an FAQ bot or a set of hardcoded answer templates. The target behavior is:

- all in-scope SicherPlan questions are answered dynamically by the OpenAI-powered assistant;
- backend tools, UI catalogs, page manifests, workflow manifests, knowledge chunks, and diagnostics provide **facts and grounding context only**;
- the final natural-language answer is generated by the model, in the user’s language, using those grounded facts;
- no backend production path should return a canned/how-to answer for an in-scope user question;
- the Assistant should feel like a specialist who understands the platform, while still being safely grounded in verified platform facts and the current user’s permissions.

This task is a corrective refactor after the AI Assistant implementation tasks AI-01 through AI-30.

---

## Current Problem Observed

After the Assistant widget became visible, these issues were observed in real UI testing:

1. User asked in Persian:

   ```text
   سلام
   چطور میتونم یک کارمند جدید بسازم؟
   ```

   The answer was good and used verified UI data for `E-01 Employees Workspace`.

2. User then asked:

   ```text
   برای اینکه یک کارمند جدید بتونم بهش یک شیفت نصبت بدم چکار باید بکنم؟
   ```

   The Assistant incorrectly repeated the same “create employee” answer.

3. User then asked:

   ```text
   برای اینکه یک کارمند جدید بتونم در یک شیفت Assgin بدم چکار باید بکنم؟
   ```

   The Assistant again incorrectly repeated the “create employee” answer.

4. User asked:

   ```text
   الان میخوام یک order جدید درست کنم، دقیقا باید چکار کنم؟
   ```

   The Assistant incorrectly classified this as out of scope.

5. User asked in English:

   ```text
   I want to create a new order. how can I do?
   ```

   The Assistant returned a mock-provider placeholder instead of a real grounded answer.

These are product-breaking issues because they show that the Assistant is currently over-relying on deterministic routing and/or mock output instead of using model reasoning with grounded context.

---

## Important Product Decision

Do **not** solve this by adding more canned answers.

Do **not** create one static answer per question type.

The correct design is:

```text
User question
  -> classify/safety/language only
  -> collect grounding context from knowledge, UI manifests, route catalog, tools, diagnostics
  -> OpenAI model reasons over grounding context
  -> backend executes only approved read-only tools when requested
  -> OpenAI model generates final structured response
  -> frontend renders model-generated answer, evidence, links, and next steps
```

Catalogs and manifests are allowed and required, but they must be used as **grounding facts**, not as a final answer template.

## Required Architecture After Refactor

### 1. Backend Must Not Produce Canned In-Scope Answers

For platform-related user questions, backend code must not directly create final text like:

```text
برای ایجاد یک Employee جدید، وارد Employees Workspace شوید...
```

or:

```text
To create a new Employee, open the Employees Workspace...
```

Instead, backend should collect facts like:

```json
{
  "type": "ui_action",
  "page_id": "E-01",
  "page_title": "Employees Workspace",
  "action_label": "Create employee file",
  "location": "top-right toolbar",
  "required_permissions": ["..."],
  "verified": true
}
```

Then the model generates the final human answer.

### 2. UI/Page Help Must Become Grounding Context

Refactor page-help helpers so they return normalized grounding data, not final answers.

Expected direction:

```text
assistant.find_ui_action
assistant.get_page_help_manifest
assistant.search_page_help
assistant.search_workflow_help
```

These tools may return:

```json
{
  "facts": [
    {
      "kind": "verified_ui_action",
      "page_id": "E-01",
      "page_title": "Employees Workspace",
      "action_key": "employee.create.open",
      "action_label": "Create employee file",
      "location": "top-right toolbar",
      "opens": "Identity and personnel number",
      "required_permissions": ["..."],
      "allowed": true,
      "verified": true
    }
  ],
  "allowed_links": [
    {
      "page_id": "E-01",
      "label": "Employees Workspace",
      "path": "/admin/employees"
    }
  ],
  "missing_permissions": []
}
```

They must not return final prose answers.

### 3. Diagnostics Must Become Grounding Context

Diagnostics such as shift visibility may compute facts/findings, but the final answer should still be generated by the model.

Allowed diagnostic output:

```json
{
  "diagnostic_key": "shift_visibility",
  "facts": [
    {
      "code": "assignment_exists",
      "severity": "info",
      "evidence": "Visible assignment found in current tenant scope."
    },
    {
      "code": "shift_plan_not_released",
      "severity": "blocking",
      "evidence": "Shift plan release state is below released."
    }
  ],
  "missing_permissions": [],
  "allowed_links": []
}
```

Not allowed as production path:

```json
{
  "answer": "من Markus را پیدا کردم و مشکل این است که..."
}
```

The model should write that answer based on diagnostic facts.

### 4. OpenAI-Orchestrated Tool Loop

Implement or complete true provider/tool orchestration.

Target flow:

```text
1. AssistantService receives message.
2. It validates auth, feature flag, rate limits, message size, language, and scope.
3. It classifies only for safety/routing metadata, not final answer generation.
4. It retrieves initial knowledge/page-help context when useful.
5. It builds the OpenAI request with:
   - system instructions
   - user message
   - conversation history
   - auth summary
   - route context
   - retrieved knowledge chunks
   - grounding facts
   - available tool definitions
6. The model may request tool calls.
7. AssistantService executes each requested tool through the existing registry only.
8. Tool outputs are redacted and appended back to model context.
9. Repeat until:
   - model returns final structured response, or
   - max tool calls is reached.
10. Store final user and assistant messages.
11. Return structured response to frontend.
```

Do not let the model access SQL or direct repositories. It may only request registered tools.

### 5. Provider Responsibilities

The provider wrapper should handle OpenAI API calls, structured output, and tool-call response parsing, but it should not bypass backend policy.

The service/tool registry must own tool execution because the service has the authorization context.

Acceptable implementation models:

#### Option A — Service-owned orchestration

```text
AssistantService
  -> OpenAIProvider.generate_or_request_tools(...)
  -> ToolRegistry.execute_tool(...)
  -> OpenAIProvider.generate_with_tool_results(...)
```

#### Option B — Provider returns tool requests, service loops

```text
provider.generate(...)
  returns either:
    - final_response
    - requested_tool_calls

service executes tools
service calls provider again with tool results
```

Choose the option that best fits the current codebase, but do not fake tool execution.

### 6. Mock Provider Must Not Look Like Production

Mock mode is for tests and CI. It must not be confused with real assistant behavior.

Required behavior:

- If `SP_AI_PROVIDER=mock`, the UI or capabilities should expose provider mode clearly.
- Mock responses may be deterministic for tests, but they must not be considered production-quality answers.
- In local dev, if the developer expects real AI answers, environment must use:

```env
SP_AI_ENABLED=true
SP_AI_PROVIDER=openai
SP_OPENAI_API_KEY=...
SP_AI_STORE_RESPONSES=false
```

Add a clear diagnostic/log message if provider mode is mock.

### 7. In-Scope Classifier Must Not Reject Platform Questions

Fix classifier/routing so these are platform-related:

Persian:

```text
الان میخوام یک order جدید درست کنم، دقیقا باید چکار کنم؟
چطور یک سفارش جدید بسازم؟
چطور یک پروژه جدید ثبت کنم؟
چطور یک کارمند جدید بسازم؟
برای اینکه یک کارمند جدید بتونم در یک شیفت Assign بدم چکار باید بکنم؟
برای اینکه یک کارمند جدید بتونم بهش یک شیفت نسبت بدم چکار باید بکنم؟
```

```text

```

```text

```

Also handle common typos and mixed-language terms:


### 8. Avoid Greedy Employee-Create Routing

The previous behavior likely happened because any message containing “کارمند جدید” matched `employee.create`.

Fix specificity:

Higher priority intents:

```text
employee_assign_to_shift_workflow
  when employee/کارمند/Mitarbeiter + shift/شیفت/Schicht + assign/اختصاص/تخصیص/نسبت/Assign/Assgin are present

order_create
  when order/Auftrag/سفارش/پروژه + create/ساخت/ثبت/درست کردن/create/erstellen are present

shift_create_or_release
  when shift/شیفت/Schicht + create/release/visibility/app are present

employee_create
  only when the message is specifically about creating an employee
  and does not include shift/assignment/order workflow terms
```

Do not use these intents to return canned answers. Use them only to choose grounding context and suggested tools.

### Step 2 — Convert Page Help Rendering to Grounding Facts

Refactor `page_help.py` and related service paths:

- remove production usage of `render_verified_employee_create_answer`;
- keep helper functions only if used in tests or convert them to fact-building helpers;
- create normalized data structures for `GroundingFact`, `GroundingBundle`, or reuse existing schemas if appropriate;
- `assistant.find_ui_action` must return verified facts, not final answer text.

### Step 3 — Refactor AssistantService Message Flow

Modify `AssistantService.add_message` so that for in-scope messages:

- it does not directly return page-help answers;
- it does not directly return diagnostic prose answers;
- it gathers context and passes it to the provider;
- it executes requested tools through the registry;
- it returns the provider’s final structured response.

Out-of-scope and unsafe refusals may remain deterministic if they are policy-level, but all SicherPlan in-scope help/diagnostic/how-to questions must go through the provider.

### Step 4 — Implement or Complete Tool-Call Orchestration

If current OpenAI provider only returns requested tool calls without executing them, complete the loop in the service.

Requirements:

- execute only known registry tools;
- enforce read-only policy;
- enforce permissions;
- audit tool calls;
- redact outputs before sending back to the model;
- cap total tool calls using `SP_AI_MAX_TOOL_CALLS`;
- if max tool calls is reached, ask model to answer with available facts and explain limitations.

### Step 5 — Improve Prompt Builder

Update system instructions:

```text
You are the SicherPlan AI Assistant.
You must answer in the same language as the user.
You must produce the final answer yourself using only grounded facts from documentation, UI manifests, tool outputs, route catalog, and permitted diagnostics.
Do not treat page-help facts as canned answers.
Do not invent UI labels.
If exact UI labels are unavailable, say that the exact label is not verified.
Use backend-provided links only.
Do not perform write actions.
Do not reveal inaccessible records.
Do not confirm inaccessible record existence.
```

Add a grounding-context section that clearly separates:

```text
- User message
- Current route/page
- User permission summary
- Retrieved knowledge chunks
- Verified UI facts
- Diagnostic facts
- Allowed links
- Missing permissions
- Tool limitations
```

### Step 6 — Add Workflow Grounding

Add or expose workflow-level grounding for:

```text

shift_release_to_employee_app
```

This can come from:

- page help manifests;
- a small verified workflow manifest;
- page route catalog;

But again: this must be facts/steps grounding, not final canned prose.

### Step 7 — Provider Mode Hardening

## Non-Negotiable Constraints

- Do not add canned responses for more question types.
- Do not implement write actions.
- Do not expose OpenAI key to frontend.
- Do not generate or execute SQL from the model.
- Do not bypass existing IAM/authz.
- Do not leak tenant, HR-private, payroll-sensitive, or inaccessible data.
- Do not return unauthorized links.
- Do not break the existing widget display or conversation persistence.
- Keep mock provider for tests, but do not let it masquerade as real production AI.

1. In-scope SicherPlan questions are answered through OpenAI/provider orchestration, not hardcoded backend prose.
2. Page Help Manifest and UI Action Catalog are used as grounding facts only.

4. Tool calls requested by the model are executed through the backend registry with permission checks.
5. The Assistant correctly handles these exact user messages:
   - `چطور میتونم یک کارمند جدید بسازم؟`
