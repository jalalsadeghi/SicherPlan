# Codex Prompt — US-11-T3

## Task title

**Implement import/export and user-account linkage for employee app access**

## Objective

Enable tenant-safe employee bulk onboarding/offboarding workflows and controlled linkage between employee records and IAM user accounts so the later employee app can authenticate against the approved access model without duplicating identity truth.

## Source context

- Updated proposal: section 5.3 digital employee file imports/exports, login/app rights, and employee-app/self-service readiness; Appendix C identification/access and personnel-file fields.
- Implementation specification: `hr.employee.user_id` optional unique linkage to IAM, `integration.import_export_job` backbone, and `EmployeeAggregate` ownership.
- Uploaded Dienstplanmacher employee and app source material: employee import/export, employee login rights, self-service/app access, and employee-facing schedule/availability capabilities once login exists.
- Sprint reference: `docs/sprint/sprint-04-recruiting-and-employee-core.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-11-T1` must already provide the employee master schema and service foundation.
- `US-5-T1` and `US-5-T2` must already provide IAM user-account and login/session foundations.
- `US-6-T4` should already provide import/export job tracking and async execution seams.
- If applicant conversion from `US-10-T4` already exists, keep import/upsert rules compatible with converted employees.

## Scope of work

- Implement employee import services with stable input format(s), dry-run validation, row-level error reporting, and tenant-safe upsert rules for operational employee master data.
- Implement export services for authorized internal users, including scoped field sets that do not leak HR-private data to inappropriate roles.
- Link employees to `iam.user_account` records through an explicit service that can create, attach, detach, or reconcile employee app access without duplicating authentication state.
- Enforce one-employee-to-one-user uniqueness where the approved schema expects it, and fail safely when ambiguous or duplicate link requests occur.
- Provide localized import/export result reporting where the repo supports user-facing job summaries.
- Add tests for dry-run validation, import idempotency/upsert semantics, export field scoping, and user-link uniqueness constraints.

## Preferred file targets

- Employee import/export services under `modules/employees/*` and job integration under `modules/platform_services/integration/*` or equivalent.
- IAM linkage services or coordinators bridging `modules/employees` and `modules/iam`.
- Tests for import dry-runs, export scoping, and user-link integrity.

## Hard constraints

- Do not create a second auth table for employee app users; employee access must link to the approved IAM user-account model.
- Do not expose HR-private data in exports or import it through broad operational templates unless permission and schema explicitly allow it.
- Imports must be idempotent or safely upsert-oriented according to explicit business keys; avoid duplicate employee creation on reruns.
- Keep tenant scoping structural in services and queries so one tenant cannot import/export or link another tenant's employees.
- Do not build mobile UI here; this task is about import/export and access linkage foundations.

## Expected implementation outputs

- Employee import/export services with validation, dry-run reporting, and job-friendly execution seams.
- Controlled employee-to-IAM user linkage services for app access.
- Tests for import/export safety, data scoping, and user-link uniqueness.
- A stable access foundation for later employee app/mobile tasks.

## Non-goals

- Do not implement schedule reads, document download, or mobile screens here.
- Do not redesign login, password reset, MFA, or SSO flows already owned by IAM tasks.
- Do not build payroll export here.
- Do not bypass audit or tenant scoping just to make imports more permissive.

## Verification checklist

- Dry-run imports report row-level issues without mutating data.
- Committed imports are idempotent or safely upsert records according to approved keys.
- Exports respect role-based field visibility and tenant scope.
- User-account linkage enforces one-to-one constraints and clean error handling.
- Tests cover import safety, export scope, and access-link integrity.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
