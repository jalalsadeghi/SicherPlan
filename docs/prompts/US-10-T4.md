# Codex Prompt — US-10-T4

## Task title

**Implement one-click transfer from applicant to employee aggregate**

## Objective

Implement a safe, idempotent conversion path that turns an accepted applicant into a clean employee aggregate while preserving consent evidence, workflow provenance, and linked documents without creating duplicate truth or shadow employee records.

## Source context

- Updated proposal: section 5.3 one-click transfer of accepted applicant data into the employee file; section 6.2 step 3 where accepted applicant data is transferred into a new employee file and step 4 where further employee completion follows.
- Implementation specification: `ApplicantAggregate` converts into `EmployeeAggregate`; `hr.applicant.converted_employee_id`; `hr.employee` ownership; docs/document-link reuse rules; aggregate ownership in section 9.
- Uploaded Dienstplanmacher Bewerberverwaltung and Mitarbeiterakte material: accepted applicants are transferred directly into the new employee file, while uploaded applicant files remain part of the usable personnel record.
- Sprint reference: `docs/sprint/sprint-04-recruiting-and-employee-core.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-10-T1` and `US-10-T2` must already provide stable intake data, consent evidence, attachments, and accepted-status workflow history.
- `US-11-T1` must provide the initial employee aggregate models before the actual conversion transaction is finalized; do not invent a temporary employee schema just to make conversion compile early.
- `US-6-T1` is required for document-link reassignment or dual-link handling.
- `US-5-T4` should provide audit hooks for the conversion event.

## Scope of work

- Implement a conversion service that only allows conversion from the approved accepted/convertible applicant state.
- Map applicant data into the operational employee aggregate using only fields that genuinely belong in `hr.employee` and related approved child rows; leave later HR-private completion to the employee-file flow instead of guessing missing data.
- Create the employee record transactionally, set `converted_employee_id` on the applicant, append a conversion workflow event, and emit an audit-safe conversion event.
- Preserve applicant attachments and consent evidence through document-link reuse or additional employee-side links, but do not duplicate binary files or break applicant-side provenance.
- If address information is available and approved for employee onboarding, initialize current address/history records according to the employee aggregate rules.
- Make the conversion idempotent so repeated clicks or retries do not create duplicate employees.
- Add tests for accepted-only conversion, duplicate-click/idempotency behavior, document linkage, rollback safety, and provenance/audit expectations.

## Preferred file targets

- Recruiting-to-employee conversion services under `modules/recruiting/services/*` and/or `modules/employees/services/*` following the repo's aggregate boundary patterns.
- Transactional service or unit-of-work code coordinating applicant and employee writes.
- Tests for conversion idempotency, document provenance, and workflow/audit side effects.

## Hard constraints

- Do not create or depend on a shadow employee schema; use the real `EmployeeAggregate` from `US-11-T1`.
- Only accepted applicants may be converted.
- Conversion must be transactional and idempotent.
- Do not fabricate HR-private or payroll-sensitive values during conversion just to 'complete' the employee file.
- Do not lose applicant history, consent evidence, or attachment provenance when the employee record is created.
- Do not create IAM user accounts here; employee login/app access belongs to `US-11-T3`.

## Expected implementation outputs

- A one-click applicant conversion service with clear accepted-only rules.
- Transactional creation of employee aggregate roots from applicant data.
- Document/provenance handling that preserves applicant history while enabling employee-file continuation.
- Tests for idempotency, rollback safety, and provenance retention.

## Non-goals

- Do not build recruiter UI here; that is `US-10-T3`.
- Do not implement private-profile completion, allowances, qualifications, or login rights here.
- Do not create payroll profiles or employee app provisioning here.
- Do not move or delete applicant history after conversion.

## Verification checklist

- Only accepted applicants can be converted.
- Repeated conversion attempts do not create duplicate employee records.
- Applicant attachments and consent evidence remain traceable after conversion.
- Applicant history shows the conversion event and links to the created employee.
- Tests prove transaction rollback behavior and audit/provenance retention.

- Verification evidence: build/lint/test commands and notable outputs.
