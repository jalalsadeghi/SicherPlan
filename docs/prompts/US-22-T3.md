# Codex Prompt — US-22-T3

## Task title

**Implement document download, push notifications, calendar export, and QR/barcode display.**

## Objective

Add the next employee-app utility layer so released documents, targeted notifications, shift calendar exports, and operational QR/barcode credentials are available on mobile without violating the docs/comm ownership model or exposing unreleased and private data.

## Source context

- Updated proposal: section 5.3 employee app scope includes document download, push notifications, calendar export, and QR/barcode display; section 5.1 communication backbone and docs/media backbone; appendix C employee IDs, profile card, and QR/barcode outputs.
- Implementation specification: docs tables in section 4.2 (`docs.document`, `docs.document_version`, `docs.document_link`); communication tables (`comm.message_template`, `comm.outbound_message`, `comm.message_recipient`, `comm.delivery_attempt`); `hr.employee_id_credential` in section 4.4; generated outputs rule in section 10.
- Sprint reference: `docs/sprint/sprint-08-employee-app-and-field-i.md`.
- Cross-cutting rules: `AGENTS.md`, especially docs as centralized truth, outbox/integration discipline for provider side effects, and mobile display of released scoped data only.
- Earlier prompt context: `US-6-T1`, `US-6-T2`, `US-6-T4`, `US-12-T3`, and `US-21-T4` define the intended seams for docs, messaging, credentials, and released schedules.

## Dependencies

- `US-22-T1` must already provide the authenticated employee-app shell.
- `US-21-T4` should already expose the released schedule data that calendar export derives from.
- `US-6-T1` must already provide centralized document storage and linking.
- `US-6-T2` and `US-6-T4` should already provide the outbound messaging and integration backbone used by notifications.
- `US-12-T3` should already provide employee credentials/badges suitable for QR/barcode display.

## Scope of work

- Implement mobile document-list and download/open flows for documents that are explicitly released and scoped to the acting employee.
- Implement shift calendar export generation or sharing from released schedule data using a derived export artifact, not a second schedule source of truth.
- Display employee QR/barcode credentials from the approved credential model for operational verification and field use.
- Integrate targeted push-notification consumption in the app using the repository messaging/integration pattern, including message-open/read handling where the existing platform already supports it.
- If server-side device-token registration is missing, add only the smallest repository-consistent seam needed; do not overload business tables with ad hoc push-token storage.
- Add tests for document scoping, credential visibility, calendar-export correctness, and notification registration/receipt flows that are feasible inside the current repo.

## Preferred file targets

- Flutter files under the mobile document, notification, schedule-utility, and credential feature paths.
- Docs/comm/platform-services backend files only where scoped download or notification-consumption contracts need to be added or refined.
- Credential read DTOs or API endpoints in the employee domain only if a narrow mobile-specific read seam is missing.
- Tests for document scope, export generation, credential rendering inputs, and notification flows.

## Hard constraints

- Only released, employee-scoped documents may appear in the app; no generic docs browsing is allowed.
- Do not place push-provider calls directly inside domain write transactions; keep provider side effects aligned with the messaging/integration backbone.
- QR/barcode display must come from the real credential model and not from locally fabricated tokens.
- Calendar export must be derived from released schedule reads and must not become a shadow source of scheduling truth.
- Do not leak HR-private or finance-sensitive documents into mobile just because they exist in the docs service.

## Expected implementation outputs

- Employee mobile flows for released document listing/downloading/opening.
- Calendar export or share flow derived from released schedules.
- QR/barcode credential display based on real employee credential records.
- Push-notification consumption integrated with the approved messaging/integration pattern plus relevant tests.

## Non-goals

- Do not implement the full pay-data surface or payroll document delivery here.
- Do not build a general-purpose chat or notification center beyond the scoped employee-app needs of this story.
- Do not invent a document file store or push-token model that conflicts with the docs/comm/integration backbone.
- Do not make downloaded or exported artifacts the canonical source of schedule truth.

## Verification checklist

- [ ] The employee can access only released, scoped documents relevant to their own app context.
- [ ] Calendar export reflects released schedule data without creating a second scheduling truth.
- [ ] QR/barcode display is sourced from the approved credential model.
- [ ] Notification flows stay aligned with the comm/integration backbone and avoid unsafe direct provider coupling.
- [ ] Tests cover document scope, credential visibility, and at least the core notification/export behavior.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
