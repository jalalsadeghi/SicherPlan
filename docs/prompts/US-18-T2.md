# Codex Prompt — US-18-T2

## Task title

**Support quick copy/day-week reuse, recurring exceptions, and workforce-scope selection.**

## Objective

Extend the shift-planning engine with the copy/reuse controls that dispatchers need for real planning work, including quick reuse across days/weeks, recurring exceptions, and explicit workforce-scope selection for internal, subcontractor, or mixed planning scenarios.

## Source context

- Updated proposal: object/site quick planning, day/week copy, trade-fair repeated-shift copy, patrol day/week copy, and Appendix D “Templates and series” with recurrence rules, exceptions, and copy/reuse controls.
- Implementation specification: `ops.shift_plan.workforce_scope_code`, `ops.shift_series`, and the planning package/aggregate rules; modeling notes about temporal validation staying centralized.
- Sprint reference: `docs/sprint/sprint-06-ops-planning-foundation.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-18-T1` must already provide the base shift-plan, template, series, and shift model plus generation services.
- Keep upcoming Sprint 7 staffing work in mind because workforce scope influences later release and assignment behavior.
- If the current repository already has a normalized exception-table pattern, reuse it. If not, introduce a small planning-local extension in a migration-safe way rather than encoding exceptions inside opaque JSON.

## Scope of work

- Add services, schemas, APIs, and tests for quick copy/reuse workflows across one day, one week, or a selected planning slice, with safe conflict/duplicate handling.
- Extend shift-plan creation and update flows so workforce scope (`internal`, `subcontractor`, `mixed`) is explicit, validated, and queryable from the planning root.
- Implement recurring-exception support for generated series in a normalized, planning-owned way, such as skipped dates, cancelled occurrences, or overridden generated shifts, depending on what best fits the existing repo structure.
- Add service logic that can regenerate, re-copy, or partially reuse shifts without silently duplicating or corrupting previously generated rows.
- Provide APIs or command handlers that later Vben screens can call for copy/day-week reuse and exception maintenance.
- Add tests for copy/reuse correctness, exception handling, duplicate prevention, and workforce-scope validation.

## Preferred file targets

- `modules/planning/` services, schemas, routers, validators, and tests.
- A small additional migration only if a normalized exception entity/table is needed and no equivalent already exists.
- No large UI build-out in this task; keep frontend work to contract support only if necessary for `US-18-T3`.

## Hard constraints

- Do not encode recurring exceptions only as opaque free-form JSON if the repository lacks a clear, testable query story for that approach.
- Workforce-scope selection must remain a planning concern only; it does not authorize direct writes into HR or partner master data.
- Copy/reuse logic must not silently duplicate the same active concrete shift rows for the same plan/time window.
- Keep exception semantics auditable and understandable; do not create hidden destructive overwrites of generated planning truth.
- Do not implement staffing assignments, release-to-subcontractor flows, or validation overrides here; those land in Sprint 7.
- Any user-visible validation/error content introduced here must preserve German default and English secondary behavior.

## Expected implementation outputs

- Copy/day-week reuse services and APIs grounded in the shift-planning model.
- Explicit workforce-scope support on shift plans and related validation.
- Recurring-exception handling that is normalized, migration-safe, and test-covered.

## Non-goals

- Do not build the full Vben shift-maintenance UI here; that is `US-18-T3`.
- Do not implement planning-board performance tuning here beyond what is minimally necessary for correctness; that is `US-18-T4`.
- Do not implement staffing, assignment, or subcontractor release flows here.
- Do not bypass the planning aggregate by letting frontend code clone shifts independently.

## Verification checklist

- [ ] Dispatch-oriented copy/reuse commands can duplicate the intended shift slice without accidental duplicates or missing records.
- [ ] Workforce scope is explicit, validated, and queryable on shift plans.
- [ ] Recurring exceptions can be represented and applied without corrupting the base series model.
- [ ] Copy/reuse and exception behavior remain tenant-safe and planning-owned.
- [ ] Tests cover day/week copy behavior, duplicate-prevention logic, exception handling, and workforce-scope validation.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
