This task is **AI-12 — Knowledge Source and Chunking Infrastructure**.

## Goal

Implement the first backend infrastructure for assistant knowledge ingestion: source registration, safe source loading, file/content hashing, deterministic chunking, metadata extraction, and persistence of knowledge sources/chunks.

This task prepares the knowledge base but does **not** need to implement semantic/vector retrieval yet. The retrieval adapter is AI-13. This task should make it possible to index project documentation once and store normalized chunks so runtime questions do not need to reread full source documents.

## Mandatory Constraints

- Do not call OpenAI embeddings in this task.
- Do not implement vector search in this task unless the repository already has a ready adapter and it is trivial to wire; otherwise leave embedding/vector fields empty.
- Do not implement the final retrieval/ranking logic; that belongs to AI-13.
- Do not index operational business data such as real employees, shifts, assignments, invoices, or payroll rows.
- Do not allow arbitrary user-supplied filesystem paths.
- Do not implement a public unrestricted file ingestion endpoint.
- Do not commit large binary documents unless they already exist in the repository.
- Do not read secrets, `.env` files, credentials, JWTs, private keys, or database dumps.
- Do not change unrelated business modules.
- Keep the implementation deterministic and testable in CI.

### 2. Knowledge Source Types

```text

```

PDF and XLSX can be represented as source types in schema/config, but actual binary extraction can be deferred unless the repository already has safe extraction utilities. If deferred, the code must fail explicitly and safely with a clear message such as:

```text

```

Do not silently ignore unsupported files.

### 3. Source Registration

Implement source registration in a service/repository function.

Required source metadata:

```text
source_type
source_name
source_path
source_hash
source_version
status
last_ingested_at
created_at
updated_at
```

Behavior:

- Compute a SHA-256 hash from normalized source content or raw file bytes, depending on source type.
- If a source with same path and hash already exists and is active, skip re-ingestion.
- If a source path exists but content hash changed, create/update the source record according to the project’s model design.
- Mark failed sources as `failed` with a safe error path if the existing schema supports error details; otherwise log safely and return a failure item.

### 4. Safe Source Loading

Implement source loading with path safety.

Rules:

- Allowed roots should be configured or limited to repository documentation paths such as `docs/` and a test fixture directory.
- Reject path traversal attempts.
- Reject absolute paths outside allowed roots unless explicitly configured for internal/admin runs.
- Reject `.env`, key, certificate, credential, database dump, binary cache, and secret-like file names.
- Apply a maximum file size limit from settings or a safe constant.
- Decode text safely as UTF-8 with clear error handling.

Suggested blocked patterns:

```text
.env
*.pem
*.key
*.p12
*.pfx
*secret*
*token*
*credential*
*.sqlite
*.db
*.dump
```

This blocklist is not a replacement for allowed roots; use both.

### 5. Deterministic Chunking

Implement deterministic chunking for text-like sources.

Requirements:

- Preserve section titles when possible.
- Split by Markdown headings first.
- Split oversized sections into smaller chunks by paragraph/sentence boundaries.
- Use stable chunk indexes.
- Store enough metadata for retrieval later.
- Do not create empty chunks.
- Avoid tiny fragments unless a section is naturally short.
- Normalize whitespace but do not destroy technical identifiers, route names, endpoint paths, page IDs, or code blocks.

Suggested default limits:

```text
max_chunk_chars: 3000-5000
min_chunk_chars: 300-500 when practical
chunk_overlap_chars: 200-400 for oversized chunks
```

Use settings if AI-02 already defines chunk limits. If not, define local constants and leave settings extension for later.

### 6. Metadata Extraction

For each chunk, try to infer metadata:

```text
title
language_code
module_key
page_id
role_keys
permission_keys
token_count or approximate_token_count
```

Minimum required:

- `title` from nearest heading or source name;
- `language_code` if detectable cheaply from existing language module or source config;
- `page_id` when text contains canonical page IDs like `P-03`, `P-04`, `E-01`, `FI-01`, `ES-01`, `CP-01`, `SP-01`;
- `module_key` when obvious from page ID or file path;
- `role_keys` and `permission_keys` may remain empty unless easily inferred.

Suggested page/module mapping examples:

```text
E-01 -> employees
P-03 -> planning
P-04 -> planning
P-05 -> planning
FD-01 -> field_execution
FI-01 -> finance_actuals
FI-02 -> finance_billing
FI-03 -> finance_payroll
FI-04 -> finance_subcontractor_invoice_checks
REP-01 -> reporting
ES-01 -> employee_self_service
CP-01 -> customer_portal
SP-01 -> subcontractor_portal
```

### 7. Persist Knowledge Chunks

Use the knowledge models/tables from AI-05.

page_id

Behavior:

- Delete/replace old chunks for a changed source if the source is re-ingested.
- Keep unchanged sources untouched.
- Insert chunks in stable order.
- Keep database operations transactional.

## Acceptance Criteria

The task is complete when:

1. Knowledge package exists and follows repository conventions.
2. Text-like documentation sources can be safely loaded from allowed roots.
3. Path traversal and blocked secret-like files are rejected.
4. Source hash is computed deterministically.
5. Unchanged sources are skipped based on hash.
6. Changed sources are re-indexed safely.
7. Markdown/text content is chunked deterministically.
8. Chunks preserve headings, page IDs, route/API identifiers, and technical terms.
9. Chunk metadata includes at least title, language when available, page ID when detected, and module key when inferred.
10. Chunks are persisted to the assistant knowledge tables.
11. Unsupported file types fail explicitly and safely.
12. Operational business data is not indexed.
13. Tests run without external API calls.
14. Existing assistant tests from AI-04 to AI-11 still pass.
