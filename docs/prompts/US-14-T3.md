# Codex Prompt — US-14-T3

## Task title

**Implement compliance completeness and validity views per worker**

## Objective

Create the derived readiness layer that shows whether each subcontractor worker is deployable, what is missing or expired, and which issues are blocking versus warning, without duplicating write-side truth.

## Source context

- Updated proposal: section 5.4 mandatory documents and qualification checks, where required staff documents must be complete before a subcontractor worker can be assigned; section 6.3 step 3 before assignment and step 4 before confirmation.
- Implementation specification: validation mapping in section 6 for qualification/function match, certificate validity, and mandatory documents; reporting/read-model guidance in section 7; the Phase C partner tables in section 4.5.
- Uploaded Dienstplanmacher subcontractor source material: workers cannot be used until mandatory documents are uploaded and checked for completeness, and planning qualifications must be queryable before placement.
- Sprint reference: `docs/sprint/sprint-05-subcontractor-management.md`.
- Cross-cutting rules: `AGENTS.md`.

## Dependencies

- `US-14-T1` must already provide worker, qualification, document-proof, and credential truth.
- Use the existing document-type and compliance-rule infrastructure where available from the core/docs foundation; do not invent a second rule store if the repo already has one.
- `US-12-T1` should already provide the shared qualification-type baseline that readiness logic can evaluate.

## Scope of work

- Implement backend read models or query services that compute worker readiness based on qualification validity, required document completeness, expiry windows, and other partner-worker compliance facts already modeled in the repo.
- Differentiate blocking issues from warning issues where the rule source supports that distinction, and keep the output stable for later planning validation, portal visibility, and reporting.
- Provide worker-level and subcontractor-level summary reads, such as ready/not-ready counts, missing-document lists, expiring certificates, and missing credential indicators where relevant.
- Expose filterable API endpoints or internal service contracts that later admin UI, portal views, planning validation, and finance/reporting projections can reuse without re-implementing the logic.
- Use SQL views/materialized views only where they fit the repo's reporting/read-model style; avoid creating a second mutable status table when the values can be derived safely.
- Add tests for missing-document cases, expired qualifications, mixed block/warn scenarios, and summary aggregation behavior.

## Preferred file targets

- Partner read-model/query modules, SQL views if the repo uses them, API endpoints, and tests under `modules/subcontractors/` or `modules/reporting/` as appropriate.
- Minimal UI column additions only if required to expose the new readiness outputs in already existing workforce screens.

## Hard constraints

- Readiness should be derived from authoritative tables and rules, not maintained as a drifting write-side status cache unless the repo has a justified projection pattern.
- Do not silently weaken blocking rules into warnings for convenience in list views or portal UX.
- Do not entangle this task with actual planning release or assignment persistence; it should prepare stable readiness outputs, not perform the release decisions yet.
- Keep the rule-evaluation surface explainable: API results should indicate what is missing or expired instead of returning opaque booleans only.
- Any minimal UI additions must remain secondary; this task is primarily a backend/read-model task.

## Expected implementation outputs

- Stable readiness/readiness-summary outputs for partner workers.
- Reusable rule-evaluation/query services for later planning, portal, and reporting work.
- Tests proving missing-document, expiry, and blocking-vs-warning behavior.

## Non-goals

- Do not implement the actual assignment or release-validation command path here; that lands in the planning/validation sprints.
- Do not build the full partner portal views here; those belong to `US-15-T2` and `US-15-T4`.
- Do not redesign document-type or compliance-rule administration here.

## Verification checklist

- [ ] Workers with complete, valid requirements are marked ready and workers with missing/expired requirements are not.
- [ ] Blocking and warning issues are distinguishable in the output structure where the rule source supports it.
- [ ] Summary counts and filters behave consistently across subcontractor and worker scopes.
- [ ] The implementation derives truth from existing tables and does not create an unnecessary mutable mirror table.
- [ ] Downstream consumers can reuse the API/service contracts without re-implementing compliance logic.

## Codex response format

- A short implementation plan before touching files.
- A concise list of files created or modified.
- A summary of key design decisions and assumptions.
- Verification evidence: build/lint/test commands and notable outputs.
- Open follow-ups or blockers that should be carried into the next task.
